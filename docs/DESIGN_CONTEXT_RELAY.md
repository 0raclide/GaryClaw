## Problem Statement

Long-running gstack skills (`/qa`, `/design-review`, `/autoplan`) are iterative fix loops that consume Claude Code's 200K context window within ~15-20 iterations. The user's current workaround — manually requesting a "handoff document + next-agent blob" — works but is tedious and expertise-dependent. Neither autocompact (degrades skill instructions) nor disabling it (hard crash) is acceptable.

The insight: Claude Code is a CLI. An external harness can monitor context, checkpoint state, and relay work across fresh sessions — making skills effectively context-infinite.

## What Makes This Cool

The "whoa" factor: you start `/qa` and it just... keeps going. Across session boundaries, across context limits, without you noticing. Fifty fixes, a hundred screenshots, hours of iterative work — and the harness handles the plumbing invisibly. No more "prepare a handoff doc" — the machine does what machines should do.

## Constraints

1. **Claude Code hooks are reactive only** — cannot read tokens, trigger /compact, or start sessions
2. **Agent SDK gives per-query usage, not real-time context position** — must estimate
3. **Skills are not currently phase-structured** — SKILL.md is a monolithic instruction doc
4. **Browse binary (`$B`)** is a gstack tool that needs to be available in spawned sessions
5. **Must work with existing gstack skills** without requiring skill authors to rewrite everything
6. **Solo dev** — solution must be maintainable by one person

## Premises (Agreed)

1. Context exhaustion is the primary bottleneck (not cost, speed, or quality)
2. Cumulative token tracking from SDK `result.usage` is sufficient to estimate context fullness
3. Skills have natural phase boundaries where relay is safe (between commits, between findings)
4. Fresh session + checkpoint prompt > session resume for relay quality
5. The harness runs outside Claude Code, not inside it

## Technology Landscape

## Approaches Considered

### Approach A: Phase Splitter

**Summary:** Parse SKILL.md into phases, run each as a bounded SDK `query()` with `maxTurns`. Checkpoint between phases. Relay when cumulative tokens cross threshold.

**Effort:** M (human: ~1 week / CC: ~2 hours)
**Risk:** Medium — requires understanding skill phase structure
**Completeness:** 9/10

**How it works:**
1. GaryClaw reads the skill's SKILL.md and identifies phase boundaries (## Phase N headers)
2. For each phase, constructs a prompt that includes: phase instructions, project context (CLAUDE.md), and checkpoint state from previous phases
3. Runs the phase as `query({ prompt, options: { maxTurns: N } })`
4. After each phase, reads `result.usage` and accumulates token counts
5. Writes checkpoint: `{ phasesCompleted, discoveredIssues, fixStatus, screenshotPaths, gitState }`
6. When cumulative tokens approach threshold (e.g., 150K), starts a fresh session for the next phase with checkpoint data injected into the prompt
7. At the end, merges all phase results into a unified report

**Pros:**
- Clean phase boundaries = safe relay points
- Each phase gets fresh context = maximum quality
- Structured checkpoints = no data loss
- Skills don't need modification (phase parsing is external)

**Cons:**
- Phase detection must handle diverse skill formats
- Some skills have phases that depend on earlier context (fix loop references baseline findings)
- maxTurns is a rough proxy — some phases need 5 turns, others 50

### Approach B: Token Watchdog

**Summary:** Wrap the entire skill in a single SDK session but monitor token usage per-turn. When approaching limit, interrupt, auto-checkpoint, and start fresh.

**Effort:** S (human: ~3 days / CC: ~1 hour)
**Risk:** High — mid-operation relay is messy
**Completeness:** 7/10

**How it works:**
1. Start skill as a single `query()` call
2. After each `result` message, check cumulative tokens
3. When crossing threshold, call `query.interrupt()`
4. Ask Claude (in a fresh micro-session) to summarize current state as structured JSON
5. Start fresh session with: skill instructions + checkpoint JSON + "continue from here"

**Pros:**
- Simpler — no phase parsing needed
- Works with any skill structure
- Minimal changes to skill format

**Cons:**
- Interrupting mid-operation can leave state inconsistent (half-written file, uncommitted changes)
- The "summarize state" step itself consumes context
- Harder to ensure nothing is lost in the relay

### Approach C: Hybrid CLI + SDK

**Summary:** Bash script chains `claude -p` invocations for simple sequencing. Falls back to SDK only for phases needing tool interception.

**Effort:** S (human: ~2 days / CC: ~45 min)
**Risk:** Low — simple architecture
**Completeness:** 6/10

**How it works:**
```bash
# Phase 1: Baseline
RESULT=$(claude -p "$PHASE1_PROMPT" --output-format json --max-turns 20)
CHECKPOINT=$(echo "$RESULT" | jq '.result')

# Phase 2: Fix loop (bounded)
RESULT=$(claude -p "$PHASE2_PROMPT with context: $CHECKPOINT" --output-format json --max-turns 15)
```

**Pros:**
- Zero dependencies beyond `claude` CLI + `jq`
- Works today, immediately
- Easy to debug (each phase is an isolated invocation)

**Cons:**
- No real-time monitoring (can't interrupt mid-phase)
- No tool interception (can't programmatically approve browse operations)
- Two systems if SDK is needed for some phases
- Bash string handling for large checkpoint data is fragile

## Recommended Approach

**No recommendation yet — this is R&D.** All three approaches have merit and different tradeoff profiles. The next step is to build a proof of concept with Approach C (simplest, fastest) to validate the core concept, then graduate to Approach A (most complete) if the concept proves out.

Suggested R&D sequence:
1. **Spike with Approach C** — Can we chain two `claude -p` invocations with a checkpoint JSON between them and get coherent results?
2. **Validate token estimation** — Run a real `/qa` session, log token usage per turn, determine if cumulative tracking is accurate enough
3. **Prototype Approach A** — Build the SDK harness, test with a real skill
4. **Evaluate Approach B as fallback** — If phase parsing proves too fragile, test the watchdog pattern

## Open Questions

1. **Token budget estimation:** SDK gives `input_tokens` + `output_tokens` per query. But context window = system prompt + conversation history + tool results. Can we estimate the "effective context position" from cumulative usage, or do we need a more sophisticated model?

2. **Skill phase parsing:** SKILL.md files use `## Phase N` headers, but the fix loop (Phase 8 in QA) is internally iterative. How do we handle phases that are themselves loops?

3. **Browse binary:** The `$B` browse binary is essential for QA/design-review skills. Does the SDK's spawned process inherit the environment where `$B` is set? Or do we need to explicitly pass it via `env` options?

4. **Checkpoint schema per skill:** QA needs `{ issues[], fixes[], screenshots[] }`. Design review needs `{ findings[], scores{} }`. Do we define per-skill schemas, or use a generic `{ phase, state, notes }` format?

5. **Git state across relays:** If the harness relays mid-fix-loop, the new session needs to know what branch it's on, what's staged, what's committed. Include `git status` / `git log` in checkpoint?

6. **Skill instruction injection:** When starting a fresh session, we inject SKILL.md content into the prompt. But SKILL.md can be 1000+ lines — that's ~4K tokens of instructions eating into the fresh context. Is there a way to load it more efficiently (CLAUDE.md? settings?)?

7. **Cost control:** Each relay starts a fresh session with ~4K tokens of skill instructions + checkpoint data. Over many relays, the overhead accumulates. Is there a budget ceiling where relay becomes more expensive than the manual handoff workflow?

8. **Terminal automation as escape hatch:** The SDK can't type `/compact` into an interactive session. But what if we need that? Should GaryClaw maintain a tmux fallback for operations the SDK can't express?

## Success Criteria

- [ ] A gstack skill (`/qa` or `/design-review`) runs to completion without hitting end-of-context
- [ ] The user does not need to manually intervene for context management
- [ ] Results (bug report, design audit) are equivalent quality to a single-session run
- [ ] Relay overhead < 10% of total session cost
- [ ] Works with existing SKILL.md files without modification
