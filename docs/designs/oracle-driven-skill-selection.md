## Problem Statement

The pipeline composition system has a blind spot. `composePipeline()` in `pipeline-compose.ts` maps `(effort, priority, hasDesignDoc)` to skill sequences from a 5-skill lookup table: `[prioritize, office-hours, implement, plan-eng-review, qa]`. Commit `1a519db` fixed the immediate bug where unknown skills (like `design-review`, `browse`, `plan-design-review`) got silently stripped during intersection. But the system still can't reason about what a task *actually needs*.

A mobile UI redesign and a TypeScript refactor both get the same pipeline if they share the same effort/priority. The daemon has no concept of task nature. It doesn't know that `design-review` exists, what it does, or when it's useful. And there's no way for a human to say "I know exactly what pipeline this needs, just run it."

Two gaps:

1. **No deterministic override.** `daemon trigger qa design-review` always goes through composition, which may strip or reorder skills. There's no way to say "run exactly these skills in this order for this TODO."

2. **No skill awareness.** The Oracle can recommend pipelines via `### Recommended Pipeline` in prioritize output, but it has no structured knowledge of what skills are available beyond the 5 hardcoded ones. It can't recommend `design-review` because it doesn't know `design-review` exists.

## What Makes This Cool

The daemon stops being a fixed-pipeline runner and becomes an intelligent process selector. It reads a skill catalog, understands that `design-review` finds visual inconsistencies while `plan-eng-review` catches architecture gaps, and picks the right combination for each task. A UI polish TODO gets `design-review → implement → qa`. A refactor gets `implement → plan-eng-review → qa`. A new feature gets the full pipeline. The static table becomes a cold-start fallback, not the permanent brain.

And when you know better than the daemon, `--todo` lets you override without fighting composition logic.

## Constraints

- Static table in `pipeline-compose.ts` remains as cold-start fallback (0 outcome data, Oracle circuit breaker tripped, errors)
- No new oracle memory files. Skill catalog is a static registry, not learned state.
- Must not break existing CLI commands, trigger configs, or cron jobs
- Must be testable without SDK calls (pure functions, synthetic data)
- Skill catalog must be maintainable by hand (not auto-discovered from filesystem... yet)
- `--todo` bypass must be absolute. No intersection, no stripping, no composition.
- Oracle skill selection only activates when `parsePipelineRecommendation()` is present AND cold-start gate passes (10+ outcomes). Below that, static table.

## Premises

1. The prioritize skill is the right place to recommend pipelines, because it already has full context on the TODO item, oracle metrics, and decision outcomes. Adding a skill catalog to its prompt gives it the information it needs to select skills intelligently.

2. A static, hand-maintained skill catalog is the right starting point. Auto-discovery from `~/.claude/skills/` is the 10x version but adds filesystem I/O, parsing complexity, and permission concerns. A hardcoded registry in `src/skill-catalog.ts` ships in 30 minutes and covers the 15 skills that matter.

3. `--todo` deterministic override and Oracle skill selection are independent features that happen to fix the same root problem from opposite ends. Ship both.

4. The existing `parsePipelineRecommendation()` + Oracle override path in `job-runner.ts` is the right integration point. We're extending what's there, not building a new composition path.

5. Task nature classification for pipeline selection should be prompt-driven (let the LLM classify), not rule-based. For the `taskCategory` extension point (populated but not acted on), a lightweight regex heuristic is fine since it's a no-op field for now. When category-specific scoring matters (Approach B), the LLM should classify.

## Approaches Considered

### Approach A: Minimal — `--todo` Flag + Skill Catalog Prompt Injection

**Summary:** Add `--todo` flag to `daemon trigger` for deterministic bypass, and inject a skill catalog into the prioritize prompt so the Oracle can recommend any skill, not just the 5 hardcoded ones.

**Effort:** S
**Risk:** Low
**Pros:**
- Ships fast. `--todo` is ~30 lines across `cli.ts`, `types.ts`, `job-runner.ts`. Skill catalog is ~100 lines in a new `src/skill-catalog.ts`. Prioritize prompt injection is ~20 lines.
- Reuses the existing `parsePipelineRecommendation()` + Oracle override path entirely. No new composition logic.
- Fail-open: if skill catalog parsing fails, static table takes over. If `--todo` is not specified, existing behavior unchanged.

**Cons:**
- Skill catalog is hand-maintained. Adding a new gstack skill means editing `skill-catalog.ts`.
- No per-skill-per-task-category outcome tracking. The Oracle recommends based on general reasoning + pipeline outcomes, not "last time we used design-review on a UI task it found 8 issues."

**Reuses:** `parsePipelineRecommendation()`, Oracle override path in `job-runner.ts`, `composePipeline()` intersection logic (bypassed by `--todo`), `PipelineOutcomeRecord` (unchanged).

### Approach B: Full — Task Category Tracking + Per-Category Skill Outcomes

**Summary:** Everything in Approach A, plus: classify each TODO into a task category (visual/UX, architectural, bug-fix, refactor, performance, infra), track per-category skill outcomes in `pipeline-outcomes.jsonl`, and inject category-specific failure rates into the prioritize prompt.

**Effort:** M
**Risk:** Medium
**Pros:**
- The Oracle gets real data: "When category=visual and design-review was skipped, failure rate was 60%. When included, failure rate was 10%."
- Enables the learning loop the TODOS.md description envisions: the daemon actually learns which skills help for which task types.
- `PipelineOutcomeRecord` already has `skippedSkills`. Adding `taskCategory` is one field.

**Cons:**
- Task category classification is done by the LLM in the prioritize prompt. Classification accuracy is unverified until after 20+ jobs.
- `pipeline-outcomes.jsonl` schema change could require migration logic for existing files.
- More test surface: category classification tests, per-category scoring, prompt injection formatting.
- The Oracle already gets pipeline outcomes via `decision-outcomes.md`. Adding structured per-category data may be redundant until there's enough volume to see category-specific patterns (probably 50+ jobs).

**Reuses:** Everything in Approach A, plus `computeSkipRiskScores()` (extended with category dimension), `PipelineOutcomeRecord` (extended with `taskCategory` field).

### Approach C: Auto-Discovery — Filesystem Skill Scanning

**Summary:** Everything in Approach A, but instead of a hand-maintained catalog, scan `~/.claude/skills/gstack/*/SKILL.md` files to build the catalog dynamically. Parse each skill's name, description, and "use when" trigger patterns.

**Effort:** M
**Risk:** Medium-High
**Pros:**
- Zero maintenance. New skills appear in the catalog automatically.
- The catalog is always accurate because it's read from the source of truth.
- Could be extended to scan project-local `.claude/skills/` too.

**Cons:**
- Filesystem I/O at prioritize time. Each SKILL.md is 5-50KB. With 25+ skills, that's 200KB+ of reads per prioritize run.
- SKILL.md files are prompt templates with bash blocks, not structured data. Parsing "what this skill does" from unstructured markdown is fragile.
- Permission and path resolution across global vs. vendored vs. project-local skill directories.
- The daemon doesn't need all 25+ gstack skills in its catalog. It needs the 10-15 that are relevant to autonomous development pipelines. A curated list is better than a complete list.

**Reuses:** Everything in Approach A, plus new `src/skill-catalog.ts` with filesystem scanning.

## Recommended Approach

**Approach A: Minimal — `--todo` Flag + Skill Catalog Prompt Injection.**

Approach B's per-category tracking is the right long-term direction, but premature today. GaryClaw has ~255 Oracle decisions at 100% accuracy. The Oracle is already good at reasoning from unstructured context. Adding a skill catalog to its prompt is enough for it to make smart skill selections. Per-category outcome tracking becomes valuable at 50+ jobs per task category, which is months away.

Approach C's auto-discovery is elegant but over-engineered for now. The skill set changes maybe once per week. A hand-maintained registry with 15 entries is 5 minutes to update.

Ship A. Add B's `taskCategory` field to `PipelineOutcomeRecord` as a no-op extension point (populate it, don't act on it yet). Revisit B when there's enough category-specific data to matter.

## Design Decisions

**`--todo` matching:** Exact title match. On mismatch, CLI prints available TODO titles from TODOS.md and exits with error. Fuzzy matching via `slugify()`/Levenshtein is a follow-up, not in scope.

**`--todo` + cross-instance dedup:** `--todo` is a human override. It bypasses composition but NOT cross-instance dedup. If another instance has the same TODO claimed, the trigger returns an error. The human can stop that instance first.

**`--todo` + TODO state trimming:** `skipComposition` bypasses `composePipeline()` only. TODO state trimming (`getStartSkill`/`findNextSkill`) still applies. If you say `--todo "X" implement qa` but X is already implemented, it trims to `qa`. This is correct: state tracking prevents wasted work regardless of how the pipeline was composed.

**`--todo` with no skills:** CLI validation error. `--todo` requires at least one skill argument.

**Skill costs:** The skill catalog includes `costUsd` for Oracle prompt context ("this skill costs ~$0.80"). `SKILL_COST_USD` in `pipeline-compose.ts` stays as the authoritative source for savings calculation. Two consumers, slightly different purposes. When the skill catalog is imported by `pipeline-compose.ts` in a future cleanup, they'll unify.

**`mode` field:** Include `mode: "plan" | "exec"` in `SkillEntry` and use it in `formatSkillCatalogForPrompt()` to group skills into "Review Skills" and "Execution Skills" sections. This helps the Oracle understand that plan-mode skills produce findings (not code) and can be parallelized in future.

## Open Questions

1. Should the skill catalog token estimate be validated with actual tokenization before shipping? The 800-token estimate may be closer to 1,200-1,500 tokens for 10 entries. Either way it fits within the prioritize prompt budget, but accuracy matters for documentation.

## Success Criteria

1. `daemon trigger --todo "Oracle-Driven Skill Selection" implement qa` runs exactly `[implement, qa]` with no composition, no stripping.
2. Prioritize skill output recommends `design-review → implement → qa` for a TODO with visual/UX keywords, when the skill catalog is present in its prompt.
3. Static table still works as fallback when prioritize doesn't output a `### Recommended Pipeline` section.
4. All existing tests pass. New tests cover: `--todo` flag parsing, `skipComposition` bypass, skill catalog structure, prioritize prompt with catalog injection.
5. `taskCategory` field present on `PipelineOutcomeRecord` and populated by reflection, even though nothing reads it yet.

```typescript
function inferTaskCategory(todoTitle: string, todoDescription?: string): string {
  const text = `${todoTitle} ${todoDescription ?? ""}`.toLowerCase();
  if (/\b(ui|ux|visual|design|css|style|layout|animation|responsive)\b/.test(text)) return "visual-ux";
  if (/\b(refactor|cleanup|consolidat|dedupe?|dry)\b/.test(text)) return "refactor";
  if (/\b(architect|types\.ts|interface|module|abstraction)\b/.test(text)) return "architectural";
  if (/\b(bug|fix|regression|crash|error|broken)\b/.test(text)) return "bug-fix";
  if (/\b(perf|performance|latency|speed|cache|optimize)\b/.test(text)) return "performance";
  if (/\b(infra|deploy|ci|cd|docker|k8s|config)\b/.test(text)) return "infra";
  return "unknown";
}
```

### Migration

- `pipeline-outcomes.jsonl`: Adding `taskCategory` field is backward-compatible. Existing records without it are valid (field is optional). No migration needed.
- `Job` interface: Adding `skipComposition` is backward-compatible. Existing jobs without it default to `false` (current behavior).

### Cost Impact

- Skill catalog adds ~800 tokens to the prioritize prompt. Current prioritize prompt is ~4K tokens. This is well within budget.
- `--todo` deterministic mode has zero cost impact (bypasses composition, no new API calls).
- `taskCategory` inference is a regex check, zero API cost.

## What I noticed about how you think

- The TODOS.md entry for this item already separated the immediate fix (pass through unknown skills) from the full solution (Oracle skill selection). That's the right instinct: ship the unblock, then build the intelligence.
- The two-mode framing (deterministic override vs. Oracle analysis) is clean. They solve the same root problem from opposite ends: human-knows-best and daemon-learns-best.
- The existing Oracle pipeline override path (`parsePipelineRecommendation` + cold-start gate) is well-designed scaffolding. This design extends it rather than replacing it.
- The skill catalog as a static registry is the right call for now. Auto-discovery is sexy but premature when the skill set changes weekly at most.
