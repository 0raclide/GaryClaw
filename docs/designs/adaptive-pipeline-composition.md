# Design: Adaptive Pipeline Composition

## Problem Statement

Every TODO item runs the full 5-skill pipeline (`prioritize -> office-hours -> implement -> plan-eng-review -> qa`) costing $3-4 and 30 minutes, regardless of task complexity. A P5/XS typo fix only needs `implement -> qa` ($0.50, 5 min). With 4+ small items typically in the backlog per overnight run, this wastes $10-14 and 2+ hours of daemon time.

The existing todo-state system trims pipelines based on *prior progress* (e.g., skip office-hours if a design doc already exists). But it cannot trim based on *task characteristics* because the initial skill list is always the full pipeline.

## What Makes This Cool

The daemon becomes 4x more productive on the same budget. Instead of grinding through 2 items overnight at $4 each, it blasts through 8 items at $0.50-$2 each. You wake up to a dashboard showing 8 completions instead of 2. The daemon learns to match effort to ambition.

## Constraints

- Must not break existing trigger configs (`skills: string[]` in DaemonConfig)
- Must compose with todo-state trimming (composition happens *before* state-based trimming)
- Manual `daemon trigger qa` must bypass composition (no TODO context available)
- CLI `garyclaw run implement qa` must bypass composition (user chose explicit skills)
- Must be testable without SDK calls (pure function, synthetic data)

## Premises

1. A static lookup table mapping `(effort, priority)` to skill sequences is sufficient. No ML or historical learning needed for Phase 1.
2. The composition logic belongs in job-runner.ts `processNext()`, between TODO pre-assignment and todo-state trimming.
3. Trigger `skills` fields become a "maximum pipeline" that composition selects a subset of. Backward-compatible.
4. Only items with a `claimedTodoTitle` (and therefore parsed TODO metadata) get adaptive composition. All others run their full skill list.
5. The composition table should be a dedicated module (`src/pipeline-compose.ts`) for testability, not inlined in job-runner.

## Approaches Considered

### Approach A: Static Lookup Table (CHOSEN)

A `composePipeline()` function in a new `src/pipeline-compose.ts` module. Maps `(effort, priority, hasDesignDoc)` to a skill sequence via a hardcoded table. ~80 lines of new code plus ~15 lines of wiring in job-runner.ts.

**Effort:** S
**Risk:** Low
**Pros:**
- Dead simple, fully testable, easy to reason about
- One new module, clean separation from job-runner
- Covers 90%+ of cases with 5-6 rules

**Cons:**
- No learning from historical data
- Must manually add rules for new patterns
- Doesn't handle exotic edge cases (e.g., "XS effort but actually needs design")

### Approach B: Rule Engine with Description Analysis

Keyword-based analysis of TODO descriptions (`bug`, `typo`, `rename` -> light pipeline; `feature`, `architecture`, `design` -> full pipeline). Configurable via `.garyclaw/pipeline-rules.json`.

**Effort:** M
**Risk:** Medium
**Pros:** More granular, configurable without code changes
**Cons:** Keyword matching is fragile, more complexity, config needs documentation

### Approach C: Inline Bypass in Job Runner

15-line change directly in `processNext()`: if effort is XS/S and no design doc, strip office-hours and plan-eng-review. No new module.

**Effort:** XS
**Risk:** Low
**Pros:** Ships in 5 minutes, captures 80% of savings
**Cons:** Business logic buried in job-runner, not independently testable, not extensible

## Recommended Approach

**Approach A: Static Lookup Table.** It captures the full savings with clean architecture while staying S effort. Approach C is tempting but puts business logic inline. Approach B over-engineers for a problem with clear, static rules.

## Implementation Plan

### Step 1: New module `src/pipeline-compose.ts` (~80 lines)

The core function:

```typescript
export interface ComposeInput {
  effort: string | null;       // from TodoItem.effort: XS/S/M/L/XL
  priority: number;            // from TodoItem.priority: 1-5
  hasDesignDoc: boolean;       // design doc already exists in docs/designs/
  requestedSkills: string[];   // original trigger skills (the "maximum")
}

export interface ComposeResult {
  skills: string[];            // composed skill sequence
  reason: string;              // human-readable explanation for logging
  savings: string;             // estimated savings vs full pipeline
}

export function composePipeline(input: ComposeInput): ComposeResult
```

**Composition rules (priority order):**

| Effort | Priority | Has Design Doc | Result |
|--------|----------|----------------|--------|
| XS     | any      | no             | `implement -> qa` |
| XS     | any      | yes            | `implement -> qa` |
| S      | P4-P5    | no             | `implement -> qa` |
| S      | P4-P5    | yes            | `implement -> qa` |
| S      | P2-P3    | no             | `office-hours -> implement -> qa` |
| S      | P2-P3    | yes            | `implement -> plan-eng-review -> qa` |
| S      | P1       | any            | full pipeline (critical priority = full rigor) |
| M      | any      | no             | `prioritize -> office-hours -> implement -> plan-eng-review -> qa` |
| M      | any      | yes            | `implement -> plan-eng-review -> qa` |
| L/XL   | any      | any            | full pipeline (should have been split by prioritize) |
| null   | any      | any            | full pipeline (unknown effort, be conservative) |

**Priority scale:** P1 = highest priority (critical), P5 = lowest (nice-to-have). P1 items follow M+ effort rules (full pipeline) regardless of effort tag, because critical items deserve full design and review rigor.

**Key invariants:**
- `implement` and `qa` are ALWAYS present. Every change needs implementation and verification.
- `prioritize` is only included for M+ effort or P1 priority (XS/S items are pre-assigned by job-runner, no need to re-prioritize).
- `office-hours` is skipped when effort is XS, effort is S with P4-P5, or a design doc already exists for S/P2-P3 items. Trivial and low-priority items don't need design thinking.
- `plan-eng-review` is skipped for XS items, S items without a design doc, and S/P4-P5 items. The review overhead exceeds the risk for tiny or low-priority changes.

**Intersection with requested skills:** The composed list is intersected with `requestedSkills` to ensure we never *add* skills the trigger didn't request. If the trigger only has `["implement", "qa"]`, composition can only remove, never add.

### Step 2: Wire into job-runner.ts `processNext()` (~15 lines)

After the TODO pre-assignment block and before the todo-state trimming block. `parsedItems` is the array from `parseTodoItems()` in the pre-assignment block (already in scope at this point). For `hasDesignDoc`, we check `nextJob.designDoc` (set by pre-assignment from stored state) and also scan `docs/designs/` for a slug match:

```typescript
// ── Adaptive pipeline composition ────────────────────
if (todoTitle && nextJob.skills.length > 1) {
  try {
    const todoItem = parsedItems?.find(i => i.title === todoTitle);
    const hasDesignDoc = !!nextJob.designDoc;
    const composed = composePipeline({
      effort: todoItem?.effort ?? null,
      priority: todoItem?.priority ?? 3,
      hasDesignDoc,
      requestedSkills: nextJob.skills,
    });
    if (composed.skills.length < nextJob.skills.length) {
      d.log("info", `Adaptive composition: [${nextJob.skills.join(", ")}] -> [${composed.skills.join(", ")}] (${composed.reason})`);
      nextJob.skills = composed.skills;
    }
  } catch (err) {
    d.log("warn", `Pipeline composition failed: ${err instanceof Error ? err.message : String(err)}`);
    // Fail-open: use original skills
  }
}
```

**Integration order in `processNext()`:**
1. Pre-assignment (picks TODO item, sets `claimedTodoTitle`, parses TODOS.md into `parsedItems`)
2. **NEW: Adaptive composition** (selects minimal skill set based on item metadata from `parsedItems`)
3. TODO state trimming (reconciles state from artifacts, skips already-completed stages)
4. Build config and run pipeline

Note: composition uses `parsedItems` from step 1 (already in scope) and `nextJob.designDoc` (set by pre-assignment from stored todo-state). It does NOT depend on `reconciledState` from step 3, so the ordering is clean.

## Open Questions

1. **Should composition be configurable via `.garyclaw/composition-rules.json`?** Deferred to Phase 2. The static table covers all current use cases. Config adds complexity for zero current benefit.

2. **What about `evaluate` and `bootstrap` skills?** These are special-purpose skills that only appear in specific pipelines (dogfood, external repo). Composition only affects the standard `prioritize -> office-hours -> implement -> plan-eng-review -> qa` flow. Other skills pass through unchanged.

3. **Should the daemon log show "would have run X but composed to Y"?** Yes, the `pipeline_composed` event covers this. Important for debugging when composition makes a wrong call.

## Success Criteria

- XS effort items run `implement -> qa` (2 skills, ~$0.50, ~5 min)
- S/P4+ items run `implement -> qa` (2 skills, ~$0.50, ~5 min)
- S/P2-P3 items run 3 skills ($1-2, ~15 min)
- M items run full pipeline (5 skills, $3-4, ~30 min)
- Overnight run with 6 mixed items completes all 6 for ~$8 instead of 2 for ~$8
- Zero regressions: existing trigger configs, manual triggers, CLI runs all work unchanged
- 40+ tests covering all composition rules and edge cases
