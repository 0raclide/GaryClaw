## Problem Statement

The dashboard backend shipped (1069 lines in `dashboard-server.ts`), web assets shipped (index.html, style.css, app.js, charts.js, 2311 lines total), growth data pipeline shipped (`buildGrowthSnapshots`, `buildModuleAttribution`, `loadOrBuildGrowthCache`), and QA fixed 3 issues. Two of the original four views are fully functional (Growth, Mutations). Two have gaps:

1. **Live tab is cards, not a feed.** Shows health/budget/oracle/skill-cost cards that update via SSE. But there's no streaming activity log showing events as they happen. The SSE infrastructure broadcasts `job_update`, `budget`, `decision`, `mutation`, `taste_update`, `expertise_update` events. The client listens for all of them. But it only uses them to update stat cards. The "Twitch for AI agents" experience requires a scrolling activity feed where each event appears as a timestamped line.

2. **Mind tab missing taste and expertise.** Shows decision stats, confidence sparkline, and searchable decision log (all working). But `taste.md` (the daemon's learned preferences) and `domain-expertise.md` (researched domain knowledge) aren't rendered. The SSE file watcher already emits `taste_update` and `expertise_update` events. The client already listens for them (app.js lines 570-577). But the Mind tab renderer (`renderOracleMind()`) never fetches or displays this content.

## What Makes This Cool

The activity feed turns the Live tab into something you leave open while working. Events scroll by as the daemon thinks, decides, builds, and ships. Oracle decisions appear with confidence scores. Budget ticks update. Mutations ship. It's the "Twitch chat" for your AI agent.

The taste profile and domain expertise in the Mind tab give the daemon a personality. You see it has preferences ("prefers vitest over jest", "names tests descriptively") and knowledge areas with freshness dates. This is the "daemon's mind" content that makes the tab name honest.

## Constraints

- No new modules. All work in existing files: `dashboard-server.ts`, `app.js`, `style.css`, possibly `index.html`
- No external dependencies (vanilla JS, node builtins only)
- No build step for web assets
- Must work when daemon isn't running (historical data mode... activity feed shows "No live activity" empty state)
- All reads are fail-open (missing taste.md = empty section, not crash)
- No new REST endpoints. Extend existing `/api/state` response for taste/expertise content.

## Premises

1. **The SSE infrastructure is complete and correct.** `FILE_EVENT_MAP` in dashboard-server.ts maps all relevant files to event types. `createFileWatcher` watches `.garyclaw/` recursively. `broadcastSSE` sends typed events. The client's `connectSSE()` handles all event types. Zero backend SSE changes needed.

2. **The activity feed is 100% client-side state.** Add `state.activityLog = []` to app.js state. Each SSE event handler appends a `{ time, icon, text, type }` entry. `renderActivityFeed()` builds a scrollable list. Max 200 entries, oldest pruned. No backend changes for this feature.

3. **Taste and expertise content needs one server-side addition.** Add `tasteProfile: string` and `domainExpertise: string` fields to the `/api/state` response in `loadDashboardData()`. Read via `safeReadText()` from oracle-memory directories. This is ~10 lines in dashboard-server.ts. On SSE `taste_update`/`expertise_update` events, the client re-fetches `/api/state` to get fresh content (same pattern as the existing `mutation` handler which re-fetches `/api/mutations`).

4. **Simple markdown rendering covers the content.** Both `taste.md` and `domain-expertise.md` use predictable structure: `##` headings, `- ` bullet lists, `**bold**`, YAML frontmatter (domain-expertise only). A 40-line regex-based renderer handles these patterns. No full markdown parser needed.

5. **All changes are in 3-4 files.** `dashboard-server.ts` (~15 lines for taste/expertise in API response), `app.js` (~200 lines for activity feed + taste/expertise rendering + domain expertise per-topic parser), `style.css` (~50 lines for activity feed + freshness badge styles), optionally `index.html` (DOM containers for activity feed section if needed).

## Recommended Approach

**Approach B: Full Completion.** The activity feed is the hook, the taste/expertise content completes the mind. Together they turn "half the views work" into "all four views deliver." Approach A leaves the Live tab underwhelming. Approach C is cool but scope creep.

## Architecture

### 1. Server: Add Taste/Expertise to `/api/state`

In `dashboard-server.ts`, modify `loadDashboardData()` to include taste and expertise content:

```typescript
// After building DashboardData, add oracle mind content
const memConfig = defaultMemoryConfig(projectDir);

// Taste: prefer global (user-wide preferences) over project-specific
// because taste is personal style that carries across all projects
const tasteProfile = safeReadText(join(memConfig.globalDir, "taste.md"))
  ?? safeReadText(join(memConfig.projectDir, "taste.md"))
  ?? "";

// Expertise: prefer project (domain knowledge specific to this codebase)
// over global because expertise is researched per-project context
const domainExpertise = safeReadText(join(memConfig.projectDir, "domain-expertise.md"))
  ?? safeReadText(join(memConfig.globalDir, "domain-expertise.md"))
  ?? "";
```

Return these as additional fields on the response object. The `DashboardData` type in `types.ts` does NOT need modification... these are dashboard-server-specific response extensions. Use a local response type or spread operator.

**Truncation:** Cap each string at 10K chars to prevent oversized responses.

### 2. Client: Activity Feed in Live Tab

Add to `app.js`:

**State:**
```javascript
state.activityLog = [];  // Array of { time: string, icon: string, text: string, type: string }
```

**Activity log population:** Each SSE event handler appends an entry:
- `job_update` -> "Job {status}: {skills}" (icon: briefcase)
- `budget` -> "Budget: ${amount} spent today" (icon: dollar)
- `decision` -> 'Oracle: "{chosen}" ({confidence}/10)' (icon: brain)
- `mutation` -> "Mutation {outcome}: {todoTitle}" (icon: DNA)
- `taste_update` -> "Taste profile updated" (icon: sparkle)
- `expertise_update` -> "Domain expertise updated" (icon: book)

**Cap:** Max 200 entries. Prune oldest when exceeded.

**Rendering:** `renderActivityFeed()` builds a scrollable `<div class="activity-feed">` with timestamped lines. Each line has: `<span class="activity-time">HH:MM:SS</span> <span class="activity-icon">{icon}</span> <span class="activity-text">{text}</span>`.

**Auto-scroll:** Scroll to bottom on new entries UNLESS user has scrolled up (detect via `scrollTop + clientHeight < scrollHeight - 50`). Pulsing green dot next to "LIVE ACTIVITY" header when connected.

**Empty state:** When `activityLog.length === 0` and SSE is connected, show "Watching for events..." with a pulsing dot. When disconnected, show "Reconnecting..." in amber.

**Integration with existing Live tab:** Activity feed renders ABOVE the existing stat cards. The cards remain (they're useful). The feed adds the streaming dimension.

**SSE re-fetch for taste/expertise:** The existing `taste_update` and `expertise_update` SSE handlers in app.js (lines 570-577) already trigger `renderOracleMind()` on the active Mind tab. Update these handlers to also re-fetch `/api/state` so the fresh taste/expertise content is loaded:
```javascript
source.addEventListener("taste_update", function () {
  fetch("/api/state").then(function (r) { return r.json(); }).then(function (data) {
    state.dashboardData = data;
    if (state.activeTab === "mind") renderOracleMind();
  }).catch(function () { /* best-effort */ });
});
```
Same pattern for `expertise_update`. This is the same approach the `mutation` handler uses (line 561: re-fetches `/api/mutations` on mutation events).

### 3. Client: Taste + Expertise in Mind Tab

Add two sections to `renderOracleMind()` in `app.js`, rendered AFTER the decision log:

**Taste Profile section:**
- Card with title "Taste Profile"
- Content: `renderSimpleMarkdown(state.dashboardData.tasteProfile)`
- If empty: "No taste profile learned yet."

**Domain Expertise section:**
- Card with title "Domain Expertise"
- Content: `renderSimpleMarkdown(state.dashboardData.domainExpertise)`
- Parse YAML frontmatter per topic for freshness dates
- Display freshness badges: green (< 14 days), amber (14-30 days), red (> 30 days)
- If empty: "No domain expertise researched yet."

**Simple markdown renderer (`renderSimpleMarkdown`):**
```javascript
function renderSimpleMarkdown(md) {
  if (!md) return "";
  var html = md
    .replace(/^---[\s\S]*?---\n?/m, "")           // strip YAML frontmatter
    .replace(/^### (.+)$/gm, "<h4>$1</h4>")        // h3 -> h4 (avoid hierarchy clash)
    .replace(/^## (.+)$/gm, "<h3>$1</h3>")         // h2 -> h3
    .replace(/\*\*(.+?)\*\*/g, "<strong>$1</strong>") // bold
    .replace(/^- (.+)$/gm, "<li>$1</li>")           // bullets
    .replace(/\n{2,}/g, "<br>")                     // paragraph breaks
    .trim();
  // Wrap contiguous <li> runs in <ul> (don't merge non-adjacent lists)
  html = html.replace(/((?:<li>.*?<\/li>\s*)+)/g, "<ul>$1</ul>");
  return html;
}
```

**Known limitation:** The regex-based renderer handles the predictable structure of taste.md and domain-expertise.md (headings, bullets, bold). It won't handle nested lists, code blocks, or links. These don't appear in either file today. If they appear in the future, the content still renders (just without formatting), which is acceptable fail-open behavior.

**Domain expertise per-topic frontmatter parser (`renderDomainExpertise`):**

`domain-expertise.md` contains multiple topic sections, each with its own YAML frontmatter:
```markdown
## Topic Name
---
lastResearched: 2026-03-28
source: web-search
---
Content here...
```

The renderer needs a dedicated function that splits on `## ` headings, extracts per-section frontmatter, and emits freshness badges:

```javascript
function renderDomainExpertise(md) {
  if (!md) return '<div class="text-dim">No domain expertise researched yet.</div>';
  // Split on ## headings, preserving the heading text
  var sections = md.split(/^## /gm).filter(Boolean);
  var html = "";
  for (var i = 0; i < sections.length; i++) {
    var lines = sections[i].split("\n");
    var title = lines[0].trim();
    var body = lines.slice(1).join("\n");
    // Extract per-section YAML frontmatter
    var lastResearched = null;
    var fmMatch = body.match(/^---\n([\s\S]*?)\n---/m);
    if (fmMatch) {
      var dateMatch = fmMatch[1].match(/lastResearched:\s*(\S+)/);
      if (dateMatch) lastResearched = dateMatch[1];
      body = body.replace(/^---\n[\s\S]*?\n---\n?/m, "");
    }
    // Freshness badge
    var badge = "";
    if (lastResearched) {
      var age = (Date.now() - new Date(lastResearched).getTime()) / 86400000;
      var cls = age < 14 ? "freshness-fresh" : age < 30 ? "freshness-aging" : "freshness-stale";
      var label = age < 14 ? "fresh" : age < 30 ? "aging" : "stale";
      badge = ' <span class="freshness-badge ' + cls + '">' + label + "</span>";
    }
    html += "<h3>" + escapeHtml(title) + badge + "</h3>";
    html += renderSimpleMarkdown(body);
  }
  return html;
}
```

### 5. Data Flow Summary

```
[daemon writes to .garyclaw/ files]
  → fs.watch in dashboard-server.ts detects change
  → mapFileToEventType() classifies the change
  → broadcastSSE() sends typed event to all clients
  → app.js SSE handler:
      1. Updates stat cards (existing behavior, unchanged)
      2. Appends to state.activityLog (NEW)
      3. Calls renderActivityFeed() to update Live tab (NEW)
  → For taste/expertise: /api/state includes content on page load
      + taste_update/expertise_update SSE events trigger re-fetch (existing handlers)
```

## Open Questions

1. **Should the activity feed persist across page refreshes?** Current design: no, the feed starts empty and fills from SSE events. Historical events are visible via Mutations and Mind tabs. Keeping the feed ephemeral is simpler and matches the "live stream" metaphor.

2. **Should taste/expertise be lazy-loaded?** Current design: included in `/api/state` response (loaded on page open). These are typically small files (< 5K each). Including them avoids an extra fetch and simplifies the client.
