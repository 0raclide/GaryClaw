## Problem Statement

The TODO says: "After the first dogfood run on an external repo, evaluate bootstrap output quality." NihontoWatch has had multiple dogfood runs. The claim verification infrastructure is fully built (`extractClaudeMdClaims`, `verifyClaudeMdClaims`, `extractCommandClaims`, `extractTestDirectoryClaims` — 71 tests across evaluate-claims.test.ts and evaluate-semantic-validation.test.ts).

Running `analyzeBootstrapQuality()` against NihontoWatch produces **score: 44/100** (below the 50-point enrichment threshold). But analysis of the 17 failed claims reveals **two bugs in the evaluation system itself** that inflate the failure count by ~10 claims. The real score should be closer to 58-65.

The evaluation is grading its own homework wrong. Before we can trust the quality gate to make enrichment decisions, we need to fix the verifier.

## What Makes This Cool

GaryClaw ran its evaluation on real data, and the evaluation found problems... with itself. The test_directory verifier has a scoping bug that fails every per-feature test count claim. The tech_stack verifier doesn't understand indirect dependencies (Supabase = PostgreSQL). Fixing these makes the quality gate trustworthy, which means the enrichment loop actually works, which means bootstrap gets better every run.

Self-debugging AI tooling. The system found its own bug by exercising itself.

### Failed Claims — Analysis

**Bug 1: test_directory count verification (8 false positives)**

NihontoWatch's CLAUDE.md lists per-feature test counts in tables:
- `| Tests (40) |` — 40 tests for the weekly digest feature
- `| Tests (159) |` — 159 tests for private offers
- `| Tests (93) |` — 93 tests for collector onboarding

`extractTestDirectoryClaims()` extracts each as `test-count:40`, `test-count:159`, etc. Then `verifyClaudeMdClaims()` in the `case "test-count":` branch calls `countTestFiles(projectDir, testPattern)` which counts ALL test files in the repo (375). Each per-feature count is compared against 375. All 8 fail.

This is wrong. These are per-feature counts, not repo-wide totals. The verifier doesn't understand context.

**Bug 2: Indirect dependency blindness (1 false positive)**

NihontoWatch claims "PostgreSQL" as its database. The verifier checks for `pg` or `@types/pg` in package.json. Neither exists because NihontoWatch uses Supabase (`@supabase/supabase-js`), which connects to PostgreSQL internally. The claim is factually correct but the verifier can't see through the abstraction.

**4 reverse coverage "failures" (not bugs, but debatable)**

- React: Not mentioned in CLAUDE.md because it says "Next.js 15+". Next.js implies React. Debatable whether this should be a separate mention.
- Vitest: Genuinely missing from CLAUDE.md prose (it's in scripts but not documented). Real gap.
- Playwright: Same — in devDeps, not mentioned in docs. Real gap.
- ESLint: Same pattern. Real gap.

**8 file_path failures (real)**

8 files referenced in CLAUDE.md don't exist: `src/app/api/track/route.ts`, `src/app/collection/page.tsx`, `src/contexts/CollectionQuickViewContext.tsx`, `src/components/dealer/AccentColorPicker.tsx`, `src/app/vault/CollectionPageClient.tsx`, `tests/api/messaging/offer`, `tests/components/messaging/Offer`, `src/lib/supabase/listingVideos.ts`. These are genuine stale references in the hand-maintained CLAUDE.md.

## Constraints

- Fixes go in `src/evaluate.ts`. No new modules.
- Must remain pure TypeScript, zero Claude calls.
- Backward compatible: existing `BootstrapEvaluation` type, existing test suite.
- Must not increase quality gate latency measurably.
- NihontoWatch is the only real-world validation target, but fixes must be general.

## Premises

1. The test_directory count verifier comparing per-feature counts against repo-wide totals is a bug, not a design choice. Per-feature counts are the more common pattern in real CLAUDE.md files.
2. Indirect dependency resolution (Supabase → PostgreSQL) is worth adding for the most common cases (Supabase→PG, Prisma→PG, Drizzle→PG), not for arbitrary transitive deps.
3. The 4 reverse coverage failures (React/Vitest/Playwright/ESLint not mentioned) are real gaps in NihontoWatch's CLAUDE.md, not evaluation bugs. The verifier is correct here.
4. The 8 stale file paths are real failures. These are documentation drift in a hand-maintained file.
5. Fixing these two bugs is the right scope. The evaluation infrastructure is otherwise solid.

## Approaches Considered

### Approach A: Fix Verifier Bugs (Minimal Viable)

**Summary:** Fix the two bugs in `verifyClaudeMdClaims()`. For test counts, skip per-feature count verification entirely (mark as `verified: true` with evidence "per-feature count, aggregate verification only") and only verify if a single aggregate count is close to the total. For indirect deps, extend the existing `KNOWN_FRAMEWORKS["PostgreSQL"]` entry to include Supabase/Prisma/Drizzle packages.

**Effort:** S
**Risk:** Low
**Pros:**
- ~40 lines of changes in evaluate.ts
- Fixes the two concrete bugs found in real data
- Corrects the quality gate for NihontoWatch immediately
- No new abstractions or modules
**Cons:**
- Skipping per-feature count verification loses some signal
- Extending KNOWN_FRAMEWORKS with indirect deps is a manual list that grows over time
**Reuses:** Existing `KNOWN_FRAMEWORKS`, `countTestFiles()`, `verifyClaudeMdClaims()`

### Approach B: Context-Aware Test Count Verification

**Summary:** Instead of skipping per-feature counts, extract the directory context from the surrounding table/prose. When CLAUDE.md says `| Tests (40) | tests/lib/digest/*.test.ts |`, extract the glob pattern, count matching files, and compare against 40. Falls back to aggregate-only when no directory context is found.

**Effort:** M
**Risk:** Medium
**Pros:**
- Verifies per-feature counts correctly
- Catches real documentation drift in per-feature claims
- More accurate quality scoring
**Cons:**
- ~120 lines of new parsing logic for table context extraction
- Fragile: depends on CLAUDE.md table format consistency
- Over-engineers for the current need (1 dogfood target)
- Regex parsing of markdown tables is inherently brittle

### Approach C: Aggregate-Only With Tolerance Band

**Summary:** Replace all per-feature count claims with a single aggregate claim. Sum all extracted counts and compare against total. Use a generous tolerance band (±30%) to account for the fact that per-feature counts may overlap or exclude some test files. Add the KNOWN_ALIASES fix from Approach A.

**Effort:** S
**Risk:** Low
**Pros:**
- Simple, testable, no fragile table parsing
- Still catches wildly wrong test counts (off by 2x+)
- Tolerance band handles the overlap/exclusion ambiguity
- Ships fast
**Cons:**
- Loses per-feature granularity
- Sum of per-feature counts may legitimately exceed total (shared test files)
- 30% tolerance is a guess

## Recommended Approach

**Approach A: Fix Verifier Bugs.** The per-feature count verification is fundamentally ambiguous (counts may overlap, may be per-directory, may be per-feature-area). Skipping it in favor of aggregate-only is the honest thing to do. The PostgreSQL indirect dep fix extends the existing `KNOWN_FRAMEWORKS` entry (no new map, no new lookup path). Only the packages with observed real-world false negatives are added (Supabase, Prisma, Drizzle). Adding MySQL/MongoDB/Redis/SQLite speculatively is YAGNI. Ships in S effort, directly fixes the two bugs found in real data.

If per-feature accuracy matters later, Approach B can be layered on top. But right now, the priority is making the quality gate trustworthy, not making it maximally granular.

## Implementation Details

### 1. Test Count Verification Fix

In `verifyClaudeMdClaims()`, the `case "test_directory"` branch, `test-count:` sub-case:

**Current:** Each `test-count:N` claim is compared individually against `countTestFiles(projectDir)` (repo-wide total) inside the `.map()` loop. No cross-claim visibility.

**Fixed:** Add a pre-pass before the `.map()` loop to count how many `test-count` claims exist. Reference that count inside the per-claim case. This gives the verifier cross-claim visibility without changing the function signature.

```typescript
// In verifyClaudeMdClaims(), BEFORE the return claims.map(...) line:
const testCountClaimCount = claims.filter(
  c => c.type === "test_directory" && c.claimed.startsWith("test-count:")
).length;

// Inside the .map() callback, case "test_directory", test-count sub-case:
if (claim.claimed.startsWith("test-count:")) {
  const claimedCount = parseInt(claim.claimed.slice("test-count:".length), 10);

  if (testCountClaimCount > 1) {
    // Multiple per-feature counts: skip individual verification.
    // These are per-feature breakdowns, not repo-wide totals.
    let actualTotal: number;
    try {
      actualTotal = countTestFiles(projectDir, testPattern);
    } catch {
      actualTotal = 0; // surface the count anyway
    }
    verified.verified = true;
    verified.evidence = `per-feature count (${actualTotal} total test files in repo)`;
    break;
  }

  // Single aggregate count: verify against actual total with 20% tolerance
  let actualTotal: number;
  try {
    actualTotal = countTestFiles(projectDir, testPattern);
  } catch {
    verified.verified = true;
    verified.evidence = "could not count test files, verification deferred";
    break;
  }
  const tolerance = Math.max(3, Math.ceil(claimedCount * 0.2));
  if (Math.abs(actualTotal - claimedCount) <= tolerance) {
    verified.verified = true;
    verified.evidence = `actual count: ${actualTotal} (within tolerance of claimed ${claimedCount})`;
  } else {
    verified.verified = false;
    verified.evidence = `claimed ${claimedCount}, found ${actualTotal} (±20% tolerance failed)`;
  }
}
```

Key behaviors:
- **Multiple `test-count` claims** (per-feature pattern): each marked `verified: true` with evidence showing the aggregate repo count for context.
- **Single `test-count` claim within 20% tolerance**: verified as correct aggregate.
- **Single `test-count` claim outside 20% tolerance**: fails with explicit evidence string.
- **`countTestFiles` throws**: deferred (marked verified), same as current behavior. This preserves the existing error-handling contract.
- **Evidence string format**: `"per-feature count (N total test files in repo)"` for multi-count, `"claimed N, found M (±20% tolerance failed)"` for single-count failures. Test assertions should match these exact formats.

### 2. Extend KNOWN_FRAMEWORKS for Indirect Dependencies

`KNOWN_FRAMEWORKS` already has `["PostgreSQL", ["pg", "@types/pg"]]` at line 83 of evaluate.ts. The `tech_stack` verification case looks up claim names in this map. The fix is simple: extend the existing PostgreSQL entry to include packages that imply PostgreSQL usage (Supabase, Prisma, Drizzle, etc.). No new map, no new lookup path, no structural change.

```typescript
// Before (evaluate.ts line 83):
["PostgreSQL", ["pg", "@types/pg"]],

// After — add indirect dependency packages:
// Supabase, Prisma, Drizzle all connect to PostgreSQL internally.
// Only packages with observed real-world false negatives are added.
["PostgreSQL", ["pg", "@types/pg", "@supabase/supabase-js", "@prisma/client", "prisma", "drizzle-orm"]],
```

The existing `tech_stack` verification logic in `verifyClaudeMdClaims()` is unchanged. It already does `frameworkPackages.filter(pkg => depSet.has(pkg))`. Adding `@supabase/supabase-js` to the PostgreSQL entry means a Supabase project now matches when "PostgreSQL" is claimed.

**Why not a separate KNOWN_ALIASES map?** The reviewer caught that `KNOWN_FRAMEWORKS.get("PostgreSQL")` already returns a value, so any KNOWN_ALIASES fallback would never fire for this key. Extending the existing entry is simpler, requires zero logic changes, and the verification path stays one lookup.

**Scope guard:** Only the PostgreSQL entry is modified. Other databases (MySQL, MongoDB, Redis, SQLite) are not touched because there are no observed false negatives for them. Add entries when real dogfood data surfaces a false negative, not speculatively.

## Open Questions

1. **Should per-feature counts contribute to an aggregate check?** Summing them risks double-counting (shared test files appear in multiple features). Current design skips this — aggregate only uses `countTestFiles()`. Acceptable for v1.
2. **KNOWN_FRAMEWORKS indirect dep ambiguity:** `drizzle-orm` is now listed under PostgreSQL, but Drizzle also supports MySQL and SQLite. If a future repo claims "MySQL" with `drizzle-orm` in deps, the PostgreSQL entry would false-positive. Acceptable for now (only one dogfood target), but worth noting if extending to other databases later.
3. **Should we re-run evaluation on NihontoWatch after the fix?** Yes. A fixture-based test in CI validates the fix permanently. A one-time manual run against the real NihontoWatch repo confirms the fixture is representative.
