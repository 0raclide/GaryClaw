# Design: Route parseBatchOracleResponse console.warn Through Callbacks

Generated by /office-hours on 2026-03-29

## Problem Statement

4 `console.warn` calls in `parseBatchOracleResponse` (oracle.ts:468-490) go to stderr. In daemon mode, stderr is invisible... warnings don't appear in daemon logs, the decision audit trail, or the dashboard. These warnings fire on batch parsing fallback paths, which are exactly the edge cases operators need visibility into when debugging oracle behavior.

## What Makes This Cool

Not cool in a "whoa" way. Cool in a "your daemon stops being a black box" way. These 4 warnings are the only diagnostic output for batch oracle fallback parsing. When parsing degrades from array to individual JSON to complete fallback, right now you have zero visibility in daemon mode. After this fix, every fallback path is observable through the event system like everything else.

## Constraints

- Zero behavioral change. Fallback logic stays identical.
- No breaking changes to `parseBatchOracleResponse` return type.
- Must not regress the 32 existing oracle-batch tests + 4 regression-1 tests.
- `parseBatchOracleResponse` is a pure function. Keep it mostly pure... callback is optional, not required.
- Follow existing codebase patterns (see orchestrator.ts:628-630 for the established approach).

## Premises

1. **The `onWarn` callback pattern is correct for this.** The orchestrator already uses `callbacks.onEvent({ type: "assistant_text", text: ... })` for reflection failures (orchestrator.ts:628). Same pattern applies here.
2. **An optional callback parameter is better than changing the return type.** Returning `{ results, warnings }` would break the return type contract and require updates across all callers and 36+ test assertions. The callback approach changes only the function signature.
3. **Scoping to these 4 warnings is correct.** There are 13 `console.warn` calls across 5 modules (ask-handler, reflection, worktree, oracle). Fixing all of them is a separate, broader effort. These 4 are the highest-value targets because they fire on the batch oracle fallback path... the exact scenario where daemon operators need observability most.
4. **`console.warn` remains the right default when no callback is provided.** Standalone CLI mode should still see warnings on stderr. The callback only replaces stderr routing when explicitly wired.

## Approaches Considered

### Approach A: Optional `onWarn` Callback Parameter (RECOMMENDED)

Add `onWarn?: (msg: string) => void` as the third parameter to `parseBatchOracleResponse`. Replace 4 `console.warn(msg)` calls with `(onWarn ?? console.warn)(msg)`. Thread the callback from `askOracleBatch` (which gets it as a new optional field on `OracleBatchInput` or as a separate param), wired in `ask-handler.ts` where the event system is accessible.

```
Summary: Add optional onWarn callback, default to console.warn
Effort:  XS (4 lines changed in oracle.ts, ~6 lines wiring in ask-handler + orchestrator)
Risk:    Low (optional param, backward compatible, default preserves current behavior)
Pros:
  - Zero breaking changes. Every existing caller works without modification.
  - parseBatchOracleResponse stays a mostly-pure function (callback is side-effect, but optional).
  - Follows the exact pattern already used at orchestrator.ts:628.
Cons:
  - Callback param on a parse function is slightly impure. Purists would prefer Approach B.
  - onWarn is threaded through 3 call layers (orchestrator -> askOracleBatch -> parseBatchOracleResponse).
Reuses: OrchestratorEvent "assistant_text" type, existing callbacks.onEvent wiring.
```

### Approach B: Structured Return with Warnings Array

Change `parseBatchOracleResponse` return to `{ results: Array<...>, warnings: string[] }`. Caller collects warnings and emits them.

```
Summary: Return warnings alongside results as structured data
Effort:  S (return type change + all callers + 36+ test assertion updates)
Risk:    Medium (breaking return type change, test churn)
Pros:
  - Purest approach. No side effects in the parse function.
  - Warnings become first-class data, not fire-and-forget.
Cons:
  - Breaking change to return type. Every caller must destructure.
  - 36+ test assertions reference the current return shape.
  - Overkill for 4 warning strings that are only used for logging.
Reuses: Nothing new.
```

### Approach C: Extend OracleConfig with onWarn

Add `onWarn` to `OracleConfig`, which is already threaded through `askOracle` and `askOracleBatch`. `parseBatchOracleResponse` receives config and uses `config.onWarn`.

```
Summary: Add onWarn to the existing OracleConfig interface
Effort:  S (OracleConfig change + parseBatchOracleResponse signature change + test config updates)
Risk:    Low-Medium (non-breaking addition to interface, but config isn't currently passed to parse fn)
Pros:
  - Callback lives where all oracle configuration lives.
  - If future oracle functions need onWarn, it's already in config.
Cons:
  - parseBatchOracleResponse doesn't currently receive config. Adding it changes the function's role.
  - Couples a pure parse function to the oracle config object.
  - Heavier diff than Approach A for the same result.
Reuses: OracleConfig interface threading.
```

## Recommended Approach

**Approach A.** Smallest diff, zero breaking changes, follows the established pattern from orchestrator.ts:628. The 4 `console.warn` calls become `(onWarn ?? console.warn)(msg)` calls. Done.

## Implementation Plan

### Files to modify:

1. **`src/oracle.ts`** (4 lines changed + 1 param added)
   - Add `onWarn?: (msg: string) => void` as 3rd param to `parseBatchOracleResponse`
   - Replace 4x `console.warn(msg)` with `(onWarn ?? console.warn)(msg)`
   - In `askOracleBatch`, accept `onWarn` and pass it to `parseBatchOracleResponse`

2. **`src/ask-handler.ts`** (~3 lines)
   - In `handleBatchOracle`, construct an `onWarn` closure: `(msg: string) => config.callbacks?.onEvent({ type: "assistant_text", text: msg })`
   - Pass it through `oracle.askOracleBatch(batchInput, oracle.config, onWarn)`
   - ask-handler already receives `callbacks` from the orchestrator, so no orchestrator changes needed

3. **`src/orchestrator.ts`** (0 lines changed)
   - Verify `callbacks` is already threaded to ask-handler (it is). No changes needed.

4. **`src/ask-handler.ts` type update** (~1 line)
   - The `askOracleBatch` function reference on the oracle config object has type `(input: OracleBatchInput, config: OracleConfig) => Promise<OracleOutput[]>`. Update this type to include the optional third `onWarn` param: `(input: OracleBatchInput, config: OracleConfig, onWarn?: (msg: string) => void) => Promise<OracleOutput[]>`

### Files to add:

5. **`test/oracle-batch-warn.test.ts`** (~8-12 tests)
   - Verify each of the 4 fallback paths calls `onWarn` instead of `console.warn`
   - Verify `onWarn` is NOT called on the happy path (valid JSON array with correct length)
   - Verify default behavior (no `onWarn` provided) still calls `console.warn`
   - Verify wiring from ask-handler to onWarn callback

### Call chain after fix:

```
ask-handler.ts: constructs onWarn closure from callbacks.onEvent (already available)
  -> oracle.ts askOracleBatch(batchInput, config, onWarn): passes onWarn to parseBatchOracleResponse
    -> oracle.ts parseBatchOracleResponse(raw, questions, onWarn): calls (onWarn ?? console.warn)(msg)
```

### Existing tests unaffected:

All 32 `oracle-batch.test.ts` tests call `parseBatchOracleResponse(raw, questions)` without the third param. With `onWarn` being optional and defaulting to `console.warn`, these tests continue to pass with zero changes. The `console.warn` calls they currently trigger just continue going to stderr as before.

## Open Questions

1. **Should `onWarn` go on `OracleBatchInput` or as a separate parameter?** Decision: separate param on both `askOracleBatch` and `parseBatchOracleResponse`. It's not "oracle input" (a question for the model), it's infrastructure plumbing. `types.ts` does not need changes... `OracleBatchInput` stays unchanged.
2. **Should we add a new `OrchestratorEvent` type like `oracle_batch_warn` instead of reusing `assistant_text`?** Recommendation: reuse `assistant_text` for now. Adding a new event type means updating CLI `formatEvent`, dashboard parsing, etc. The message prefix `[oracle-batch]` already identifies the source. If we later need structured batch warning data (e.g., for dashboard metrics), we can add a typed event then.
