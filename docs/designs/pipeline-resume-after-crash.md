## Problem Statement

When the daemon restarts after a crash (macOS sleep, OOM, SIGTERM), it marks any running pipeline job as "failed" and moves on. The completed skills in that pipeline ($1-3 worth of work) are thrown away. The daemon starts a fresh pipeline on the next cron cycle, re-doing prioritize and office-hours from scratch.

Over a week of overnight operation with daily macOS sleep events, this wastes $10-20 in abandoned pipelines. The full self-improvement pipeline (`prioritize -> office-hours -> implement -> eng-review -> qa`) costs $3-4 and takes 30-40 minutes. Losing it mid-implement after $1.29 of prioritize + office-hours is real money.

## What Makes This Cool

You close your laptop at midnight. GaryClaw was mid-implement on a feature it designed for itself. You open the laptop at 8am. GaryClaw picks up exactly where it left off, skips the skills it already finished, and completes the pipeline. No lost work. No re-spending. The daemon just... handles it.

## Constraints

- Must be backward-compatible with existing `daemon-state.json` format (add fields, don't break existing)
- Must not retry infinitely. A job that crashes 3 times total (retryCount reaches 3) is marked failed
- Must work with both single-skill jobs and multi-skill pipelines
- Must work with both default instance (main repo) and named instances (worktrees)
- Completed skill checkpoint data must remain valid across daemon restarts
- Must not double-charge completed skills against the job budget

## Premises

1. The primary failure mode is macOS sleep interrupting a running pipeline, not OOM or bugs in GaryClaw itself.
2. Pipeline state in `pipeline.json` is reliable enough to resume from (`safeWriteJSON` prevents corruption via atomic rename).
3. A crashed skill's working directory is recoverable via `git stash --include-untracked` (the relay system already handles dirty state).
4. Retrying a crashed skill at most twice is sufficient. retryCount increments each crash: 1 = first retry, 2 = second retry, 3 = abandon. If retryCount > 2, it's a real bug.
5. Completed skills in a resumed pipeline don't need to re-run. Their checkpoint data and reports are still valid on disk.

## Approaches Considered

### Approach A: Minimal — Retry-on-Restart

**Summary:** Change `job-runner.ts` lines 103-111 to mark crashed `running` jobs as `queued` with a `retryCount` field instead of `failed`. On `processNext()`, detect `retryCount > 0` with pipeline state and call `resumePipeline()` instead of `runPipeline()`.

**Effort:** XS (human: ~1 day / CC: ~10 min)
**Risk:** Low
**Pros:**
- 10-line change to job-runner.ts, 5-line change to types.ts
- Leverages existing `resumePipeline()` which already handles skill status correctly
- Immediate value: stops wasting $10-20/week

**Cons:**
- No budget adjustment for completed skills (job budget counts from 0 on resume)
- No audit trail linking resumed job to original
- No crash classification (can't distinguish sleep vs. OOM vs. bug)

### Approach B: Full Crash Recovery with Budget Adjustment (RECOMMENDED)

**Summary:** Approach A plus: track completed skill costs for dashboard reporting, integrate with failure-taxonomy for crash classification, and send "recovered from crash" notification.

**Effort:** S (human: ~3 days / CC: ~20 min)
**Risk:** Low
**Pros:**
- Correct budget accounting: resumed jobs don't re-pay for completed work
- Observability: dashboard shows crash recoveries and cost saved
- Failure taxonomy integration: crash cause is classified (sleep, OOM, timeout, unknown)
- Notification on recovery: user knows the daemon recovered, not just that it ran
- Dashboard integration: health score accounts for crashes + recoveries

**Cons:**
- More moving parts than Approach A (but all are well-tested subsystems)
- Budget deduction requires reading completed skill costs from pipeline state

## Recommended Approach

**Approach B: Full Crash Recovery with Budget Adjustment.**

It's the complete version without over-engineering. Approach A works but leaves money on the table with double-charging and no observability.

## Implementation Plan

### Step 1: Add `retryCount` to Job interface

In `src/types.ts`, add to the `Job` interface:

```typescript
retryCount?: number;           // How many times this job has been retried after crash (1 = first retry, 2 = second retry)
priorSkillCostUsd?: number;    // Cost of already-completed skills (for dashboard "saved" reporting, not budget checks)
```

### Step 2: Change crash-recovery logic in job-runner.ts

Replace lines 103-111 in `createJobRunner()` (the loop that marks running jobs as failed):

```typescript
// On start, attempt to resume crashed jobs instead of marking them failed
for (const job of state.jobs) {
  if (job.status === "running") {
    const retryCount = (job.retryCount ?? 0) + 1;
    if (retryCount > 2) {
      // Too many crashes — this is a real bug, not transient
      job.status = "failed";
      job.error = `Daemon restarted ${retryCount} times — job abandoned (likely a persistent failure, not transient)`;
      job.completedAt = new Date().toISOString();
      d.log("error", `Job ${job.id} failed after ${retryCount} crash retries`);
    } else {
      // Re-queue for resume
      job.status = "queued";
      job.retryCount = retryCount;
      job.costUsd = 0;
      d.log("info", `Job ${job.id} interrupted — re-queued for resume (attempt ${retryCount}/2)`);
    }
  }
}
```

### Step 3: Detect resume in processNext()

In `processNext()`, after selecting `nextJob` and creating `jobDir`, detect if this is a retry with existing pipeline state:

**Existing functions used (already exported from `pipeline.ts`):**
- `readPipelineState(dir: string): PipelineState | null` — reads `pipeline.json` from dir, validates via `safeReadJSON`
- `resumePipeline(checkpointDir: string, config: GaryClawConfig, callbacks: OrchestratorCallbacks): Promise<void>` — resumes from persisted state, retries failed skills, skips completed ones

Add `resumePipeline` to imports and to `JobRunnerDeps` interface + `defaultDeps`:

```typescript
// In JobRunnerDeps interface:
resumePipeline: typeof resumePipeline;

// In defaultDeps:
resumePipeline,
```

Then in `processNext()`, after creating `jobDir`:

```typescript
// Detect if this is a pipeline resume (retry with existing pipeline.json)
let isPipelineResume = (nextJob.retryCount ?? 0) > 0 && nextJob.skills.length > 1;

if (isPipelineResume) {
  const pipelineState = readPipelineState(jobDir);
  if (pipelineState) {
    const completedCost = pipelineState.skills
      .filter(s => s.status === "complete" && s.report)
      .reduce((sum, s) => sum + (s.report?.estimatedCostUsd ?? 0), 0);
    nextJob.priorSkillCostUsd = completedCost;
    d.log("info", `Resuming pipeline: ${pipelineState.skills.filter(s => s.status === "complete").length}/${pipelineState.skills.length} skills already complete ($${completedCost.toFixed(3)} spent)`);
  } else {
    d.log("warn", `No valid pipeline.json found for resumed job ${nextJob.id} — falling back to fresh pipeline`);
    isPipelineResume = false;
  }
}
```

Then in the execution block:

```typescript
if (nextJob.skills.length === 1) {
  // Single-skill jobs retry from scratch (cheap, $0.30-0.50)
  if ((nextJob.retryCount ?? 0) > 0) {
    d.log("info", `Retrying single-skill job ${nextJob.id} from scratch (attempt ${nextJob.retryCount}/2)`);
  }
  await d.runSkill({ ...clawConfig, skillName: nextJob.skills[0] }, callbacks);
} else if (isPipelineResume) {
  await d.resumePipeline(jobDir, clawConfig, callbacks);
} else {
  await d.runPipeline(nextJob.skills, clawConfig, callbacks);
}
```

### Step 4: Budget adjustment for resumed jobs

No code change needed in `buildCallbacks()`. Step 2 resets `job.costUsd = 0` when re-queuing, so the resumed job accumulates only new spending. The `priorSkillCostUsd` field is for dashboard reporting only.

### Step 5: Failure taxonomy integration for crash retries

In `job-runner.ts`, in the crash-recovery loop (Step 2), when a job exceeds the retry limit (retryCount > 2), classify the crash using the existing failure-taxonomy system:

```typescript
if (retryCount > 2) {
  job.status = "failed";
  job.error = `Daemon restarted ${retryCount} times — job abandoned`;
  job.completedAt = new Date().toISOString();
  job.failureCategory = "daemon_crash";
  job.retryable = false;

  // Append failure record for observability
  const record = buildFailureRecord(
    new Error(job.error), job.id, job.skills, resolvedInstanceName
  );
  appendFailureRecord(record, checkpointDir);
  d.log("error", `Job ${job.id} failed after ${retryCount} crash retries`);
}
```

Also, in the `processNext()` catch block, when a resumed job fails again, tag it as a retry:

```typescript
if ((nextJob.retryCount ?? 0) > 0) {
  d.log("warn", `Retry ${nextJob.retryCount}/2 failed for ${nextJob.id} [${classification.category}]`);
}
```

**Note:** Distinguishing sleep vs OOM vs bug as crash cause is deferred. The daemon process is dead by the time we detect the crash, so we can't inspect the cause. The existing `classifyError` handles the error type from the catch block (if the job threw before dying), and the startup loop classifies it generically as `daemon_crash`. Future work could use heartbeat staleness or OS sleep notifications to classify, but that's a separate feature.

### Step 6: Notification on resume

Add `notifyJobResumed` to `src/notifier.ts`:

```typescript
export function notifyJobResumed(job: Job, completedSkillCount: number, config: DaemonConfig): void {
  const msg = `Recovered job ${job.id} — resuming from skill ${completedSkillCount + 1}/${job.skills.length} (attempt ${job.retryCount ?? 1}/2)`;
  notify(msg, config);
}
```

Add to `JobRunnerDeps` and `defaultDeps`. Call from `processNext()` when `isPipelineResume` is true.

### Step 6: Dashboard integration

In `src/dashboard.ts`, add crash recovery stats:

```typescript
const recoveredJobs = state.jobs.filter(j => j.status === "complete" && (j.retryCount ?? 0) > 0);
const crashRecoveries = recoveredJobs.length;
const savedCostUsd = recoveredJobs.reduce((sum, j) => sum + (j.priorSkillCostUsd ?? 0), 0);
```

Display: "Crash recoveries: N ($X.XX saved)"

**Total job cost for display** = `costUsd + (priorSkillCostUsd ?? 0)`. The `costUsd` field tracks only post-resume spending; `priorSkillCostUsd` tracks pre-crash spending. Dashboard should sum both for "total cost" columns and show `priorSkillCostUsd` separately as "saved by resume."

## Open Questions

1. **Should single-skill jobs that crash mid-relay also resume?** The relay checkpoint exists, but single-skill jobs are cheap ($0.30-0.50). For now, single-skill jobs retry from scratch. Future optimization.

2. **Should the daemon attempt resume on SIGTERM (graceful shutdown)?** Yes. The SIGTERM handler in `daemon.ts` waits up to 60s for the running job to finish. If it doesn't finish, the process exits and the job remains `status: "running"` in `daemon-state.json`. Step 2's startup loop catches this and re-queues it. No special SIGTERM handling needed... the existing "running at startup = crashed" logic covers it uniformly.

3. **What about worktree state on crash?** The relay system's `git stash --include-untracked` handles dirty state. Low risk since relay already does this on every fresh session.

## Success Criteria

- A pipeline interrupted by macOS sleep resumes from the last completed skill on daemon restart
- Completed skill costs are not double-charged against the per-job budget
- Jobs that crash 3 times are marked failed (not retried forever)
- Dashboard shows crash recovery count and saved cost
- Zero regressions in existing 1869 tests
