# Design: Failure Taxonomy in Job Runner

## Problem Statement

When GaryClaw daemon jobs fail overnight, the error is an opaque string — whatever `err.message` says. There's no way to distinguish "my project's ESLint config is broken" from "the Claude SDK timed out" from "GaryClaw has a bug in relay logic." This forces manual log reading every morning and prevents smart retry or automated triage.

The job runner's catch block (job-runner.ts:224-232) is the single convergence point where all job errors land. Classification here enables: auto-retry for transient failures, skip for project bugs, escalate for GaryClaw bugs, and structured data for the dogfood dashboard.

## What Makes This Cool

Every failure the daemon encounters gets fingerprinted and classified in real-time. Instead of "3 jobs failed" you get "1 auth expiry (auto-retryable), 1 project test failure (skip), 1 GaryClaw bug (filed to backlog)." The daemon becomes self-aware about *why* things break — and that data feeds Oracle memory, the dashboard, and eventually auto-retry.

## Constraints

- Must classify at the single catch point in job-runner.ts (eng review approved this approach)
- Classification heuristics must be conservative — "unknown" is better than misclassification
- Must not change the Job type in a breaking way (add fields, don't modify existing ones)
- Must work with errors from: orchestrator, pipeline, SDK wrapper, relay, git operations
- Structured output must be append-only JSONL for easy analysis
- No auto-retry in this phase — classification only (retry is a follow-up)

## Premises

1. The job-runner.ts catch block is the right single classification point — all job errors converge there.
2. Heuristic string/class matching is sufficient — error messages from the SDK, git, and Node.js are distinctive enough to classify without instrumentation.
3. Classification alone (without auto-retry) is valuable — it enables triage, feeds the dashboard, and builds the failure history needed for smart retry later.
4. Conservative fallback to "unknown" is better than misclassification — wrong category is worse than no category.
5. The taxonomy should be table-driven and extensible — new patterns added as real failures are observed.

## Approaches Considered

### Approach A: Inline classifyError() in job-runner.ts

Add a `classifyError(err)` function directly in job-runner.ts. Smallest possible diff (~80 lines). Classification logic lives next to where it's used.

- **Effort:** S
- **Pros:** Smallest diff, ships in one commit, no new files
- **Cons:** Mixes classification concerns with job runner, harder to unit test in isolation, bloats job-runner.ts if taxonomy grows

### Approach B: Dedicated failure-taxonomy.ts Module (RECOMMENDED)

New module `src/failure-taxonomy.ts` with table-driven classification rules. `FailureCategory` type and `FailureRecord` interface in `types.ts`. Job runner calls `classifyError()` in catch block, stores structured result on the Job object and appends to `failures.jsonl`.

- **Effort:** S
- **Pros:** Clean separation of concerns (matches project's module-per-concept pattern), table-driven rules are trivially extensible, easy to unit test independently, consistent with the 26-module architecture
- **Cons:** One more module to maintain, slightly larger diff than Approach A

### Approach C: Classification + Auto-Retry

Everything in B plus retry logic: `retryable` flag per category, re-enqueue with backoff, retry count tracking. The daemon self-heals transient failures.

- **Effort:** M
- **Pros:** Self-healing overnight runs, maximum resilience
- **Cons:** Retry loop adds complexity to job lifecycle state machine, risk of infinite retry on misclassified errors, scope creep beyond priority pick

## Recommended Approach

**Approach B — Dedicated failure-taxonomy.ts module.** Follows the project's established one-module-per-concept architecture (like `safe-json.ts`, `reflection-lock.ts`). Table-driven heuristics make it trivial to add new patterns as real failures are observed overnight. Classification alone unlocks triage and dashboard value immediately; auto-retry is a clean follow-up.

### Classification Algorithm

```
1. If err is an instance of a known Error subclass → match by errorNames
2. Get error message (err.message or String(err))
3. Get stack trace (err.stack, first 5 lines)
4. For each rule in RULES (top-to-bottom):
   a. Matchers that are undefined on a rule are skipped (not evaluated).
   b. If rule.errorNames defined → check if err.name matches any
   c. If rule.messagePatterns defined → check if message contains any (case-insensitive)
   d. If rule.stackPatterns defined → check if stack contains any (case-insensitive)
   e. If ANY of the *defined* matchers hit → return this rule's category
      (Each defined matcher is evaluated independently — OR across fields.)
5. If no rule matches → return { category: "unknown", retryable: false }
```

**Priority order matters:** Budget > Auth > Infra > SDK > GaryClaw > Project > Skill > Unknown. This ensures typed errors are caught first, then transient/retryable issues, then source attribution.

**GaryClaw-bug vs SDK-bug disambiguation:** If the stack trace mentions both `src/` (GaryClaw code) and `node_modules/@anthropic-ai` (SDK code), the SDK rule fires first (it's higher priority). This is intentional — if the SDK threw the error, it's an SDK bug even if our code called it.

**Note on OR semantics for multi-field rules:** The `sdk-bug` rule defines both `messagePatterns` and `stackPatterns`. Under OR semantics, an error containing "invalid json" from project code (not SDK) could be misclassified. To mitigate: keep `messagePatterns` on `sdk-bug` narrow and SDK-specific. In practice, generic terms like "invalid json" are rare in error messages from the orchestrator/pipeline — they're almost always from the SDK's streaming parser. If false positives emerge in real failures, tighten or split the rule.

**Stack trace paths:** The project always runs via `tsx` (unbundled TypeScript), so stack traces reference `src/*.ts` paths. If a compiled `dist/` build is ever used, add `dist/` variants to `stackPatterns` for `garyclaw-bug`.

### Integration Points

#### 1. Job Runner Catch Block (job-runner.ts:224-232)

```typescript
// BEFORE (current):
} catch (err) {
  nextJob.status = "failed";
  nextJob.completedAt = new Date().toISOString();
  nextJob.error = err instanceof Error ? err.message : String(err);
  // ...
}

// AFTER:
} catch (err) {
  nextJob.status = "failed";
  nextJob.completedAt = new Date().toISOString();
  nextJob.error = err instanceof Error ? err.message : String(err);

  // Classify the failure
  const classification = classifyError(err);
  nextJob.failureCategory = classification.category;
  nextJob.retryable = classification.retryable;

  // Append structured failure record
  const record = buildFailureRecord(err, nextJob.id, nextJob.skills, resolvedInstanceName);
  appendFailureRecord(record, checkpointDir);

  d.writeSummary(nextJob, jobDir);
  d.notifyJobError(nextJob, jobConfig);
  d.log("error", `Failed ${nextJob.id} [${classification.category}]: ${nextJob.error}`);
}
```

## Open Questions

1. **Should "daemon restarted" (stale job recovery) be classified?** Currently line 93 sets `job.error = "Daemon restarted — job was interrupted"` — should this also get a FailureRecord? Recommendation: yes. The RULES table already handles this — "daemon restarted" has its own dedicated rule as `infra-issue` with `retryable: true`, placed before the `garyclaw-bug` rule. Daemon restart is transient (not a code bug), and the job can be safely re-enqueued.
2. **Stack trace depth:** 5 lines is a guess. Should we store more for garyclaw-bug classification? Recommendation: 5 is enough for source file identification; full stack is in the daemon log.
3. **PerJobCostExceededError special handling:** Currently this saves a checkpoint before re-throwing (orchestrator.ts:375-386). Should budget-exceeded failures get any special treatment beyond classification? Recommendation: no — the checkpoint is already saved.
