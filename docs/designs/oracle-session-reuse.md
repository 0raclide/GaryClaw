# Design: Oracle Session Reuse (Latency Optimization)

Generated by /office-hours on 2026-03-29
Branch: main
Repo: 0raclide/GaryClaw
Status: APPROVED
Mode: Builder

## Problem Statement

Every Oracle decision creates a fresh SDK `query()` call with the full prompt: 7 Decision Principles, oracle memory (taste.md, domain-expertise.md, decision-outcomes.md, MEMORY.md), recent decision history, project context, and the actual question. That prompt is ~43K tokens.

At 20 decisions per job, that's 20 cold starts. Each call pays the full input token cost and 2-5 seconds of latency. The batch optimization (COMPLETE) helps when multiple questions arrive in one `AskUserQuestion` tool call, but in practice most questions arrive individually (1 per tool call). Batching only saves latency on ~20% of calls.

Session reuse sends the 43K context once. Subsequent decisions resume the same conversation with just the new question (~1K tokens). This cuts input token cost by ~95% for decisions 2-N and latency by ~50% across all Oracle calls.

## What Makes This Cool

The SDK already supports `resume` by session ID. `startSegment` in `sdk-wrapper.ts` passes `resume` through to `query()`. The orchestrator uses it for segment continuation. The Oracle just never takes advantage of it.

The `createSdkOracleQueryFn` in `oracle.ts:500-530` is a stateless function factory. It returns `(prompt: string) => Promise<string>`. Each invocation creates a fresh `query()`. The fix is making it stateful: first call creates the session, subsequent calls resume it.

## Constraints

- Must be backward compatible. `queryFn: (prompt: string) => Promise<string>` signature stays. Callers don't change.
- Must not break test isolation. Tests mock `queryFn` as a simple function. The session state is internal to the real implementation.
- Must handle session expiry/failure gracefully. If resume fails, fall back to fresh session (cold start). No user-visible errors.
- Must work with both `askOracle` (single) and `askOracleBatch` (batch). Both call `config.queryFn`.
- Per-skill scope. Each skill in a pipeline gets its own ask-handler + oracle config (line 275-289 in orchestrator.ts). Session reuse is per-skill, which is correct: different skills have different memory/context.

## Premises

1. The SDK's `resume` option works with `maxTurns: 1` queries. The session ID from the first query can be used to resume with a follow-up prompt, and the resumed session retains the full conversation history (including the 43K system context). **This is unverified and requires a spike. If the spike fails, this design is not viable and we accept the current latency as-is.** Approach C (prompt caching) doesn't solve the latency problem. The honest answer if resume doesn't work: there is no cheap optimization available, and we'd need to investigate SDK-level connection pooling or more aggressive batching as a separate effort.
2. Oracle calls within a single skill share the same prompt prefix (principles, memory, project context). Only the question and decision history differ. The prefix is ~40K tokens and stays constant.
3. The decision history grows by ~100 tokens per decision (one line per recent decision, capped at 5). This growth is negligible compared to the 40K prefix saved.
4. Resume sessions use cached input tokens, which are cheaper and faster than fresh input tokens. The SDK's caching behavior should make resumed Oracle calls significantly cheaper.
5. `extractResultData()` returns a `sessionId` field from the SDK result message. This is confirmed by the `SegmentResult` interface in `types.ts:561`.
6. Oracle calls within a skill are serial. The ask-handler `await`s each Oracle response before proceeding to the next question. `askOracleBatch` sends one API call with multiple questions, but the closure's `queryFn` is still called once. No concurrent access to the shared `sessionId`/`callCount` state.

## Approaches Considered

### Approach A: Stateful queryFn (Minimal Viable)

**Summary:** Replace the stateless `createSdkOracleQueryFn` with a stateful version that tracks the session ID. First call sends the full prompt. Subsequent calls resume the session with just the new question. Falls back to fresh session on any error.

**Effort:** S
**Risk:** Med (depends on spike validating SDK resume with maxTurns:1)
**Pros:**
- Zero API surface changes. `queryFn` signature stays `(prompt: string) => Promise<string>`.
- `askOracle`, `askOracleBatch`, ask-handler, orchestrator are untouched.
- Graceful fallback: if resume fails, next call starts fresh. No data loss.
- Session naturally scoped per-skill (orchestrator creates a new config per skill).

**Cons:**
- Hidden state in what looks like a pure function. Could surprise someone debugging.
- Session accumulates conversation history. After 20+ decisions, the context might grow enough to offset the savings. Needs a reset threshold.
- SDK resume behavior with `maxTurns:1` is unverified.

**Reuses:** `createSdkOracleQueryFn()`, `query()` import, `extractResultData()`.

### Approach B: Explicit Oracle Session Manager

**Summary:** New `OracleSession` class that manages the session lifecycle explicitly. Exposes `ask(prompt)` and `reset()`. Wired into `OracleConfig` as a replacement for `queryFn`.

**Effort:** M
**Risk:** Med
**Pros:**
- Explicit session lifecycle. Clear when sessions are created, reused, and reset.
- Can expose session metrics (reuse count, cache hit rate, total tokens saved).
- Easier to test the session management logic in isolation.

**Cons:**
- Changes `OracleConfig` interface. Every caller needs updating.
- New class + new interface = more code, more test surface.
- The session management logic is simple enough that a class is overkill.

**Reuses:** `query()`, `extractResultData()`.

### Approach C: Prompt Prefix Caching (No Session Reuse)

**Summary:** Instead of SDK session reuse, rely on the API's automatic prompt caching. The 43K prefix is identical across calls, so the API should cache it after the first call and serve subsequent calls from cache.

**Effort:** XS
**Risk:** Low
**Pros:**
- Zero code changes in GaryClaw. Just rely on the API's built-in caching.
- No hidden state. No session management. No failure modes.

**Cons:**
- API prompt caching is a cost optimization, not a latency optimization. The full prompt still gets sent over the wire every time.
- Cache hits depend on exact prefix matching. If decision history changes (it does, every call), the cache key may not match.
- We're already paying for cache_creation_input_tokens on the first call. This approach doesn't reduce the number of API calls or the per-call overhead.
- Doesn't address the 2-5 second latency per decision.

## Recommended Approach

**Approach A: Stateful queryFn.** Smallest change, biggest win. The queryFn signature doesn't change, so all callers are untouched. The hidden state is justified because the stateful behavior is purely an optimization. If it breaks, the fallback is exactly the current behavior (cold start). Add a JSDoc comment explaining the session reuse and the fallback.

### 1. Refactor createSdkOracleQueryFn to Track Session

In `src/oracle.ts`, replace the current stateless factory:

```typescript
export function createSdkOracleQueryFn(
  env: Record<string, string>,
): (prompt: string) => Promise<string> {
  let sessionId: string | null = null;
  let callCount = 0;
  const MAX_REUSE = 25; // Reset session after 25 decisions to bound context growth

  return async (prompt: string): Promise<string> => {
    const { query } = await import("@anthropic-ai/claude-agent-sdk");
    const { extractResultData } = await import("./sdk-wrapper.js");

    // Reset session periodically to bound context growth
    if (callCount >= MAX_REUSE) {
      sessionId = null;
      callCount = 0;
    }

    const isResume = sessionId !== null;
    let result = "";
    let newSessionId: string | null = null;

    try {
      const gen = query({
        prompt: isResume ? buildResumePrompt(prompt) : prompt,
        options: {
          maxTurns: 1,
          env,
          permissionMode: "bypassPermissions",
          allowDangerouslySkipPermissions: true,
          canUseTool: async () => ({ behavior: "deny" as const, message: "Oracle does not allow tool use" }),
          ...(isResume && sessionId ? { resume: sessionId } : {}),
        },
      });

      for await (const msg of gen) {
        if (msg.type === "result") {
          const resultData = extractResultData(msg);
          if (resultData) {
            newSessionId = resultData.sessionId || null;
            if (resultData.subtype === "success") {
              result = resultData.resultText;
            }
          }
        }
      }

      // Update session tracking on success
      if (result && newSessionId) {
        sessionId = newSessionId;
        callCount++;
      } else if (isResume) {
        // Resume failed — reset and retry as cold start
        sessionId = null;
        callCount = 0;
        return await createSdkOracleQueryFn(env)(prompt); // One retry
      }
    } catch {
      // Any error — reset session and retry cold
      if (isResume) {
        sessionId = null;
        callCount = 0;
        return await createSdkOracleQueryFn(env)(prompt);
      }
      // Cold start also failed — propagate
      throw new Error("Oracle query failed");
    }

    return result;
  };
}
```

**Key design decisions:**
- `MAX_REUSE = 25`: After 25 decisions in one session, the conversation history is ~25K tokens of Q&A. Reset to avoid approaching context limits. 25 is a safety bound, not a frequently-triggered optimization. Most skills have 15-20 decisions total, so the reset rarely fires. The observability metric "average call count before reset" is for anomaly detection, not routine monitoring.
- **Single retry on resume failure:** If resume returns empty/error, reset session and retry cold. If cold also fails, propagate the error (existing behavior).
- **Retry pattern (preferred implementation):** Use a `while` loop with a `retried` flag instead of recursive closure creation. Clearer, no throwaway closures:
  ```typescript
  let retried = false;
  while (true) {
    const isResume = sessionId !== null;
    try {
      // ... query logic ...
      if (result && newSessionId) { sessionId = newSessionId; callCount++; }
      else if (isResume && !retried) { sessionId = null; callCount = 0; retried = true; continue; }
      break;
    } catch {
      if (isResume && !retried) { sessionId = null; callCount = 0; retried = true; continue; }
      throw new Error("Oracle query failed");
    }
  }
  ```
  The loop body runs at most twice: once as resume (may fail), once as cold start (throws on failure). The code in Section 1 above uses the recursive pattern for readability, but the loop pattern is preferred for implementation.

### 2. Prompt Section Ordering + Resume Strategy

**Prompt layout produced by `buildOraclePrompt()`:**

```
[buildOraclePromptPrefix output:]
  "You are a decision-making oracle..."
  ## Decision Principles           ← ~500 tokens
  ## Current Context               ← ~100 tokens
  ## Taste Profile                 ← ~1K tokens (if present)
  ## Domain Expertise              ← ~2K tokens (if present)
  ## Decision Outcomes             ← ~1K tokens (if present)
  ## Project Memory                ← ~2K tokens (if present)
  ## Recent Decisions (last 5)     ← ~500 tokens

[buildOraclePrompt appends:]
  ## Question                      ← ~200 tokens
  ## Options                       ← ~300 tokens
  Response format instructions     ← ~200 tokens
```

**Decision history grows with each call.** On a resumed session, the previous decisions are already in the conversation transcript (Claude can see every prior Q&A). Re-injecting the "Recent Decisions" section would duplicate them.

**Design decision: strip everything before `## Question` on resume.** The resumed session retains the full prefix from the first call. Only the question and options sections change between calls. This avoids duplication and keeps the resume prompt at ~700 tokens.

### 3. Build Resume Prompt Helper

```typescript
/** Shared constant — must match the heading in buildOraclePrompt() */
export const ORACLE_QUESTION_MARKER = "## Question\n";

function buildResumePrompt(fullPrompt: string): string {
  // The full prompt is: prefix (principles/memory/context/history) + question + options
  // On resume, the prefix is already in the session context from the first call.
  // Strip everything before ## Question, keeping only the question + options + format.
  const idx = fullPrompt.indexOf(ORACLE_QUESTION_MARKER);
  if (idx === -1) {
    // Can't find the question section — send the full prompt (safe fallback).
    // This is invisible but safe: the model just gets a redundant prefix.
    return fullPrompt;
  }
  return "New decision needed:\n\n" + fullPrompt.slice(idx);
}
```

**Important:** Export `ORACLE_QUESTION_MARKER` as a constant shared between `buildOraclePrompt()` and `buildResumePrompt()`. Add a test that `buildOraclePrompt()` output always contains this marker. This prevents the marker from drifting silently.

**Batch calls and session reuse:** `askOracleBatch` with N>1 questions uses a different prompt structure (`## Questions` plural, JSON array response format). Session reuse does NOT apply to batch calls. Batch calls always send the full prompt and reset the session afterward. This avoids two problems: (a) `buildResumePrompt` can't find `## Question` in a batch prompt, and (b) a resumed single call after a batch call would see a structurally different conversation history. Since batching is already a latency optimization (~20% of calls), combining both optimizations adds complexity for negligible gain. Implementation: if `callCount > 0` and this is detected as a batch prompt (contains `## Questions\n`), reset `sessionId = null` before the call.

**Decision history on resume:** Decision history growth within the session is handled naturally: each prior Oracle Q&A is visible in the resumed conversation transcript. No re-injection of the "Recent Decisions" section is needed because all decisions within a skill flow through this same session. The `decisionHistory` array in the ask-handler grows with each decision, but the resumed conversation already contains the full Q&A pairs, which is richer context than the 5-line summary.

### 4. Wire Session Reset on Skill Boundary

The orchestrator already creates a fresh `OracleConfig` per skill (line 275-289 in `orchestrator.ts`). Each call to `createSdkOracleQueryFn(config.env)` returns a new closure with fresh `sessionId = null`. No additional wiring needed. Session reuse is automatically scoped per-skill.

### 5. Observability

Add an optional `onOracleSessionEvent` callback to track reuse:

```typescript
export interface OracleSessionEvent {
  type: "session_created" | "session_resumed" | "session_reset" | "resume_fallback";
  callCount: number;
  sessionId?: string;
}
```

Wire into the existing event system so dashboard/logs can report:
- How many Oracle calls reused a session vs. cold started
- How often resume fallback triggered (expect <5%, mostly from session expiry)
- Average call count before reset (expect "never" for most skills. This metric is for anomaly detection, not routine monitoring, since MAX_REUSE=25 exceeds typical skill decision counts)

### What Doesn't Change

- `OracleConfig` interface: `queryFn` signature stays `(prompt: string) => Promise<string>`.
- `askOracle()` and `askOracleBatch()`: unchanged. They call `config.queryFn(prompt)`.
- `buildOraclePrompt()` and `buildOraclePromptPrefix()`: unchanged. They still build the full prompt. The resume logic strips it down.
- `createAskHandler()` and orchestrator wiring: unchanged.
- All existing tests: unchanged. Test mocks provide `queryFn` as a simple function. The session state only exists in the real `createSdkOracleQueryFn`.

## Open Questions

1. **Does SDK resume with maxTurns:1 retain the full conversation context?** This is the critical unknown. The spike must answer this before implementation. If resume starts a "new" conversation despite the session ID, the approach doesn't work.
2. **Does the resumed session use cached tokens?** If the API re-processes the full context on resume (just without re-sending it over the wire), the latency savings may be less than expected. The spike should measure `cache_read_input_tokens` on the resumed call.
3. **What happens when the session expires?** The SDK may have a session TTL. If the session expires between Oracle calls (e.g., a long-running skill with 10 minutes between decisions), the resume will fail. The fallback handles this, but frequent fallbacks would negate the optimization.
4. **Should MAX_REUSE be configurable?** Probably not in v1. 25 is a reasonable default. If dashboard data shows sessions hitting the limit frequently, add a config knob later.
