# Design: Pre-Merge Validation Gate

## Problem Statement

The auto-merge flow in GaryClaw has zero pre-merge validation. When a named daemon instance completes a pipeline (prioritize → implement → QA), `mergeWorktreeBranch()` rebases and ff-only merges to main without running tests, checking types, or verifying the build. Merge failures are silently logged as warnings while the job is marked "complete." There is no rollback mechanism if merged code breaks main.

This is the biggest quality gap in autonomous operation. On 2026-03-28, the daemon ran 8 QA passes chasing its own commits because bad code landed on main unchecked. Each self-triggering cascade costs $3-5. Over a night of parallel instances, that's $30-60 wasted.

The current auto-merge block in `processNext()` in `job-runner.ts`:
```typescript
const result = mergeWorktreeBranch(jobConfig.projectDir, resolvedInstanceName, baseBranch);
if (result.merged) {
  d.log("info", `Auto-merge: merged ${result.commitCount ?? 0} commit(s)...`);
} else {
  d.log("warn", `Auto-merge failed: ${result.reason}`);
  // ❌ No action taken — job already marked complete
}
```

## What Makes This Cool

The daemon becomes truly safe for overnight autonomous operation. You start it, go to sleep, and wake up to a main branch where every single merged commit passed the full test suite. The merge audit trail shows exactly what landed, when, and which instance merged it. If something goes wrong, the dashboard shows merge health over time.

"It ran 8 hours overnight, merged 12 features, and every single one passed tests before landing."

## Constraints

- Must not break existing merge flow for instances that don't have test suites
- Must be fast enough to not bottleneck parallel instance throughput (target: <90s per merge gate)
- Must work with the existing mkdir-based merge lock. **Lock timeout must accommodate test duration:** current `MERGE_LOCK_TIMEOUT_MS` is 60s, but test gate adds up to 120s. Increase lock timeout to `testTimeout + 60s` (default: 180s) so other instances waiting for the lock don't time out during test execution.
- Must be configurable (some projects may want merge-without-tests for speed)
- Must not require external CI infrastructure — runs locally via `npm test` or configured command

## Premises

1. Bad code landing on main is the highest risk in autonomous daemon operation — a single broken merge cascades into wasted QA cycles and budget.
2. Running `npm test` before merge catches the majority of regressions (GaryClaw's 1896 tests cover all 34 modules).
3. The test suite runs fast enough (~30-60s) that a pre-merge gate doesn't meaningfully delay the pipeline.
4. Merge failures should fail the job, not be silently logged — the current silent-warning behavior masks operational problems.
5. The insertion point is between rebase and ff-only merge in `mergeWorktreeBranch()` — after rebase succeeds, run tests on the rebased branch before merging to main.

## Approaches Considered

### Approach A: Minimal Test Gate

**Summary:** Add `npm test` execution between rebase and ff-only merge in `mergeWorktreeBranch()`. If tests fail, abort merge, return `{ merged: false, reason: "Tests failed" }`. In `job-runner.ts`, treat merge failure as job failure.

**Effort:** S (human: ~2 days / CC: ~20 min)
**Risk:** Low
**Pros:**
- Smallest possible diff — ~40 lines across 2 files
- Prevents the most damaging failure mode (broken code on main)
- Zero new dependencies

**Cons:**
- No rollback mechanism if tests pass pre-merge but something breaks post-merge
- No audit trail of what was merged
- No dashboard visibility into merge health
- Merge failure marks job as failed with no retry intelligence

### Approach B: Test Gate + Merge Audit + Failure Intelligence (RECOMMENDED)

**Summary:** Test gate (Approach A) plus: structured merge audit log (`.garyclaw/daemons/{name}/merge-audit.jsonl`), merge failure → job retry with `retryable: true` category, dashboard merge health stats, configurable test command via daemon config. The merge gate becomes a first-class subsystem with observability.

**Effort:** M (human: ~1 week / CC: ~45 min)
**Risk:** Low-Med
**Pros:**
- Complete validation gate with full observability
- Audit trail enables debugging "what broke main?"
- Dashboard shows merge health trends (% passed, avg test time, common failure patterns)
- Merge failures are retryable — the daemon can re-attempt after the next job fixes the issue
- Configurable test command supports repos with custom test scripts

**Cons:**
- More moving parts than Approach A
- Dashboard changes require `dashboard.ts` updates + new aggregation function
- Merge failure adds a new failure taxonomy category and error class

### Approach C: Test Gate + Auto-Fix Loop

**Summary:** If pre-merge tests fail, instead of aborting, run a mini QA-fix cycle on the worktree branch. Extract failing test output, build a fix prompt, run a single orchestrator segment to fix the failures, then re-run tests. Max 2 fix attempts before abort.

**Effort:** L (human: ~2 weeks / CC: ~2 hours)
**Risk:** High
**Pros:**
- The daemon self-heals test failures before they become merge failures
- Could dramatically reduce the number of jobs that "waste" their pipeline run
- Closes the loop between "implement broke something" and "fix it"

**Cons:**
- High complexity — needs to invoke orchestrator from within the merge gate
- Risk of infinite fix loops (fix introduces new failure → fix that → new failure)
- Budget implications — each fix attempt costs tokens
- The QA skill already does this in the pipeline — this duplicates that responsibility

## Recommended Approach

**Approach B: Test Gate + Merge Audit + Failure Intelligence.**

It hits the sweet spot: prevents the damage (test gate), explains what happened (audit log), enables recovery (retryable failures), and shows trends (dashboard). Approach A is too minimal for a system running 12+ merges per day. Approach C is too risky and duplicates QA skill responsibility.

## Implementation Order

### Step 1: MergeValidationConfig + test gate in worktree.ts

Add a `MergeValidationConfig` interface:
```typescript
interface MergeValidationConfig {
  /** Command to run for validation (default: "npm test") */
  testCommand?: string;
  /** Timeout in ms (default: 120000 = 2 min) */
  testTimeout?: number;
  /** Skip validation entirely (default: false) */
  skipValidation?: boolean;
}
```

Extend the `MergeResult` interface with new optional fields:
```typescript
interface MergeResult {
  merged: boolean;
  commitCount?: number;
  reason?: string;
  testOutput?: string;      // NEW: truncated stderr from test run
  testDurationMs?: number;  // NEW: how long tests took
}
```

In `mergeWorktreeBranch()`, add an optional `validation?: MergeValidationConfig` parameter. After the `git rebase baseBranch` call succeeds in `mergeWorktreeBranch()`, before the `git merge --ff-only` call:

1. If `validation.skipValidation`, skip to ff-only merge
2. Run `validation.testCommand ?? "npm test"` in the worktree directory (not main repo)
3. Capture stdout/stderr and measure duration
4. If exit code !== 0, leave the rebased branch as-is (rebase already succeeded, the commits are on the branch), skip the ff-only merge, and return `{ merged: false, reason: "Pre-merge tests failed", testOutput: stderr, testDurationMs }`. The branch stays rebased but unmerged. The next pipeline run will produce new commits on top.
5. If exit code === 0, proceed to ff-only merge

**Important:** Do NOT `git rebase --abort` on test failure. The rebase has already completed successfully at this point. The rebased commits are on the branch. We simply skip the merge step and leave the branch in its rebased state.

The test runs ON THE REBASED BRANCH in the worktree. This catches regressions from both the implementation AND the rebase.

**Key insight:** Run tests in the worktree directory (`getWorktreePath()`), not `repoDir`. The worktree has the rebased code. The main repo is on `baseBranch`. Running tests in the worktree validates the exact code that will be merged.

**Dynamic merge lock timeout:** The caller in `job-runner.ts` must pass `validation.testTimeout + 60_000` (default: 180,000ms) to `acquireMergeLock()` so other instances waiting for the lock don't time out during test execution. `acquireMergeLock` already accepts a `timeoutMs` parameter.

**Prerequisite: `node_modules` in worktree.** The worktree already has `node_modules` because the daemon runs skills (implement, QA) in the worktree directory, which installs dependencies. The test gate runs AFTER the pipeline completes, so `node_modules` is guaranteed to exist. If a project doesn't install deps during pipeline execution, the test command will fail and the merge will be blocked (correct behavior — untestable code shouldn't auto-merge).

### Step 2: Merge audit log in worktree.ts

After each merge attempt (success or failure), append a structured record:

```typescript
interface MergeAuditEntry {
  timestamp: string;
  instanceName: string;
  branch: string;
  baseBranch: string;
  commitCount: number;
  merged: boolean;
  reason?: string;
  testsPassed?: boolean;
  testDurationMs?: number;
  testOutput?: string;  // truncated to 2000 chars
  jobId?: string;
}
```

Write to `.garyclaw/daemons/{name}/merge-audit.jsonl` using `safeWriteJSON` append pattern (actually just `appendFileSync` since JSONL is append-only).

### Step 3: Merge failure → retryable job failure in job-runner.ts

Replace the current silent-warning pattern:

```typescript
// BEFORE (current)
if (!result.merged) {
  d.log("warn", `Auto-merge failed: ${result.reason}`);
}

// AFTER
if (!result.merged) {
  // Merge failure is a retryable job failure
  throw new MergeFailedError(result.reason, result.testOutput);
}
```

Create `MergeFailedError` class with a unique `name` property (`"MergeFailedError"`). Add to failure taxonomy in `failure-taxonomy.ts`:
- Category: `"merge-failed"` (kebab-case, matching the existing convention: `"garyclaw-bug"`, `"project-bug"`, etc.)
- Retryable: `true`
- Match via `errorNames` array (highest priority in classification), NOT `messagePatterns`, to prevent accidental misclassification by the existing `"project-bug"` rule which already matches "tests failed"
- Notification: includes test output snippet (first 500 chars)

**Note on retry semantics:** The `retryable: true` field is advisory metadata, not an automatic retry trigger. The existing crash-recovery retry logic in job-runner.ts only applies to jobs found in `status: "running"` on daemon restart. A `MergeFailedError` thrown during `processNext()` will be caught by the catch block at line 340, which marks the job as `"failed"`. The job will NOT be automatically re-queued. Instead, the next pipeline cycle (triggered by cron or git push) will implement a fix and attempt the merge again. The `retryable` field signals to the dashboard and notifications that this failure is self-healing, not a permanent error.

### Step 4: Dashboard merge health stats in dashboard.ts

Add `aggregateMergeStats()` function:
- Read `merge-audit.jsonl` from all instance dirs
- Compute: total merges, success rate, avg test duration, common failure reasons
- Add to `computeHealthScore()` as a 5th signal. Current weights: jobs 40%, oracle 25%, budget 20%, circuit breaker 15%. New weights: jobs 35%, oracle 20%, budget 20%, circuit breaker 10%, merge health 15%

Add merge health row to `formatDashboard()`:
```
| Merge Health | 12 merged, 2 failed (85%) | avg test: 34s |
```

### Step 5: Daemon config for test command

Add to daemon config schema (validated in `daemon.ts`):

interface DaemonConfig {

  merge?: {
    testCommand?: string;    // default: "npm test"

Pass through to `mergeWorktreeBranch()` call in job-runner.ts.

## Open Questions

1. **Test timeout default:** 120s is generous for 1896 tests. Should we measure actual test duration and set timeout to 2x the p95? Or just use 120s and let users override?
2. **Partial test failure:** If 1895/1896 tests pass and 1 flaky test fails, should the merge gate block? Option: configurable failure threshold (default: 0 = all must pass).

## Success Criteria

- Zero broken merges to main: every commit on main after this ships passes the full test suite
- Merge failures are visible: dashboard shows merge health, failures have audit trail
- Merge failures are recoverable: retryable category means the daemon can self-heal
- No manual intervention needed: overnight operation produces only clean merges
- Test gate adds <90s overhead per merge (GaryClaw's test suite target: ~45s)

## What I noticed about how you think

- The project has 1896 tests and 34 source modules, but the merge gate — the last line of defense before code hits main — has zero validation. This gap stands out because everything else in the system (Oracle with 7 principles, reflection with Levenshtein matching, failure taxonomy with 8 categories) is so carefully designed. The merge gate is the one place where "it probably works" is the standard instead of "we verified it works."
- The self-commit filtering, sleep-resilient cron, and pipeline resume features all shipped in the last 2 days. The pace of improvement is real, not aspirational.
- The session 3 observation about "$30-60/day wasted" on self-triggering cascades shows the project tracks actual cost impact, not just theoretical improvements. That specificity is rare.
- The advisory lock pattern (mkdir-based) appears 3 times (reflection lock, merge lock, claim lock). This codebase has a consistent "pick a simple pattern and reuse it" philosophy. The test gate should follow the same principle: simple, reusable, testable.
