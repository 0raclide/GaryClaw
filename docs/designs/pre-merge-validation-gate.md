## Problem Statement

The auto-merge flow in GaryClaw has zero pre-merge validation. When a named daemon instance completes a pipeline (prioritize -> implement -> QA), `mergeWorktreeBranch()` rebases and ff-only merges to main without running tests, checking types, or verifying the build. Merge failures are silently logged as warnings while the job is marked "complete." There is no rollback mechanism if merged code breaks main.

On 2026-03-28, the daemon ran 8 QA passes chasing its own commits because bad code landed on main unchecked. Each self-triggering cascade costs $3-5. Over a night of parallel instances, that's $30-60 wasted.

The current auto-merge block in `processNext()` in `job-runner.ts` (lines 355-368):
```typescript
// Auto-merge: named instances merge their branch to main after successful jobs
if (jobConfig.worktreePath && resolvedInstanceName !== "default") {
  try {
    const baseBranch = resolveBaseBranch(jobConfig.projectDir);
    const result = mergeWorktreeBranch(jobConfig.projectDir, resolvedInstanceName, baseBranch);
    if (result.merged) {
      d.log("info", `Auto-merge: merged ${result.commitCount ?? 0} commit(s)...`);
    } else {
      d.log("warn", `Auto-merge failed: ${result.reason}`);
      // No action taken, job already marked complete
    }
  } catch (err) {
    d.log("warn", `Auto-merge error: ${err instanceof Error ? err.message : String(err)}`);
  }
}
```

## What Makes This Cool

The daemon becomes truly safe for overnight autonomous operation. You start it, go to sleep, and wake up to a main branch where every single merged commit passed the full test suite. The merge audit trail shows exactly what landed, when, and which instance merged it.

"It ran 8 hours overnight, merged 12 features, and every single one passed tests before landing."

## Constraints

- Must not break existing merge flow for instances that don't have test suites
- Must be fast enough to not bottleneck parallel instance throughput (target: <90s per merge gate)
- Must work with the existing mkdir-based merge lock. Lock timeout must accommodate test duration: current `MERGE_LOCK_TIMEOUT_MS` is 60s, but test gate adds up to 120s. Dynamic lock timeout = `testTimeout + 60s` (default: 180s).
- Must be configurable (some projects may want merge-without-tests for speed)
- Must not require external CI infrastructure, runs locally via `npm test` or configured command
- Job completion status must remain independent of merge outcome. The pipeline ran successfully. Merge is a separate post-completion step.

## Premises

1. Bad code landing on main is the highest risk in autonomous daemon operation. A single broken merge cascades into wasted QA cycles and budget ($30-60 on 2026-03-28).
2. Running `npm test` before merge catches the majority of regressions (2046+ tests cover all 34 modules).
3. The test suite runs fast enough (~30-60s) that a pre-merge gate doesn't meaningfully delay the pipeline.
4. Merge outcomes should be tracked events, not silent warnings. But merge failure should NOT retroactively fail a completed job. The job (prioritize -> implement -> QA) succeeded. The merge is a separate concern. Re-running an entire pipeline for a merge conflict wastes $3-5.
5. The insertion point is between rebase and ff-only merge in `mergeWorktreeBranch()`. After rebase succeeds at worktree.ts:348, run tests on the rebased branch before the ff-only merge at line 366.

## Approaches Considered

### Approach A: Minimal Test Gate

**Summary:** Add `npm test` execution between rebase and ff-only merge in `mergeWorktreeBranch()`. If tests fail, abort merge, return `{ merged: false, reason: "Tests failed" }`.

**Effort:** XS (human: ~1 day / CC: ~10 min)
**Risk:** Low
**Pros:**
- Smallest possible diff (~30 lines in 1 file)
- Prevents the most damaging failure mode
- Zero new dependencies

**Cons:**
- No audit trail of what was merged
- No dashboard visibility into merge health
- Hardcoded `npm test`, no project-level config
- Merge failures still silently logged in job-runner.ts
- Completeness: 4/10

### Approach B: Test Gate + Merge Audit + Dashboard (RECOMMENDED)

**Summary:** Test gate (Approach A) plus: structured merge audit log (`.garyclaw/daemons/{name}/merge-audit.jsonl`), dashboard merge health as 5th signal, configurable test command via daemon config, `"merge-failed"` failure taxonomy category, dynamic merge lock timeout.

**Effort:** S (human: ~2 days / CC: ~25 min)
**Risk:** Low
**Pros:**
- Complete validation gate with full observability
- Audit trail enables debugging "what broke main?"
- Dashboard shows merge health trends (pass rate, avg test time, failure patterns)
- Configurable test command supports repos with custom test scripts
- Properly separates job success from merge outcome
- Completeness: 9/10

**Cons:**
- Touches 6 source files
- ~70 new tests across 4 test files
- Reweights existing dashboard health score formula

### Approach C: Test Gate + Auto-Fix Loop

**Summary:** If pre-merge tests fail, run a mini QA-fix cycle on the worktree branch. Max 2 fix attempts before abort.

**Effort:** L (human: ~2 weeks / CC: ~2 hours)
**Risk:** High
**Pros:**
- Self-healing test failures
- Could save $3-5 per failed merge

**Cons:**
- Invoking orchestrator from within the merge gate is architecturally risky
- Infinite fix loop risk
- Budget implications per fix attempt
- Duplicates QA skill responsibility
- Completeness: 10/10 (but at disproportionate effort/risk)

## Recommended Approach

**Approach B: Test Gate + Merge Audit + Dashboard.**

Hits the sweet spot: prevents the damage (test gate), explains what happened (audit log), and shows trends (dashboard). Approach A is too bare for a system running 12+ merges per day. Approach C duplicates QA skill responsibility. This is a reimplementation from worker-2's existing design, not greenfield.

## Implementation Order

### Step 1: MergeValidationConfig + test gate in worktree.ts

Add a `MergeValidationConfig` interface:
```typescript
export interface MergeValidationConfig {
  /** Command to run for validation (default: "npm test") */
  testCommand?: string;
  /** Timeout in ms (default: 120000 = 2 min) */
  testTimeout?: number;
  /** Skip validation entirely (default: false) */
  skipValidation?: boolean;
}
```

Extend the existing `MergeResult` interface (worktree.ts:22-26) with new optional fields:
```typescript
export interface MergeResult {
  merged: boolean;
  reason?: string;
  commitCount?: number;
  testsPassed?: boolean;       // NEW: undefined if tests not run, true/false otherwise
  testOutput?: string;         // NEW: truncated stderr from test run (max 2000 chars)
  testDurationMs?: number;     // NEW: how long tests took
}
```

Add an optional `validation?: MergeValidationConfig` parameter to `mergeWorktreeBranch()`. After the `git rebase baseBranch` call succeeds (worktree.ts:348) and before the `git merge --ff-only` call (worktree.ts:366):

1. If `validation?.skipValidation` is true, skip to ff-only merge
2. If `validation` is undefined/null, skip to ff-only merge (backward compatible, no tests by default)
3. Run `validation.testCommand ?? "npm test"` in the worktree directory via `execFileSync` with `timeout: validation.testTimeout ?? 120_000`
4. Capture stdout+stderr and measure duration with `Date.now()` delta
5. If exit code !== 0: leave the rebased branch as-is (rebase already succeeded), skip the ff-only merge, return `{ merged: false, reason: "Pre-merge tests failed", testsPassed: false, testOutput: stderr.slice(0, 2000), testDurationMs }`
6. If exit code === 0: set `testsPassed = true`, `testDurationMs`, proceed to ff-only merge

**Important:** Do NOT `git rebase --abort` on test failure. The rebase completed successfully. The commits are on the branch. We just skip the merge step.

**Key insight:** Run tests in the worktree directory (`getWorktreePath()` or the `wtDir` already computed at line 345), not `repoDir`. The worktree has the rebased code. The main repo is on `baseBranch`.

**Test command execution:** Use `execSync` (not `execFileSync`) with `{ cwd: wtDir, timeout, stdio: 'pipe', shell: true }` to support compound commands like `npm run lint && npm test`. This is the one exception to the module's `execFileSync`-only rule (documented in the header comment at line 9), because the test command is user-configured and may contain pipes or `&&`. The security tradeoff is acceptable: the command comes from the daemon config file, not from untrusted input.

**Prerequisite: `node_modules` in worktree.** The worktree already has `node_modules` because the daemon runs skills (implement, QA) in the worktree directory, which installs dependencies. The test gate runs AFTER the pipeline completes, so `node_modules` exists. If a project doesn't install deps during pipeline execution, the test command will fail and the merge will be blocked (correct behavior).

### Step 3: Update job-runner.ts auto-merge block

Replace the current auto-merge block (job-runner.ts:355-368) with:

```typescript
// Auto-merge: named instances merge their branch to main after successful jobs
if (jobConfig.worktreePath && resolvedInstanceName !== "default") {
  try {
    const baseBranch = resolveBaseBranch(jobConfig.projectDir);
    const mergeConfig = jobConfig.merge ?? {};
    const result = mergeWorktreeBranch(
      jobConfig.projectDir,
      resolvedInstanceName,
      baseBranch,
      {
        validation: mergeConfig.skipValidation
          ? { skipValidation: true }
          : {
              testCommand: mergeConfig.testCommand,
              testTimeout: mergeConfig.testTimeout,
            },
        jobId: nextJob.id,
      },
    );

    if (result.merged) {
      d.log("info", `Auto-merge: merged ${result.commitCount ?? 0} commit(s) from garyclaw/${resolvedInstanceName} to ${baseBranch}` +
        (result.testDurationMs ? ` (tests: ${Math.round(result.testDurationMs / 1000)}s)` : ""));
    } else {
      d.log("warn", `Auto-merge blocked: ${result.reason}` +
        (result.testOutput ? `\n${result.testOutput.slice(0, 500)}` : ""));
      // Notify on merge failure (not a job failure, but worth alerting)
      d.notifyMergeBlocked?.(nextJob, result, jobConfig);
    }
  } catch (err) {
    d.log("warn", `Auto-merge error: ${err instanceof Error ? err.message : String(err)}`);
  }
}
```

**Add `notifyMergeBlocked` to `notifier.ts`:** Follow the existing `notifyJobError` pattern. Export a function that takes `(job: Job, mergeResult: MergeResult, config: DaemonConfig)` and sends a macOS notification with the instance name, branch, and reason (test failure vs rebase conflict). Add to `JobRunnerDeps` interface in job-runner.ts as an optional method: `notifyMergeBlocked?: (job: Job, result: MergeResult, config: DaemonConfig) => void`.

**Key change from worker-2's design:** No `MergeFailedError` thrown. No job status change. The job stays "complete" because the pipeline succeeded. Merge outcome is tracked via audit log and dashboard, not by failing the job.

```typescript
// Compute dynamic lock timeout: testTimeout + 60s buffer (default: 180s)
const lockTimeout = options?.validation?.skipValidation
  ? MERGE_LOCK_TIMEOUT_MS   // no tests = use default 60s
  : (options?.validation?.testTimeout ?? 120_000) + 60_000;

if (!acquireMergeLock(repoDir, lockTimeout)) {
  // ...
}
```

This ensures other instances waiting for the lock will wait long enough for both rebase + test execution to complete.

## Open Questions

1. **Test timeout default:** 120s is generous for 2046 tests. Should we measure actual test duration and set timeout to 2x the p95? For now, 120s is safe and users can override.
2. **Partial test failure:** If 2045/2046 tests pass and 1 flaky test fails, the merge gate blocks. No configurable threshold yet. Flaky test detection is a separate concern, not needed for v1.
3. **Notification for merge blocked:** The `d.notifyMergeBlocked?.()` call needs a new method on the daemon context. Pattern: same as `notifyJobComplete` but with merge result details. If method doesn't exist, skip (optional chaining handles this).

## Success Criteria

- Zero broken merges to main: every commit on main after this ships passes the full test suite
- Merge outcomes visible: dashboard shows merge health, failures have audit trail
- Backward compatible: existing configs without `merge` section work exactly as before (no tests, merge proceeds)
- Test gate adds <90s overhead per merge (target: ~45s for GaryClaw's test suite)
- Job completion status unaffected: a successful pipeline stays "complete" regardless of merge outcome

## What I noticed about how you think

- The project has 2046 tests and 34 source modules, but the merge gate has zero validation. This gap stands out because everything else (Oracle with 7 principles, reflection with Levenshtein matching, failure taxonomy with 8 categories) is so carefully designed. The merge gate is the one place where "it probably works" is the standard.
- Worker-2's original design wanted merge failure to retroactively fail the job. But the code tells a different story: auto-merge runs at line 355, after the job is marked complete at line 332. Treating merge failure as job failure would re-run a $3-5 pipeline for a rebase conflict. The refined design separates these concerns: job success is about the pipeline, merge success is about landing the code.
- The advisory lock pattern (mkdir-based) appears 3 times now (reflection lock, merge lock, claim lock). This codebase picks simple patterns and reuses them. The test gate follows the same principle.
- The self-improvement loop this design doc feeds into is remarkable. The daemon will implement this design, QA it, and then the test gate will validate itself during its own merge.
