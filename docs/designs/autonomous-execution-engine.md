# CEO Plan: GaryClaw — Autonomous Skill Execution Engine

## Vision

### 10x Check
GaryClaw as a **development daemon** — a persistent process that watches your project, queues skills, runs them autonomously in parallel, and presents a unified dashboard. You push code, go to sleep. GaryClaw runs /qa, catches 12 bugs, fixes them, runs /design-review, polishes 8 visual issues, runs /review, resolves 3 findings, creates a PR. You wake up to a summary: "14 fixes shipped, 3 taste decisions for your review, PR ready."

### Platonic Ideal
A **transparent autonomy layer** where the user feels like they have a tireless junior engineer who shares their taste. The key feeling: *trust*. Every decision has a legible rationale. Taste decisions are surfaced, not hidden. The audit trail is complete — you can replay any decision. It knows when it's uncertain and says so. Quality never degrades across session boundaries — fix #47 is as good as fix #1.

## Core Problem (Validated)

Two bottlenecks kill long-running gstack skills:
1. **Context exhaustion** — even at 1M tokens (~830K usable after system prompt, tool definitions, and compaction buffer), iterative fix loops with screenshots, source reads, edits, and commits fill the window. Confirmed still happening post-1M window.
2. **Human attention** — skills ask 15-30 AskUserQuestion prompts. Each one blocks the process waiting for human input. For truly autonomous development, an AI decision-maker must handle these.

Compaction API (beta) doesn't solve this — it summarizes away SKILL.md instructions, degrading quality. Fresh session + checkpoint prompt is better.

## Architecture Decision

**Approach A: Agent SDK Harness** — chosen over CLI Pipe (can't intercept mid-turn) and Hybrid (two mechanisms to maintain).
- `@anthropic-ai/claude-agent-sdk` for structured JSON communication
- `canUseTool` callback intercepts AskUserQuestion mid-turn (see Interception Mechanism below)
- `result.usage` for token tracking (see Token Tracking Strategy below)
- `resume` / `forkSession` for session management
- Decision oracle: separate Opus 4.6 API call per AskUserQuestion, using the 6 Decision Principles

**Plan A: canUseTool with updatedInput (confirmed by SDK docs)**
```
1. SDK's query() runs the skill (via settingSources or Skill tool)
2. Claude calls AskUserQuestion tool
3. canUseTool fires → call Decision Oracle (Phase 2) or surface to human (Phase 1b) → get answer
4. Return { behavior: 'allow', updatedInput: { questions: input.questions, answers: { "question": "chosen_label" } } }
5. AskUserQuestion executes with pre-filled answer, returns immediately
```

**Fallbacks (documented, not actively planned):**
- **B1: AsyncIterable input injection** — SDK accepts `prompt: AsyncIterable<SDKUserMessage>`. Inject tool_result after detecting AskUserQuestion in output stream.
- **B2: Terminal automation (node-pty)** — Spawn Claude Code in pseudo-terminal, type answers when AskUserQuestion appears.
- **B3: Prompt-level pre-answering** — Include decision rules in prompt to eliminate AskUserQuestion calls entirely. Simplest but loses audit trail.

### Token Tracking Strategy

**Execution model: bounded query() segments.** The SDK reports usage on `ResultMessage` at the end of each `query()` call, not per-turn mid-loop. GaryClaw runs skill execution as a series of bounded `maxTurns` segments. Between segments, it checks token usage and decides whether to continue or relay.

```
Segment 1 (maxTurns: 15) → ResultMessage.usage → check threshold
  ↓ under threshold
Segment 2 (maxTurns: 15) → ResultMessage.usage → check threshold
  ↓ OVER threshold (>700K input_tokens)
Checkpoint → Relay → Fresh session → Segment 3...
```

**Key rules:**
- **Context size formula:** `input_tokens + cache_creation_input_tokens + cache_read_input_tokens` from the latest `AssistantMessage.message.usage` (NOT just `input_tokens` alone — that undercounts by excluding cached tokens like system prompt, tool defs, SKILL.md)
- **Context window denominator:** `modelUsage.contextWindow` from `ResultMessage` gives the model's max context window size
- **Per-turn monitoring IS available** — `AssistantMessage.message.usage` carries per-turn token counts, enabling mid-segment monitoring (not just end-of-segment)
- **Relay threshold:** When computed context size exceeds ~700K (85% of 830K usable), trigger checkpoint and relay
- **Default maxTurns:** 15 per segment (enough for a fix iteration, not so much that context explodes)
- **Trend tracking:** Log per-turn context size to detect growth rate and predict relay timing

**Env var passthrough CONFIRMED:** SDK `query()` accepts `env: Record<string, string>` option (fixed in v0.2.73). Pass `$B` browse binary via `env: { ...process.env, B: "/path/to/browse" }`.

**SPIKE #2 must verify:** (a) Confirm per-turn usage includes all three token fields as documented, (b) whether `resume` with `settingSources` preserves skill instructions through compaction (if yes, resume may be viable alternative to fresh sessions)

### The 6 Decision Principles (from /autoplan)

Source: `~/.claude/skills/gstack/autoplan/SKILL.md`, lines 387-401

1. **Choose completeness** — Ship the whole thing. Pick the approach that covers more edge cases.
2. **Boil lakes** — Fix everything in the blast radius. Auto-approve expansions that are in blast radius AND < 1 day CC effort.
3. **Pragmatic** — Two options fix the same thing? Pick the cleaner one. 5 seconds choosing, not 5 minutes.
4. **DRY** — Duplicates existing functionality? Reject. Reuse what exists.
5. **Explicit over clever** — 10-line obvious fix > 200-line abstraction.
6. **Bias toward action** — Merge > review cycles > stale deliberation. Flag concerns but don't block.

**Conflict resolution:** CEO phases → P1+P2 dominate. Eng phases → P5+P3 dominate. Design phases → P5+P1 dominate.

### Decision Oracle Specification

**Input contract:**
```typescript
interface OracleInput {
  question: string;        // The AskUserQuestion text
  options: Option[];       // The available choices
  skillName: string;       // Which skill is running (e.g., "qa", "design-review")
  skillPhase: string;      // Current phase context
  projectContext: string;  // CLAUDE.md content (abbreviated)
  decisionHistory: Decision[];  // Prior decisions this session
  principles: string;     // The 6 principles above
}
```

**Output contract:**
```typescript
interface OracleOutput {
  choice: string;          // Selected option label
  confidence: number;      // 1-10 scale
  rationale: string;       // One sentence explaining why
  principle: string;       // Which principle drove the decision
  isTaste: boolean;        // True if confidence < 6 or multiple viable options
  escalate: boolean;       // True if security/destructive/contradicts skill recommendation
}
```

**Oracle prompt structure:** "You are a decision-making agent for an autonomous development tool. Given the following question from a {skillName} skill in phase {skillPhase}, select the best option using these principles: {principles}. Consider the project context and prior decisions. Return your answer as JSON."

**SPIKE REQUIRED:** Run the oracle against 20+ historical AskUserQuestion prompts from real skill runs to validate decision quality before relying on it in production.

## Terminology

- **Milestone** — GaryClaw's own implementation phases (1a, 1b, 2, 3, 4). Numbered.
- **Skill phase** — A gstack skill's internal phases (e.g., /qa "Phase 8: Fix Loop"). Identified by `## Phase N` headers in SKILL.md files.
- **Relay** — The act of checkpointing state and starting a fresh Claude Code session.

## Skill Phase Detection

Skill phases are identified by parsing `## Phase N` or `## Step N` headers in SKILL.md files. The GaryClaw harness does NOT need to deeply understand phases — it treats the skill as a black box and only needs to identify **safe relay points** (between commits, between findings). The token monitor triggers relay; the checkpoint captures current state regardless of which skill phase is active. No SKILL.md format changes required.

## Checkpoint Schema

```typescript
interface Checkpoint {
  version: 1;
  timestamp: string;              // ISO 8601
  skillName: string;              // e.g., "qa"
  skillPhase: string;             // e.g., "Phase 8: Fix Loop"
  phaseIndex: number;             // 0-based
  phasesCompleted: string[];      // Names of completed phases

  // Discovered work
  issues: Issue[];                // Found issues with status
  findings: Finding[];            // Non-issue observations

  // Decision history (session-level)
  decisions: Decision[];          // All auto-decisions made so far

  // Git state (session-level)
  gitBranch: string;
  gitHead: string;                // Commit SHA at checkpoint time
  uncommittedChanges: boolean;

  // Token state (session-level)
  tokenUsage: {
    lastInputTokens: number;      // Most recent segment's input_tokens (≈ context size)
    totalOutputTokens: number;    // Cumulative output tokens generated
    sessionCount: number;         // How many relay sessions so far
    estimatedCostUsd: number;
  };

  // File references (session-level)
  screenshotPaths: string[];      // Screenshots taken
  artifactPaths: string[];        // Generated files (reports, logs)
}

interface Issue {
  id: string;                     // e.g., "QA-001"
  severity: "critical" | "high" | "medium" | "low" | "cosmetic";
  description: string;
  filePath?: string;
  screenshotPath?: string;
  status: "open" | "fixed" | "skipped" | "deferred";
  fixCommit?: string;             // Git SHA of the fix commit
}

interface Finding {
  description: string;
  category: string;               // e.g., "design", "performance", "accessibility"
  actionTaken?: string;
}
```

**Relay prompt optimization:** When generating the relay prompt from a checkpoint, use a tiered strategy:
- **Open issues:** Full details (description, file path, screenshot path)
- **Last 5 fixed issues:** Full details (so the new session knows what was recently fixed)
- **Older fixed issues:** One-line summary each (e.g., "QA-007: Fixed nav z-index in header.tsx")
- **Findings:** Full text (typically short)
- **Decisions:** Last 5 full, older ones summarized

Target: relay prompt under 10K tokens (excluding SKILL.md loaded via settingSources).

## Failure Modes and Recovery

| Failure Mode | Detection | Recovery | User Sees |
|---|---|---|---|
| SDK connection error | query() throws | Retry 2x with backoff, then save checkpoint and exit | "Connection lost. Run `garyclaw resume` to continue." |
| Decision oracle timeout | API call exceeds 30s | Fall back to heuristic (pick option marked "Recommended") | Logged as low-confidence decision |
| Decision oracle hallucination | Response doesn't match any option | Parse fuzzy match; if no match, escalate to human or pick first option | Logged as escalated decision |
| Corrupt checkpoint | JSON parse error on resume | Start from last valid checkpoint (keep 2 rotations) | "Checkpoint corrupted. Resuming from previous checkpoint." |
| Git state conflict | Uncommitted changes at relay | `git stash` before relay, `git stash pop` after | Logged in checkpoint |
| Claude refuses tool call | Tool call denied in SDK | Log and skip; continue with next phase | Decision logged as "refused" |
| Process killed mid-fix | No detection (crash) | `garyclaw resume` reads checkpoint | Last committed state is safe; current fix may be partial |
| Token estimate wrong (relay too late) | Context error from API | Aggressive relay threshold (85% not 95%) | Relay triggers early enough to avoid this |
| Skill instructions too large for relay | >10K tokens of SKILL.md | Summarize SKILL.md for relay prompt; keep full version in CLAUDE.md | Quality may degrade — log and flag |

## Authentication

**Uses Max plan billing via CLAUDE_CODE_OAUTH_TOKEN** (not API key pay-per-token).

Setup: `claude setup-token` → export `CLAUDE_CODE_OAUTH_TOKEN=<token>`

**Risk:** This is a community-confirmed workaround ([Issue #559](https://github.com/anthropics/claude-agent-sdk-python/issues/559), closed as Completed), not officially supported by Anthropic. [Issue #6536](https://github.com/anthropics/claude-code/issues/6536) was closed as Not Planned. The workaround works today but could break if Anthropic restricts OAuth tokens to the interactive CLI.

**Mitigation:** GaryClaw verifies auth at startup before any SDK call. If the OAuth token is rejected, it displays a clear error with instructions to re-run `claude setup-token` or fall back to `ANTHROPIC_API_KEY`.

## Landscape Context

- Claude Code 1M context window is GA (~830K usable after system prompt, tool defs, compaction buffer) — buys headroom but doesn't solve the structural problem for iterative skills
- Compaction API (beta) does server-side summarization but compresses away skill instructions
- Agent Teams (experimental) — isolated context windows with peer messaging, but no session resumption
- Letta/MemGPT — virtual context management for general LLM agents, not Claude Code specific
- No existing tool solves autonomous skill execution with context relay for Claude Code
