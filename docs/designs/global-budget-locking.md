# Design: Global Budget Locking — Prevent Lost Updates in Parallel Instances

Generated by /office-hours on 2026-03-30
Branch: main
Repo: 0raclide/GaryClaw
Status: APPROVED
Mode: Builder

## Problem Statement

Parallel daemon instances (`--parallel 5`) share a single `global-budget.json` file for cost tracking. The `updateGlobalBudget()` function in `daemon-registry.ts` does an unsynchronized read-modify-write:

```
Instance A: read {totalUsd: 5.0}
Instance B: read {totalUsd: 5.0}   // same stale value
Instance A: write {totalUsd: 5.5}  // adds 0.5
Instance B: write {totalUsd: 5.3}  // adds 0.3, clobbers A's update
```

Result: `totalUsd` is 5.3 instead of 5.8. Instance A's $0.50 job cost is silently lost. Over an overnight run with dozens of jobs across 5 instances, budget tracking drifts by dollars. This allows budget overruns (the daily limit check reads stale totals) and makes the dashboard unreliable.

The same race affects `setGlobalRateLimitHold()` and `clearGlobalRateLimitHold()`, though those are less likely to collide (rate limits are rare events).

Real bug: ISSUE-005, deferred from 2026-03-30 run6.

## What Makes This Cool

The fix uses the exact same mkdir-based advisory lock pattern already proven in `reflection-lock.ts`. Same POSIX atomicity guarantee, same PID-based stale detection, same timeout polling. Copy the pattern, apply it to a different file. The codebase already solved this problem once.

## Constraints

- XS effort (priority pick scored effort efficiency 10/10)
- Must not break existing single-instance behavior (default instance, no parallel)
- Lock must be fast (microsecond-scale hold time, just read-modify-write)
- Fail-open: lock timeout logs warning, proceeds with unprotected write
- No new npm dependencies

## Premises

1. The read-modify-write on global-budget.json is unprotected and causes real lost updates in parallel mode.
2. The mkdir-based lock pattern from reflection-lock.ts is proven and sufficient for this use case.
3. Three mutating functions need protection: `updateGlobalBudget`, `setGlobalRateLimitHold`, `clearGlobalRateLimitHold`.
4. Read-only calls (`readGlobalBudget`) do NOT need locks. Stale reads cause at most one extra job before budget enforcement catches up. That's the existing acceptable margin.
5. Lock scope is narrow (just the file I/O), not the entire job lifecycle.
6. Fail-open on lock timeout. Lost updates are bad; crashed jobs are worse.

## Approaches Considered

### Approach A: Dedicated budget-lock.ts module (CHOSEN)
New `src/budget-lock.ts` module mirroring the reflection-lock API but targeting `.garyclaw/.budget-lock/`. Wrap the three mutating functions in acquire/release.

- Effort: XS
- Risk: Low
- Pros: Clean module boundary, dedicated lock directory, mirrors proven pattern, easy to test in isolation
- Cons: One new file (tiny, ~60 lines)

### Approach B: Generic file-lock utility + DRY refactor
Extract the mkdir lock pattern into a shared `file-lock.ts` utility. Refactor both reflection-lock.ts and budget-lock to use it.

- Effort: S
- Risk: Low-Med (touches working reflection-lock code)
- Pros: DRY, single lock implementation
- Cons: Violates XS constraint, refactoring risk on stable code

### Approach C: Inline lock in daemon-registry.ts
Add lock functions directly in daemon-registry.ts.

- Effort: XS
- Risk: Low
- Pros: Zero new files, fastest to implement
- Cons: DRY violation with reflection-lock.ts, harder to test in isolation

## Recommended Approach

**Approach A.** Clean module boundary, proven pattern, no refactoring risk. The DRY violation with reflection-lock.ts is acceptable. If GaryClaw grows more lock sites (3+), a future consolidation pass can extract a shared utility then. Two independent lock modules is fine for now.

### Implementation Plan

#### 1. New file: `src/budget-lock.ts` (~60 lines)

```typescript
/**
 * Budget Lock — file-based advisory lock for global-budget.json writes.
 *
 * Prevents concurrent read-modify-write on global-budget.json from
 * parallel daemon instances. Uses mkdir (atomic on POSIX) as the
 * locking primitive, same pattern as reflection-lock.ts.
 */

import { mkdirSync, rmSync, writeFileSync, readFileSync, existsSync } from "node:fs";
import { join } from "node:path";

const LOCK_DIR_NAME = ".budget-lock";
const PID_FILE_NAME = "pid";
const DEFAULT_TIMEOUT_MS = 5_000;  // 5s (shorter than reflection-lock's 30s)
const POLL_INTERVAL_MS = 50;       // 50ms (faster poll for microsecond-hold lock)

export function acquireBudgetLock(checkpointDir: string, timeoutMs?: number): boolean
export function releaseBudgetLock(checkpointDir: string): void
export function isBudgetLocked(checkpointDir: string): boolean
```

The optional `timeoutMs` parameter (default 5s) is required for tests that need short timeouts to avoid slow test runs, matching the pattern in `reflection-lock.test.ts`.

Key differences from reflection-lock.ts:
- **Shorter timeout** (5s vs 30s). Budget writes take microseconds. If we're waiting 5s, something is very wrong.
- **Faster poll** (50ms vs 500ms). The lock is held for so briefly that 500ms polling would add unnecessary latency.
- **Lock directory:** `{checkpointDir}/.budget-lock/` where `checkpointDir` is `.garyclaw/`. So the full path is `.garyclaw/.budget-lock/`. The constant `LOCK_DIR_NAME = ".budget-lock"` is joined with the `checkpointDir` argument, same as how reflection-lock joins with `oracleMemoryDir`.

Internal helpers (`tryCreateLockDir`, `isOwnLock`, `isStaleLock`) are identical to reflection-lock.ts.

#### 2. Modify: `src/daemon-registry.ts` — wrap three functions

**`updateGlobalBudget()`** — wrap the existing read-modify-write in acquire/release:

```typescript
export function updateGlobalBudget(
  checkpointDir: string,
  addCostUsd: number,
  instanceName: string,
): GlobalBudget {
  const locked = acquireBudgetLock(checkpointDir);
  if (!locked) {
    // Fail-open: log warning, proceed without lock
    console.warn("budget-lock: timeout acquiring lock for updateGlobalBudget, proceeding without lock");
  }
  try {
    const budget = readGlobalBudget(checkpointDir);
    budget.totalUsd += addCostUsd;
    budget.jobCount += 1;
    // ... per-instance tracking (unchanged) ...
    const filePath = join(checkpointDir, GLOBAL_BUDGET_FILE);
    safeWriteJSON(filePath, budget);
    return budget;
  } finally {
    if (locked) releaseBudgetLock(checkpointDir);
  }
}
```

**`setGlobalRateLimitHold()`** — same pattern (acquire, read, conditionally write, release). Note: even though `rateLimitResetAt` uses extend-only semantics (only updates if new time is later), locking is still required because the function reads the entire `GlobalBudget` object and writes it back. Without a lock, the write clobbers concurrent changes to `totalUsd` and `jobCount` fields in the same JSON file, even though the `rateLimitResetAt` field itself would have been fine.

**`clearGlobalRateLimitHold()`** — same pattern. Same rationale: the read-modify-write touches the entire JSON object.

### Edge Cases

1. **Default instance (no parallel):** Lock still works, just never contended. Zero overhead (mkdir succeeds immediately).
2. **Lock held during crash:** PID-based stale detection recovers on next attempt. Same as reflection-lock.
3. **Doctor cleanup:** `--fix` removes stale `.budget-lock/` just like it handles `.reflection-lock/`.
4. **Date rollover during locked write:** The read inside the lock sees the fresh date and resets. No issue.
5. **safeWriteJSON failure inside lock:** `finally` block ensures lock is always released. The write failure propagates normally.

### What NOT to do

- Don't lock `readGlobalBudget()`. Stale reads are acceptable for budget headroom checks. Locking reads would serialize all instances' `processNext()` calls, adding latency for zero safety benefit.
- Don't use `flock()`. Node.js doesn't have built-in `flock`, and the mkdir pattern is already proven in this codebase.
- Don't refactor reflection-lock.ts. It works. Two similar-but-independent lock modules is the right tradeoff for XS scope.
