## Problem Statement

GaryClaw's backlog is exhausted. Health score 100, 1725 tests passing, Oracle accuracy 100% across 239 decisions. No remaining item scores above 5.0/10. The bootstrap skill exists (`src/bootstrap.ts`) and the full pipeline (`bootstrap → prioritize → implement → qa`) is wired up, but it has never been exercised on a real external repo.

The missing piece isn't running the pipeline... that command already works:
```bash
npx tsx src/cli.ts run bootstrap prioritize implement qa --autonomous --project-dir /path/to/target
```

The missing piece is **what happens after**. When the pipeline finishes on an external repo, GaryClaw produces artifacts (.garyclaw/ dir, decisions.jsonl, checkpoint files, dashboard report) but nobody reads them. There's no structured evaluation of:

1. **Bootstrap quality** — Was the generated CLAUDE.md accurate? Did TODOS.md produce items scoring >5.0 in prioritize?
2. **Oracle performance** — Where did the Oracle struggle? What decision patterns emerged?
3. **Pipeline health** — How many relays? What was the context growth rate? Did skills chain correctly?
4. **GaryClaw improvement candidates** — What should we build next based on what went wrong?

The evaluate skill closes this gap. It runs after the dogfood pipeline, reads all artifacts, and produces a structured campaign report that feeds directly back into GaryClaw's own prioritize skill.

## What Makes This Cool

You run GaryClaw on an unfamiliar repo overnight. In the morning you don't just see what it fixed in that repo... you see what it learned about *itself*. "Bootstrap missed the monorepo structure (CLAUDE.md didn't mention workspace packages). Oracle made 4 low-confidence decisions about test frameworks it hadn't researched. The implement skill relayed 3 times on a 2-step task because browse-heavy steps triggered aggressive context growth."

Those observations become GaryClaw backlog items automatically. The system generates its own next sprint of work.

## Constraints

- Must run as a regular GaryClaw skill, composable with the existing pipeline: `bootstrap → prioritize → implement → qa → evaluate`
- Must also work standalone: `garyclaw run evaluate --project-dir /path/to/target` (reads existing .garyclaw/ artifacts)
- Must not modify the target repo's source files. Evaluation artifacts go into the target's `.garyclaw/` dir (evaluation report). Improvement candidates for GaryClaw itself are returned via pipeline context handoff, and a post-pipeline hook in `cli.ts` (which captures `process.cwd()` at startup before `--project-dir` redirects) appends them to GaryClaw's own TODOS.md. This avoids cross-repo writes from within a skill.
- Must handle the case where the pipeline partially failed (some skills completed, others didn't)
- Must work within existing budget system. Evaluation itself is cheap... mostly reading and synthesizing JSON/markdown files, minimal Claude turns needed.
- Output format must be consumable by GaryClaw's own prioritize skill (TODOS.md format for improvement candidates)

## Premises

1. The bootstrap pipeline works end-to-end but has never been tested on a real external repo. First dogfood run is the immediate priority.
2. The highest-value output from dogfooding isn't the fixes to the target repo... it's the GaryClaw improvement opportunities revealed by the run.
3. Existing infrastructure (reflection.ts, dashboard.ts, auto-research.ts) captures raw data. What's missing is the synthesis that converts cross-run data into actionable GaryClaw backlog items.
4. A single-repo evaluator is the right scope. Multi-repo campaign orchestration is a follow-on that requires real single-repo data first.
5. The first dogfood run will unblock the "Evaluate Bootstrap Output Quality" TODO that's currently blocked.

## Approaches Considered

### Approach A: Post-Run Evaluator Skill (Minimal Viable)

**Summary:** New `evaluate` skill that runs after the dogfood pipeline, reads all .garyclaw/ artifacts from the target repo, and produces a structured campaign report + GaryClaw improvement candidates.

**Effort:** S
**Risk:** Low
**Pros:**
- Chains into existing pipeline: `bootstrap → prioritize → implement → qa → evaluate`
- Single new module (`src/evaluate.ts`) following existing skill patterns (like implement.ts, prioritize.ts)
- Unblocks the "Evaluate Bootstrap Output Quality" TODO immediately
- Pure TypeScript analysis (reads files, computes metrics) + one Claude segment for synthesis

**Cons:**
- Single-repo only, no cross-repo synthesis
- Evaluation quality depends on Claude's ability to self-assess
- First run has no baseline to compare against

**Reuses:** pipeline.ts skill chaining, dashboard.ts aggregation patterns, reflection.ts outcome analysis, safe-json.ts for file I/O

### Approach B: Campaign Runner with Cross-Repo Synthesis

**Summary:** New `garyclaw campaign` command that runs the full dogfood pipeline across multiple repos sequentially, then synthesizes patterns across all runs.

**Effort:** M
**Risk:** Med
**Pros:**
- Cross-repo patterns reveal systemic GaryClaw weaknesses
- Auto-generates TODOS.md entries for GaryClaw itself

**Cons:**
- Over-engineers before single-repo dogfood is validated
- Cross-repo synthesis logic is speculative until we see real data
- More complex CLI surface area

### Approach C: Daemon Self-Improvement Loop (Ideal Architecture)

**Summary:** Daemon mode extension where GaryClaw autonomously dogfoods on a curated repo list, evaluates results, generates its own backlog items, and implements them.

**Effort:** L
**Risk:** High
**Pros:**
- Full closed-loop self-improvement
- Maximum "whoa" factor

**Cons:**
- Requires Approach A working first
- Self-modifying code without guard rails is risky
- Need human review gate before self-modifications land

## Recommended Approach

**Approach A: Post-Run Evaluator Skill.** Ship the evaluator, run it on one real repo, and use the actual data to decide if B or C is worth building. A is a prerequisite for both anyway.

## Implementation Details

### New Module: `src/evaluate.ts`

Follows the same pattern as `src/implement.ts` and `src/prioritize.ts`:

```typescript
// src/evaluate.ts — Post-dogfood evaluation and GaryClaw improvement extraction

export interface EvaluationReport {
  targetRepo: string;
  timestamp: string;
  bootstrap: BootstrapEvaluation;
  oracle: OracleEvaluation;
  pipeline: PipelineEvaluation;
  improvements: ImprovementCandidate[];
}

export interface BootstrapEvaluation {
  claudeMdExists: boolean;
  claudeMdSizeTokens: number;
  claudeMdHasSections: string[];         // e.g. ["Architecture", "Tech Stack", "Test Strategy"]
  claudeMdMissingSections: string[];     // expected but absent
  todosMdExists: boolean;
  todosMdItemCount: number;
  todosMdItemsAboveThreshold: number;    // items that scored >5.0 in prioritize
  qualityScore: number;                  // 0-100
  qualityNotes: string[];
}

export interface OracleEvaluation {
  totalDecisions: number;
  lowConfidenceCount: number;            // confidence < 6
  escalatedCount: number;
  averageConfidence: number;
  topicClusters: { topic: string; count: number; avgConfidence: number }[];
  researchTriggered: boolean;
}

export interface PipelineEvaluation {
  skillsRun: string[];
  skillsCompleted: string[];
  skillsFailed: string[];
  totalRelays: number;
  totalCostUsd: number;
  totalDurationSec: number;
  contextGrowthRate: number;             // average across segments
  adaptiveTurnsUsed: boolean;
}

export interface ImprovementCandidate {
  title: string;
  priority: "P2" | "P3" | "P4";         // P1 reserved for human-escalated items only
  effort: "XS" | "S" | "M";
  category: "bootstrap" | "oracle" | "pipeline" | "skill" | "relay";
  description: string;
  evidence: string;                      // specific data from the dogfood run
}
```

### Key Functions

#### `analyzeBootstrapQuality(projectDir: string): Promise<BootstrapEvaluation>`

Reads the generated CLAUDE.md and TODOS.md from the target repo. Checks:
- **Structural completeness:** Does CLAUDE.md have Architecture, Tech Stack, Test Strategy sections?
- **Factual accuracy:** Extract dependency names from package.json/requirements.txt/go.mod (pure TS, JSON/text parsing). Extract tech stack mentions from CLAUDE.md via keyword matching against a known-frameworks list. Compute overlap: tech stack coverage = (mentioned frameworks found in deps) / (total deps). Flag deps >5% of project that CLAUDE.md doesn't mention.
- **TODOS.md viability:** Were any items picked by prioritize? What scored above 5.0?
- **Token efficiency:** Is CLAUDE.md within a reasonable size (2K-10K tokens)?

Pure TypeScript. No Claude calls needed for structural checks.

**Known-frameworks list:** Hardcoded `Map<string, string[]>` of canonical names to aliases, similar to `HEAVY_TOOLS` in token-monitor.ts. Examples:
```typescript
const KNOWN_FRAMEWORKS = new Map([
  ["React", ["react", "react-dom", "@types/react"]],
  ["Next.js", ["next", "@next/font", "next-auth"]],
  ["Vitest", ["vitest", "@vitest/coverage-v8"]],
  ["Jest", ["jest", "@types/jest", "ts-jest"]],
  ["Express", ["express", "@types/express"]],
  ["Supabase", ["@supabase/supabase-js", "@supabase/ssr"]],
  ["Tailwind CSS", ["tailwindcss", "@tailwindcss/forms"]],
  ["Prisma", ["prisma", "@prisma/client"]],
  // ... extend as needed, ~30-50 entries
]);
```
Match is case-insensitive on both dependency names and CLAUDE.md content.

**Quality score formula:** Weighted sum of 4 dimensions:
- Structural completeness: 40 pts (10 pts per expected section: Architecture, Tech Stack, Test Strategy, Usage)
- Factual accuracy: 30 pts (coverage ratio * 30, where coverage = frameworks mentioned in CLAUDE.md / frameworks found in deps)
- TODOS.md viability: 20 pts (items above 5.0 threshold / total items * 20, or 0 if no items)
- Token efficiency: 10 pts (10 if 2K-10K tokens, 5 if 1K-2K or 10K-20K, 0 otherwise)

Each `analyze*` function returns a default/empty evaluation with `qualityNotes: ["No artifacts found — {filename} missing"]` when expected files are absent. The overall report still generates with partial data.

#### `analyzeOraclePerformance(projectDir: string): Promise<OracleEvaluation>`

Reads `.garyclaw/decisions.jsonl` from the target repo. Computes:
- Decision confidence distribution
- Topic clustering (reuses `extractTopicKeywords` from auto-research.ts)
- Escalation patterns
- Whether auto-research was triggered

Pure TypeScript. Reuses existing auto-research keyword extraction.

#### `analyzePipelineHealth(projectDir: string): Promise<PipelineEvaluation>`

Reads `.garyclaw/pipeline.json` and checkpoint files. Computes:
- Skills run vs. completed vs. failed
- Relay count per skill
- Context growth rate: computed by calling `computeGrowthRate()` from token-monitor.ts on each checkpoint's `turnUsages` array, then averaging across segments. If checkpoint data is missing turnUsages (older format), falls back to `totalTokens / turnCount` estimate.
- Cost and duration (from pipeline.json `costUsd` and timestamps)
- Adaptive turns effectiveness (from `adaptive_turns` events in daemon-state.json if available)

Pure TypeScript. Reuses `computeGrowthRate()` from token-monitor.ts and aggregation patterns from dashboard.ts.

#### `extractObviousImprovements(report: EvaluationReport): ImprovementCandidate[]`

Pure function. Deterministic extraction of improvement candidates from metrics thresholds:
- Bootstrap missing sections → P2 improvement candidate
- Low-confidence Oracle clusters (3+ decisions, avg confidence < 6) → P3 research gap candidate
- TODOS.md items with hallucinated paths (path doesn't exist on disk) → P2 bootstrap guard candidate
- Relay count > 3 for a single skill → P3 context efficiency candidate

These are the "obvious" candidates that don't need Claude. Returns `ImprovementCandidate[]`.

#### `buildEvaluatePrompt(report: EvaluationReport): string`

Assembles the evaluation data into a prompt for Claude. Pattern follows `buildImplementPrompt()` and `buildPrioritizePrompt()`. The prompt instructs Claude to output improvement candidates as a JSON array within a `<improvements>` fenced block:

```json
[
  { "title": "...", "priority": "P3", "effort": "S", "category": "bootstrap", "description": "...", "evidence": "..." }
]
```

The orchestrator runs this prompt as a normal segment.

#### `parseClaudeImprovements(output: string): ImprovementCandidate[]`

Extracts the `<improvements>` JSON block from Claude's segment output. Validates each entry against the `ImprovementCandidate` interface (required fields, valid priority/effort/category enum values). Malformed entries are dropped with a warning. Returns the parsed array.

After parsing, merge with `extractObviousImprovements()` results. Dedup: normalized Levenshtein distance < 0.3 on titles (reusing the threshold from reflection.ts). When duplicates exist, keep the version with more specific evidence.

#### `writeEvaluationReport(projectDir: string, report: EvaluationReport): Promise<void>`

Writes two outputs:
1. `{targetRepo}/.garyclaw/evaluation-report.json` — structured data
2. `{targetRepo}/.garyclaw/evaluation-report.md` — human-readable markdown

#### `formatImprovementCandidates(candidates: ImprovementCandidate[]): string`

Pure function. Formats candidates as TODOS.md entries matching the contract expected by `parseTodoItems()` in prioritize.ts:

```markdown
## {priority}: {title}

**What:** {description}

**Why:** {evidence}

**Effort:** {effort} (human: ~{est} / CC: ~{est})
**Depends on:** Nothing
**Added by:** evaluate skill on {date}
```

This format matches existing TODOS.md entries exactly (heading level 2 with priority prefix, bold field labels, effort with both scales). `parseTodoItems()` parses on `## P{n}:` headings.

**Cross-repo write strategy:** The evaluate skill does NOT write to GaryClaw's TODOS.md directly. Instead, it returns the formatted candidates in its pipeline context handoff (via the `findings` field in the pipeline state). A post-pipeline step in `cli.ts` (which knows its own repo root via the original `process.cwd()` captured at startup) reads the evaluate skill's findings and appends them to GaryClaw's TODOS.md using `fs.appendFile` with the atomic read-append-write pattern. This avoids cross-repo file writes from within a skill running in a `--project-dir` context.

When running standalone (no pipeline), the evaluate skill writes candidates to `{projectDir}/.garyclaw/improvement-candidates.md` for manual transfer.

### Pipeline Integration

In `src/pipeline.ts`, add `evaluate` to the known skill list. When chained:

```bash
npx tsx src/cli.ts run bootstrap prioritize implement qa evaluate --autonomous --project-dir /path/to/target
```

The evaluate skill receives the pipeline context handoff from qa (issues found, fixes applied, decisions made) and also reads the .garyclaw/ artifacts directly.

### CLI Integration

```bash

npx tsx src/cli.ts run bootstrap prioritize implement qa evaluate --autonomous --project-dir /path/to/target

# Run standalone on a repo that already has .garyclaw/ artifacts
```

## Open Questions

1. **Target repo selection:** Which external repo should the first dogfood run target? Ideal: a real TypeScript/Node.js project with actual issues, not too large (< 500 files), not too small (> 20 files).
2. **Improvement candidate threshold:** Should candidates auto-append to GaryClaw's TODOS.md, or require human review? Recommendation: auto-append with "Added by: evaluate" tag so prioritize can score them normally.
3. **Baseline tracking:** After multiple dogfood runs, should we track evaluation scores over time? Probably yes, but defer to campaign runner (Approach B).

## Success Criteria

- [ ] Evaluate skill runs successfully after the dogfood pipeline on one external repo

- [ ] At least 1 GaryClaw improvement candidate is identified and added to TODOS.md
- [ ] The improvement candidate scores >5.0 in prioritize (proving the feedback loop works)
- [ ] "Evaluate Bootstrap Output Quality" TODO is unblocked and resolved

## What I noticed about how you think

- The priority pick correctly identified that the backlog is exhausted and reframed the problem from "what feature to build" to "how to generate new features to build." That's a system-level insight, not a feature-level one.
- The scoring rubric (weighted dimensions, threshold gate) prevented low-value busywork from masquerading as progress. "Oracle Decision Batching at 4.9/10 is below threshold" is a hard call that most teams would override.
- The recommendation to dogfood before building more features shows restraint. Most projects at this stage would keep adding features to avoid the discomfort of real-world testing.
- The blocked dependency tracking (Memory-Informed Adaptive Scheduling needs 50+ jobs) is honest about what you don't have yet instead of hand-waving it.
