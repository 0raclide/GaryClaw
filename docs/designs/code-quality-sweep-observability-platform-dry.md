## Problem Statement

Eight `console.warn` calls across ask-handler.ts (2), reflection.ts (4), and worktree.ts (2) are invisible when GaryClaw runs as a daemon. The daemon captures events via `callbacks.onEvent()`, but raw `console.warn` goes to stderr of the daemon process, which nobody reads. This means overnight runs that hit lock timeouts, corrupt JSONL, failed stash pops, or lost decision logs produce zero diagnostic output in the daemon log.

Additionally: `todo-state.ts:232` shells out to `head -5` (platform-dependent, unnecessary subprocess), `pipeline.ts:515` uses fragile regex path stripping instead of an explicit config field, `detectArtifacts` calls `resolveBaseBranchSafe()` twice (2 redundant git subprocesses), and `slugify()` has no Unicode/edge-case test coverage.

## What Makes This Cool

Nothing flashy. This is the kind of work that makes the difference between "the daemon ran overnight and we have no idea what happened" and "the daemon ran overnight and here's exactly what went wrong at 3:47 AM." Debuggability is the foundation of trust in autonomous systems.

## Constraints

- Zero new dependencies
- All changes are backward-compatible (new params are optional, default to current behavior)
- No interface changes visible to CLI users
- Tests must cover all new paths
- Each fix is independently mergeable (no ordering dependencies between the 8 implementation items)

## Approaches Considered

### Approach A: Direct Inline Fixes
Add `onWarn?: (msg: string) => void` parameter directly to each function that currently calls `console.warn`. Replace `head -5` inline. Cache `resolveBaseBranchSafe()` with a local variable.

- Effort: S
- Risk: Low
- Pros: Smallest diff, no new exports, no shared abstractions
- Cons: Pattern is duplicated (`const warn = onWarn ?? console.warn`) in 3+ modules, no grepable callsite marker

### Approach B: Shared Warn Utility + Config Field (RECOMMENDED)
Extract a 2-line `resolveWarnFn` helper for consistent warn routing. Add `rootCheckpointDir` to `GaryClawConfig`. Apply all 8 fixes using these shared patterns.

- Effort: S
- Risk: Low
- Pros: Grepable pattern, consistent across all modules, `rootCheckpointDir` eliminates regex guesswork
- Cons: One more export (2-line function), marginal indirection

## Recommended Approach

**Approach B.** The shared `resolveWarnFn` helper makes every warn callsite discoverable via grep, and `rootCheckpointDir` on the config type is the right long-term fix for the fragile regex. The delta over Approach A is ~10 lines of shared utility code.

## Implementation Plan

### Fix 1: Shared warn utility

**File:** `src/types.ts` (or inline in each module, but types.ts is zero-import and already holds shared interfaces)

```typescript
/** Resolve a warn function — routes to callback if provided, else console.warn. */
export type WarnFn = (msg: string) => void;
export function resolveWarnFn(onWarn?: WarnFn): WarnFn {
  return onWarn ?? console.warn;
}
```

This is the same pattern as `oracle.ts:601` (`const warn = onWarn ?? console.warn`) but named and importable.

### Fix 2: Route ask-handler console.warn through callbacks

**File:** `src/ask-handler.ts`

Add `onWarn` to `AskHandlerConfig` (already exists at line 41). Thread it into `writeDecisionLog` and `writeEscalatedLog`:

- `writeDecisionLog(path, decision)` → `writeDecisionLog(path, decision, warn)` where `warn = resolveWarnFn(config.onWarn)`
- `writeEscalatedLog(path, decision, oracleResult)` → `writeEscalatedLog(path, decision, oracleResult, warn)`

Both functions currently catch errors and call `console.warn`. Replace with the `warn` parameter.

**Wiring:** Already wired. `orchestrator.ts:275` already passes `onWarn: (msg) => callbacks.onEvent({ type: "assistant_text", text: msg })` into the ask-handler config. The ask-handler config already has the `onWarn` field at line 41. The missing piece is that `writeDecisionLog` and `writeEscalatedLog` don't use it.

**Event type rationale:** `assistant_text` is used deliberately (not `warn` or `diagnostic`) because there is no `warn` event type in the `OrchestratorEvent` discriminated union, and adding one would require wiring it through CLI formatEvent, daemon logger, and dashboard aggregation. The `assistant_text` event type is already handled everywhere and appears in daemon logs. These warnings are low-frequency (error paths only), so the semantic mismatch is acceptable. A future sweep could add a `warn` event type, but that's scope creep for this batch.

### Fix 4: Route worktree.ts console.warn through callbacks

**File:** `src/worktree.ts`

Two callsites:

1. `mergeWorktreeBranch` line 485: stash pop failure after merge
2. `listWorktrees` line 541: worktree list failure

Add `onWarn?: WarnFn` to `MergeOptions` (or as a direct parameter). For `listWorktrees`, add as optional second parameter: `listWorktrees(repoDir: string, onWarn?: WarnFn)`.

**Wiring — specific callsites:**

The `console.warn` fallback is acceptable for CLI and doctor contexts where stderr is visible to the user. Only daemon-mode callers (job-runner.ts and daemon.ts) need explicit `onWarn` wiring.

### Fix 7: Cache `resolveBaseBranchSafe()` in detectArtifacts

The two calls are in separate `try` blocks (line 262 inside section 2's `try`, line 275 inside section 3's `try`), so both `const baseBranch` declarations compile but each spawns a subprocess. The fix is a simple hoist to the top of `detectArtifacts`, before section 2:

  // 3. Commits on main ... uses baseBranch (was line 275)

No conditional restructuring needed. Both consumers already handle errors independently via their own `try/catch` blocks.

### Fix 8: Harden slugify with Unicode edge case tests

**File:** `test/todo-state.test.ts` (new test cases in existing file)

Add test cases for:
- Emoji titles: `"🚀 Launch Feature"` → `"launch-feature"`
- CJK characters: `"修复 Bug 在 Dashboard"` → `"bug-dashboard"` (CJK stripped by `[^a-z0-9]+` regex, Latin words preserved)
- All-special-characters: `"!!!"` → `""` (empty string — needs graceful handling)
- Very long Unicode: 200+ char title with mixed scripts → truncated at 80 chars
- Only emoji: `"🎉🎊🎈"` → `""` (empty — needs graceful handling)

**Required slugify fix:** If the title is all non-Latin characters, `slugify` returns `""`. This causes state file collision at `.garyclaw/todo-state/.json`. Add a guard: if slug is empty after processing, return a deterministic hash-based fallback:

```typescript
if (!slug) {
  // djb2 hash — deterministic, no crypto dependency, stable across platforms/versions
  let hash = 5381;
  for (let i = 0; i < title.length; i++) {
    hash = ((hash << 5) + hash + title.charCodeAt(i)) >>> 0;
  }
  return `item-${hash.toString(36)}`;
}
```

djb2 is chosen because: (a) deterministic across all platforms and Node versions (pure integer math), (b) no import needed, (c) produces short alphanumeric output via base-36 encoding, (d) collision risk is acceptable for the ~100 TODO item scale this operates at.
