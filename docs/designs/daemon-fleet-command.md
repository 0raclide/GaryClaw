## Problem Statement

Launching parallel daemon instances takes 30+ minutes of manual orchestration per session. Each instance requires: checking for stale state, clearing dead PIDs, resetting orphaned budget entries, creating the instance, configuring it, starting it, and verifying it's running. Every session starts with 15+ minutes of cleanup from the previous session's dead instances. Monitoring what 5 parallel workers are doing requires reading raw logs (100+ `daemon log` calls per session). Even checking a single instance's progress requires `daemon status` which shows "Running: yes" and a job ID but not *what skill is executing* or *which TODO it's working on*. Three features collapse this into one command with full visibility.

## What Makes This Cool

`daemon start --parallel 5` and walk away. Five workers spin up with isolated worktrees, auto-cleaned state, validated budget, and staggered starts. `daemon status --all` shows a compact table where each row tells you: what TODO is being worked on, which skill is running, how far along it is, and how many commits it's made. One command to launch. One command to see everything. Zero cleanup rituals.

## Constraints

- Must be backward-compatible with existing `daemon start --name <n>` workflow
- Named instances follow existing `worker-{N}` convention
- Budget validation must happen before any instance launches (fail-fast)
- Auto-cleanup must not kill running healthy instances
- Live status must work without IPC when instances are stopped (read from disk)
- All three sub-features are independently testable and deployable
- No new dependencies

## Premises

1. The three sub-features share surface area (cli.ts, daemon.ts, daemon-registry.ts) but each is independently implementable
2. `doctor --fix` already contains the cleanup logic. Auto-cleanup on boot is wiring, not new code.
3. Live status data (current skill, pipeline index, claimed TODO) already exists on disk in `pipeline.json` and `daemon-state.json`. It needs to be surfaced through IPC.
4. `--parallel N` creates instances named `worker-1` through `worker-N`. No new naming scheme.
5. Budget pre-validation is mandatory: N * perJobCostLimitUsd must fit within remaining dailyCostLimitUsd.
6. Staggered instance starts (1s delay) prevent N simultaneous git worktree creations from racing.

## Approaches Considered

### Approach A: CLI-Only Orchestration (Minimal)

**Summary:** `--parallel N` is a for-loop in cli.ts that forks N daemon processes sequentially. Auto-cleanup calls `runDoctor({fix: true})` inline before the first fork. Live status reads `pipeline.json` directly from each instance directory in `displayAllInstances()`. No IPC changes.

**Effort:** S
**Risk:** Low
**Pros:**
- Smallest diff (~150 lines across cli.ts + doctor.ts)
- No IPC protocol changes, no daemon.ts changes
- Live status works even when instances are stopped

**Cons:**
- File-based status reads are racey (partial writes to pipeline.json)
- No commit count without shelling out to `git log` from the CLI process
- Status data is stale by up to 5s (processInterval)

### Approach B: Full IPC Enrichment (Recommended)

**Summary:** `--parallel N` launches with budget pre-validation and staggered starts (1s delay between forks). Auto-cleanup logic extracted from doctor.ts as `runAutoCleanup()` callable function. IPC status response enriched with `pipelineProgress` field containing current skill name, skill index/total, claimed TODO title, elapsed time, and commit count from worktree. `displayAllInstances()` queries live instances via IPC, falls back to disk for stopped instances.

**Effort:** M
**Risk:** Low
**Pros:**
- IPC status is always fresh (no stale file reads)
- Commit count comes from the daemon process (already has worktree path)
- Budget pre-validation prevents wasted worktree creation
- Staggered starts prevent git worktree race conditions
- `runAutoCleanup()` is reusable for future daemon health automation

**Cons:**
- IPC protocol change (additive, backward-compatible)
- `displayAllInstances()` needs async IPC calls per running instance (but parallel)
- ~250 lines total across 4 files

### Approach C: Watch Mode with Live Refresh

**Summary:** Same as Approach B, plus a `daemon watch` command that refreshes status every 2s in the terminal (like `watch` or `htop`). Uses ANSI cursor positioning to update in-place.

**Effort:** L
**Risk:** Medium
**Pros:**
- True real-time monitoring without repeated `daemon status` calls
- Impressive demo experience

**Cons:**
- Terminal compatibility issues (Windows, non-TTY)
- Extra ~100 lines for the watch loop
- Diminishing returns over Approach B for daemon pipeline use

## Recommended Approach

**Approach B: Full IPC Enrichment.** The IPC enrichment is ~30 lines in daemon.ts and makes every `daemon status` call genuinely useful. Budget pre-validation prevents the failure mode where 3 of 5 instances launch before discovering budget is exhausted. Staggered starts are a 3-line addition that prevents a real race condition. Approach C's watch mode is cool but deferrable, and `daemon status --all` in a 5s loop achieves 90% of the value.

## Implementation

### Sub-feature 1: `daemon start --parallel N`

**Files:** `src/cli.ts`, `src/daemon-registry.ts`

**CLI parsing:**
- New flag `--parallel N` on `daemon start` subcommand (mutually exclusive with `--name`)
- `N` must be 2-10 (1 is just `daemon start`, >10 is resource abuse)
- Creates instances named `worker-1` through `worker-N`

**Budget pre-validation in cli.ts before any fork:**

Note: this is an optimistic single-job check ("can each new instance afford at least one job?"), not a full-session reservation. Instances may run multiple jobs per day.

```typescript
const globalBudget = readGlobalBudget(checkpointDir);
const remaining = config.budget.dailyCostLimitUsd - globalBudget.totalUsd;
// Subtract already-running instances from N (they already have budget allocated)
const alreadyRunning = countRunningWorkers(checkpointDir, N);  // check worker-1..N PIDs
const newInstances = N - alreadyRunning;
const needed = newInstances * config.budget.perJobCostLimitUsd;
if (needed > remaining) {
  error: `Budget insufficient for ${newInstances} new instances. Remaining: $${remaining}, needed: $${needed}`
  exit(1)
}
```

**Launch sequence:**
1. Run auto-cleanup (sub-feature 2). This clears dead budget entries first so step 2 sees accurate remaining budget.
2. Budget pre-validation (fail-fast before any fork)
3. For i = 1..N:
   a. Check if `worker-{i}` is already running (PID alive). If so, skip with warning, continue.
   b. Fork daemon process with `--instance worker-{i}`
   c. Wait 1s (stagger to prevent worktree race)
   d. Verify PID file written: poll `instanceDir(checkpointDir, "worker-{i}")/daemon.pid` every 100ms up to 3s. On timeout: print warning "worker-{i} failed to start, check log", continue to next instance (do not abort fleet).
   e. Print instance info line
4. Print summary table showing launched/skipped/failed counts

**Trigger auto-enqueue:** Each instance auto-enqueues the self-improvement pipeline on start (same as current behavior via triggers). The existing priority claiming mechanism ensures they pick different TODOs.

**Edge cases:**
- Instance `worker-3` already exists and is running: skip it, print warning, continue
- Instance `worker-3` exists but is dead: auto-cleanup already handled it in step 1, proceed
- All N instances already running: print status table, exit 0
- Some instances fail to start: launch continues for remaining instances, summary shows failures

### Sub-feature 2: Auto-Cleanup on Boot

**Files:** `src/doctor.ts`, `src/daemon.ts`

**Extract from doctor.ts:**
```typescript
export async function runAutoCleanup(options: {
  projectDir: string;
  skipAuth: boolean;
  dailyCostLimitUsd?: number;
  maxJobsPerDay?: number;
}): Promise<{ cleaned: string[] }>
```

This runs the existing doctor checks with `fix: true` and returns what was cleaned:
- Stale PID files (dead processes)
- Orphaned worktrees (no matching live instance). Full removal including worktree directory. Branch is preserved in git for manual merge investigation. On re-launch, `createWorktree()` reuses the existing branch if present (existing behavior in worktree.ts), or creates a new one if the branch was manually deleted.
- Stuck reflection locks (stale mkdir locks)
- Dead instance budget entries in global-budget.json
- Orphaned TODO state files (stale >2h, instance dead)

**Error handling:** Each cleanup category is independent and fail-open. If stale PID removal succeeds but worktree removal fails (e.g., git lock error), log a warning and continue. Partial cleanup never blocks daemon start. The `cleaned` array tracks what succeeded; failures are logged but not returned as errors.

**Wiring in daemon.ts `startDaemon()`:**
```typescript
// After config load, before PID write
const cleaned = await runAutoCleanup({
  projectDir: config.projectDir,
  skipAuth: true,  // auth checked separately
  dailyCostLimitUsd: config.budget.dailyCostLimitUsd,
  maxJobsPerDay: config.budget.maxJobsPerDay,
});
if (cleaned.length > 0) {
  log("info", `Auto-cleanup: ${cleaned.join(", ")}`);
}
```

**Wiring in cli.ts for `--parallel`:**
```typescript
// Must run before budget pre-validation — clears dead budget entries so remaining is accurate
await runAutoCleanup({ projectDir, skipAuth: true, ... });
```

**Guard:** Auto-cleanup only touches instances where `isPidAlive()` returns false. Running instances are never touched. The existing doctor checks already have this guard.

### Sub-feature 3: Live Pipeline Status

**Files:** `src/daemon.ts` (IPC handler), `src/cli.ts` (display), `src/types.ts` (IPCResponse enrichment)

**IPC status response enrichment:**

Add `pipelineProgress` to the status response data:

```typescript
interface PipelineProgress {
  currentSkill: string;       // e.g. "implement"
  skillIndex: number;         // 0-based
  totalSkills: number;        // total in pipeline
  claimedTodoTitle: string | null;  // from job.claimedTodoTitle
  elapsedSeconds: number;     // since job started
  commitCount: number;        // commits on worktree branch since creation
}
```

**In `buildIPCHandler()` status case:**

The `readPipelineState()` function is the existing export from `pipeline.ts`, which reads `pipeline.json` via `safeReadJSON()`. The current `buildIPCHandler()` signature is `(runner, startTime, projectDir?, parentCheckpointDir?)`. It needs a new `instDir` parameter added (the instance directory path, already computed in `startDaemon()` at line 385). This is a one-line signature change.

```typescript
// After finding runningJob (instDir is from the handler closure)
let pipelineProgress: PipelineProgress | null = null;
if (runningJob) {
  // Read pipeline.json from the job's checkpoint dir
  const jobDir = join(instDir, "jobs", runningJob.id);
  const pipelineState = readPipelineState(jobDir);  // from pipeline.ts
  if (pipelineState) {
    const currentSkill = pipelineState.skills[pipelineState.currentSkillIndex];
    pipelineProgress = {
      currentSkill: currentSkill?.skillName ?? "unknown",
      skillIndex: pipelineState.currentSkillIndex,
      totalSkills: pipelineState.skills.length,
      claimedTodoTitle: runningJob.claimedTodoTitle ?? null,
      elapsedSeconds: Math.floor((Date.now() - new Date(runningJob.startedAt).getTime()) / 1000),
      commitCount: cachedCommitCount,  // updated async, see below
    };
  }
}
```

**`getWorktreeCommitCount()` helper (async to avoid blocking IPC event loop):**
```typescript
import { execFile } from "node:child_process";
import { promisify } from "node:util";
const execFileAsync = promisify(execFile);

async function getWorktreeCommitCount(worktreePath?: string, projectDir?: string): Promise<number> {
  if (!worktreePath) return 0;
  try {
    // Always use worktreePath for resolveBaseBranch (it's the git repo context).
    // For default instance (no worktree), commitCount is always 0 since there's no branch to count.
    const base = resolveBaseBranch(worktreePath);
    const { stdout } = await execFileAsync("git", ["-C", worktreePath, "rev-list", "--count", `${base}..HEAD`],
      { encoding: "utf-8", timeout: 3000 });
    return parseInt(stdout.trim(), 10) || 0;
  } catch { return 0; }
}
```

**Commit count caching:** To avoid a git subprocess on every IPC status call, the daemon caches `commitCount` and refreshes it every 10s via `setInterval`. The cached value is used in the IPC response. Staleness of up to 10s is acceptable for a status display.

```typescript
// In startDaemon(), after creating the IPC handler:
let cachedCommitCount = 0;
const commitCountTimer = setInterval(async () => {
  cachedCommitCount = await getWorktreeCommitCount(config.worktreePath);
}, 10_000);
// Clear on shutdown alongside processTimer
```

**Display in `displayAllInstances()`:**

For running instances, query IPC in parallel:

```
NAME             STATUS    SKILL           TODO                        TIME    COMMITS
```

For single-instance `daemon status`:


**Fallback for stopped instances:** Read `daemon-state.json` from disk to show last job info. No pipeline progress (it's done).

**Fallback for IPC timeout:** Show "running (IPC timeout)" status, fall back to disk read.

## Resolved Design Decisions

1. **`--parallel` pipeline override:** No for MVP. The trigger config already defines the skill pipeline. Each instance inherits the same trigger config and the priority claiming mechanism differentiates them by TODO selection, not by pipeline shape.

2. **`daemon stop --all` merge behavior:** Yes, attempt merge for each named instance sequentially. Uses the existing `daemon stop --name X` logic in a loop (which already does ff-only merge and leaves diverged branches for manual merge). Sequential ordering is safe because each instance has its own branch, and the existing merge lock prevents concurrent merge attempts if stop signals arrive out of order.

3. **Auto-cleanup worktree depth:** Full cleanup (worktrees + PIDs + budget + locks). A dead instance whose branch was not merged has left stale state that will never self-resolve. The branch persists for manual merge investigation even after the worktree is removed (git branches are independent of worktrees). This matches the existing `daemon stop --cleanup` behavior.

## Success Criteria

1. `daemon start --parallel 5` launches 5 instances in under 30 seconds (vs 30+ minutes manual)
2. `daemon status --all` shows pipeline progress for all running instances in one table
3. Zero manual cleanup needed between sessions (auto-cleanup handles all stale state)
4. Budget pre-validation prevents partial fleet launches
5. All existing `daemon start --name X` workflows continue to work unchanged
