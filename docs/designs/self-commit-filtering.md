# Design: Self-Commit Filtering in Git Poller

Generated by /office-hours on 2026-03-28
Branch: garyclaw/cron-fix
Repo: 0raclide/GaryClaw
Status: APPROVED
Mode: Builder

## Problem Statement

The daemon's git poller fires on every HEAD change, including commits made by the daemon itself. This creates a self-chasing cascade: implement commits code, poller detects HEAD change, fires QA, QA commits fixes, poller detects again, fires more QA. On 2026-03-28, the daemon ran 8 QA passes (~$6) chasing its own tail when 2-3 would have sufficed. At 12 cron cycles/day, that's $30-60/day in pure waste.

The fix: tag daemon commits with `GIT_COMMITTER_EMAIL=garyclaw-daemon@local` (already verified in spike, 2026-03-27), then teach the git poller to check committer identity before triggering. If every commit in the `oldHead..newHead` range is from the daemon, skip the trigger.

## What Makes This Cool

The daemon stops feeding itself. One env var + one git log check eliminates the single biggest dollar waste in the system. It's the kind of fix where you look at the before/after cost graphs and think "why didn't we do this on day one."

## Constraints

- Must not break external commit detection (human pushes, other CI agents)
- Must be testable without real git repos (injectable deps pattern already established in `GitPollerDeps`)
- `GIT_COMMITTER_EMAIL` must propagate through the Agent SDK to Claude's bash tool (spike confirmed this works)
- Mixed commits (some daemon, some human) must still trigger the poller
- The `GARYCLAW_DAEMON_EMAIL` constant must be shared between `sdk-wrapper.ts` and `triggers.ts` to avoid drift

## Premises

1. **The git poller self-triggering is the primary cost waste.** Evidence: 8 QA passes on 2026-03-28 at ~$0.75 each, observed in daemon logs. The cascade pattern is well-documented.
2. **`GIT_COMMITTER_EMAIL` is a reliable, non-destructive marker.** The spike proved it propagates through the SDK. Committer email is metadata, not author identity. `git log --format=%ce` reads it accurately. No downstream tooling (GitHub, CI) breaks when committer != author.
3. **Filtering all-daemon commits is safe.** The pipeline already runs QA as its final skill. Regressions from daemon commits are caught in-pipeline. The poller only needs to fire for external commits.
4. **The debounce mechanism is orthogonal.** Self-commit filtering answers "who made this commit?" while debounce answers "has HEAD stabilized?" They compose naturally. Debounce still fires after the filter passes.

## Approaches Considered

### Approach A: Inline Filter (Minimal Viable)

**Summary:** Add `getCommitEmails()` to `triggers.ts` and a filter check inside `createGitPoller`'s `poll()` function. Add `GIT_COMMITTER_EMAIL` to `buildSdkEnv()`. Two files changed, one new function, one constant.

**Effort:** XS (human: ~30 min / CC: ~5 min)
**Risk:** Low
**Pros:**
- Smallest possible diff. Three surgical changes across two files.
- Follows existing patterns exactly (`GitPollerDeps` for injectability, `buildSdkEnv` for env management).
- Easy to review, easy to revert.

**Cons:**
- The email string `"garyclaw-daemon@local"` becomes a cross-module contract. If someone changes it in one place and not the other, filtering silently breaks.
- No config-level override. If a user wants a different email marker, they'd need to edit source.

**Reuses:** `GitPollerDeps` injection pattern, `buildSdkEnv()` env construction, `execFileSync` pattern from `getGitHead()`.

### Approach B: Configurable Filter with Shared Constant (Ideal Architecture)

**Summary:** Same core mechanism as Approach A, but: (1) extract `GARYCLAW_DAEMON_EMAIL` as a named constant exported from a shared location, (2) add `selfCommitEmail` optional field to `GitPollTrigger` type for config-level override, (3) log filtered triggers at debug level so users can verify filtering is working.

**Effort:** S (human: ~1 hr / CC: ~10 min)
**Risk:** Low
**Pros:**
- Shared constant eliminates drift risk between sdk-wrapper and triggers.
- Debug logging makes the invisible visible. When someone wonders "why didn't the poller fire?", the log says "Skipped: all 3 commits from garyclaw-daemon@local".
- Config override future-proofs for multi-daemon setups where each instance might use a different email.

**Cons:**
- Slightly larger diff. The config override field requires a type change in `types.ts` and validation in `daemon.ts`.
- The config override may be YAGNI. Currently all instances use the same email.

**Reuses:** Same as Approach A, plus `DaemonConfig` validation pattern, debug logging pattern from `createDaemonLogger`.

### Approach C: Commit Metadata Tag (Creative/Lateral)

**Summary:** Instead of email-based filtering, add a custom git trailer (`GaryClaw-Commit: true`) to the commit message via a git hook or prompt instruction. The poller checks for the trailer instead of the committer email.

**Effort:** M (human: ~2 hr / CC: ~20 min)
**Risk:** Medium
**Pros:**
- Doesn't touch committer identity at all. Cleaner separation of concerns.
- Trailers are visible in `git log`, making it obvious which commits are daemon-generated.

**Cons:**
- Requires the daemon to control the commit message format, which means prompt engineering. Claude decides what to commit. If it doesn't include the trailer, filtering fails silently.
- `git log --format=%ce` is one syscall. Parsing trailers from commit messages is heavier and more fragile.
- The `GIT_COMMITTER_EMAIL` spike is already done and verified. This approach throws away that work.

**Reuses:** None of the existing env passthrough infrastructure.

## Recommended Approach

**Approach B: Configurable Filter with Shared Constant.**

Approach A is tempting (smallest diff), but the shared constant and debug logging are worth the extra 5 minutes of CC time. The constant prevents a class of bugs (email string drift) that would be invisible until someone wastes hours debugging why filtering stopped working. The debug logging turns "why didn't my poller fire?" from a mystery into a grep.

Approach C is clever but fragile. Relying on Claude to include a specific trailer in every commit message is a prompt engineering problem with no good failure mode. The email approach is mechanical and guaranteed.

## Implementation Plan

### Step 1: Shared Constant

Add to `src/sdk-wrapper.ts` (it's the env construction module, natural home):

```typescript
/** Email marker for daemon-generated commits. Used by buildSdkEnv() to tag commits
 *  and by the git poller to filter self-triggered events. */
export const GARYCLAW_DAEMON_EMAIL = "garyclaw-daemon@local";
```

### Step 2: Tag Daemon Commits

In `src/sdk-wrapper.ts` `buildSdkEnv()`:

```typescript
export function buildSdkEnv(
  processEnv: Record<string, string | undefined>,
): Record<string, string> {
  const { ANTHROPIC_API_KEY: _, ...rest } = processEnv;
  const env: Record<string, string> = {};
  for (const [key, value] of Object.entries(rest)) {
    if (value !== undefined) {
      env[key] = value;
    }
  }
  // Tag daemon commits so the git poller can filter self-triggers
  env.GIT_COMMITTER_EMAIL = GARYCLAW_DAEMON_EMAIL;
  env.GIT_COMMITTER_NAME = "GaryClaw Daemon";
  return env;
}
```

Note: this sets `GIT_COMMITTER_EMAIL` on ALL SDK sessions, not just daemon ones. This is intentional. CLI runs (`garyclaw run qa`) also benefit from being identifiable. If this becomes a problem, gate it behind an `options.isDaemon` flag later.

### Step 3: Add `getCommitEmails()` to triggers.ts

Import the shared constant at the top of `triggers.ts`: `import { GARYCLAW_DAEMON_EMAIL } from "./sdk-wrapper.js";`

```typescript
/**
 * Get committer emails for commits in a range.
 * Returns empty array on error or if range is invalid.
 */
export function getCommitEmails(
  projectDir: string,
  oldHead: string,
  newHead: string,
): string[] {
  try {
    const output = execFileSync(
      "git",
      ["log", "--format=%ce", `${oldHead}..${newHead}`],
      { cwd: projectDir, encoding: "utf-8", timeout: 5000, stdio: ["ignore", "pipe", "ignore"] },
    ).trim();
    if (!output) return [];
    return output.split("\n").filter(Boolean).slice(0, 100); // Cap to prevent memory spike on pathological ranges
  } catch {
    return []; // On error, don't filter (safe default: trigger fires)
  }
}
```

Key design decision: on error, return empty array. An empty array means "we don't know who committed" which means the filter passes and the trigger fires. This is the safe default. Silent suppression of triggers is worse than an extra QA run.

### Step 4: Filter in `createGitPoller`

In the `poll()` function, after detecting HEAD change and before starting the debounce timer:

```typescript
// Check if all commits are self-made (daemon's own commits)
const selfEmail = config.selfCommitEmail ?? GARYCLAW_DAEMON_EMAIL;
const emails = d.getCommitEmails(projectDir, lastHead, head);
if (emails.length > 0 && emails.every(e => e === selfEmail)) {
  d.log("debug", `Git poll: skipping self-commits (${emails.length} commits from ${selfEmail})`);
  lastHead = head; // Update HEAD so we don't re-check these commits
  return;
}
```

Critical: `lastHead = head` must happen even when skipping. Otherwise the next poll re-checks the same range and either re-skips (wasting a git log call per interval) or, worse, accumulates a range that eventually includes external commits mixed with old daemon commits.

### Step 5: Update Types

In `src/types.ts`, add optional field to `GitPollTrigger`:

```typescript
export interface GitPollTrigger {
  type: "git_poll";
  intervalSeconds: number;
  skills: string[];
  branch?: string;
  debounceSeconds?: number;
  selfCommitEmail?: string;  // Override daemon email for filtering (default: GARYCLAW_DAEMON_EMAIL)
}
```

### Step 6: Update `GitPollerDeps` for Testability

```typescript
export interface GitPollerDeps {

  // ... existing timer deps

```

Wire the default in `defaultDeps` (same location as the existing `getHead` default in `createGitPoller`). Tests inject a stub that returns controlled email arrays.

### Step 7: Tests

New test file: `test/triggers-self-commit.test.ts` (dedicated file per the project's convention of one concern per test file)

| Test case | What |
|-----------|------|
| All daemon commits → skip | `getCommitEmails` returns `["garyclaw-daemon@local", "garyclaw-daemon@local"]` → trigger not called |
| Mixed commits → fire | `["garyclaw-daemon@local", "human@example.com"]` → trigger called |
| All external commits → fire | `["human@example.com"]` → trigger called |
| Empty emails (error) → fire | `[]` → trigger called (safe default) |
| Custom selfCommitEmail config | Config override respected |
| lastHead updated on skip | After skipping, next poll doesn't re-check same range |
| GARYCLAW_DAEMON_EMAIL in buildSdkEnv | `buildSdkEnv()` output contains `GIT_COMMITTER_EMAIL` key |

## Open Questions

1. **Should CLI runs also get the daemon email?** Step 2 sets it unconditionally. This means `garyclaw run qa` on a developer's machine also tags commits with `garyclaw-daemon@local`. Pro: consistency, all GaryClaw commits are identifiable. Con: a human running `garyclaw run qa` might want their own committer identity preserved. Recommendation: set it unconditionally for now. If someone complains, gate it behind an `options.isDaemon` flag at that point.

## Success Criteria

1. A daemon pipeline run (prioritize -> implement -> qa) does NOT trigger the git poller for its own commits.
2. A human `git push` to the same branch DOES trigger the git poller.
3. Mixed scenarios (daemon + human commits in same range) trigger the poller.
4. `git log --format="%ce"` on daemon commits shows `garyclaw-daemon@local`.
5. All existing trigger tests pass unchanged.
6. New tests cover all 7 cases in the test plan above.
7. Cost per overnight cycle drops from ~$6 to ~$3 (one pipeline run without the QA echo).
