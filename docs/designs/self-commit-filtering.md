# Design: Self-Commit Filtering in Git Poller

Generated by /office-hours on 2026-03-28
Branch: garyclaw/batching

Status: APPROVED
Mode: Builder

## Problem Statement

The daemon's git poller triggers on every HEAD change, including commits the daemon made itself. This creates a self-chasing cascade: implement commits code, auto-merge advances HEAD on main, the poller detects the change, fires QA, QA commits fixes, the poller detects again, fires more QA. On 2026-03-28, the daemon ran 8 QA passes (~$6) chasing its own commits when 2-3 would have sufficed. At 12 cron cycles/day, that's $30-60/day burned on redundant self-triggered jobs.

The pipeline already runs QA as its final skill, so daemon-triggered QA from the poller is pure waste. External commits (human pushes) should still trigger normally.

## What Makes This Cool

Zero-dollar savings with zero risk. The poller becomes self-aware, knowing which commits are "mine" vs "theirs." The daemon stops chasing its own tail and only wakes up when a human pushes code. The spike for `GIT_COMMITTER_EMAIL` propagation through the Agent SDK is already verified (2026-03-27), so this is just wiring.

## Constraints

- XS effort budget. The fix touches 3 files max.
- Must be fully testable with synthetic data (no SDK calls in tests).
- Must not break the git poller for external commits.
- Must not affect cron-triggered jobs (only git_poll triggers).
- Named instances with worktrees auto-merge to the base branch. The poller watches the base branch. Both paths must be covered.
- The `GIT_COMMITTER_EMAIL` must propagate through the SDK env to the Claude Code subprocess, which performs the actual git operations.

## Premises

1. **Every daemon commit flows through `buildSdkEnv()`** — all git operations happen inside the SDK subprocess, so setting `GIT_COMMITTER_EMAIL` in the env object guarantees every daemon commit is tagged. Verified by spike.
2. **The poller has old/new HEAD at trigger time** — `poll()` in `triggers.ts` already tracks `lastHead` and `pendingHead`. Adding a committer check between those two SHAs is a natural extension.
3. **Daemon-only commit ranges should be silently skipped** — if every commit between old HEAD and new HEAD was authored by `garyclaw-daemon@local`, the trigger adds no value. QA already ran in-pipeline.
4. **Mixed ranges (some daemon, some human commits) should still trigger** — if a human pushed between two daemon auto-merges, the poller must fire. The filter checks ALL commits in the range, not just the tip.
5. **The auto-merge in job-runner.ts is the primary self-commit source** — `mergeWorktreeBranch()` calls `git merge --ff-only`, which advances HEAD. The commit was originally made in the worktree by the SDK subprocess with the daemon's env.

## Approaches Considered

### Approach A: Committer Email Filter (recommended)

**Summary:** Set `GIT_COMMITTER_EMAIL=garyclaw-daemon@local` in `buildSdkEnv()`. In the git poller, after detecting a HEAD change, run `git log --format=%ce {oldHead}..{newHead}` and skip the trigger if every committer email is `garyclaw-daemon@local`.

**Effort:** XS (human: ~30 min / CC: ~5 min)
**Risk:** Low

**Pros:**
- Stateless from the poller's perspective. No shared state between job-runner and poller.
- Independently testable. `getCommitAuthors()` is a pure function on git data.
- The spike already verified that `GIT_COMMITTER_EMAIL` propagates through the SDK correctly.
- Works for both default instance (direct commits) and named instances (worktree auto-merge).

**Cons:**
- Daemon commits become invisible to the git poller. If a daemon commit introduces a regression that the in-pipeline QA missed, the poller won't re-trigger QA.
- Mitigation: the pipeline always ends with QA, so regressions are caught before merge. And cron triggers still fire independently.

**Reuses:** `buildSdkEnv()` (env construction), `createGitPoller()` (existing deps injection pattern), `getGitHead()` (existing git command pattern).

### Approach B: SHA Tracking via Shared State

**Summary:** After auto-merge in `job-runner.ts`, read the new HEAD SHA and store it in a `Set<string>` shared with the git poller. The poller skips triggers where `newHead` is in the set.

**Effort:** S (human: ~1 day / CC: ~15 min)
**Risk:** Medium

**Pros:**
- No environment variable dependency. Works even if something else makes commits.
- Exact match on SHA, no false positives possible.

**Cons:**
- Creates coupling between job-runner and triggers (shared mutable state).
- The SHA set needs lifecycle management (clear on daemon restart, handle rebase SHA changes).
- Doesn't handle the case where the SDK subprocess makes multiple commits in a single run, since we'd only track the final merge commit SHA.
- More complex to test — requires coordinating state between two subsystems.

### Approach C: Time-Based Suppress Window

**Summary:** After auto-merge, suppress all git poll triggers for N seconds (e.g., 60s). Any HEAD change during the window is ignored.

**Effort:** XS (human: ~15 min / CC: ~3 min)
**Risk:** High

**Pros:**
- Dead simple. One boolean flag and a timer.

**Cons:**
- Could miss real external commits during the suppress window. If a human pushes code 30 seconds after a daemon merge, that push gets silently dropped.
- The suppress duration is a guess. Too short and self-commits still trigger. Too long and external commits are missed.
- Brittle to timing changes (e.g., if auto-merge takes longer than expected).

## Recommended Approach

**Approach A: Committer Email Filter.** It's the design the codebase already prepared for (spike verified), it's stateless, it handles mixed commit ranges correctly, and it's the smallest diff.

## Implementation Plan

### Step 1: Tag daemon commits (sdk-wrapper.ts)

In `buildSdkEnv()`, inject `GIT_COMMITTER_EMAIL`:

```typescript
export function buildSdkEnv(
  processEnv: Record<string, string | undefined>,
): Record<string, string> {
  const { ANTHROPIC_API_KEY: _, ...rest } = processEnv;
  const env: Record<string, string> = {};
  for (const [key, value] of Object.entries(rest)) {
    if (value !== undefined) {
      env[key] = value;
    }
  }
  // Tag all daemon commits so the git poller can filter self-commits
  env.GIT_COMMITTER_EMAIL = "garyclaw-daemon@local";
  env.GIT_COMMITTER_NAME = "GaryClaw Daemon";
  return env;
}
```

Both `EMAIL` (for filtering) and `NAME` (for human-readable `git log` output) are set.

### Step 2: Add committer check function (triggers.ts)

Export a new function that reads committer emails for a commit range:

```typescript
export const DAEMON_COMMITTER_EMAIL = "garyclaw-daemon@local";

export function getCommitAuthors(
  projectDir: string,
  oldHead: string,
  newHead: string,
): string[] {
  try {
    const output = execFileSync(
      "git",
      ["log", "--format=%ce", `${oldHead}..${newHead}`],
      {
        cwd: projectDir,
        encoding: "utf-8",
        timeout: 5000,
        stdio: ["ignore", "pipe", "ignore"],
      },
    ).trim();
    if (!output) return [];
    return output.split("\n").filter(Boolean);
  } catch {
    return []; // On error, don't filter (let the trigger fire)
  }
}

export function isAllDaemonCommits(authors: string[]): boolean {
  if (authors.length === 0) return false; // No commits = don't filter
  return authors.every((email) => email === DAEMON_COMMITTER_EMAIL);
}
```

### Step 3: Wire filter into git poller (triggers.ts)

Add `getCommitAuthors` to `GitPollerDeps` and check it in `poll()`:

```typescript
export interface GitPollerDeps {
  getHead: (projectDir: string, branch?: string) => string | null;
  getCommitAuthors: (projectDir: string, oldHead: string, newHead: string) => string[];
  log: LogFn;
  // ... timers unchanged
}
```

In the debounce callback, before calling `onTrigger`:

```typescript
debounceTimer = d.setTimeout(() => {
  debounceTimer = null;
  if (pendingHead !== null && pendingHead !== lastHead) {
    // Check if all commits in the range are from the daemon itself
    const authors = d.getCommitAuthors(projectDir, lastHead!, pendingHead);
    if (isAllDaemonCommits(authors)) {
      d.log("info", `Git poll: skipping self-commit range ${lastHead?.slice(0, 7)}..${pendingHead.slice(0, 7)} (${authors.length} daemon commit(s))`);
      lastHead = pendingHead;
      pendingHead = null;
      return;
    }

    const detail = `HEAD changed: ${lastHead?.slice(0, 7)} -> ${pendingHead.slice(0, 7)}`;
    lastHead = pendingHead;
    pendingHead = null;
    onTrigger(config.skills, detail);
  }
}, debounceMs);
```

## Open Questions

1. **Should `GIT_COMMITTER_EMAIL` be overridable?** If a user sets their own `GIT_COMMITTER_EMAIL` in their environment, should `buildSdkEnv()` respect it or always override? Recommendation: always override. The daemon must tag its commits for filtering to work, and there's no legitimate reason for a user to set a different committer email in daemon mode.

2. **Should `GIT_COMMITTER_NAME` also be set?** Setting `GIT_COMMITTER_NAME=GaryClaw Daemon` makes daemon commits identifiable in `git log` output. Low cost, nice-to-have. Recommendation: yes, set both.

3. **Should the filter be configurable?** A `selfCommitFilter: boolean` flag in `daemon.json` could let users disable it. Recommendation: no. This is a correctness fix, not a preference. Always filter.
