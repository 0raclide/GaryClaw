## Problem Statement

GaryClaw is a learning development daemon that runs autonomously — prioritizing, designing, implementing, reviewing, and fixing code overnight. The entire interaction model is CLI + scattered markdown/JSON files. Checking status requires 3-5 terminal commands across multiple data sources. The developer ends up asking Claude "what's the status?" because synthesizing the information manually is painful. There's no visual feedback — no way to feel the daemon working without actively polling it.

## Demand Evidence

The developer ran the daemon for 30+ hours, made 160+ commits, spent $40 in daemon costs. They asked "what's the status?" at least 10 times during our session. Each time required running daemon status, daemon log, git log, npm test, and reading pipeline reports. They said: "It's painful." They use Claude as a human dashboard — that's the strongest demand signal: they built a workaround for the missing UI.

## Status Quo

Current workflow to check daemon status:
1. `npx tsx src/cli.ts daemon status` — job info, cost, oracle health
2. `npx tsx src/cli.ts daemon log --tail 50` — what's happening now
3. `git log --oneline -20` — what commits landed
4. `npm test | tail -5` — are tests passing
5. Read `.garyclaw/priority.md` — what's next
6. Read pipeline reports in `.garyclaw/daemons/*/jobs/*/pipeline-report.md`

That's 6 commands, 4 different data formats, and mental synthesis to answer "is it working?"

## Target User & Narrowest Wedge

**Target user:** The developer who built GaryClaw — and soon, any developer running GaryClaw as a background daemon on their projects.

**Narrowest wedge rejected.** The developer explicitly said "The full native app — don't narrow it." The reasoning: with CC+gstack, the effort equation is different. The full vision ships in hours, not months.

## Constraints

- macOS only (the daemon uses osascript for notifications, the user is on macOS)
- No modifications to GaryClaw's daemon process — the dashboard reads existing data files passively
- Separate repo from GaryClaw (TypeScript) — the dashboard is a Tauri (Rust + web) project
- Must include an embedded terminal for escape-hatch CLI access
- The daemon can't build or improve this (different language/toolchain) — at least not initially

## Premises

1. **Tauri, not native Swift.** Web tech (HTML/CSS/JS/Svelte) in a Tauri shell. Same-ish ecosystem affinity. Ships in days. Tradeoff: not 100% native feel.
2. **Menu bar app with popover window.** Click icon, get dropdown panel. Cmd+D for larger window. Matches quick-glance usage pattern.
3. **Data layer already exists.** All data in JSON/markdown files under `.garyclaw/`. The app watches these files and reacts. No API, no server.
4. **Separate repo.** Dashboard is its own Tauri project. Reads GaryClaw's directory. Distribution: brew or .dmg from GitHub Releases.
5. **Embedded terminal is critical.** xterm.js in webview. Opens to project dir. "Trigger Skill" buttons that type commands for you.

### Architecture

```
garyclaw-dashboard/              (new Tauri + Svelte repo)
  src-tauri/                     (Rust backend)
    src/
      main.rs                    Tauri entry, system tray, window management
      watcher.rs                 notify crate — watch .garyclaw/ directory
      parser.rs                  Read/parse daemon-state.json, metrics.json, etc.
  src/                           (Svelte frontend)
    stores/
      daemon.ts                  Reactive store: DaemonState from file watcher
      budget.ts                  Reactive store: GlobalBudget
      metrics.ts                 Reactive store: OracleMetrics
      pipeline.ts                Reactive store: current pipeline from pipeline.json
      priority.ts                Reactive store: priority.md parsed
      skills.ts                  Reactive store: available gstack skills
    components/
      MenuBarPopover.svelte      Main popover container with tabs
      PipelineView.svelte        Pipeline chain visualization (nodes + arrows)
      JobList.svelte             Job queue list with status badges
      BudgetView.svelte          Budget bars + cost charts
      SkillInventory.svelte      RPG 3x4 grid with tiers + equip
      PriorityQueue.svelte       Ranked backlog items
      Terminal.svelte            xterm.js embedded terminal
      AmbientStrip.svelte        Transparent edge window — HUD
      CommitFeed.svelte          Live commit stream
    App.svelte                   Root component, tab router
```

### Components

**1. Ambient Strip (always visible)**
- 60px transparent borderless window, right screen edge, always-on-top
- Interaction model: `ignoresMouseEvents = true` (click-through) for the body, but a 16px "grab zone" at the top that accepts clicks to expand the full popover. Requires Tauri plugin or native Swift bridge for `NSPanel` with mixed hit-testing — **spike this first** before committing to the strip.
- Green pulse = running, amber flash = error, dim = idle
- Issues fixed count, pipeline progress bar, cost
- Clicking the grab zone opens the full popover

**2. Menu Bar Popover (click to open)**
- System tray icon with status color
- 420x600 popover with 5 tabs:
  - Pipeline: skill chain visualization, "Now Building" card, recent commits
  - Jobs: job queue list (running/queued/completed/failed), cost per job
  - Budget: daily spend, remaining, per-instance breakdown, 7-day trend
  - Skills: RPG inventory grid, drag-to-equip pipeline composition
  - Terminal: xterm.js, pre-cd to project dir, "Trigger" buttons

**3. RPG Skill Inventory**
- 3x4 grid of gstack skills
- Tiers: Legendary (purple border) / Rare (blue) / Common (gray)
- Equipped state = blue glow + listed in pipeline bar
- Drag skills to the "EQUIPPED" bar to compose pipelines
- Click skill for description + recent usage stats

**4. Priority Queue**
- Reads `.garyclaw/priority.md`
- Ranked list: title, score, effort tag
- Top item highlighted
- Grayed items = blocked dependencies

**5. Repo Selector (Phase 2)**
- Simple path config field in Settings (not a scanning UI)
- Default: auto-detect from first `.garyclaw/` found in common locations (`~/Desktop`, `~/Developer`, `~/Projects`)
- Switching repos updates all stores
- Full multi-project UI with auto-scanning deferred to Phase 2

### Data Flow

**Existing asset:** GaryClaw already has `src/dashboard.ts` (335 lines) with a `DashboardData` interface that aggregates jobs, oracle metrics, budget, and health score. It writes `dogfood-report.md` after every job. The Tauri app should consume this, not rebuild the aggregation.

**Strategy:** Have GaryClaw also write `.garyclaw/dashboard-data.json` (structured JSON) alongside the markdown report. The Tauri app reads this single file for the summary view, plus individual files for detail views.

**Actual file paths:**
```
.garyclaw/
  dashboard-data.json            ← NEW: structured DashboardData for Tauri
  dogfood-report.md              ← Existing: human-readable markdown
  global-budget.json             ← Shared budget across instances
  priority.md                    ← Current priority pick
  oracle-memory/
    metrics.json                 ← Oracle accuracy, confidence, circuit breaker
    decision-outcomes.md         ← Recent outcomes
  daemons/
    {instance}/
      daemon-state.json          ← Per-instance job queue and status
      daemon.log                 ← Per-instance log
      daemon.sock                ← IPC socket (for trigger buttons)
      jobs/{jobId}/
        pipeline.json            ← Per-job pipeline state
        pipeline-report.md       ← Per-job results
```

File watcher (Rust `notify` crate with FSEvents on macOS) → parse changed file → emit Tauri event → Svelte store updates → UI re-renders. Event-driven, not polling — `notify` uses native OS filesystem events.

### Error & Empty States

| State | UI Behavior |
|-------|-------------|
| No `.garyclaw/` directory | Show "No GaryClaw project found. Select a directory." with repo selector. |
| No daemon running | Show dashboard data from last run (files persist). "Daemon offline" badge. Start button. |
| Stale PID file | Show "Daemon may have crashed. Run `garyclaw doctor --fix`." button. |
| Corrupt/partial JSON | Use `safeReadJSON` pattern — if parse fails, show last known good state with "stale" indicator. |
| No jobs yet | Empty state: "No jobs have run yet. Trigger one from the terminal tab." |
| No priority.md | Priority tab shows "Run `garyclaw run prioritize` to pick the next item." |

### Skill Discovery

Read `~/.claude/skills/gstack/*/SKILL.md` to populate the inventory. Parse YAML frontmatter for name, description, allowed-tools. Map to tiers based on heuristic:
- Legendary: skills that modify code (qa, implement, design-review)
- Rare: skills that produce analysis (eng-review, ceo-review, office-hours, cso)
- Common: utility skills (ship, research, investigate, browse)

## Open Questions

1. **Keyboard shortcut registration** — Cmd+D to open dashboard. Tauri supports global shortcuts but they can conflict with other apps. May need a Settings panel for custom shortcuts.
2. **Multiple daemon instances** — the popover needs to show all instances (default + named). Tab per instance? Or unified view with instance labels?
3. **Skill equip persistence** — when you drag skills to compose a pipeline in the inventory, where is that saved? A local `.garyclaw/pipelines.json`? Or just fires the daemon trigger command?
4. **Dark mode only?** — the wireframe is dark. Should we support light mode for macOS auto-switching?

## Success Criteria

1. Open the app, see the daemon's current status in < 1 second (no commands needed)
2. The ambient strip shows running/idle/error state in peripheral vision
3. Pipeline visualization shows which skill is active with real-time progress
4. Embedded terminal works for `garyclaw daemon trigger` without leaving the app
5. RPG inventory shows all installed gstack skills with equip-to-pipeline
6. Priority queue shows what's next without reading markdown files
7. Works with multiple `.garyclaw/` directories (repo selector)

## Technical Spikes (before full implementation)

1. **Ambient strip hit-testing** — Can Tauri create a transparent, always-on-top window with mixed click-through and clickable zones? May require a native Swift bridge or Tauri plugin. Spike: create a minimal Tauri window with `transparent: true`, `always_on_top: true`, `decorations: false` and test mouse event behavior on macOS.

2. **PTY in Tauri** — xterm.js needs a PTY backend. Since Tauri doesn't bundle Node.js, use `portable-pty` (Rust crate). Spike: spawn a shell from `src-tauri/`, pipe stdin/stdout via Tauri commands/events to xterm.js in the webview.

3. **System tray popover positioning** — Tauri v2 system tray supports icons and menus but not rich popovers. Need `WindowBuilder` with `decorations(false)` + `position()` relative to tray icon bounds. Spike: get tray icon screen coordinates and position a borderless window below it.

## Trigger Buttons

Trigger buttons in the Terminal and Skills tabs send IPC requests via the Unix domain socket at `.garyclaw/daemons/{instance}/daemon.sock`, using the existing `IPCRequest` protocol (`{ type: "trigger", skills: [...], designDoc?: string }`). This is direct IPC, not shelling out. The Rust backend connects to the socket, sends the JSON request, and returns the response to the Svelte UI.

For the RPG inventory "equip and run" flow: dragging skills to the EQUIPPED bar composes a pipeline. Clicking "Run Pipeline" sends a trigger IPC request with the composed skill list. The composition is ephemeral (not saved) unless explicitly named via a "Save Pipeline" button that writes to `.garyclaw/pipelines.json`.

## Phasing

**Phase 1 (ship first):** Menu bar popover with Pipeline + Jobs + Budget + Terminal tabs. File watcher. System tray icon with status color. No ambient strip, no RPG inventory. This solves the core problem: "stop asking Claude what's the status."

**Phase 2 (add delight):** Ambient edge strip (after spike #1 proves feasibility). RPG skill inventory with tiers and equip-to-pipeline. Priority queue view. Saved pipeline templates.

**Phase 3 (expand):** Multi-repo selector with auto-scanning. Notification center integration. Confidence calibration charts. Historical trend views.
