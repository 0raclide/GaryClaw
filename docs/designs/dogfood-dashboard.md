## Problem Statement

When GaryClaw runs overnight jobs autonomously, understanding daemon health requires reading raw logs and cross-referencing multiple files: `daemon-state.json` for job status, `metrics.json` for oracle accuracy, `global-budget.json` for cost, per-job reports for issues/relays. There's no single-glance view of "is GaryClaw healthy?" After 7+ jobs run overnight, you want to open one file and know: how many jobs ran, what succeeded/failed, what the oracle decided, how much it cost, and whether anything needs attention.

## What Makes This Cool

One file tells you everything. Instead of `cat daemon-state.json | jq '.jobs | length'` and `cat metrics.json` and reading 7 job summaries, you open `.garyclaw/dogfood-report.md` and see a health score, job table, oracle stats, cost breakdown by instance, and a "top concerns" section that surfaces the single most actionable insight. The dashboard regenerates after every job — it's always current.

## Constraints

- Must wire into job-runner completion path (eng review directive — not a separate polling mechanism)
- Must also be readable via `garyclaw dashboard` CLI command (read-only, just prints the file)
- Must not slow down job completion (dashboard generation must be < 100ms)
- Must work for both single-instance and multi-instance (parallel daemon) setups
- All data sources already exist — no new data collection needed
- Output format: markdown (consistent with existing reports, readable in terminal/IDE)
- Must handle missing/corrupt data gracefully (first run, partial state)

## Premises

1. The job-runner.ts completion path is the right hook point — all job completions (success and failure) converge there, and `writeSummary()` is already called at that point.
2. All data needed exists in three files: `daemon-state.json` (jobs), `metrics.json` (oracle), `global-budget.json` (cost) — no per-job report parsing needed for the MVP dashboard.
3. A single computed health score (0-100) makes the dashboard scannable without interpreting 8 metrics individually.
4. Markdown is the right format — consistent with `report.md`, `summary.md`, `priority.md`, and readable everywhere.
5. Historical trend tracking (sparklines, week-over-week) is out of scope for MVP but the data model should not preclude adding `dashboard-history.jsonl` later.

## Approaches Considered

### Approach A: Minimal — Single Function in Job Runner

**Summary:** Add a `writeDashboard()` function directly in `job-runner.ts` that reads state + metrics + global budget and writes `dogfood-report.md`. Called after `writeSummary()`.

**Effort:** XS
**Risk:** Low
**Pros:**
- Fewest files changed (1 source module + CLI wire-up)
- No new module to test independently
- Ships in ~15 min

**Cons:**
- Bloats job-runner.ts (already 383 lines)
- Aggregation logic untestable without mocking entire job runner
- Health score formula buried in a file about job queuing

### Approach B: Dedicated Module with Pure Functions

**Summary:** New `src/dashboard.ts` module with pure aggregation functions (`aggregateJobStats()`, `aggregateOracleStats()`, `computeHealthScore()`, `formatDashboard()`). Called from job-runner completion. CLI `garyclaw dashboard` reads and prints the file. All functions take data as input — no file I/O in the core logic.

**Effort:** S
**Risk:** Low
**Pros:**
- Pure functions = trivially testable with synthetic data (same pattern as every other GaryClaw module)
- Clean separation: dashboard.ts knows about aggregation, job-runner.ts knows about jobs
- Health score formula isolated and tunable
- Functions reusable for future JSON API endpoint or TUI

**Cons:**
- One more module (27th source file)
- Slightly more wiring (import in job-runner + CLI)

**Reuses:** `DaemonState`/`Job`/`OracleMetrics`/`GlobalBudget` types from `types.ts`, `safeReadJSON` from `safe-json.ts`, `readMetrics` from `oracle-memory.ts`

### Approach C: Dashboard with History Tracking

**Summary:** Same as B, plus append a JSON snapshot to `dashboard-history.jsonl` after each generation. Enables trend display (last 7 days sparkline, week-over-week comparison).

**Effort:** S-M
**Risk:** Medium — history file growth, formatting complexity for ASCII sparklines
**Pros:**
- Trend visibility ("oracle accuracy improving", "cost per job decreasing")
- Enables future `garyclaw dashboard --trend` command

**Cons:**
- Scope creep for an S-effort item
- Sparkline rendering adds complexity with limited terminal compatibility
- Can be added later by layering on Approach B (append one line to JSONL is a 5-line diff)

## Recommended Approach

**Approach B: Dedicated Module with Pure Functions.** It follows the established GaryClaw pattern (pure functions, dependency injection, synthetic test data), keeps job-runner.ts focused, and ships in ~20 minutes. History tracking (Approach C) layers on trivially later.

## Implementation Order

### 1. Types (types.ts)

Add `DashboardData` interface:

```typescript
export interface DashboardData {
  generatedAt: string;
  healthScore: number;           // 0-100
  topConcern: string | null;     // Most actionable insight, null if healthy
  jobs: {
    total: number;
    complete: number;
    failed: number;
    queued: number;
    running: number;
    successRate: number;         // 0-100
    totalCostUsd: number;
    avgCostPerJob: number;
    avgDurationSec: number;
    failureBreakdown: Record<string, number>;  // FailureCategory → count
  };
  oracle: {
    totalDecisions: number;
    accuracyPercent: number;
    confidenceAvg: number;       // avg of last 20
    circuitBreakerTripped: boolean;
  };
  budget: {
    dailyLimitUsd: number;
    dailySpentUsd: number;
    dailyRemaining: number;
    jobCount: number;
    maxJobsPerDay: number;
    byInstance: Record<string, { totalUsd: number; jobCount: number }>;
  };
  instances: string[];           // Active instance names
}
```

### 2. Core Module (src/dashboard.ts)

**Pure functions — all take data as input, no file I/O:**

- `aggregateJobStats(jobs: Job[]): DashboardData["jobs"]` — filter jobs where `enqueuedAt` starts with today's UTC date string (YYYY-MM-DD), consistent with `todayDateStr()` in job-runner.ts. Count by status, success rate, avg cost, avg duration, failure category breakdown.
- `aggregateOracleStats(metrics: OracleMetrics): DashboardData["oracle"]` — extract accuracy, avg confidence from trend array, circuit breaker status.
- `aggregateBudgetStats(globalBudget: GlobalBudget, config: BudgetConfig): DashboardData["budget"]` — daily spend vs limit, per-instance breakdown. Note: `budget.jobCount` is sourced from `GlobalBudget.jobCount` which reflects cost-incurring completions only (accurate for budget tracking purposes); the total job count including failures is in `jobs.total`.
- `computeHealthScore(data: Omit<DashboardData, "healthScore" | "topConcern" | "generatedAt">): { score: number; topConcern: string | null }` — weighted formula (see below).
- `formatDashboard(data: DashboardData): string` — markdown output.
- `buildDashboard(state: DaemonState, metrics: OracleMetrics, globalBudget: GlobalBudget, config: DaemonConfig): DashboardData` — orchestrator that calls the above.

**File I/O wrapper (thin, for production use):**

- `generateDashboard(instanceDir: string, parentDir: string | undefined, config: DaemonConfig): void` — reads files from specific locations, calls `buildDashboard`, writes `dogfood-report.md` to the parent `.garyclaw/` dir. File locations: `daemon-state.json` at `{instanceDir}/daemon-state.json`, `metrics.json` via `readMetrics(defaultMemoryConfig(config.projectDir))` (per-project oracle-memory dir), `global-budget.json` at `{parentDir}/global-budget.json` (falls back to local-only budget if `parentDir` is undefined).

### 3. Health Score Formula

Weighted 0-100 score:

| Signal | Weight | Score Logic |
|--------|--------|-------------|
| Job success rate | 40% | `successRate` (0-100 directly) |
| Oracle accuracy | 25% | `accuracyPercent` (0-100 directly). If 0 decisions, score = 100 (no signal) |
| Budget headroom | 20% | `(dailyRemaining / dailyLimitUsd) * 100`, capped at 100 |
| No circuit breaker | 15% | 100 if not tripped, 0 if tripped |

Final score = weighted sum, rounded to nearest integer.

**Top concern selection** (first matching rule wins):
1. Circuit breaker tripped → "Oracle memory disabled — accuracy below 60%"
2. Success rate < 50% → "More jobs failing than succeeding — check failure breakdown"
3. Budget headroom < 10% → "Daily budget nearly exhausted (${remaining} remaining)"
4. Any `garyclaw-bug` failures → "GaryClaw bug detected in {N} job(s) — check logs"
5. Success rate < 80% → "{N} job(s) failed today — review failure categories"
6. Null (healthy)

### 5. Job Runner Wire-Up (job-runner.ts)

In `processNext()`, after `writeSummary()` in both the success and failure paths, call:

```typescript
try {
  generateDashboard(checkpointDir, parentCheckpointDir, currentConfig);
} catch (err) {
  d.log("warn", `Dashboard generation failed: ${err instanceof Error ? err.message : String(err)}`);
}
```

Wrapped in try/catch so dashboard errors never affect job completion. The dashboard is a best-effort side effect.

### 6. CLI Command (cli.ts)

Add `garyclaw dashboard` subcommand:

```typescript
case "dashboard":
  const reportPath = join(checkpointDir, "dogfood-report.md");
  if (existsSync(reportPath)) {
    console.log(readFileSync(reportPath, "utf-8"));
  } else {
    console.log("No dashboard yet — run a daemon job first.");
  }
  break;
```

Read-only. Just prints the file. No generation logic in the CLI path.

## Open Questions

1. **Dashboard location for named instances:** Should each instance write its own dashboard to `.garyclaw/daemons/{name}/dogfood-report.md`, or should there be one unified dashboard at `.garyclaw/dogfood-report.md`? **Recommendation:** One unified dashboard — it reads global state, not instance-local state. Per-instance data appears in the "By Instance" table.
2. **Job time window:** Should the dashboard show "today" (midnight-to-midnight UTC) or "last 24 hours"? **Recommendation:** Match the existing `dailyCost.date` field — midnight-to-midnight UTC, consistent with budget enforcement.

## Success Criteria

- `garyclaw dashboard` prints a readable markdown report with health score, job stats, oracle stats, and budget breakdown
- Dashboard regenerates automatically after every daemon job (success or failure)
- Health score correctly reflects system state (100 when healthy, degrades with failures/budget exhaustion/circuit breaker)
- ~29 unit tests passing with synthetic data
- Dashboard generation < 100ms (pure aggregation of in-memory data + one file write)
- Graceful degradation: missing metrics.json → oracle section shows "No data", missing global-budget.json → budget section shows local-only data

## Distribution Plan

Internal module — no separate distribution needed. Ships as part of GaryClaw's existing npm/tsx execution model. The `garyclaw dashboard` CLI command is the user interface.
