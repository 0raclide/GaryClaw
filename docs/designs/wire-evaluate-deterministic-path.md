## Problem Statement

The evaluate skill has a complete, tested set of pure TypeScript analysis functions that produce `improvement-candidates.md` deterministically. None of them are called at runtime. The pipeline currently asks Claude to write that file via prompt instructions, which is best-effort. The `cli.ts` post-pipeline hook reads `improvement-candidates.md` to append improvements to GaryClaw's TODOS.md. If Claude doesn't write the file, the self-improvement loop is broken.

The functions that exist and are tested (72 tests across 3 files):
- `analyzeBootstrapQuality(projectDir)` -> `BootstrapEvaluation`
- `analyzeOraclePerformance(projectDir)` -> `OracleEvaluation`
- `analyzePipelineHealth(projectDir)` -> `PipelineEvaluation`
- `extractObviousImprovements(report)` -> `ImprovementCandidate[]`
- `parseClaudeImprovements(output)` -> `ImprovementCandidate[]` (parses `<improvements>` XML block)
- `deduplicateImprovements(obvious, claude)` -> `ImprovementCandidate[]` (Levenshtein dedup)
- `writeEvaluationReport(projectDir, report)` -> writes JSON + MD + improvement-candidates.md
- `formatImprovementCandidates(candidates)` -> TODOS.md-compatible markdown

The gap: `pipeline.ts` line 319-323 calls `buildEvaluatePrompt` and `runSkillWithPrompt`, then moves on. No post-skill hook runs the deterministic analysis or captures Claude's output for merging.

## What Makes This Cool

This closes a feedback loop that makes GaryClaw literally smarter after every run. The daemon runs `evaluate` on external repos, finds its own weaknesses via deterministic metrics, writes them to TODOS.md, and `prioritize` picks them up next cycle. Right now that loop is broken because it depends on Claude remembering to write a file. Wiring the TS path makes it guaranteed.

## Constraints

- `runSkillWithPrompt` returns `Promise<void>`. No output is returned.
- Assistant text is emitted as `{ type: "assistant_text", text }` events via callbacks but not stored anywhere.
- The `Checkpoint` interface stores issues, findings, decisions, and token usage, but not assistant text.
- `pipeline.ts` passes the same `callbacks` object to all skills. Wrapping must be scoped to evaluate only.
- The evaluate skill might relay (long sessions on large repos), so text accumulates across segments.
- `buildEvaluatePrompt` already runs all analysis functions internally to build the prompt. Those same functions need to run again post-skill (with potentially updated `.garyclaw/` data from Claude's writes during the segment).

## Premises

1. The deterministic TS analysis should run in `pipeline.ts` as a post-skill hook, not inside the orchestrator. Pipeline owns skill lifecycle and already reads checkpoints post-completion.
2. Claude's `<improvements>` output should be captured by wrapping callbacks with a text accumulator, scoped to the evaluate skill only.
3. The merge flow is: obvious (from metrics) + Claude (parsed from `<improvements>`) -> deduplicated -> `writeEvaluationReport()`. This exercises all tested functions.
4. Items 2A (effort scale fix) and 3A (atomic TODOS.md append) from the original TODO are already implemented. Only 1A (wire TS analysis) and 4A (tests) remain.
5. Re-running analysis functions post-skill is correct because Claude may have modified `.garyclaw/` files during its segment (e.g., writing pipeline state, creating files). The post-skill analysis captures the final state. Prompt-time and post-skill analysis may produce different results; the post-skill run is authoritative.

## Approaches Considered

### Approach A: Accumulate Assistant Text in Pipeline Callback (CHOSEN)

Wrap `callbacks` with a text accumulator before calling `runSkillWithPrompt` for evaluate. After the skill completes, parse accumulated text for `<improvements>`, run deterministic analysis, merge via `deduplicateImprovements`, and write via `writeEvaluationReport`.

**Effort:** S
**Risk:** Low
**Pros:**
- No changes to orchestrator.ts, types.ts, or checkpoint format
- Text accumulation scoped to evaluate skill only (other skills unaffected)
- Exercises the full tested merge pipeline (obvious + Claude + dedup)
**Cons:**
- Accumulated text could be large on long sessions (but we only parse it, never persist it)
- Adds ~40 lines to pipeline.ts

### Approach B: Deterministic-Only (Skip Claude's Output)

Run only `extractObviousImprovements` post-skill. Don't capture Claude's text at all. `parseClaudeImprovements` and `deduplicateImprovements` go unused at runtime.

**Effort:** XS
**Risk:** Low
**Pros:**
- Simplest possible change (~20 lines)
- Zero new state accumulation
- Guaranteed improvement-candidates.md
**Cons:**
- Loses Claude's subjective improvement suggestions
- Two tested functions (`parseClaudeImprovements`, `deduplicateImprovements`) remain dead code
- Defeats the purpose of having Claude analyze the evaluation data in the first place

### Approach C: Checkpoint Text Field

Add `assistantOutput?: string` to the Checkpoint interface. Orchestrator accumulates text and persists it. Pipeline reads it post-skill.

**Effort:** S
**Risk:** Medium
**Pros:**
- Clean data path through existing checkpoint mechanism
- Useful for other skills later
**Cons:**
- Checkpoint bloat (50K+ tokens of text per skill)
- Changes types.ts (Checkpoint interface), orchestrator.ts, and checkpoint.ts
- Checkpoint is for relay state, not output capture. Mixing concerns.

## Recommended Approach

**Approach A: Accumulate Assistant Text in Pipeline Callback.**

It's the right trade: captures Claude's output without polluting shared infrastructure, keeps changes scoped to pipeline.ts, and exercises every tested function. The text accumulator is a closure, not persistent state. It exists for the duration of one skill and gets garbage collected.

## Implementation Plan

### Step 1: Create callback wrapper function in pipeline.ts

Add a helper that wraps `OrchestratorCallbacks` with text accumulation:

```typescript
/**
 * Wrap callbacks to accumulate assistant text for post-skill analysis.
 * Returns the wrapped callbacks and a getter for accumulated text.
 */
function createTextAccumulatingCallbacks(
  callbacks: OrchestratorCallbacks,
): { wrapped: OrchestratorCallbacks; getAccumulatedText: () => string } {
  const textChunks: string[] = [];

  const wrapped: OrchestratorCallbacks = {
    ...callbacks,
    onEvent(event: OrchestratorEvent) {
      if (event.type === "assistant_text") {
        textChunks.push(event.text);
      }
      // Always forward to original callbacks
      callbacks.onEvent(event);
    },
    // onAskUser is inherited from spread above
  };

  return {
    wrapped,
    getAccumulatedText: () => textChunks.join(""),
  };
}
```

### Step 3: Add `runPostEvaluateAnalysis` to evaluate.ts

New exported function that runs the full deterministic pipeline:

```typescript
/**
 * Post-evaluate deterministic analysis. Intentionally synchronous —
 * all I/O is fs.writeFileSync via writeEvaluationReport.
 * Called by pipeline.ts after the evaluate segment completes.
 * Runs all TS analysis, parses Claude's <improvements>, merges, deduplicates,
 * and writes the final evaluation report + improvement-candidates.md.
 *
 * Note on multi-relay text: if the evaluate skill relays, the accumulated
 * claudeOutput contains text from ALL segments. If a relay boundary splits
 * mid-<improvements> block, the first match may be incomplete and fail to
 * parse. To handle this, parseClaudeImprovements should be called with a
 * "last valid match" strategy: iterate all <improvements> blocks and return
 * the last one that successfully JSON.parses. This is a defensive fix to
 * the existing function (see Step 3a below).
 */
export function runPostEvaluateAnalysis(
  projectDir: string,
  claudeOutput: string,
): EvaluationReport {
  // 1. Run deterministic analysis
  const bootstrap = safeAnalyze(() => analyzeBootstrapQuality(projectDir), defaultBootstrapEvaluation());
  const oracle = safeAnalyze(() => analyzeOraclePerformance(projectDir), defaultOracleEvaluation());
  const pipeline = safeAnalyze(() => analyzePipelineHealth(projectDir), defaultPipelineEvaluation());

  // 2. Extract obvious improvements from metrics
  const partialReport: EvaluationReport = {
    targetRepo: projectDir,
    timestamp: new Date().toISOString(),
    bootstrap,
    oracle,
    pipeline,
    improvements: [],
  };
  const obvious = safeAnalyze(() => extractObviousImprovements(partialReport), []);

  // 3. Parse Claude's <improvements> output
  const claudeImprovements = parseClaudeImprovements(claudeOutput);

  // 4. Merge + deduplicate
  const merged = deduplicateImprovements(obvious, claudeImprovements);

  // 5. Build final report and write to disk
  const report: EvaluationReport = { ...partialReport, improvements: merged };
  writeEvaluationReport(projectDir, report);

  return report;
}
```

Where `safeAnalyze` is a small error-boundary helper:

```typescript
function safeAnalyze<T>(fn: () => T, fallback: T): T {
  try { return fn(); } catch { return fallback; }
}
```

### Step 3a: Harden `parseClaudeImprovements` for multi-block input

The existing `parseClaudeImprovements` uses `.match()` which returns the first `<improvements>` block. If a relay splits mid-block, the first match is incomplete and fails JSON.parse, returning `[]` and losing valid improvements from the second block. Fix by iterating all matches and returning the last one that parses:

```typescript
export function parseClaudeImprovements(output: string): ImprovementCandidate[] {
  const blocks = [...output.matchAll(/<improvements>([\s\S]*?)<\/improvements>/g)];
  if (blocks.length === 0) return [];

  // Try each block from last to first, return first that parses
  for (let i = blocks.length - 1; i >= 0; i--) {
    try {
      const parsed = JSON.parse(blocks[i][1].trim());
      if (!Array.isArray(parsed)) continue;
      // ... existing validation logic ...
      return valid;
    } catch {
      continue;
    }
  }
  return [];
}
```

## Open Questions

1. **Should `runPostEvaluateAnalysis` overwrite files Claude wrote during the segment?** Yes. The deterministic path is the source of truth. If Claude wrote `improvement-candidates.md` during the segment, we overwrite it with the merged version. The merged version includes Claude's `<improvements>` anyway, so nothing is lost.

2. **What if Claude's segment fails (error/max_turns)?** The pipeline marks the skill as failed and throws. The post-evaluate hook never runs. This is correct: if the segment didn't complete, we don't have Claude's output to merge. The deterministic analysis could still run standalone, but that's a separate concern (resilience enhancement, not this PR).

3. **Memory pressure from text accumulation?** For a typical evaluate segment, Claude's output is ~5-20K characters. Even a multi-relay session would be ~100K characters. This is negligible. The accumulator is a closure that gets GC'd after the skill completes.

4. **Why is `runPostEvaluateAnalysis` synchronous?** All underlying I/O (`writeFileSync`, `readFileSync`, `mkdirSync`) is synchronous. `writeEvaluationReport` calls `safeWriteJSON` and `safeWriteText` which use sync fs operations. Making it async would add complexity for no benefit. If the underlying I/O ever moves to async, the call site in pipeline.ts should add `await`.

## Success Criteria

1. `improvement-candidates.md` is written deterministically after every evaluate skill run, regardless of what Claude does during the segment.
2. The file contains both metric-based (obvious) and Claude-suggested improvements, deduplicated.
3. `cli.ts` post-pipeline hook successfully reads the file and appends to TODOS.md.
4. All existing evaluate tests pass unchanged.
