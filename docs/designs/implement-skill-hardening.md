## Problem Statement

The implement skill (`src/implement.ts`) has three unresolved findings from eng review job-1774487394584 that degrade autonomous run quality:

1. **Silent failure on missing implementation order.** `extractImplementationOrder()` returns `[]` when the section is missing. The prompt then says "follow the implementation order exactly" with no steps listed. Claude improvises, producing unpredictable results.

2. **Bloated review context.** `formatReviewContext()` dumps ALL review decisions including "accepted/resolved" confirmations. A pipeline with 42 decisions from two reviews wastes ~3K tokens on entries like "Use DI? -> Yes (confidence: 9/10)" that need no action.

3. **Dynamic import failure mode.** `pipeline.ts` line 308 uses `await import("./implement.js")` which delays error detection. A syntax error in implement.ts only surfaces 45+ minutes into a pipeline run, after review skills already completed. Since implement.ts has zero external dependencies, the lazy-loading benefit is zero.

Two other findings from the same review are ALREADY COMPLETE:
- loadDesignDoc tests: `test/implement-loaddesigndoc.regression-1.test.ts` (7 tests)
- Pipeline implement dispatch tests: `test/pipeline-implement.test.ts` (4 tests)

The step tracking TODO is already captured in TODOS.md.

## Implementation Order

1. **Convert dynamic import to static in pipeline.ts** (~5 lines changed)
   - Replace `const { buildImplementPrompt } = await import("./implement.js")` with a top-level static `import { buildImplementPrompt } from "./implement.js"`
   - Catches syntax/dependency errors at startup instead of 45 minutes into a pipeline
   - Static import is acceptable because implement.ts has no heavy dependencies (only `node:fs`, `node:path`, local types). If that changes in the future, revert to dynamic import.
   - Test: existing `pipeline-implement.test.ts` tests should still pass (they already mock the module)

2. **Add validation to extractImplementationOrder** (~15 lines + 5 tests)
   - When `extractImplementationOrder()` returns empty AND a design doc exists, inject a warning into the prompt
   - New function: `validateImplementationOrder(steps: string[], hasDesignDoc: boolean): string | null` returns a warning message or null
   - Call from `buildImplementPrompt()` after extraction, inject warning into prompt if non-null
   - Prompt-only warning is sufficient for XS scope. No orchestrator event plumbing needed. The warning is visible in the checkpoint and decisions.jsonl.
   - Known limitation: the existing regex (`/## Implementation [Oo]rder\n/`) may miss headings with trailing spaces or CRLF line endings. Not in scope for this change, but noted for future hardening.
   - Tests: empty steps with design doc (warning), empty steps without design doc (no warning), non-empty steps (no warning), warning message format, warning injected into prompt

3. **Filter formatReviewContext to actionable decisions** (~10 lines + 8 tests)
   - Add optional `actionableOnly: boolean` parameter to `formatReviewContext(skills, options?)`
   - When `actionableOnly` is true, filter decisions using this heuristic:
     - **Include** decisions where `confidence <= 7` (low confidence = needs attention)
     - **Include** decisions where `chosen` text contains action keywords: "add", "fix", "change", "create", "remove", "switch", "replace"
     - **Exclude** everything else (high-confidence confirmations like "Keep as-is", "Skip", "No change")
     - **Always include** all findings and all issues (always actionable regardless of confidence)
   - Check action keywords in BOTH `d.chosen` AND `d.question` text. A decision like "Should we add retry logic?" answered "Yes" has the action keyword in the question, not the answer. Checking both fields catches this.
   - This filter is best-effort and errs on inclusion. A decision like "No, don't add Redis" contains "add" and will be included, which is correct (it's an actionable architectural choice). False positives (including too much) are acceptable. False negatives (filtering out something actionable) are the real risk, so the confidence threshold is moderately inclusive (<=7 covers the majority of non-escalated decisions, since the Oracle escalates at <6).
   - Tests: all decisions pass through when `actionableOnly` is false/undefined, high-confidence "accepted" decisions filtered, low-confidence decisions kept, action-keyword decisions kept regardless of confidence, confidence exactly 7 included, confidence exactly 8 excluded, findings always included, issues always included

4. **Wire actionable filter into buildImplementPrompt** (~2 lines)
   - Pass `{ actionableOnly: true }` when calling `formatReviewContext()` from `buildImplementPrompt()`
   - The default (`actionableOnly` unset) preserves backward compatibility for any other callers
