## Problem Statement

The bootstrap skill generates CLAUDE.md and TODOS.md for external repos. The existing claims system (`extractClaudeMdClaims` + `verifyClaudeMdClaims`) catches hallucinated tech stack names, missing file paths, wrong test frameworks, and phantom entry points. But it misses two categories of errors that waste downstream tokens:

1. **Hallucinated commands.** CLAUDE.md says `npm run dev` but there's no `dev` script in package.json. Or `npx tsx src/server.ts` but `src/server.ts` doesn't exist. The implement skill copies these commands into generated code. The QA skill runs them and burns context on "command not found" errors.

2. **Wrong test directory/count claims.** CLAUDE.md says "tests live in `__tests__/`" but the real directory is `test/`. Or claims "150 test files" when there are 42. The implement skill generates tests in the wrong directory and the QA skill reports phantom failures.

The quality gate already triggers re-bootstrap on low scores and feeds failed claims into `buildEnrichedBootstrapPrompt()`. Extending the claims system with these two new types plugs the remaining gaps with zero new infrastructure.

## What Makes This Cool

The claims pipeline is already built. Adding command and test-directory validation is extending a working system, not building something new. The enrichment prompt already formats failed claims by type. New claim types get correction prompts for free. Five new functions, two new claim types, zero new modules.

## Constraints

- Extend existing `ClaimType` union and claims pipeline. No new validation layer.
- Pure TypeScript. Zero Claude calls. All validation is deterministic.
- Must not break existing `BootstrapEvaluation` interface (optional fields only).
- All tests use synthetic data in tmp directories.
- Must work on any Node/TypeScript repo. Non-Node repos get deferred verification (existing pattern).

## Premises

1. The existing claims pipeline (`extractClaudeMdClaims` → `verifyClaudeMdClaims` → `buildEnrichedBootstrapPrompt`) is the right extension point. Building parallel validators (the worker-4 approach) duplicates extraction logic and scoring infrastructure.
2. Two new claim types (`command` and `test_directory`) close the remaining semantic gaps. The existing `file_path`, `tech_stack`, `test_framework`, and `entry_point` types handle the rest.
3. The quality score weighting doesn't need rebalancing. The current split (structural 30 + claims 20 + framework coverage 20 + TODOS 20 + efficiency 10) already weights claims appropriately. More claim types naturally increase the denominator and make the claims sub-score more representative.
4. The enrichment prompt in `buildEnrichedBootstrapPrompt()` already formats failed claims by `c.type` in a switch statement. Adding `case "command"` and `case "test_directory"` is 6 lines.

## Approaches Considered

### Approach A: Extend Claims Pipeline (Minimal Viable)

**Summary:** Add `command` and `test_directory` to `ClaimType`. Add extraction logic to `extractClaudeMdClaims()` and verification logic to `verifyClaudeMdClaims()`. Add two switch cases to `buildEnrichedBootstrapPrompt()`. Done.

**Effort:** S
**Risk:** Low
**Pros:**
- Zero new infrastructure. Uses the existing extraction → verification → scoring → enrichment pipeline.
- Failed command/test-directory claims automatically appear in re-bootstrap correction prompts.
- Existing test patterns in `test/evaluate-claims.test.ts` and `test/evaluate-claims.regression-1.test.ts` serve as templates.

**Cons:**
- `extractClaudeMdClaims()` grows larger (already ~120 lines, would add ~60 more).
- Single function doing 6 types of extraction is approaching "too many concerns." But not there yet.

**Reuses:** `extractClaudeMdClaims()`, `verifyClaudeMdClaims()`, `buildEnrichedBootstrapPrompt()`, `findTestDir()` from bootstrap.ts, `ClaimType` union.

### Approach B: Dedicated Validator Functions (Worker-4 Design)

**Summary:** Four new functions (`validateFileReferences`, `validateCommands`, `validateTechStackClaims`, `validateTestClaims`) returning dedicated interfaces, wired into `analyzeBootstrapQuality()` as a second validation pass. New fields on `BootstrapEvaluation`. Rebalanced quality score.

**Effort:** M
**Risk:** Med
**Pros:**
- Each validator is a clean, focused function.
- Dedicated interfaces (FileRefValidation, CommandValidation, etc.) are self-documenting.

**Cons:**
- Duplicates extraction logic already in `extractClaudeMdClaims()` (file paths, tech stack).
- Needs a separate scoring layer + weight rebalancing.
- Doesn't automatically integrate with the enrichment prompt. Needs separate wiring.
- Worker-4 built this and it conflicted. More code = more merge surface.

### Approach C: Hybrid — Extract Helpers, Keep Claims Pipeline

**Summary:** Factor the extraction logic for commands and test directories into helper functions (`extractCommandClaims`, `extractTestDirectoryClaims`), then call them from `extractClaudeMdClaims()`. Best of both worlds: clean separation + unified pipeline.

**Effort:** S
**Risk:** Low
**Pros:**
- Keeps `extractClaudeMdClaims()` manageable by delegating to helpers.
- Helpers are independently testable.
- Still uses the unified claims → verify → score → enrich pipeline.
- Smallest diff, lowest merge risk.

**Cons:**
- Slightly more files to touch than Approach A (but same files, just more organized).

## Recommended Approach

**Approach C: Hybrid.** Factor extraction into helpers, keep the unified pipeline. This is the smallest change that closes the gap. The helpers are independently testable and keep `extractClaudeMdClaims()` from becoming a 200-line monster.

## Implementation Details

### 1. Extend ClaimType

### 2. Command Claim Extraction

New helper in `src/evaluate.ts`:

```typescript
export function extractCommandClaims(claudeMdContent: string): ClaudeMdClaim[]
```

**Logic:**
1. Parse fenced code blocks (` ```bash `, ` ```sh `, ` ``` ` with no language tag).
2. For each line in a code block, extract verifiable commands:
   - `npm run <script>` / `yarn <script>` / `pnpm <script>` → claim type `command`, claimed = `npm-script:<script>`
   - **Special case:** `npm test` → `npm-script:test`, `npm start` → `npm-script:start`. These are shorthand for `npm run test` / `npm run start` but don't match the `npm run <script>` pattern. Need their own regex branch: `/^npm\s+(test|start)\b/`.
   - `npx tsx <path>` / `npx ts-node <path>` / `node <path>` → claim type `command`, claimed = `file-command:<path>`
   - `npx <package>` (e.g., `npx vitest`) → skip (package existence is a different check)
3. Skip lines that are clearly not commands: comments (`#`), variable assignments (`FOO=`), `cd`, `mkdir`, `git`, `echo`, pure flags.
4. Deduplicate by claimed value.

**Example extractions:**
- `npm run dev` → `{ type: "command", claimed: "npm-script:dev", evidence: "", verified: false }`
- `npx tsx src/cli.ts run qa` → `{ type: "command", claimed: "file-command:src/cli.ts", evidence: "", verified: false }`
- `npm test` → `{ type: "command", claimed: "npm-script:test", evidence: "", verified: false }`
- `npm install` → skipped (no verifiable reference)

### 3. Command Claim Verification

In `verifyClaudeMdClaims()`, add a `case "command"` branch:

```typescript
case "command": {
  if (claim.claimed.startsWith("npm-script:")) {
    const scriptName = claim.claimed.slice("npm-script:".length);
    // Special case: "npm test" maps to "test" script
    const pkgPath = join(projectDir, "package.json");
    if (!existsSync(pkgPath)) {
      verified.verified = true;
      verified.evidence = "no package.json, verification deferred";
      break;
    }
    try {
      const pkg = JSON.parse(readFileSync(pkgPath, "utf-8"));
      const scripts = pkg.scripts ?? {};
      if (scriptName in scripts) {
        verified.verified = true;
        verified.evidence = `script exists: "${scripts[scriptName]}"`;
      } else {
        verified.verified = false;
        verified.evidence = `no "${scriptName}" script in package.json (available: ${Object.keys(scripts).join(", ")})`;
      }
    } catch {
      verified.verified = true;
      verified.evidence = "package.json parse error, verification deferred";
    }
  } else if (claim.claimed.startsWith("file-command:")) {
    const filePath = claim.claimed.slice("file-command:".length);
    if (existsSync(join(projectDir, filePath))) {
      verified.verified = true;
      verified.evidence = "file exists";
    } else {
      verified.verified = false;
      verified.evidence = "file does not exist";
    }
  }
  break;
}
```

### 4. Test Directory Claim Extraction

New helper in `src/evaluate.ts`:

```typescript
export function extractTestDirectoryClaims(claudeMdContent: string): ClaudeMdClaim[]
```

**Logic:**
1. Look for test directory mentions: patterns like `` `test/` ``, `` `tests/` ``, `` `__tests__/` ``, `` `spec/` `` in prose (not inside tree listings).
2. Look for test file count claims: patterns like "N test files", "N tests", "N test files" where N is a number. Create a claim with claimed = `test-count:<N>`.
3. One claim per unique directory mention, one claim per count mention.

**Example extractions:**
- "Tests live in `test/`" → `{ type: "test_directory", claimed: "test-dir:test", evidence: "", verified: false }`
- "42 test files" → `{ type: "test_directory", claimed: "test-count:42", evidence: "", verified: false }`

### 5. Test Directory Claim Verification

In `verifyClaudeMdClaims()`, add a `case "test_directory"` branch:

```typescript
case "test_directory": {
  if (claim.claimed.startsWith("test-dir:")) {
    const dirName = claim.claimed.slice("test-dir:".length);
    const dirPath = join(projectDir, dirName);
    if (existsSync(dirPath)) {
      verified.verified = true;
      verified.evidence = "directory exists";
    } else {
      // Check common alternatives (all sync — no async needed)
      const alternatives = ["test", "tests", "__tests__", "spec", "specs"];
      const found = alternatives.filter(d => existsSync(join(projectDir, d)));
      verified.verified = false;
      verified.evidence = found.length > 0
        ? `directory does not exist (found instead: ${found.join(", ")})`
        : "directory does not exist (no test directory found)";
    }
  } else if (claim.claimed.startsWith("test-count:")) {
    const claimedCount = parseInt(claim.claimed.slice("test-count:".length), 10);
    // Count actual test files using walkFileTree (static ESM import at file top)
    // Import: import { walkFileTree } from "./bootstrap.js";
    // (evaluate.ts already depends on bootstrap.ts via analyzeBootstrapQuality)
    const testPattern = /\.(test|spec)\.(ts|tsx|js|jsx|mjs)$/;
    let actualCount = 0;
    try {
      const { files } = walkFileTree(projectDir);
      actualCount = files.filter(f => testPattern.test(f)).length;
    } catch {
      verified.verified = true;
      verified.evidence = "could not count test files, verification deferred";
      break;
    }
    // Allow 20% tolerance (bootstrap may have counted differently)
    const tolerance = Math.max(3, Math.ceil(claimedCount * 0.2));
    if (Math.abs(actualCount - claimedCount) <= tolerance) {
      verified.verified = true;
      verified.evidence = `actual count: ${actualCount} (within tolerance of claimed ${claimedCount})`;
    } else {
      verified.verified = false;
      verified.evidence = `actual count: ${actualCount}, claimed: ${claimedCount} (off by ${Math.abs(actualCount - claimedCount)})`;
    }
  }
  break;
}
```

**Important:** `verifyClaudeMdClaims()` is currently synchronous. The test-count verification needs `walkFileTree()` which is also synchronous (uses `readdirSync`). Import it at the top of evaluate.ts (already imports from bootstrap.ts via `analyzeBootstrapQuality` → `bootstrap.ts`). Use a direct import, not dynamic require.

### 8. Update formatEvaluationReport()

Add a "Claim Verification Details" subsection in the Bootstrap Quality section that shows failed claims grouped by type:

```typescript
const failedClaims = (report.bootstrap.claims ?? []).filter(c => !c.verified);
if (failedClaims.length > 0) {
  lines.push("**Failed Claims:**");
```

### What Doesn't Change

- `BootstrapEvaluation` interface: no new fields needed. The existing `claims`, `claimsVerified`, `claimsTotal`, `claimVerificationScore` fields already capture everything.
- Quality score weighting: stays at structural (30) + claims (20) + framework coverage (20) + TODOS (20) + efficiency (10). More claim types increase the denominator, making the 20-point claims sub-score more representative without rebalancing.
- `analyzeBootstrapQuality()` flow: unchanged. It calls `extractClaudeMdClaims()` → `verifyClaudeMdClaims()` → counts verified. The new claim types flow through automatically.

## Open Questions

1. **Should `npm test` extract as `npm-script:test`?** Yes. `npm test` is shorthand for `npm run test`. The verification checks `scripts.test` in package.json. Same for `npm start` → `scripts.start`.
2. **What about `bun test` / `bun run <script>`?** Add as a variant alongside npm/yarn/pnpm. Low effort, catches Bun-based repos.
3. **Should test file count tolerance be configurable?** Not now. 20% or 3 files (whichever is larger) is generous enough. Tune if NihontoWatch data says otherwise.

## What I noticed about how you think

- The priority pick correctly identified that the worker-4 design was validated but the implementation conflicted. Rather than trying to resurrect the branch, reimplementing from the design is the right call. Fresh code on main, no merge debt.
- The existing claims system is better infrastructure than worker-4 realized. Extending it with two new types is less code, less risk, and gets the enrichment prompt integration for free. The worker-4 design proposed a parallel validation layer because the claims system didn't exist yet when that design was written.
- The 20% test count tolerance is a smart detail. Bootstrap might count `*.test.ts` while the actual repo has a mix of `.test.ts` and `.spec.ts`. Exact count matching would produce false negatives.
