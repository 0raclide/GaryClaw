# Design: Bootstrap Quality Gate — Self-Healing Cold Start

## Problem Statement

The bootstrap skill generates CLAUDE.md and TODOS.md for cold-start repos, but there's no quality checkpoint between bootstrap and the rest of the pipeline. If bootstrap produces a thin or inaccurate CLAUDE.md (missing sections, wrong framework coverage) or a shallow TODOS.md (items that all score below 5.0 in prioritize), the entire downstream pipeline (prioritize → implement → qa) wastes $2-5 of compute working on garbage input.

`analyzeBootstrapQuality()` in evaluate.ts already scores bootstrap output on a 0-100 scale. But it only runs at the end of the pipeline in the evaluate skill. By then, the damage is done.

The dogfood-bootstrap.md design doc explicitly called out Approach B (QA pre-scan before bootstrap) as the follow-up if bootstrap quality is insufficient. This design implements that follow-up as a self-healing pipeline quality gate.

## What Makes This Cool

Bootstrap runs on a repo it has never seen. The quality gate catches that CLAUDE.md missed the monorepo structure and only scored 35/100. Instead of proceeding with bad data, the pipeline automatically inserts a QA pre-scan, feeds those findings into a second bootstrap pass, and produces artifacts that score 72/100. The system corrected its own cold-start failure without human intervention.

## Constraints

- Must not add new skills. Uses existing `analyzeBootstrapQuality()` from evaluate.ts and existing QA skill.
- Must be a pipeline.ts enhancement, not a standalone feature.
- Enrichment retry capped at 1 attempt. If the enriched bootstrap still scores below threshold, continue anyway (fail-open) with a warning event.
- QA pre-scan runs without CLAUDE.md context (by definition, the bootstrap output failed). The QA skill must handle this gracefully.
- The quality gate is optional. A new config flag `bootstrapQualityGate` (default: true for dogfood pipelines, false otherwise) controls whether the gate runs.
- Must not break existing `bootstrap → prioritize → implement → qa` pipelines. If the gate is disabled or scoring throws, the pipeline continues normally.

## Premises

1. `analyzeBootstrapQuality()` already scores bootstrap output (structural 40pts + accuracy 30pts + viability 20pts + efficiency 10pts). 72 tests cover all codepaths. No new scoring logic needed.
2. A quality score below 50 indicates bootstrap output too weak for productive downstream work. This threshold is already used in evaluate.ts (line 515) for flagging improvement candidates.
3. QA findings from a cold repo provide real evidence (test failures, lint errors, broken builds, security issues) that enrich the second bootstrap pass. This is Approach B from dogfood-bootstrap.md.
4. Dynamic skill insertion in pipeline.ts is the right mechanism. The pipeline already has per-skill dispatch branches and tracks state per-skill.
5. One enrichment retry is sufficient. If two bootstrap passes can't produce score >= 50, the repo is genuinely hard and the pipeline should proceed with what it has.

## Approaches Considered

### Approach A: Pipeline Quality Gate (Minimal — Fail-Open Warning)

**Summary:** After bootstrap, call `analyzeBootstrapQuality()`. If score < 50, emit a warning event and continue. No enrichment.

**Effort:** XS
**Risk:** Low
**Pros:**
- ~20 lines in pipeline.ts
- Can't break anything (fail-open)
- Evaluate skill sees the warning downstream
**Cons:**
- Documents the problem but doesn't fix it
- Prioritize/implement still waste compute on bad input

### Approach B: Quality Gate + Auto-Enrichment (Recommended)

**Summary:** After bootstrap, run quality gate. If score < 50, dynamically run QA → re-bootstrap with enriched context. Capped at 1 retry.

**Effort:** S
**Risk:** Medium
**Pros:**
- Self-healing: bad bootstrap → QA scan → enriched re-bootstrap → better output
- Uses existing QA skill and bootstrap skill, no new modules
- Retry cap prevents infinite loops
- The demo: "the system corrected its own cold-start failure"
**Cons:**
- Adds ~100 lines across pipeline.ts + bootstrap.ts + dashboard.ts
- QA on a cold repo may generate noise
- Doubles bootstrap cost when triggered (~$1.50-2.50 extra, capped by maxRelaySessions:1 on QA pre-scan)

### Approach C: Threshold-Only Skip (Fail-Fast)

**Summary:** If score < 50, skip the rest of the pipeline. Mark job as "bootstrap-insufficient." Next run auto-includes QA pre-scan.

**Effort:** XS
**Risk:** Low
**Pros:**
- Zero wasted compute
- ~15 lines
**Cons:**
- Gives up immediately, no self-correction
- Requires a second daemon run for enrichment

## Recommended Approach

**Approach B: Quality Gate + Auto-Enrichment.** It's the only option that fixes bad bootstrap in the same pipeline run. ~103 lines across 5 modules. The retry cap means the worst case is 2x bootstrap cost, not runaway.

## Implementation

### 1. Config Flag in types.ts (~3 lines)

Add `bootstrapQualityGate?: boolean` to `GaryClawConfig` in types.ts. Default: `undefined`, which the pipeline treats as `true` (opt-out, not opt-in). Users pass `--no-bootstrap-gate` to disable. The check `config.bootstrapQualityGate !== false` means the gate runs unless explicitly disabled.

```typescript
// In GaryClawConfig:
bootstrapQualityGate?: boolean; // Enable quality gate after bootstrap (default: undefined = enabled)
```

### 2. Quality Gate in pipeline.ts (~50 lines)

After the bootstrap skill dispatch completes (pipeline.ts:331), add a quality gate check. The enrichment runs QA and re-bootstrap as inline side-effects within the existing bootstrap dispatch branch, using the same `runSkillWithPrompt()` helper that already exists in pipeline.ts (line 457). The QA pre-scan is capped at `maxRelaySessions: 1` to limit cost.

```typescript
// After bootstrap's runSkillWithPrompt completes (inside the skillName === "bootstrap" branch):
if (config.bootstrapQualityGate !== false) {
  const { analyzeBootstrapQuality, BOOTSTRAP_QUALITY_THRESHOLD } = await import("./evaluate.js");

  try {
    const bootstrapEval = analyzeBootstrapQuality(config.projectDir);

    callbacks.onEvent({
      type: "bootstrap_quality_check",
      qualityScore: bootstrapEval.qualityScore,
      missingSections: bootstrapEval.claudeMdMissingSections,
      notes: bootstrapEval.qualityNotes,
    });

    if (bootstrapEval.qualityScore < BOOTSTRAP_QUALITY_THRESHOLD && !state.bootstrapEnriched) {
      callbacks.onEvent({
        type: "assistant_text",
        text: `\n[Quality Gate] Bootstrap scored ${bootstrapEval.qualityScore}/100 (threshold: ${BOOTSTRAP_QUALITY_THRESHOLD}). Running QA pre-scan for enrichment...\n`,
      });

      state.bootstrapEnriched = true;
      savePipelineState(state, config.checkpointDir); // Persist for crash recovery

      // Run QA pre-scan (capped at 1 relay session to limit cost)
      const qaCheckpointDir = join(config.checkpointDir, "skill-enrichment-qa");
      const qaConfig: GaryClawConfig = {
        ...config,
        skillName: "qa",
        checkpointDir: qaCheckpointDir,
        maxRelaySessions: 1,  // Cost cap: 1 session max for pre-scan
      };
      await runSkillWithPrompt(qaConfig, callbacks,
        "Run the /qa skill on this repo. Focus on: test failures, broken builds, missing dependencies, and obvious code issues. This is a pre-scan for bootstrap enrichment.");

      // Re-run bootstrap with QA findings injected
      const { buildEnrichedBootstrapPrompt } = await import("./bootstrap.js");
      const enrichedPrompt = await buildEnrichedBootstrapPrompt(config, qaCheckpointDir, config.projectDir);
      const rebootstrapConfig: GaryClawConfig = {
        ...config,
        skillName: "bootstrap",
        checkpointDir: join(config.checkpointDir, "skill-enrichment-bootstrap"),
      };
      await runSkillWithPrompt(rebootstrapConfig, callbacks, enrichedPrompt);

      // Re-check quality (informational only, fail-open regardless)
      const enrichedEval = analyzeBootstrapQuality(config.projectDir);
      callbacks.onEvent({
        type: "bootstrap_quality_recheck",
        qualityScore: enrichedEval.qualityScore,
        previousScore: bootstrapEval.qualityScore,
      });
    }
  } catch (gateErr) {
    // Fail-open: if the quality gate itself crashes, continue the pipeline
    callbacks.onEvent({
      type: "assistant_text",
      text: `\n[Quality Gate] Scoring failed: ${gateErr instanceof Error ? gateErr.message : String(gateErr)}. Continuing pipeline.\n`,
    });
  }
}
```

Note: `runSkillWithPrompt()` already exists in pipeline.ts (line 457). It wraps `runOrchestrator()` with a custom initial prompt. No new function abstraction needed.

### 3. Enriched Bootstrap Prompt in bootstrap.ts (~40 lines)

New export `buildEnrichedBootstrapPrompt()` that:
1. Calls existing `analyzeCodebase()` to get fresh codebase analysis
2. Reads the QA checkpoint report via existing `readCheckpoint()` from checkpoint.ts
3. Reads the existing (failed) CLAUDE.md via `readFileSync` with try/catch
4. Builds a prompt with enriched context

All helper functions used already exist:
- `analyzeCodebase()` — bootstrap.ts (existing)
- `readCheckpoint()` — checkpoint.ts (existing)
- `estimateTokens()` — checkpoint.ts (existing, used for truncation)
- `buildFileTreeString()` — bootstrap.ts (existing, for `formatAnalysis`)

```typescript
import { readCheckpoint } from "./checkpoint.js";

export async function buildEnrichedBootstrapPrompt(
  config: GaryClawConfig,
  qaCheckpointDir: string,
  projectDir: string,
): Promise<string> {
  const analysis = await analyzeCodebase(projectDir);

  // Read existing CLAUDE.md (the one that failed the quality gate)
  let existingClaudeMd = "";
  try {
    existingClaudeMd = readFileSync(join(projectDir, "CLAUDE.md"), "utf-8");
  } catch { /* missing file is fine */ }

  // Truncate to 5K tokens if large
  if (estimateTokens(existingClaudeMd) > 5000) {
    const lines = existingClaudeMd.split("\n");
    while (estimateTokens(lines.join("\n")) > 5000 && lines.length > 10) {
      lines.pop();
    }
    existingClaudeMd = lines.join("\n") + "\n[... truncated]";
  }

  // Read QA findings from checkpoint (issues, findings from the pre-scan)
  const qaCheckpoint = readCheckpoint(qaCheckpointDir);
  const qaFindings = qaCheckpoint?.relayPrompt ?? "No QA findings captured.";

  // Format codebase analysis using existing buildFileTreeString
  const fileTree = analysis.fileTree ?? buildFileTreeString(projectDir);

  return `You are improving a CLAUDE.md that scored below the quality threshold.

## Current CLAUDE.md (needs improvement)
${existingClaudeMd || "(empty — bootstrap produced no output)"}

## QA Pre-Scan Findings
${qaFindings}

## File Tree
${fileTree}

## Package Dependencies
${analysis.packageJson ?? "No package.json found."}

## Instructions
Rewrite CLAUDE.md to address the quality gaps. Include:
- All 4 required sections: Architecture, Tech Stack, Test Strategy, Usage
- Every framework/library found in package.json
- Specific file paths and module descriptions
- Any issues found by the QA pre-scan

Also update TODOS.md to include QA findings as backlog items with proper P1-P5 priorities.

Write the updated files now.`;
}
```

### 4. Pipeline State Extension (~3 lines)

Add `bootstrapEnriched?: boolean` to `PipelineState` in types.ts. This prevents infinite enrichment loops (the re-bootstrap skips the gate when `bootstrapEnriched` is true). The field is persisted via `savePipelineState()` so crash recovery knows enrichment already happened.

### 5. Constants (~2 lines)

Extract the existing magic number `50` at evaluate.ts:515 into a named constant:

```typescript
/** Minimum bootstrap quality score (0-100) to proceed without enrichment */
export const BOOTSTRAP_QUALITY_THRESHOLD = 50;
```

Defined in evaluate.ts alongside the existing `EXPECTED_SECTIONS` constant. The existing `if (report.bootstrap.qualityScore < 50)` at line 515 should reference this constant too.

### 6. Pipeline Event Types

Two new event types for observability:
- `bootstrap_quality_check` — emitted after every bootstrap, carries qualityScore + notes
- `bootstrap_quality_recheck` — emitted after enrichment, carries both scores for delta comparison

### 7. Dashboard Integration (~10 lines)

Add enrichment stats to `formatDashboard()` in dashboard.ts:
- Count of enrichment triggers (from `bootstrap_quality_check` events where score < threshold)
- Average score improvement (from `bootstrap_quality_recheck` events: enrichedScore - previousScore)
- Displayed as a new row: `| Bootstrap Enrichment | {count} triggered, avg +{delta} pts |`

Read enrichment data from pipeline state files in `.garyclaw/` (same pattern as crash recovery stats).

### 8. Pipeline Resume Interaction

Enrichment checkpoint directories (`skill-enrichment-qa`, `skill-enrichment-bootstrap`) are nested under the bootstrap skill's checkpoint dir. If the daemon crashes mid-enrichment:
- `bootstrapEnriched: true` is already persisted in pipeline state
- On resume, the bootstrap skill is re-run (normal resume behavior)
- The `bootstrapEnriched` flag prevents a second enrichment attempt
- This matches the retry-cap semantics: 1 enrichment attempt max

### 9. QA Pre-Scan on Cold Repos

The QA pre-scan runs via `runSkillWithPrompt()` which calls `runOrchestrator()` which spawns a Claude session with the `/qa` skill loaded from gstack's SKILL.md. The gstack QA skill reads CLAUDE.md for context but does not require it. When CLAUDE.md is thin or incomplete:
- QA still runs `npm test`, checks for lint errors, scans for security issues
- Findings are less contextual but still provide real evidence (test failures, build errors)
- The `maxRelaySessions: 1` cap means the pre-scan costs ~$0.50-1.00 max

This is validated by the QA skill's behavior on NihontoWatch's first run, where QA ran successfully after bootstrap generated a thin initial CLAUDE.md. **Soft assumption:** If QA produces confused output on a cold repo, the enriched bootstrap still has the original codebase analysis as its primary source. QA findings are additive, not the sole input.

## Success Criteria

- Quality gate runs after every bootstrap in a pipeline (observable via `bootstrap_quality_check` event)
- When bootstrap scores < 50, QA pre-scan + re-bootstrap runs automatically
- Enriched bootstrap produces a higher quality score than the original (verifiable from `bootstrap_quality_recheck` event)
- Pipeline continues even if enrichment doesn't improve the score (fail-open)
- Zero regression in existing `bootstrap → prioritize → implement → qa` pipelines when gate is enabled
