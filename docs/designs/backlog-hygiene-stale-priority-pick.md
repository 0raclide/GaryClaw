## Approaches Considered

### Approach A: TODOS.md Hygiene Only
Mark implement-skill-hardening items and Dashboard Adaptive Turns Stats as complete in TODOS.md. Update CLAUDE.md test counts.

- Effort: XS (5 minutes)
- Risk: Low
- Pros: Accurate backlog, prevents future stale picks, smallest possible diff
- Cons: Doesn't add any new capability
- Reuses: Existing TODOS.md strikethrough pattern

### Approach B: Hygiene + Staleness Guard in Prioritize
Add a pre-check to the prioritize skill that greps for implementation evidence before scoring. When a design doc has Status: APPROVED, scan for the key functions/tests it specifies before including it as a candidate.

- Effort: S (CC: ~30 min)
- Risk: Medium — heuristic matching could false-positive on partial implementations
- Pros: Prevents stale picks systematically, not just this instance
- Cons: Adds complexity to prioritize.ts, heuristic matching is fragile, premature optimization when the real fix is just keeping TODOS.md current

### Approach C: Hygiene + Oracle Decision Batching Design
Fix TODOS.md, then run a fresh office-hours session for Oracle Decision Batching (the actual next P3 item). Two design docs in one pipeline run.

- Effort: S (CC: ~30 min for the batching design)
- Risk: Low
- Pros: Moves the project forward on real work
- Cons: Scope creep beyond the original task. Batching deserves its own focused session.

## Recommended Approach

**Approach A: TODOS.md Hygiene Only.** The simplest fix for the simplest problem. The prioritize skill works correctly. Its inputs are stale. Fix the inputs.

Approach B is premature. The root cause is "someone shipped code without updating TODOS.md." The fix is "update TODOS.md." Building an automated staleness detector is the kind of automation that costs more to maintain than the manual 2-minute task it replaces.
