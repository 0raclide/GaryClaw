## Problem Statement

20+ of the last 50 Oracle decisions are semantically identical. The #1 pattern: "GaryClaw is a CLI tool with no web UI, what should QA do?" answered with "Run the test suite" every single time. Each of these is a ~40K-token API call costing ~$0.05 and 2-5s latency. With costs trending up (+4%, $0.08/job), this is the highest-ROI optimization available.

Project Type Awareness (shipped today, 7 commits) reduces the frequency of these questions by injecting project context into skill prompts. But the cache addresses the general pattern: any question answered identically 5+ times should be auto-resolved at zero cost. Other repeated patterns exist beyond "no web UI" (e.g., "commit my changes" for dirty working trees, "clean tree" decisions).

## What Makes This Cool

The daemon gets smarter with every run. Not in a vague ML way. In a "I've answered this exact question 20 times, I'm not asking again" way. The cache is the daemon's short-term memory for operational decisions. It turns cost into a step function instead of a linear one: the first few runs pay full price, then common decisions become free forever.

Combined with the existing oracle memory (taste.md, decision-outcomes.md) and oracle session reuse, this completes the decision cost reduction stack. Session reuse cuts per-call token cost by ~95% for decisions 2-N in a session. The cache eliminates the call entirely for known answers.

## Constraints

- Must integrate with existing ask-handler.ts interception flow (the `canUseTool` callback)
- Must not break oracle session reuse (session state must remain valid when cache is used)
- Must be invalidatable when reflection detects a failure outcome for a cached answer
- Must produce decisions.jsonl entries indistinguishable from real Oracle decisions (for audit trail)
- Cache must survive daemon restarts (warm start from decision-outcomes.md)
- Must be testable without SDK calls (same isolation boundary as oracle.ts)

## Premises

1. **Semantic matching via keyword normalization, not exact strings.** The same question appears with 10+ different wordings. Stripping variable content (timestamps, branch names, file counts, file lists) and matching on a normalized keyword bag is the right tradeoff between accuracy and complexity.

2. **5+ identical answers is the right caching threshold.** Conservative enough to avoid caching coincidences. With 100% oracle accuracy and 20+ hits on the main pattern, 5 is safe.

3. **Reflection-based invalidation is sufficient.** No time-based expiry. If a cached answer produces a "failure" outcome, clear that entry. The decision-outcomes.md already tracks what worked.

4. **The cache intercepts in ask-handler.ts, before the Oracle call.** Short-circuit before `askOracle()` / `askOracleBatch()`, not inside them. This preserves oracle session state (no session for a cached decision).

5. **The cache is in-memory with warm start from disk.** Read decision-outcomes.md at startup to pre-populate. Runtime cache updates stay in memory. No additional disk writes for cache state (decisions.jsonl already records everything needed for reconstruction).

## Approaches Considered

### Approach A: Keyword Bag Normalization (Recommended)

**Summary:** Normalize questions by stripping variable tokens (numbers, file paths, branch names, timestamps) and reducing to a sorted keyword bag. Cache key = hash of sorted keywords + sorted option labels. Cache populated from decision-outcomes.md entries where 5+ decisions share the same normalized key and identical chosen answer.

**Effort:** S
**Risk:** Low
**Pros:**
- Simple, deterministic, fully testable
- No ML, no embeddings, no external dependencies
- Handles the actual observed patterns (all variations of "CLI tool no web UI" normalize to the same keywords)
- In-memory Map lookup is O(1), zero latency

**Cons:**
- Aggressive normalization might collapse genuinely different questions into the same key (false positive)
- New question patterns that don't appear in decision-outcomes.md won't cache until they accumulate 5+ hits at runtime

**Reuses:** `levenshteinDistance` from reflection.ts for similarity fallback, `readDecisionOutcomes` from oracle-memory.ts for warm start, `Decision` type from types.ts

### Approach B: Levenshtein Clustering

**Summary:** Instead of keyword bags, cluster questions using normalized Levenshtein distance (already in reflection.ts). Two questions with distance < 0.3 after stripping variable tokens are "the same question." Cache maps cluster centroids to answers.

**Effort:** S
**Risk:** Medium
**Pros:**
- More nuanced matching (handles typos, minor wording changes)
- Already have Levenshtein in the codebase

**Cons:**
- O(n) comparison against all centroids on every question (vs O(1) hash lookup)
- Centroid selection is non-trivial: which question represents the cluster?
- Levenshtein is character-level, not semantic. "Should I run the test suite?" and "What QA approach works for a CLI?" are semantically similar but Levenshtein-distant
- MAX_LEVENSHTEIN_LENGTH (500) truncation could cause false matches on long questions

### Approach C: Two-Tier Cache (Keyword Bag + Levenshtein Fallback)

**Summary:** Primary: keyword bag hash (Approach A). Fallback: Levenshtein clustering (Approach B) for questions that miss the primary cache. Best accuracy, higher complexity.

**Effort:** M
**Risk:** Low
**Pros:**
- Catches patterns that keyword bag misses
- Best cache hit rate

**Cons:**
- 2x the code, 2x the testing surface
- The keyword bag alone handles ~90% of the observed patterns
- The fallback adds latency on misses (Levenshtein scan before Oracle call)

## Recommended Approach

**Approach A: Keyword Bag Normalization.** The observed patterns all normalize cleanly with keyword bags. The main offender ("CLI tool no web UI, what should QA do?") has 10+ wording variants that all share the keywords `{cli, tool, web, ui, qa}`. No need for Levenshtein complexity when simple normalization handles the actual data. If edge cases emerge later, Approach C is an incremental upgrade.

## Implementation Plan

### New Module: `src/oracle-cache.ts`

```
Exports:
- normalizeQuestion(question: string): string
  → Strip numbers, file paths (/foo/bar/...), branch names, timestamps
  → Lowercase, split into words, sort, deduplicate
  → Return joined as space-separated string

- normalizeOptions(options: {label: string}[]): string
  → Sort option labels alphabetically, join with "|"

- computeCacheKey(normalizedQuestion: string): string
  → The cache key is the normalized question string itself (already sorted,
    deduped, lowercased). No hashing needed — the string is short enough
    for a Map key directly.
  → **Question-only keying:** Options are NOT part of the cache key.
    Rationale: DecisionOutcome in decision-outcomes.md does not store
    options, so warm-start entries would never match runtime lookups if
    options were included in the key. Question-only matching is safe
    because the same normalized question always maps to the same semantic
    decision — the options are presentation variants of the same choice.
    If two genuinely different decisions share the same normalized question
    but different option sets, the 5-hit threshold prevents false caching
    (the `chosen` answers would differ, blocking promotion).

- OracleCache class:
  - constructor(config: { minHits: number })
  - warmFromOutcomes(outcomes: DecisionOutcome[]): void
    → Parse DecisionOutcome entries (each has question, chosen, confidence,
      principle fields from the structured markdown in decision-outcomes.md)
    → Group by computeCacheKey(normalizeQuestion(question))
    → For groups where `minHits`+ entries share the same `chosen` answer,
      add to cache with that answer and the most recent principle
    → DecisionOutcome type already exists in types.ts (line 426);
      readDecisionOutcomes() already parses the markdown format
  - lookup(question: string, options: {label, description}[]): CachedDecision | null
    → computeCacheKey(normalizeQuestion(question)) → Map.get
    → Option labels are NOT part of the key (see computeCacheKey rationale)
  - record(question: string, options: {label, description}[], chosen: string): void
    → Hit counts tracked per (cacheKey, chosen) pair. Increment count for this
      specific answer. When any single answer reaches minHits, promote to cache.
      If a different answer later reaches minHits for the same key, overwrite
      the cached entry (the new answer has now proven itself). Mixed answers
      below threshold coexist without promoting.
  - invalidate(question: string, options: {label, description}[]): void
    → Remove cache entry for this key
  - stats(): { entries: number, totalHits: number, totalMisses: number }

- CachedDecision interface:
  - chosen: string
  - confidence: 10  // Always 10: empirically validated N+ times. Never triggers escalation.
  - rationale: "Cached: answered identically N times (originally P7)"
  - principle: string  // Most recent real principle from warm-start data
  - hitCount: number

Cache size: bounded by ~50 entry rolling window in decision-outcomes.md for
warm start, plus runtime accumulations bounded by daemon lifetime (one skill
run). No eviction strategy needed — Map entries are ~200 bytes each.
```

### Integration: `src/ask-handler.ts`

Add `oracleCache?: OracleCache` to `AskHandlerConfig.oracle`.

In `handleOracleBatch()` and the serial Oracle path, check cache before calling `askOracle()`:

```
const cached = oracleCache.lookup(question, options);
if (cached) {
  // Build Decision from cached result
  // Log to decisions.jsonl with source: "cache"
  // Emit oracle_cache_hit event
  // Skip Oracle API call
  return cached;
}
// ... existing Oracle flow ...
// After successful Oracle call:
oracleCache.record(question, options, result.choice);
```

**Partial-batch behavior:** When a batch of N questions arrives, check each against the cache individually. Questions with cache hits are resolved immediately. Remaining uncached questions are collected and sent as a single `askOracleBatch()` call (or `askOracle()` if only one remains). This preserves batching efficiency for novel questions while eliminating API calls for known answers. The answers array is assembled from both cached and fresh results in original question order.

### Integration: `src/reflection.ts`

After detecting a "failure" outcome for a decision, call `oracleCache.invalidate(question, options)` to prevent the failed answer from being reused. The orchestrator passes the `OracleCache` instance to the reflection runner via an invalidation callback `(question: string, options: {label: string}[]) => void`, keeping the reflection module decoupled from cache internals.

### Integration: `src/orchestrator.ts`

Initialize `OracleCache` once per skill run. Warm from `readDecisionOutcomes()`. Pass to ask-handler config. Per-skill scope matches oracle session reuse scope (orchestrator creates fresh queryFn per skill).

### Integration: `src/types.ts`

Add `OracleCacheEvent` type:
```typescript
export interface OracleCacheEvent {
  type: "oracle_cache_hit" | "oracle_cache_miss" | "oracle_cache_invalidated";
  cacheKey?: string;
  question?: string;
  chosen?: string;
  hitCount?: number;
}
```

### Integration: `src/cli.ts`

Add `formatEvent` case for `oracle_cache_hit` events in the CLI event display.

Add to daemon config:


### Normalization Strategy

Strip these patterns before keyword extraction:
- File paths: `/foo/bar/baz.ts` → removed
- Branch names: `` `main` ``, `` `garyclaw/worker-1` `` → removed
- Numbers: "2,978 vitest tests", "184 test files" → removed
- Timestamps: ISO dates → removed
- Quoted strings longer than 20 chars → removed (backticks, single quotes, or double quotes; likely file contents or error messages)
- Common filler: "the", "a", "an", "is", "are", "on", "in", "for", "to", "with", "and", "or", "your", "this", "that", "it" → removed

What remains: the semantic core. "CLI tool no web UI QA skill browser testing" → `{browser, cli, qa, skill, testing, tool, ui, web}`

**False-positive guard:** If normalization strips more than 70% of the original question's word count, the question is too variable-heavy for safe caching. Skip cache lookup and record for that question (fall through to Oracle). This prevents two genuinely different questions from collapsing to the same key when they differ only in stripped content (file paths, error messages, etc.). Example: "How should I handle the error in `src/foo.ts`?" strips to `{error, handle, should}` (3 of 9 words kept = 33%) — below threshold, skipped.

### Event Flow

```
canUseTool("AskUserQuestion", input)
  → for each question:
      → oracleCache.lookup(question, options)
        → HIT:  emit oracle_cache_hit, build Decision, skip Oracle
        → MISS: call askOracle(), emit oracle_cache_miss, oracleCache.record()
  → return updatedInput with answers
```

## Open Questions (All Resolved)

1. **Should cache warm from ALL decision-outcomes.md entries or just recent ones?** RESOLVED: Warm from all entries in the rolling window (~50). The `readDecisionOutcomes()` function in oracle-memory.ts already returns the full rolling set parsed via `parseDecisionOutcomes()`. The 5-hit threshold naturally filters: only genuinely repeated patterns qualify. Patterns below `minHits` accumulate at runtime and promote when they cross the threshold.

2. **Should cache entries carry the original OracleOutput or synthesize new ones?** RESOLVED: Synthetic rationale with original principle. Store `"Cached: answered identically N times (originally P7)"` preserving the most recent real principle from the warm-start data. This makes cache hits obvious in decision logs without requiring consumers to understand a new field.

3. **Should the Decision logged for cache hits have a `source` field?** RESOLVED: No type change. Encode in the rationale string prefix `"Cached:"` (see #2). The `Decision` interface in types.ts is imported by 15+ modules. The rationale prefix is grep-able for analytics (`grep "Cached:" decisions.jsonl`). Dashboard can count cache hits from the rationale pattern.

## Success Criteria

- Zero Oracle API calls for the "CLI tool no web UI" pattern after warm start
- oracle_cache_hit events visible in CLI and dashboard
- Cache invalidation works: inject a "failure" outcome, verify cache entry removed
- No regression in oracle session reuse (cache bypasses session, doesn't corrupt it)
- Measurable cost reduction: dashboard shows per-job Oracle spend dropping
