## Problem Statement

The bootstrap skill (`src/bootstrap.ts`) generates CLAUDE.md and TODOS.md for external repos. `analyzeBootstrapQuality()` in `evaluate.ts` checks structural completeness (section presence, framework coverage, token efficiency) but never validates that the content is actually accurate. It confirms "Architecture section exists" but not "Architecture section describes the right modules." After multiple NihontoWatch dogfood runs, there's still no formal quality gate that catches semantic inaccuracies in bootstrap output.

The consequence: bad bootstrap output propagates through the entire pipeline. If CLAUDE.md mis-describes the test framework, the implement skill writes tests wrong. If TODOS.md contains busywork items, the prioritize skill wastes budget on low-value work. Garbage in, garbage out across 4 downstream skills.

## What Makes This Cool

GaryClaw grades its own homework. After bootstrapping a repo, it cross-references the generated CLAUDE.md against the actual codebase and says "I claimed this repo uses Vitest but actually it uses Jest" or "I mentioned 12 source modules but only 8 exist." Self-correcting AI tooling.

## Constraints

- Pure TypeScript extension to `evaluate.ts`. No new modules, no new CLI commands.
- Zero Claude calls. All validation is deterministic cross-checking.
- Must extend existing `BootstrapEvaluation` interface (add new fields, don't break existing ones).
- Must work on any repo, not just NihontoWatch.
- Must be testable with synthetic data (no SDK calls in tests).
- Existing `qualityScore` (0-100) weighting must be rebalanced to include new semantic checks.

## Premises

1. `analyzeBootstrapQuality()` has the right structure but only checks syntax (section exists) not semantics (section is accurate). Extending it is the right approach vs. building something new.
2. Deterministic cross-checks (file references valid, commands exist, deps mentioned) are sufficient for a first-pass quality audit. Claude-powered semantic grading is a follow-on, not required now.
3. NihontoWatch's bootstrap output is the only real-world test case available. The design should work for any repo but can be validated against NihontoWatch specifically.
4. The output should feed into the evaluate skill's existing reporting, not create a standalone artifact.
5. This is pure TypeScript extension — no new modules, no new CLI commands, no Claude calls.

## Approaches Considered

### Approach A: Deterministic Cross-Checks (Minimal Viable)

**Summary:** Add 4-5 pure TypeScript validation functions to `evaluate.ts` that cross-reference CLAUDE.md content against the actual codebase. File reference validation, command verification, dependency accuracy, test directory claims.

**Effort:** S
**Risk:** Low
**Pros:**
- Zero Claude calls, zero cost, runs in <1 second
- Extends existing `analyzeBootstrapQuality()` with same pattern
- Immediately testable with NihontoWatch as ground truth

**Cons:**
- Can't catch semantic inaccuracies (e.g., "uses React for rendering" when it actually uses Svelte)
- Limited to pattern matching, not comprehension

**Reuses:** `analyzeBootstrapQuality()`, `detectTechStack()`, `walkFileTree()`, `computeFrameworkCoverage()`

### Approach B: Semantic Grading with Claude (Ideal Architecture)

**Summary:** After deterministic checks, run a Claude turn that reads the generated CLAUDE.md alongside the `CodebaseAnalysis` and produces a semantic accuracy score.

**Effort:** M
**Risk:** Med
**Pros:**
- Catches semantic errors that pattern matching misses
- Produces natural language quality feedback

**Cons:**
- Costs tokens per evaluation
- Harder to test deterministically

### Approach C: Ground Truth Snapshot + Regression

**Summary:** Save `CodebaseAnalysis` as `.garyclaw/bootstrap-snapshot.json`. On subsequent runs, diff against the snapshot. Track quality trends.

**Effort:** S
**Risk:** Low
**Pros:**
- Creates regression baseline automatically
- Enables quality trend tracking

**Cons:**
- Only useful after 2+ runs on the same repo
- First run has no baseline

## Recommended Approach

**Approach A: Deterministic Cross-Checks.** It directly answers the question "is bootstrap output accurate?" with zero cost and immediate testability. The other approaches are good follow-ons but this is the right first step.

## Implementation Details

### New Validation Functions

All pure TypeScript functions added to `evaluate.ts`. All sub-interfaces (`FileRefValidation`, `CommandValidation`, `TechStackValidation`, `TestClaimValidation`) go in `types.ts` alongside `BootstrapEvaluation`, since they are referenced from it.

**Missing-artifact handling:** If CLAUDE.md does not exist, `analyzeBootstrapQuality()` already returns early with `qualityScore: 0`. The new validation functions are only called when `claudeMdExists === true`. If `package.json` is missing, `validateCommands()` and `validateTechStackClaims()` return accuracy 1.0 with zero items (nothing to validate against). `validateTestClaims()` returns `frameworkMatch: true` and `null` for both claimed and actual framework when package.json is absent.

#### 1. `validateFileReferences(claudeMdContent: string, projectDir: string): FileRefValidation`

Extract file paths mentioned in CLAUDE.md and check each against actual file system. Three extraction passes:

1. **Backtick paths:** Regex `` /`([^`]+)`/g `` — primary source of file references
2. **Markdown links:** Regex `/\[([^\]]*)\]\(([^)]+)\)/g` — extract the URL target (group 2), keep only relative paths (no `http`)
3. **Table cell paths:** Regex `/\|\s*`([^`]+)`\s*\|/g` — backtick-quoted content inside table cells (already caught by pass 1, but this ensures table-specific formatting doesn't break extraction)

```typescript
interface FileRefValidation {
  totalRefs: number;       // files referenced in CLAUDE.md
  validRefs: number;       // files that actually exist
  invalidRefs: string[];   // files mentioned but missing
  accuracy: number;        // validRefs / totalRefs (0-1)
}
```

**Filtering:** After all three passes, deduplicate and filter candidates to those containing at least one `/` OR ending in a known source extension (`.ts`, `.js`, `.json`, `.md`, `.yml`, `.yaml`, `.css`, `.html`). This excludes bare command names like `` `npm` `` and config keys like `` `maxTurns` ``. Use `existsSync(join(projectDir, candidate))` for each. If `totalRefs === 0`, return `accuracy: 1.0` (nothing to validate). Note: generated/built files (e.g., `dist/index.js`) may not exist on a clean checkout. This is expected and will count as invalid. A future refinement could check `.gitignore` to distinguish "missing" from "not checked in."

#### 2. `validateCommands(claudeMdContent: string, projectDir: string): CommandValidation`

Extract shell commands from code blocks in CLAUDE.md. For `npm`/`npx`/`yarn`/`pnpm` commands, verify the script exists in package.json. For file-referencing commands, verify the file exists.

```typescript
interface CommandValidation {
  totalCommands: number;    // commands found in code blocks
  validCommands: number;    // commands that reference real scripts/files
  invalidCommands: string[]; // commands referencing missing scripts/files
  accuracy: number;         // validCommands / totalCommands (0-1)
}
```

**Detection pattern:** Parse fenced code blocks (` ```bash `, ` ```sh `, ` ``` `). Extract lines matching `npm run X`, `npx tsx X`, `yarn X`, `pnpm X` patterns. Only count commands with a verifiable reference (a script name or file path). Commands with no verifiable reference (e.g., `npm install`, `git clone`, `cd foo`) are excluded from both `totalCommands` and `validCommands`. Cross-reference script names against `package.json` `scripts` keys. Cross-reference file paths against `existsSync()`. If `package.json` is missing, skip script validation and only validate file-referencing commands. If `totalCommands === 0`, return `accuracy: 1.0`.

#### 3. `validateTechStackClaims(claudeMdContent: string, projectDir: string): TechStackValidation`

Run `detectTechStack(projectDir, files, packageJsonContent)` from `bootstrap.ts` against the actual codebase. Requires calling `walkFileTree(projectDir)` first for the file list, and reading `package.json` if it exists. Compare detected stack against what CLAUDE.md claims. Find frameworks/tools mentioned in CLAUDE.md that aren't in the real stack, and real stack items missing from CLAUDE.md.

```typescript
interface TechStackValidation {
  detectedStack: string[];    // actual tech stack from codebase
  claimedStack: string[];     // tech mentioned in CLAUDE.md
  missingFromDoc: string[];   // in codebase but not in CLAUDE.md
  phantomClaims: string[];    // in CLAUDE.md but not in codebase
  accuracy: number;           // overlap / union (Jaccard similarity)
}
```

**Detection pattern:** For ground truth, call `detectTechStack(projectDir, files, packageJsonContent)` from `bootstrap.ts` (already exported, accepts a file list from `walkFileTree()` and optional package.json content). For claimed stack, scan CLAUDE.md for names from the same canonical list used by `detectTechStack()`: the language keywords (`typescript`, `javascript`, `python`, etc.) and framework names from `KNOWN_FRAMEWORKS` in `evaluate.ts` (line ~44, a `ReadonlyMap<string, readonly string[]>` with 30+ entries like React, Next.js, Vitest, Prisma, etc.). Case-insensitive match. Jaccard similarity = |intersection| / |union|. Edge cases: if both sets are empty, return `accuracy: 1.0`. If one set is empty and the other is not, `accuracy: 0.0` (union is non-empty, intersection is empty). This is correct: if the codebase uses 5 frameworks and CLAUDE.md mentions none, that's a real failure. The `walkFileTree()` call can be cached if `analyzeBootstrapQuality()` calls multiple validators. For a first pass, calling it once and passing the result to `validateTechStackClaims()` is sufficient.

#### 4. `validateTestClaims(claudeMdContent: string, projectDir: string): TestClaimValidation`

If CLAUDE.md mentions test counts, test frameworks, or test directories, verify against actual test files on disk.

```typescript
interface TestClaimValidation {
  claimedTestDir: string | null;      // test dir mentioned in CLAUDE.md
  actualTestDir: string | null;       // test dir found on disk
  testDirMatch: boolean;
  claimedTestFramework: string | null; // e.g., "Vitest", "Jest"
  actualTestFramework: string | null;  // from package.json devDependencies
  frameworkMatch: boolean;
  testFileCount: number;              // actual test files found
}
```

**Detection pattern:** Look for "test" directory mentions, framework names (Vitest/Jest/Mocha/Playwright), test file patterns (`*.test.ts`, `*.spec.ts`). Cross-reference against `findTestDir()` from `bootstrap.ts` and package.json devDependencies.

### Extended BootstrapEvaluation Interface

Add new fields to `BootstrapEvaluation` in `types.ts`:

```typescript
export interface BootstrapEvaluation {
  // ... existing fields ...

  // New semantic validation fields
  fileRefValidation?: FileRefValidation;
  commandValidation?: CommandValidation;
  techStackValidation?: TechStackValidation;
  testClaimValidation?: TestClaimValidation;
  semanticAccuracy?: number;  // 0-100, weighted composite of all validations (optional for backward compat)
}
```

### Rebalanced Quality Score

Current weighting (0-100):
- Structural completeness: 40 pts (section presence)
- Factual accuracy: 30 pts (framework coverage)
- TODOS.md viability: 20 pts (items above threshold)
- Token efficiency: 10 pts

New weighting (0-100):
- Structural completeness: 20 pts (section presence) — halved, less important now
- Semantic accuracy: 35 pts (new composite: file refs 10, commands 5, tech stack 15, test claims 5)
- Framework coverage: 20 pts (known frameworks mentioned in CLAUDE.md vs. package.json deps) — renamed from "factual accuracy" to avoid confusion with "semantic accuracy"
- TODOS.md viability: 15 pts (items above threshold)
- Token efficiency: 10 pts

The semantic accuracy score is the weighted average of the four validation accuracies.

### Integration Points

1. **`analyzeBootstrapQuality()`** — extend to call all 4 validation functions and compute `semanticAccuracy`
2. **`formatEvaluationReport()`** — add a "Bootstrap Semantic Validation" section to the markdown report showing each validation result
3. **`buildEvaluatePrompt()`** — include semantic validation results in the prompt context so Claude can comment on accuracy gaps
4. **`BootstrapEvaluation` type** — add optional fields (backward-compatible, existing code won't break)
