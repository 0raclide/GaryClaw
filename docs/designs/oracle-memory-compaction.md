## Problem Statement

`decision-outcomes.md` is 34K bytes (451 lines, 50 entries) and growing ~1K/day. Each entry's Question field contains 300-500 chars of full Oracle prompt context (verbatim question text with branch names, file paths, commit SHAs, and long descriptions). This file feeds into every Oracle call via four consumers:

1. **Memory injection** (`readOracleMemory()`) — raw content injected into Oracle prompt
2. **Cache warm-start** (`warmFromOutcomes()`) — parses all entries at skill startup
3. **Reflection** (`readDecisionOutcomes()`) — reads all entries for outcome mapping
4. **Pipeline counting** (`countPipelineOutcomes()`) — regex scans for "PIPELINE OUTCOME:" lines

The 12K token budget (ORACLE_MEMORY_BUDGETS.decisionOutcomes = 12,000) means ~48K chars max. At current growth rate, truncation from the front starts within weeks, silently dropping the oldest pattern data.

## What Makes This Cool

The normalization algorithm already exists in `oracle-cache.ts` (`normalizeQuestion()`). It strips file paths, timestamps, numbers, quoted strings, and filler words, reducing a 400-char question to a 40-char keyword bag. Reusing it for compaction means we get 10x size reduction per entry with zero new algorithm design. The system literally gets more efficient the longer it runs — every batch of 50 entries gets compacted down to ~10% of its original size when it rolls over.

## Constraints

- Must not break any of the four consumers listed above
- Must not change the `DecisionOutcome` type interface
- Must reuse existing `normalizeQuestion()` — no new normalization code
- Must preserve full question text for recent entries (human readability for debugging)
- Must be backward-compatible with existing decision-outcomes.md files (parse both compacted and uncompacted)

## Premises

1. **Growth is in entry verbosity, not count.** Rolling window is capped at 50 entries. Each entry is ~680 chars avg because Question fields contain full Oracle prompt context. Compacting Question text gives 5-10x reduction.
2. **`normalizeQuestion()` is the right algorithm.** Already battle-tested in oracle-cache.ts with 41 tests. Strips variable tokens, reduces to sorted keyword bag.
3. **Age-gated compaction preserves debugging.** Last 10 entries keep full text. Entries 11-50 get compacted. When entries roll to Patterns section, they're already compact.
4. **Patterns section is already compact but unbounded.** `summarizeOutcomePatterns()` produces ~5-10 lines per batch. Needs a fixed cap to be truly bounded.
5. **All consumers tolerate compacted questions.** Cache warm-start normalizes anyway. Pipeline counting matches on "PIPELINE OUTCOME:" prefix. Reflection matches by decisionId. Memory injection benefits from smaller content.

## Approaches Considered

### Approach A: Age-Gated Question Compaction Only
- Add `compactDecisionOutcomes()` to oracle-memory.ts
- On rolling write, entries at positions 0 through N-10 get Question replaced with `normalizeQuestion(question)`
- Prefix compacted questions with `[compact] ` marker for parseability
- ~40 lines new code, 8 tests
- Pros: Smallest diff, ships fastest, 5-8x reduction
- Cons: Patterns section still unbounded (minor risk, currently ~5 lines per batch)

### Approach B: Compaction + Patterns Budget Cap
- Everything in A, plus cap Patterns section to PATTERNS_BUDGET_TOKENS (2,000 tokens = ~8K chars)
- When Patterns exceeds budget, keep only the most recent summary block
- ~80 lines, 15 tests
- Pros: Truly bounded forever, handles years of autonomous operation
- Cons: Slightly more complex Patterns truncation

### Approach C: Full Entry Dedup via Cache Keys
- Use `computeCacheKey()` to detect duplicate normalized entries
- Merge duplicates into single entries with `count` field
- Pros: Could reduce 50→15 unique entries if many repeat
- Cons: Changes data model, breaks `parseDecisionOutcomes` contract, more invasive, untested interaction with reflection outcome mapping

## Recommended Approach

**Approach B: Compaction + Patterns Budget Cap**

Rationale: The Question compaction is where 90% of the value lives, but the Patterns cap is cheap insurance (5 lines of code) that guarantees the file is bounded regardless of how long the daemon runs. Total effort is ~80 lines + 12 tests, all within blast radius of existing code.

### Implementation Plan

#### 1. Add `compactOutcomeEntry()` to oracle-memory.ts (~15 lines)

```typescript
import { normalizeQuestion } from "./oracle-cache.js";

const COMPACT_MARKER = "[compact] ";

/**
 * Strip the [compact] prefix from a question string if present.
 * Used by warmFromOutcomes() to get a clean keyword bag for cache key computation.
 */
export function stripCompactMarker(question: string): string {
  return question.startsWith(COMPACT_MARKER) ? question.slice(COMPACT_MARKER.length) : question;
}

/**
 * Compact a decision outcome entry by normalizing its question text.
 * Compacted entries are prefixed with "[compact] " for parseability.
 *
 * Idempotent: entries already prefixed with COMPACT_MARKER are returned unchanged.
 * parseDecisionOutcomes() preserves the prefix in the question field, so
 * read -> parse -> compact -> write produces stable output.
 */
export function compactOutcomeEntry(outcome: DecisionOutcome): DecisionOutcome {
  if (outcome.question.startsWith(COMPACT_MARKER)) return outcome; // already compacted
  return {
    ...outcome,
    question: COMPACT_MARKER + normalizeQuestion(outcome.question),
  };
}
```

**Round-trip invariant:** `parseDecisionOutcomes()` preserves the `[compact]` prefix in the `question` field. `compactOutcomeEntry()` checks for this prefix and returns unchanged, so read -> parse -> compact -> write is idempotent.

**Cache key interaction:** `warmFromOutcomes()` in oracle-cache.ts calls `normalizeQuestion(outcome.question)` on parsed entries. For compacted entries, the question is already a normalized keyword bag but with the `[compact]` prefix. Calling `normalizeQuestion()` on `"[compact] approach cli garyclaw..."` would include "compact" as a keyword, producing a different cache key than normalizing the original question. To fix this, `warmFromOutcomes()` must call `stripCompactMarker()` before normalizing. This is a one-line change in oracle-cache.ts.

**Module dependency:** This introduces a dependency from `oracle-memory.ts -> oracle-cache.ts` (for `normalizeQuestion`). The reverse direction must remain indirect (wired via orchestrator.ts) to avoid circular imports. `oracle-cache.ts` gains a dependency on `oracle-memory.ts` for `stripCompactMarker`, but this can be avoided by having `warmFromOutcomes()` do its own inline prefix strip (or by moving `stripCompactMarker` to a shared utils location). Recommended: inline the strip in `warmFromOutcomes()` to keep the dependency one-directional.

#### 2. Add `compactDecisionOutcomes()` to oracle-memory.ts (~20 lines)

```typescript
const RECENT_KEEP_FULL = 10; // last 10 entries keep full question text

/**
 * Compact older entries in a decision outcomes array.
 * Last RECENT_KEEP_FULL entries keep full question text.
 * Older entries get question normalized to keyword bag via normalizeQuestion().
 */
export function compactDecisionOutcomes(outcomes: DecisionOutcome[]): DecisionOutcome[] {
  if (outcomes.length <= RECENT_KEEP_FULL) return outcomes;

  const cutoff = outcomes.length - RECENT_KEEP_FULL;
  const compacted = outcomes.slice(0, cutoff).map(compactOutcomeEntry);
  const recent = outcomes.slice(cutoff);
  return [...compacted, ...recent];
}
```

#### 3. Wire into `writeDecisionOutcomesRolling()` (~3 lines changed)

In the existing function, compact entries before writing:

```typescript
export function writeDecisionOutcomesRolling(
  config: OracleMemoryConfig,
  outcomes: DecisionOutcome[],
): void {
  const MAX_ENTRIES = 50;
  const compacted = compactDecisionOutcomes(outcomes); // NEW

  if (compacted.length <= MAX_ENTRIES) {
    writeDecisionOutcomes(config, formatDecisionOutcomes(compacted)); // changed
    return;
  }

  const recent = compacted.slice(-MAX_ENTRIES);
  const older = compacted.slice(0, -MAX_ENTRIES);
  const patterns = summarizeOutcomePatterns(older);

  // Cap patterns section to budget (truncateToTokenBudget keeps the tail,
  // which works for Patterns because the most recent summary block appears last)
  const cappedPatterns = truncateToTokenBudget(patterns, PATTERNS_BUDGET_TOKENS); // NEW

  const content = `# Decision Outcomes\n\n## Patterns (from ${older.length} older decisions)\n${cappedPatterns}\n\n## Recent Outcomes (${recent.length})\n${formatOutcomeEntries(recent)}`;
  writeDecisionOutcomes(config, content);
}
```

### Size Impact

Current state (50 entries, no compaction):
- Avg question length: ~350 chars
- Per-entry overhead (timestamp, chosen, confidence, principle, outcome, job): ~330 chars
- Avg entry size: ~680 chars
- Total: 50 × 680 = ~34K chars = ~8.5K tokens

After compaction (10 full + 40 compacted):
- Full entries: 10 × 680 = 6,800 chars
- Compacted entries: 40 × 370 = 14,800 chars (question drops from ~350 to ~40, overhead stays ~330)
- Total: ~21,600 chars = ~5.4K tokens

**36% reduction in file size. Bounded at ~7.4K tokens permanently.**

With Patterns cap (2K tokens) + Recent (5.4K tokens) = 7.4K tokens max. That's 62% of the 12K budget, leaving permanent headroom.
