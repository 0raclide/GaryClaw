## Problem Statement

SDK-bug is the #1 failure category in GaryClaw (13 failures: 8 in prioritize, 3 in implement, 2 in qa). These are transient API/SDK errors — protocol errors, stream disconnects, 429s, 503s — that crash the `for await (const msg of segment)` loop in `orchestrator.ts` (line 363). When this happens, the entire job fails, wasting the pipeline cost up to that point ($2-5 per failure).

The error path today:
1. SDK throws during segment iteration (line 363)
2. Only `PerJobCostExceededError` is caught (line 474) — everything else propagates
3. Error hits job-runner's main catch, which classifies via `classifyError()`
4. Job marked "failed" with `retryable: true` for sdk-bug category
5. The `retryable` flag is set but nothing actually retries — it's only used for crash-recovery re-queuing on daemon restart

The existing failure taxonomy already correctly classifies these errors:
- `sdk-bug`: protocol error, stream error, chunk parsing, stack traces in `node_modules/@anthropic-ai`
- `infra-issue`: ECONNRESET, ETIMEDOUT, 429, 503, socket hang up

Both categories have `retryable: true`. The classification exists. The retry doesn't.

## What Makes This Cool

Overnight runs become dramatically more reliable. Instead of waking up to 5 failed jobs from transient API hiccups, you wake up to 5 completed jobs that each silently recovered from a blip. The daemon absorbs network instability the way a good server absorbs packet loss — invisibly.

At $2-5 per wasted pipeline, 13 failures = $26-65 burned. This fix costs nothing at runtime and saves that every week.

## Constraints

- XS effort: ~50 lines of new code in orchestrator.ts, ~5 lines in failure-taxonomy.ts
- No new modules, no cross-module interface changes
- Must preserve all accumulated state (monitor, issue tracker, checkpoints, codebase summary)
- Must not retry non-transient errors (budget exceeded, auth failures, project bugs)
- Must emit observable events so dashboard/logs can track retries
- Retry backoff must not exceed 60s (the API will either recover in 30s or it's down)

## Premises

1. **Transient SDK errors are recoverable if retried after a brief wait.** The 13 sdk-bug failures are network/stream issues, not logic bugs. A 30s wait lets the API stabilize.
2. **The retry belongs in the orchestrator's segment loop**, not the job runner. The segment has accumulated state (monitor, issue tracker, observations) that should be preserved, not rebuilt from scratch.
3. **A single retry with 30s backoff is sufficient.** If the same error recurs after 30s, it's not transient — propagate normally. The job runner's crash recovery handles persistent outages.
4. **The existing `classifyError()` already identifies retryable errors.** We reuse it as-is with a thin `isTransientError()` wrapper.
5. **This change touches only `orchestrator.ts`** (retry logic) and `failure-taxonomy.ts` (export `isTransientError` helper). One new event type in `types.ts`, one new format case in `cli.ts`.

## Approaches Considered

### Approach A: Segment-Level Try-Catch Wrapper (CHOSEN)

Wrap the `for await` loop in a retry helper. On error, call `classifyError()`. If retryable, wait 30s, emit a retry event, and restart the segment with `resume: sessionId` (continuing the existing SDK session). If not retryable, propagate normally.

- **Effort:** XS (~50 lines orchestrator.ts, ~5 lines failure-taxonomy.ts, ~60 lines tests)
- **Risk:** Low
- **Pros:** Minimal diff, reuses existing classification, preserves all state, observable
- **Cons:** Retries with same session — if session state is corrupted, retry fails the same way (but then propagates normally, which is correct)

### Approach B: Fresh Segment on Retry

Same retry wrapper, but on retry clear sessionId and start a fresh segment with relay prompt. Avoids corrupted-session issues.

- **Effort:** S (~80 lines, needs mid-segment relay prompt generation)
- **Risk:** Medium
- **Pros:** Clean session avoids session-corruption edge case
- **Cons:** Loses in-progress turns, more complex, needs relay prompt generation outside normal relay boundaries

## Recommended Approach

**Approach A.** It handles 90%+ of transient errors (network/stream) with minimal code. The corrupted-session edge case (Approach B) is rare — if it shows up in failure data, add fresh-segment retry later.

## Implementation

### 1. New helper in `failure-taxonomy.ts`

```typescript
/**
 * Check if an error is transient and safe to retry at the segment level.
 * Uses classifyError() under the hood — retryable sdk-bug and infra-issue
 * categories qualify. Budget, auth, and project errors do not.
 */
export function isTransientError(err: unknown): boolean {
  const { category, retryable } = classifyError(err);
  // Only retry transient categories — not auth (handled by rate-limit hold)
  // or merge-failed (thrown post-segment, never hits this path).
  if (!retryable) return false;
  return category === "sdk-bug" || category === "infra-issue";
}
```

### 2. Retry wrapper in `orchestrator.ts`

Inside the segment loop (around lines 360-490), wrap the `for await` in a try-catch:

```typescript
const MAX_SEGMENT_RETRIES = 1;
const SEGMENT_RETRY_DELAY_MS = 30_000;

// ... existing segment setup ...

let segmentRetries = 0;
let segmentSucceeded = false;

while (!segmentSucceeded) {
  try {
    const segment = startSegment({
      prompt: segmentIndex === 0 ? currentPrompt : "Continue.",
      maxTurns: adaptiveMaxTurns,
      cwd: config.projectDir,
      env: config.env,
      settingSources: config.settingSources,
      canUseTool: askHandler.canUseTool,
      ...(segmentIndex > 0 && sessionId ? { resume: sessionId } : {}),
    });

    // ... existing for-await loop (unchanged) ...

    segmentSucceeded = true; // Normal exit from for-await
  } catch (err) {
    if (err instanceof PerJobCostExceededError) {
      // Budget errors are never retried — save checkpoint and propagate
      // (existing behavior, unchanged)
      throw err;
    }

    if (segmentRetries < MAX_SEGMENT_RETRIES && isTransientError(err)) {
      segmentRetries++;
      const errMsg = err instanceof Error ? err.message : String(err);
      callbacks.onEvent({
        type: "segment_retry",
        sessionIndex,
        segmentIndex,
        attempt: segmentRetries,
        maxRetries: MAX_SEGMENT_RETRIES,
        error: errMsg,
        delayMs: SEGMENT_RETRY_DELAY_MS,
      });

      // Wait before retry (abort-aware)
      if (config.abortSignal?.aborted) throw err;
      await new Promise<void>((resolve) => {
        const timer = setTimeout(resolve, SEGMENT_RETRY_DELAY_MS);
        config.abortSignal?.addEventListener("abort", () => {
          clearTimeout(timer);
          resolve();
        }, { once: true });
      });

      // Check abort after sleep
      if (config.abortSignal?.aborted) throw err;

      // Retry: loop back to startSegment with same parameters
      continue;
    }

    // Non-transient or exhausted retries — propagate
    throw err;
  }
}
```

### 3. New event type

Add `segment_retry` to the OrchestratorEvent union in `types.ts`:

```typescript
| { type: "segment_retry"; sessionIndex: number; segmentIndex: number;
    attempt: number; maxRetries: number; error: string; delayMs: number }
```

### 4. CLI display

In `cli.ts` `formatEvent()`, add a case:

```typescript
case "segment_retry":
  return `\x1b[33m⟳ Retrying segment ${e.segmentIndex} (attempt ${e.attempt}/${e.maxRetries}) after transient error: ${truncate(e.error, 60)}. Waiting ${e.delayMs / 1000}s...\x1b[0m`;
```

### 5. Constants

Export `MAX_SEGMENT_RETRIES` and `SEGMENT_RETRY_DELAY_MS` from orchestrator.ts for testability.

### Key Behaviors

- **Preserves state:** The retry loop is inside the segment loop. Monitor, issue tracker, pending observations, checkpoints — all remain valid. Only the current segment's iteration restarts.
- **Resume on retry:** If `sessionId` is set (segment > 0 or previous segment set it), the retry resumes the existing SDK session. The SDK's resume protocol replays from the server's last acknowledged state, so partially-processed turns are not duplicated. Tool calls within a turn are atomic from the SDK's perspective — a stream error mid-turn means the turn was not committed server-side. If the session itself is corrupted (rare), the resume will fail and propagate normally.
- **Budget-safe:** `PerJobCostExceededError` is explicitly excluded from retry (thrown before the retry check). Budget tracking continues accurately across retries.
- **Observable:** The `segment_retry` event lets dashboard track retry frequency. If retries spike, something bigger is wrong.
- **Abort-safe:** The abort signal check at line 312 still runs before each segment attempt. Additional abort checks before and after the sleep.
- **No double-checkpoint:** If the segment throws before any result message, no checkpoint was written. The retry starts clean.

### What Does NOT Change

- `failure-taxonomy.ts` rules — no new categories, no reordering
- `job-runner.ts` error handling — still catches propagated errors normally
- `types.ts` FailureCategory — no new categories
- `pipeline.ts` — pipeline resume is a separate mechanism (for daemon crashes, not segment errors)
- All existing tests — behavior is additive
