# Design: Dogfood Bootstrap — Cold Start GaryClaw on External Repos

## Problem Statement

GaryClaw's backlog is exhausted. Health score 97/100, 1669 tests passing, Oracle accuracy 100% across 232 decisions. No item scores above 5.0/10. The system is complete but untested against real-world codebases it wasn't built alongside.

The prioritize skill, pipeline runner, implement skill, QA skill, and Oracle all assume the target project has CLAUDE.md (project context), TODOS.md (backlog), and design docs. This is the cold-start problem: GaryClaw can't operate on a repo that doesn't already have GaryClaw artifacts.

The fix: a `garyclaw dogfood <path>` command that bootstraps those artifacts by analyzing the target codebase, then runs the standard `prioritize -> implement -> qa` pipeline. This closes the self-improvement loop and generates the real-world feedback needed to discover GaryClaw's next wave of high-impact improvements.

## What Makes This Cool

You point GaryClaw at any repo and go to sleep. It reads the code, figures out what's wrong, writes a backlog, picks the top item, implements it, QA-tests the result, and produces a report. The entire prioritize-implement-qa cycle that currently requires hand-authored CLAUDE.md and TODOS.md now works from a cold start.

The deeper payoff: every dogfood run generates oracle decisions, codebase summaries, and reflection data that feed back into GaryClaw's own improvement. The system literally gets smarter about how to work on unfamiliar codebases.

## Constraints

- Must not modify the target repo's existing files (CLAUDE.md, TODOS.md) if they already exist. Bootstrap only fills gaps.
- Must work within existing daemon budget system (global budget, per-job cost limits).
- Must produce artifacts the existing pipeline can consume. No new artifact formats.
- The bootstrap step runs as a regular GaryClaw skill, composable with the existing pipeline.
- Target repo must be a local git repo (no direct GitHub URL cloning in v1).
- Bootstrap prompt must fit within a single segment. The `analyzeCodebase()` function runs in TypeScript (not as Claude tool calls), pre-gathering file listings and key file contents with a **50K token budget** for injected content. Tiered gathering strategy: (1) always include package.json, tsconfig.json, CI config, lock file metadata (~2K tokens), (2) README up to 5K tokens, (3) file tree listing up to 5K tokens, (4) sample source files up to remaining budget, prioritized by: test files first (reveal project structure), then entry points, then largest files. If total exceeds budget, truncate lowest-priority files. Claude only needs to synthesize and write the two output files (~5-10 turns).
- Degenerate repos (binary-heavy, zero source files, unsupported languages) should produce a minimal CLAUDE.md with what's known and an empty TODOS.md with a note explaining why no items were generated. The skill should not error. For permission errors on individual files, skip silently and note in analysis. For symlink loops, use `fs.realpathSync` with cycle detection. For repos with >1000 source files, use the sampling strategy above (never read all files).
- Idempotency gate is in TypeScript, not the prompt: `buildBootstrapPrompt()` checks for existing CLAUDE.md/TODOS.md and omits write instructions for files that already exist. Claude never receives instructions to write files that are already present.

## Premises

1. GaryClaw's backlog is genuinely exhausted. The remaining P3-P5 items aren't worth prioritizing over new discovery work.
2. The highest-value next step is dogfooding on an external target project, not adding more features to GaryClaw itself.
3. The bootstrap problem (cold start on a new repo without CLAUDE.md/TODOS.md) is the primary blocker. The pipeline primitives already work.
4. The dogfood target should be a real, maintained repo with actual issues to find.
5. Distribution is not a concern yet. This is a CLI command for the project owner.

## Approaches Considered

### Approach A: Bootstrap Skill (Minimal Viable)

**Summary:** Add a single new skill (`bootstrap`) that generates CLAUDE.md and TODOS.md for any repo by analyzing its structure, then chains into the existing pipeline.

**Effort:** S
**Risk:** Low
**Pros:**
- Reuses 100% of existing pipeline, orchestrator, oracle, and QA infrastructure
- Single new module (`src/bootstrap.ts`) following the implement.ts/prioritize.ts pattern
- `garyclaw run bootstrap prioritize implement qa --autonomous` works immediately
**Cons:**
- Bootstrap quality depends entirely on Claude's ability to read a codebase in one pass
- Generated TODOS.md may be shallow (obvious lint issues, not architectural problems)
- No special handling for different project types (web app vs. CLI vs. library)
**Reuses:** pipeline.ts skill chaining, prioritize.ts prompt pattern, implement.ts design doc discovery

### Approach B: Two-Phase Bootstrap with QA Pre-Scan

**Summary:** Phase 1 runs QA on the raw repo (no CLAUDE.md needed, QA skill works on any codebase). Phase 2 uses QA findings + codebase analysis to generate richer CLAUDE.md and TODOS.md. The QA pre-scan provides real evidence for the backlog instead of guessing from structure alone.

**Effort:** M
**Risk:** Medium
**Pros:**
- TODOS.md is grounded in actual QA findings, not structural guessing
- Two-pass approach catches more: structure analysis + behavioral testing
- QA pre-scan can discover test gaps, broken builds, security issues
**Cons:**
- QA on a cold repo may generate noise (no context about intentional tradeoffs)
- Two full pipeline runs (QA + bootstrap) doubles the context/cost budget
- QA skill's prompts assume CLAUDE.md exists for project context
**Reuses:** qa skill (with relaxed CLAUDE.md requirement), pipeline chaining

### Approach C: Dogfood Command with Smart Target Selection

**Summary:** A `garyclaw dogfood` meta-command that doesn't just bootstrap one repo, but picks the best target from a set of local repos. Scans `~/Projects/` (or a configured path), scores each repo by: last commit recency, test coverage signal, presence of CI, number of open issues. Picks the highest-signal target, bootstraps it, and runs the full pipeline. After each dogfood run, the target selection improves based on which repos yielded the most useful findings.

**Effort:** L
**Risk:** High
**Pros:**
- Fully autonomous end-to-end: GaryClaw picks what to work on AND does the work
- Target selection learning closes the feedback loop
- "Set it and forget it" daemon mode for continuous improvement across multiple repos
**Cons:**
- Over-engineered for the current need (find ONE target, learn from it)
- Target scoring heuristics are speculative without dogfood data
- Multi-repo management adds complexity to daemon, worktree, and budget systems
**Reuses:** daemon registry (multi-instance), oracle memory (learning), prioritize (scoring)

## Recommended Approach

**Approach A: Bootstrap Skill.** It's the smallest diff that unlocks the entire dogfood loop. One new module, composable with the existing pipeline. The first dogfood runs will reveal whether the generated TODOS.md is shallow (driving Approach B as a follow-up) or whether target selection matters (driving Approach C). Build the simple thing, learn from it, iterate.

Approach B's QA pre-scan is a natural follow-up if bootstrap quality is insufficient, and it can be added as a pipeline prefix (`garyclaw run qa bootstrap prioritize implement qa`) without any new infrastructure.

## Open Questions

1. **CLAUDE.md generation quality:** Can Claude produce a useful CLAUDE.md from a single-pass codebase analysis? The existing CLAUDE.md for GaryClaw is extremely detailed and was hand-curated. The bootstrap version will be thinner. Is "thin but correct" sufficient for the pipeline to work?

2. **TODOS.md depth:** Structural analysis finds obvious things (missing tests, no CI, outdated deps). Architectural issues (wrong abstraction, performance bottleneck, security gap) require deeper understanding. How deep does the bootstrap need to go?

3. **First dogfood target:** Which repo should be the first target? Ideal properties: TypeScript or JavaScript (same ecosystem), has tests (so QA can compare), actively maintained (so improvements are useful), medium complexity (100-500 files).

4. ~~**Bootstrap idempotency:**~~ Resolved: TypeScript-level gate in `buildBootstrapPrompt()` checks for existing files and omits write instructions. No ambiguity.

## Success Criteria

- `garyclaw run bootstrap prioritize implement qa --autonomous --project-dir /path/to/target` completes without errors
- Generated CLAUDE.md is accurate enough for the pipeline to produce meaningful results (no hallucinated file paths, correct tech stack identification)
- Generated TODOS.md contains at least 3 actionable items scored above 5.0/10 by prioritize
- At least 1 implement + QA cycle produces a real improvement (test added, bug fixed, or code quality improved)
- The dogfood run generates oracle memory entries that are useful for future runs on the same repo

## Distribution Plan

Not applicable. This is a CLI command for the project owner, extending the existing `garyclaw run` interface. No new distribution channel needed.

## Implementation Order

### Step 1: `src/bootstrap.ts` — Core Module
Create the bootstrap module. The pattern: export a `buildBootstrapPrompt()` function that returns a string prompt. pipeline.ts dispatches it via `runSkillWithPrompt()` with a canUseTool limited to Write/Read. Same pattern as implement.ts and prioritize.ts.

**`CodebaseAnalysis` interface:**
```typescript
export interface CodebaseAnalysis {
  techStack: string[];              // e.g. ["typescript", "node", "vitest"]
  packageJson: string | null;       // raw content, truncated to 2K tokens
  tsConfig: string | null;          // raw content
  ciConfig: string | null;          // .github/workflows/*.yml or similar
  readme: string | null;            // truncated to 5K tokens
  fileTree: string;                 // `find`-style listing, truncated to 5K tokens
  testDir: string | null;           // path to test directory if found
  sourceFiles: Record<string, string>; // sampled source file contents, within token budget
  existingClaudeMd: boolean;        // true if CLAUDE.md already exists (skip generation)
  existingTodosMd: boolean;         // true if TODOS.md already exists (skip generation)
  totalSourceFiles: number;         // count of all source files found
  totalTokensGathered: number;      // tokens used by gathered content
}
```

Key functions:
- `analyzeCodebase(projectDir: string): Promise<CodebaseAnalysis>`: **runs in TypeScript** (not as Claude tool calls). Globs for key files with the tiered gathering strategy (see Constraints). 50K token budget. Returns the `CodebaseAnalysis` object. Handles permission errors (skip + note), symlink cycles (`realpathSync`), and large repos (sampling, never reads all files).
- `buildBootstrapPrompt(config: GaryClawConfig, prevSkills: PipelineSkillEntry[], projectDir: string): Promise<string>`: calls `analyzeCodebase()`, then assembles the analysis into a prompt. If `existingClaudeMd` is true, omits CLAUDE.md write instructions. If `existingTodosMd` is true, omits TODOS.md write instructions. If both exist, returns a no-op prompt that tells Claude "nothing to bootstrap."

**TODOS.md format requirement:** The prompt must instruct Claude to generate TODOS.md entries in `parseTodoItems()` format:
```markdown
## P{N}: {Title}

**What:** {description}
**Why:** {reason}
**Effort:** {XS|S|M|L|XL}
**Depends on:** {deps or "Nothing"}
```
This format is parsed by `parseTodoItems()` in prioritize.ts. Each `## P{N}:` heading becomes a `TodoItem`. Without this exact format, the prioritize skill fails silently.

### Step 2: Pipeline Integration
Add "bootstrap" case to pipeline.ts skill dispatch (~line 290), following the prioritize pattern: build prompt via `buildBootstrapPrompt()`, dispatch via `runSkillWithPrompt()`. The bootstrap skill writes CLAUDE.md and TODOS.md to the target project directory. Subsequent pipeline skills (prioritize, implement) read these files normally.

### Step 3: Unit Tests
- `test/bootstrap.test.ts`: `analyzeCodebase` with mock filesystems, `buildBootstrapPrompt` output validation, edge cases (existing CLAUDE.md, empty repo, monorepo, binary-heavy repo, zero source files)
- Integration: verify pipeline dispatch works with bootstrap skill

### Step 4: CLI + Docs
No new CLI flags needed. `garyclaw run bootstrap` works via existing skill routing. Update CLAUDE.md usage section to document the `bootstrap prioritize implement qa` pipeline.

### Step 5: First Dogfood Run
Pick target repo. Run `garyclaw run bootstrap prioritize implement qa --autonomous --project-dir <target>`. Analyze results: did CLAUDE.md/TODOS.md produce useful prioritize scores? Did implement produce valid changes? Did QA catch real issues? Feed learnings back into GaryClaw's own TODOS.md as new P2/P3 items.

## What I noticed about how you think

- The priority pick analysis was thorough and honest. Scoring every item, showing the math, and concluding "nothing crosses 5.0" takes discipline. Most people would have inflated a score to have something to build.
- "The backlog needs new items from dogfooding GaryClaw on a real target project" is the right conclusion. The system is healthy. The next risk isn't code quality, it's whether the tool actually works on codebases it wasn't co-developed with.
- 1669 tests, 97/100 health score, 100% oracle accuracy across 232 decisions. This is a tool that was built with craft, not speed. The test-to-source ratio is unusually high for a side project.
- The gap between "works on GaryClaw's own repo" and "works on any repo" is the real test. Everything so far has been validated in a controlled environment where the CLAUDE.md was hand-written by someone who knows every module. The bootstrap skill is where the rubber meets the road.
