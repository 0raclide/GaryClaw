# Design: Dashboard Adaptive Turns Stats

Generated by /office-hours on 2026-03-28
Branch: main
Repo: 0raclide/GaryClaw
Status: APPROVED
Mode: Builder

## Problem Statement

The adaptive maxTurns feature shipped (commits 5b251aa..558986f, 26+7 tests) and emits `adaptive_turns` events every segment with `maxTurns`, `reason`, `sessionIndex`, and `segmentIndex`. The dogfood dashboard aggregates job/oracle/budget stats but has zero visibility into adaptive turns behavior. You can't tell if the 2.5x heavy tool multiplier is calibrated, whether the min/max clamps are too aggressive, or how often segments fall back to the configured default.

This was called out as Open Question #2 in the adaptive-max-turns design doc: "Should adaptive turns be reported in the dashboard? Probably yes (avg adaptive turns per job), but defer to a separate change."

## What Makes This Cool

The observability loop closes. After this ships, GaryClaw can look at its own dashboard and see: "Heavy tool multiplier fired 12 times today, average predicted turns was 5.2, and 30% of segments used the fallback default." That's the data you need to decide if 2.5x should be 3x, or if the min clamp of 3 is too low. Self-aware automation.

## Constraints

- XS effort. Pure wiring, no new architectural patterns.
- No new files. Extend existing types, functions, and tests.
- Must follow existing dashboard patterns (dedicated type, aggregation function, format section).
- Collection must not add overhead to the hot path (event callback only).
- Backward compatible: dashboard must render cleanly when no adaptive turns data exists (pre-existing jobs, `--no-adaptive` jobs).

## Premises

1. The `adaptive_turns` event already carries all needed data. No new event types required.
2. Collection happens in the job-runner's `onEvent` callback, which already processes orchestrator events for cost tracking.
3. Dashboard value comes from aggregate stats (avg/min/max, fallback %, heavy tool count), not per-segment detail.
4. Per-segment detail stays in the CLI log. The dashboard is the aggregate view.
5. Without this data, the 2.5x multiplier and min/max clamps remain uncalibrated.

## Approaches Considered

### Approach A: Extend Job + Inline Aggregation

**Summary:** Add `adaptiveTurnsStats` to the `Job` interface. Collect in job-runner's onEvent. Extend `aggregateJobStats` to compute the roll-up.

**Effort:** XS
**Risk:** Low
**Pros:**
- Smallest diff (no new types beyond a stats shape on Job)
- Stats travel with the job naturally

**Cons:**
- Mixes adaptive turns data into the already-busy Job type (13 fields)
- `aggregateJobStats` grows more complex (already computes 10 metrics)
- Harder to test the adaptive turns aggregation in isolation

### Approach B: Dedicated Stats Type + New Aggregation Function

**Summary:** New `AdaptiveTurnsJobStats` interface on Job for per-job collection. New `AdaptiveTurnsStats` interface on `DashboardData`. New `aggregateAdaptiveTurnsStats()` pure function in dashboard.ts. New section in `formatDashboard`.

**Effort:** XS
**Risk:** Low
**Pros:**
- Follows the exact pattern of `aggregateOracleStats()` and `aggregateBudgetStats()`
- Clean separation: each dashboard section has its own aggregation function
- Easy to test in isolation
- `DashboardData` gains a typed `adaptiveTurns` field (like `oracle`, `budget`, `jobs`)

**Cons:**
- Slightly more lines of code than Approach A (new interface, new function)
- One more field on DashboardData

### Approach C: JSONL Event Log + Post-Hoc Read

**Summary:** Write `adaptive_turns` events to `.garyclaw/adaptive-turns.jsonl`. Dashboard reads the file and aggregates at render time.

**Effort:** S
**Risk:** Medium
**Pros:**
- Preserves raw per-segment event data for future analysis
- Decouples collection from job lifecycle

**Cons:**
- New file I/O in the hot path (event callback writes to disk every segment)
- Dashboard currently reads structured JSON, not event logs (new pattern)
- File rotation/cleanup needed
- Overkill for aggregate stats

## Recommended Approach

**Approach B: Dedicated Stats Type + New Aggregation Function.**

It follows the existing dashboard architecture exactly. Every dashboard section (jobs, oracle, budget) has: a dedicated interface on DashboardData, a dedicated `aggregate*Stats()` pure function, and a dedicated section in `formatDashboard`. Adaptive turns should be the fourth instance of this pattern.

## Implementation Plan

### Step 1: Types (src/types.ts)

Add per-job collection type and dashboard aggregate type:

```typescript
// Per-job adaptive turns stats (collected during job execution)
export interface AdaptiveTurnsJobStats {
  segmentCount: number;           // Total segments in this job
  adaptiveCount: number;          // Segments that used growth-rate prediction
  fallbackCount: number;          // Segments that used configured default ("no growth data")
  clampedCount: number;           // Segments forced to minTurns ("already at/past target")
  heavyToolActivations: number;   // Segments where heavy tool multiplier fired
  minTurns: number | null;        // Lowest adaptive maxTurns predicted (null until first event)
  maxTurns: number;               // Highest adaptive maxTurns predicted
  totalTurns: number;             // Sum of all maxTurns (for computing avg)
}
```

**Note on `minTurns: number | null`:** Using `null` instead of `Infinity` because `JSON.stringify(Infinity)` produces `null` anyway. Explicit `null` initialization means the field survives serialization to `daemon-state.json` and back without corruption. The aggregation function handles `null` by skipping that job's min contribution.

Add to `Job` interface:
```typescript
adaptiveTurnsStats?: AdaptiveTurnsJobStats;  // undefined for pre-existing jobs
```

Add to `DashboardData`:
```typescript
adaptiveTurns: {
  totalSegments: number;          // Across all today's jobs
  adaptiveSegments: number;       // Used growth-rate prediction
  fallbackSegments: number;       // Used configured default
  clampedSegments: number;        // Forced to minTurns (context budget exhausted)
  heavyToolActivations: number;   // Heavy tool multiplier firings (subset of adaptiveSegments)
  avgTurns: number;               // Average maxTurns across all segments
  minTurns: number;               // Lowest maxTurns seen today
  maxTurns: number;               // Highest maxTurns seen today
  adaptiveRate: number;           // % of segments using adaptive (0-100)
};
```

**Counting semantics:** `adaptiveSegments + fallbackSegments + clampedSegments === totalSegments`. The `heavyToolActivations` counter is orthogonal to that three-way split. In practice, `"heavy tool"` only appears in the adaptive reason string (the fallback branch returns before heavy tool multiplier is applied), so `heavyToolActivations <= adaptiveSegments` always holds.

### Step 2: Collection (src/job-runner.ts)

In the job-runner's `onEvent` callback (which already handles `cost_update` events), add a handler for `adaptive_turns`:

```typescript
case "adaptive_turns": {
  // Initialize stats on first event (null minTurns survives JSON serialization)
  if (!currentJob.adaptiveTurnsStats) {
    currentJob.adaptiveTurnsStats = {
      segmentCount: 0, adaptiveCount: 0, fallbackCount: 0, clampedCount: 0,
      heavyToolActivations: 0, minTurns: null, maxTurns: 0, totalTurns: 0,
    };
  }
  const stats = currentJob.adaptiveTurnsStats;
  stats.segmentCount++;
  stats.totalTurns += event.maxTurns;
  stats.minTurns = stats.minTurns === null
    ? event.maxTurns
    : Math.min(stats.minTurns, event.maxTurns);
  stats.maxTurns = Math.max(stats.maxTurns, event.maxTurns);

  // Parse reason string to classify segment type
  if (event.reason.includes("no growth data")) {
    stats.fallbackCount++;
  } else if (event.reason.includes("already at/past target")) {
    stats.clampedCount++;   // Context budget exhausted, forced to minTurns
  } else {
    stats.adaptiveCount++;  // Normal growth-rate prediction
  }
  if (event.reason.includes("heavy tool")) {
    stats.heavyToolActivations++;
  }
  break;
}
```

**Key design decision:** Parse the `reason` string rather than adding new fields to the event. The reason string is already structured with three variants from `computeAdaptiveMaxTurns`:
- `"no growth data yet, using configured default"` → fallback
- `"already at/past target (N >= M)"` → clamped (context budget exhausted)
- `"growth N tok/turn (heavy tool: x2.5), budget M tok, predicted P, clamped to C"` → adaptive

This avoids changing the event type, which would require updating the orchestrator, CLI display, and all existing tests.

**Reason string contract:** The parser depends on three substrings: `"no growth data"`, `"already at/past target"`, and `"heavy tool"`. These are produced by `computeAdaptiveMaxTurns()` in `src/token-monitor.ts`. Add a test that imports the function, feeds it representative inputs, and asserts the reason strings contain these substrings. This pins the contract between producer and consumer without making either brittle.

### Step 3: Aggregation (src/dashboard.ts)

New pure function following the existing pattern:

```typescript
export function aggregateAdaptiveTurnsStats(
  jobs: Job[],
  todayStr?: string,
): DashboardData["adaptiveTurns"] {
  const today = todayStr ?? new Date().toISOString().slice(0, 10);
  const todayJobs = jobs.filter(j => j.enqueuedAt.startsWith(today));
  const withStats = todayJobs.filter(j => j.adaptiveTurnsStats);

  if (withStats.length === 0) {
    return {
      totalSegments: 0, adaptiveSegments: 0, fallbackSegments: 0,
      clampedSegments: 0, heavyToolActivations: 0, avgTurns: 0,
      minTurns: 0, maxTurns: 0, adaptiveRate: 0,
    };
  }

  let totalSegments = 0, adaptiveSegments = 0, fallbackSegments = 0;
  let clampedSegments = 0, heavyToolActivations = 0, totalTurns = 0;
  let globalMin: number | null = null;
  let globalMax = 0;

  for (const job of withStats) {
    const s = job.adaptiveTurnsStats!;
    totalSegments += s.segmentCount;
    adaptiveSegments += s.adaptiveCount;
    fallbackSegments += s.fallbackCount;
    clampedSegments += s.clampedCount;
    heavyToolActivations += s.heavyToolActivations;
    totalTurns += s.totalTurns;
    // null-safe min: skip jobs where minTurns was never set
    if (s.minTurns !== null) {
      globalMin = globalMin === null ? s.minTurns : Math.min(globalMin, s.minTurns);
    }
    if (s.maxTurns > globalMax) globalMax = s.maxTurns;
  }

  return {
    totalSegments,
    adaptiveSegments,
    fallbackSegments,
    clampedSegments,
    heavyToolActivations,
    avgTurns: totalSegments > 0 ? totalTurns / totalSegments : 0,
    minTurns: globalMin ?? 0,
    maxTurns: globalMax,
    adaptiveRate: totalSegments > 0 ? (adaptiveSegments / totalSegments) * 100 : 0,
  };
}
```

Add an "Adaptive Turns" section to `formatDashboard`, between Oracle and Budget:

### Step 5: Wire into buildDashboard

```typescript
// In buildDashboard(), after the existing aggregation calls:
const adaptiveTurns = aggregateAdaptiveTurnsStats(state.jobs, todayStr);

// computeHealthScore call must include adaptiveTurns (required by Omit type):
const { score, topConcern } = computeHealthScore({ jobs, oracle, budget, adaptiveTurns, instances });

// Return object gains the new field:
return {
  generatedAt: new Date().toISOString(),
  healthScore: score,
  topConcern,
  jobs,
  oracle,
  budget,
  adaptiveTurns,   // ← new

```

Adding `adaptiveTurns` to `DashboardData` automatically flows into `computeHealthScore`'s input type via the existing `Omit<DashboardData, "healthScore" | "topConcern" | "generatedAt">`. No signature change needed. Zero weight initially.

## Scope Boundaries

- **CLI-mode runs are not collected.** Collection only covers daemon-mode jobs. CLI-mode (`garyclaw run`) adaptive turns events are already displayed in real-time via the CLI's `adaptive_turns` case and are not persisted to the dashboard. This is intentional: the dashboard is a daemon observability tool.
- **`--no-adaptive` jobs are silently excluded.** Jobs run with `--no-adaptive` emit zero `adaptive_turns` events, so `adaptiveTurnsStats` stays `undefined`. The dashboard section is skipped when `totalSegments === 0`, which correctly covers both pre-existing jobs (no stats field) and `--no-adaptive` jobs (no events emitted). These two zero-states are indistinguishable in the aggregate, which is acceptable because neither has adaptive data to report.

## Open Questions

1. **Should adaptive turns data factor into health score?** Probably not yet. There's no clear "unhealthy" threshold (what's a bad adaptive rate?). Add the data first, establish baselines, then decide if a health score weight makes sense. For now, zero weight.
2. **Should the reason string parsing be extracted to a utility?** If more consumers need to parse adaptive_turns reasons, yes. For now, inline in job-runner is fine. One consumer, one parse site.
