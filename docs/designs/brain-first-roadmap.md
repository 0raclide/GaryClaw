## Vision

### 10x Check
GaryClaw as a **learning development daemon** — it doesn't just run skills, it gets
smarter every run. Push code. Go to sleep. The daemon runs QA, remembers that approach X
failed last Tuesday, applies your taste preferences, draws on researched domain knowledge,
and makes decisions a senior engineer would respect. You wake up to results that feel like
*your* work, not automation output.

### Differentiator
No other tool in the 2026 agent landscape combines: file-based memory (no RAG/vector DB),
post-job reflection, taste profiles, domain expertise research, and decision quality
tracking — all injected into a 1M context window at <4% utilization. This is architecturally
superior to retrieval-heavy approaches at this memory scale (validated by A-Mem/Zettelkasten
SOTA research).

## Approach: Brain-First Sequencing

Creative Oracle (Phase 5) first, then Scheduling (Phase 4b), then Parallel Daemons.
Ships the moat first. Oracle improvements benefit ALL subsequent daemon runs.

**Roadmap Amendment:** CLAUDE.md currently lists Phase 4b as the immediate next step.
This plan deliberately resequences: Phase 5 (Oracle) ships before Phase 4b (Scheduling).
CLAUDE.md must be updated to reflect this new sequence when implementation begins.

## Accepted Scope (added to this plan)

### Phase 5a: Memory Infrastructure + Enhanced Oracle Prompt
- `src/oracle-memory.ts` — read/write oracle memory files, size budget enforcement, corruption recovery
- Two-layer memory: `~/.garyclaw/oracle-memory/` (global) + `.garyclaw/oracle-memory/` (per-project)
  - `~/.garyclaw/` is a **new** global GaryClaw directory (distinct from `~/.gstack/` used by gstack itself). Created by `garyclaw oracle init` (Phase 5d). Not to be confused with per-project `.garyclaw/`.
  - Per-project overrides global on conflict; global provides shared domain knowledge fallback
  - **Scoping rule:** `decision-outcomes.md` is **per-project only** (project-specific failures shouldn't poison other projects). Only `domain-expertise.md` (general knowledge) and `taste.md` (personal preferences) live in the global layer.
- Oracle prompt injects: taste.md + domain-expertise.md + decision-outcomes.md + MEMORY.md
- **Memory file integrity:** On JSON parse error or malformed content, rename to `.bak`, treat as empty, log warning. Same 2-rotation pattern as checkpoint.ts.
- **Oracle.ts refactor:** `askOracle()` gains an optional `OracleMemory` parameter. When present, memory files are injected into the prompt between principles and recent decisions. When absent (memory not yet initialized), existing behavior is preserved — full backward compatibility.
- "Other" response parsing: When oracle returns `"choice": "Other"` with an `otherProposal` field, the `canUseTool` callback returns `updatedInput` with the proposal as a free-text answer string. The skill sees it as a custom "Other" selection. **Schema change:** `OracleOutput` gains `otherProposal?: string`; `parseOracleResponse` extracts it from the JSON response.
- 7th Decision Principle: "Local evidence trumps general knowledge — if we tried X and it failed, prefer alternatives even if X is theoretically SOTA. decision-outcomes.md takes precedence over domain-expertise.md."

### Phase 5b: Post-Job Reflection + Quality Tracking
- `src/reflection.ts` — post-job reflection runner, memory file updates
- **Reflection sandboxing:** Runs as a separate `query()` call where `canUseTool` allows only `Write` tool calls targeting paths within `oracle-memory/` directories. Path validation: resolve with `path.resolve()` and confirm the resolved path starts with the target oracle-memory directory prefix — prevents traversal attacks (`../../etc/passwd`). All other tool calls are denied. Working directory set to the oracle-memory dir.
- Decision quality tracking: compare decisions against job outcomes
  - **"Reopened" detection:** An issue is considered reopened when a later job's `IssueTracker` finds an issue matching the same `filePath` + similar `description` (normalized Levenshtein distance — `edit_distance / max(len_a, len_b)` — below 0.3, i.e. 70%+ similar). Checked during reflection by cross-referencing the current job's issues against `decision-outcomes.md` entries.
  - **Migration:** Pre-existing `decisions.jsonl` entries (from before Phase 5b) have no outcome data. They are excluded from quality metrics — only post-reflection entries with outcome tracking contribute to accuracy scores.
- Rolling `decision-outcomes.md` (~50 entries, patterns summarized from older)
- `oracle-memory/metrics.json` — decision accuracy, confidence trends
- Metrics displayed in `daemon status` and notifications

### Phase 5c: Domain Expertise Research
- `src/researcher.ts` — web search + synthesis agent
- `garyclaw research <topic>` CLI command
- **Scope boundaries:** Max 10 web searches per research session. 5-minute time budget. Output schema: YAML frontmatter (`last_researched`, `topic`, `search_count`) + 4 structured sections (Current SOTA, What Works, What Doesn't, Key References). Hard cap: 20K tokens for domain-expertise.md.
- Research freshness: `last_researched` timestamp in domain-expertise.md frontmatter
- Staleness check: 14-day default, auto-refresh on 3+ low-confidence decisions
- Graceful degradation when WebSearch unavailable: skip research, log warning, use existing expertise file
- **Timeout behavior:** On 5-minute timeout, save whatever has been synthesized so far with `partial: true` in frontmatter. Partial results are usable — better than nothing.

### Phase 5d: Taste Seeding
- `garyclaw oracle init` — creates memory directory + template files
- taste.md template with clear instructions

### Phase 5e: Taste Inference (NEW — expansion #2, conditional on 5b metrics)
- **Gate:** Only ship if Phase 5b quality metrics show measurable improvement (decision accuracy trending up). If metrics are flat or negative, defer.
- Analyze decision patterns after 50+ decisions using frequency analysis:
  - Example patterns: "User chose explicit approach over clever 8/10 times" → suggest "Prefer explicit over clever"
  - Example: "User always approves test cleanup" → suggest "Prefer cleaning up stale test artifacts"
  - Example: "User defers database migration changes" → suggest "Be cautious with schema changes"
- Propose taste.md updates with human approval via notification
- Never auto-apply — all suggestions human-gated

### Phase 4b: Cron Scheduling + Config Reload
- CronTrigger alongside GitPollTrigger in triggers.ts
- SIGHUP handler for config hot-reload
  - **Reload boundary:** Config changes apply only to future job enqueues and new poller intervals. In-flight jobs keep the config snapshot from when they started (same pattern as `updateBudget()` in job-runner.ts).
- Static cron rules (adaptive scheduling deferred)

### Parallel Daemons
- `src/daemon-registry.ts` — instance coordination
- Per-instance directories under `.garyclaw/daemons/{name}/`
- Global budget in `.garyclaw/global-budget.json`
  - **Locking strategy:** Last-writer-wins semantics. Read → write to `.tmp` → rename (atomic on POSIX). Two concurrent writers both succeed — last rename wins. This is intentionally not locked: worst case is one cycle slightly over-budget, which is acceptable for a cost guard that exists to prevent runaway spending, not enforce exact limits.
- Cross-instance dedup via daemon-state.json scanning
- `--name` flag + `daemon list` CLI
- Zero breaking changes (backward-compatible default instance)

## Review Additions (accepted during CEO review sections)

### From Error & Rescue Map (Section 2)
- All oracle-memory.ts file I/O wrapped in try/catch. On EACCES/ENOSPC: log error, degrade to no-memory mode.
- Invalid cron expressions: validate at config load, skip bad triggers with warning, start daemon with remaining valid triggers.

### From Security Review (Section 3)
- Sanitize memory file content before Oracle prompt injection. Strip known prompt injection patterns, cap per-file size.

### From Code Quality (Section 5)
- Extract `src/safe-json.ts` with `safeReadJSON()`/`safeWriteJSON()`. Both checkpoint.ts and oracle-memory.ts import from it. Eliminates DRY violation.

### From Data Flow Edge Cases (Section 4)
- Corrupt daemon-state.json during cross-instance scan: try/catch, skip that instance.
- Corrupt daemon.json on SIGHUP reload: keep old config, log warning.

### From Observability (Section 8)
- Oracle degradation notification: if Oracle falls back to no-memory mode, send macOS notification.
- `daemon status` shows Oracle health: memory file ages, decision accuracy score, last reflection timestamp.

### From Outside Voice (Cross-Model Tensions)
- **Circuit breaker:** If Oracle accuracy drops below 60%, auto-disable memory injection and notify. Prevents bad reflection data from compounding.
- **`--no-memory` flag:** CLI flag + `disable_memory` daemon config option. Manual kill switch for Oracle memory.
- **Reflection lock:** File-level lock for oracle-memory writes during reflection. Required before Parallel Daemons ships. Prevents concurrent reflection clobber.
- **Validation gate:** Ship 5a+5b, validate over 10+ jobs, then decide on 5c/5d/5e. Cron (4b) ships in parallel.
- **Latency documentation:** ~40-100s Oracle overhead per job (20 decisions × 2-5s each). ~3-5% of typical 30-min job. Batching as future optimization.
- **7th Principle ranking:** P7 ranks below P1 (completeness) in CEO phases, above P3 (pragmatic). In Eng phases: P7 ranks between P5 (explicit) and P3. Added to conflict resolution hierarchy.
- **"Other" safety:** Skills receiving unexpected free-text "Other" answers should have a catch-all handler. Document in oracle-memory integration guide. Skills without "Other" handling fall back to first option with a warning.

## Token Budget Analysis

Oracle prompt at steady state, per-file breakdown:

| Component | Budget | Hard Cap | Truncation Strategy |
|-----------|--------|----------|---------------------|
| System + 7 principles | ~600 | n/a (static) | None |
| `taste.md` | ~3K | 4K tokens | Oldest entries trimmed |
| `domain-expertise.md` | ~20K | 20K tokens | Sections summarized |
| `decision-outcomes.md` | ~10K | 12K tokens | Rolling window: 50 entries, older → patterns |
| `MEMORY.md` | ~5K | 6K tokens | Prune stale state |
| Recent decisions (5) | ~1K | n/a | Fixed window |
| Question + options | ~500 | n/a | From skill |
| **Total** | **~40K** | **~43K max** | **4.3% of 1M context** |

Cost per decision: ~$0.006 (40K input + ~200 output at Opus 4.6 rates).
Cost per reflection: ~$0.03 (60K input [report + memory] + ~2K output).
At 20 decisions + 1 reflection per job: ~$0.15 Oracle overhead — negligible vs. skill execution.

## Temporal Interrogation Summary
- First-run: empty memory files with graceful degradation (no crash on missing files)
- Reflection sandboxing: separate SDK query with restricted Write access
- Taste inference threshold: 50 decisions minimum (configurable)
- Quality tracking: fixed=success, skipped/deferred=neutral, reopened=failure
- Research unavailable: graceful degradation, log warning, use existing expertise
- Global budget: advisory locking (read-update-write), acceptable worst case
- Migration: auto-migrate flat .garyclaw/ to daemons/ subdirs on first start
