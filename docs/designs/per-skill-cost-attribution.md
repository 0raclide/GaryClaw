## Problem Statement

Costs are trending up 4% ($2.04→$2.13/job) but we can't tell which skills within a pipeline consume the most. The prioritize skill's `measureRecentImpact()` computes average cost per job, but a job running `prioritize → implement → qa` could have implement costing $1.50 and qa costing $0.30. Without per-skill attribution, cost optimization is guesswork. The priority pick explicitly says "prioritize measurement, profiling, or optimization work over new features."

## What Makes This Cool

The data already exists. `PipelineSkillEntry.report.estimatedCostUsd` is populated by `buildSkillReport()` in pipeline.ts (line 578-610) after every skill completes. It reads from `checkpoint.tokenUsage.estimatedCostUsd`. It just never surfaces anywhere useful.

The real payoff: once per-skill costs are on the `Job` interface, every consumer (dashboard, prioritize, reflection, pipeline composition) can use real data instead of the hardcoded guesses in `skill-catalog.ts`.

## Constraints

- S effort (~100 lines of production code + 15 tests)
- Zero new data collection. Pure aggregation of existing `RunReport.estimatedCostUsd`
- Backward compatible. `skillCosts` is optional on `Job`. Old jobs without it degrade gracefully
- No auto-calibration of skill-catalog.ts in this scope (that's a follow-up)

## Premises

1. **The data already exists** — `PipelineSkillEntry.report.estimatedCostUsd` is populated by `buildSkillReport()` in pipeline.ts line 578-610, which reads from the orchestrator's accumulated `checkpoint.tokenUsage.estimatedCostUsd`. No new collection infrastructure needed. Verified in source.
2. **Dashboard is the right first consumer** — Seeing the data is prerequisite to acting on it. Feeding it back into pipeline composition is a follow-up.
3. **Per-job `costUsd` on `Job` interface stays as-is** — We add a new `skillCosts` field alongside the aggregate. All existing consumers unaffected.
4. **Pipeline jobs are the interesting case** — Single-skill jobs have trivial attribution (one skill = full job cost). Multi-skill pipelines are where this matters.
5. **Sum-of-parts invariant** — `job.costUsd` may exceed the sum of `skillCosts` values if a skill crashes mid-way (orchestrator records partial session cost but `buildSkillReport()` never runs). Consumers should treat `skillCosts` as "cost of skills that completed successfully" and `job.costUsd` as the authoritative total.
6. **Failed jobs get no skillCosts** — `skillCosts` is only populated after `nextJob.status = "complete"`. Failed jobs (`status !== "complete"`) do not populate `skillCosts`. Partial pipeline costs for failed jobs are only available via `job.costUsd`.

## Approaches Considered

### Approach A: Dashboard-Only Aggregation
Read `PipelineSkillEntry.report.estimatedCostUsd` from pipeline.json at dashboard generation time. ~50 lines. Downside: no persistence on Job, so prioritize.ts and other consumers can't use it without re-reading pipeline.json. Data coupling between dashboard and pipeline internals.

### Approach B: Job-Level Persistence + Dashboard (CHOSEN)
Add `skillCosts: Record<string, number>` to `Job` interface. Populate after job completes from pipeline state. Dashboard aggregates from `Job.skillCosts`. Enables all consumers. ~100 lines + 15 tests.

### Approach C: Full Feedback Loop
Everything in B, plus auto-calibrate `skill-catalog.ts` from rolling averages. Higher risk (oscillation), M effort. Deferred.

## Recommended Approach

**Approach B: Job-Level Persistence + Dashboard.**

#### Step 2: Extract skill costs from pipeline state in job-runner.ts

Add a pure function `extractSkillCosts()`:

```typescript
export function extractSkillCosts(pipelineState: PipelineState): Record<string, number> {
  const costs: Record<string, number> = {};
  for (const entry of pipelineState.skills) {
    if (entry.report?.estimatedCostUsd != null) {
      costs[entry.skillName] = entry.report.estimatedCostUsd;
    }
  }
  return costs;
}
```

Wire into the job completion path at ~line 993-995, after `nextJob.status = "complete"`:

```typescript
// Per-skill cost attribution: extract from pipeline state
if (nextJob.skills.length > 1) {
  const pipeState = readPipelineState(jobDir);
  if (pipeState) {
    nextJob.skillCosts = extractSkillCosts(pipeState);
  }
}
```

This adds zero I/O for single-skill jobs and one `readPipelineState` call for pipelines (pipeline.json is already in the job dir, so it's a local file read).

**Note on pipeline resume:** When a pipeline resumes after crash, `resumePipeline()` reads existing state. The completed skills from the prior run already have their reports. After resume completes, `extractSkillCosts()` captures all skills (both pre-crash and post-resume). This is correct because `job.costUsd` also reflects the full pipeline cost via the `pipeline_complete` event.

**Note on auto-fix jobs:** Auto-fix re-attempts create a new `Job` with fresh `skillCosts`. No collision with the original pipeline's skill keys.

#### Step 3: Add `aggregateSkillCostStats()` to dashboard.ts

New pure function:

```typescript
export interface SkillCostStats {
  skillName: string;
  totalCostUsd: number;
  runCount: number;
  avgCostUsd: number;
  minCostUsd: number;
  maxCostUsd: number;
}

export function aggregateSkillCostStats(
  jobs: Job[],
  todayStr?: string,
): SkillCostStats[] {
  const today = todayStr ?? new Date().toISOString().slice(0, 10);
  const todayJobs = jobs.filter(
    (j) => j.enqueuedAt.startsWith(today) && j.status === "complete" && j.skillCosts,
  );

  const bySkill = new Map<string, number[]>();
  for (const job of todayJobs) {
    for (const [skill, cost] of Object.entries(job.skillCosts!)) {
      const existing = bySkill.get(skill) ?? [];
      existing.push(cost);
      bySkill.set(skill, existing);
    }
  }

  return Array.from(bySkill.entries())
    .map(([skillName, costs]) => ({
      skillName,
      totalCostUsd: costs.reduce((s, c) => s + c, 0),
      runCount: costs.length,
      avgCostUsd: costs.reduce((s, c) => s + c, 0) / costs.length,
      minCostUsd: Math.min(...costs),
      maxCostUsd: Math.max(...costs),
    }))
    .sort((a, b) => b.totalCostUsd - a.totalCostUsd);
}
```

## Open Questions

1. **Should skill-catalog.ts costUsd be auto-calibrated from actuals?** Deferred to a follow-up. The data will be available once this ships. A simple rolling average with a staleness window would work.
2. **Should prioritize.ts use per-skill costs for scoring?** Yes, eventually. The `measureRecentImpact()` function could compute cost trends per skill category instead of per job. Consumer change, not a data change.
