## Problem Statement

Costs are trending up 4% but the system has no per-skill diagnostic. The basic aggregation exists (event collection in job-runner.ts near line 1680, `aggregateSkillCostStats()` in dashboard.ts near line 103, formatting in dashboard.ts near line 671 — line numbers approximate, may shift with concurrent changes). But trend detection — comparing recent per-skill costs to previous windows and flagging >15% increases — is missing. And the entire feature has zero tests.

The prioritize skill's impact measurement signal says "Costs are trending up. Prioritize measurement." Per-skill cost attribution with trend detection is that measurement.

## What Makes This Cool

You can see *which skill* is getting more expensive and by how much. Not "costs went up 4%" but "implement costs went up 22% while qa is flat." That's the difference between guessing and knowing.

## Constraints

- S effort (production code is ~80% done, trend detection is ~30 lines of pure function)
- Follow established dashboard patterns (aggregation → formatting → health score integration)
- All functions under test are pure (no I/O mocking needed for core logic)
- `aggregateSkillCostStats` already exists and works — extend, don't rewrite
- Trend window: all jobs with skillCosts (not just today), grouped into recent vs previous

## Premises

1. **Basic aggregation is complete and correct** — `aggregateSkillCostStats()` groups today's jobs by skill, computes totals/avg/min/max, sorts descending. Event collection in job-runner.ts populates `Job.skillCosts` from `pipeline_skill_complete` events. Dashboard renders the table.
2. **Trend detection is the missing piece** — TODOS.md specifies "compare last 10 jobs' per-skill costs to previous 10, flag >15% increases." This doesn't exist anywhere in the codebase.
3. **Tests don't exist** — `test/dashboard-skill-costs.test.ts` is missing. Zero coverage.
4. **Trend detection is pure** — takes a Job array, returns per-skill trend data. Same pattern as every other dashboard aggregation function.

## Approaches Considered

### Approach A: Trend Detection + Tests + CLAUDE.md (Complete Feature)
Add `computeSkillCostTrends()` comparing recent vs previous job windows. Extend dashboard formatting to flag increases. Write 22 tests. Update CLAUDE.md.
- Effort: S
- Risk: Low
- Pros: Completes feature as spec'd. Dashboard becomes diagnostic. Enables targeted optimization.
- Cons: Slightly more than tests-only.

### Approach B: Tests-Only (Skip Trend Detection)
Write tests for existing aggregation. Skip trends.
- Effort: XS
- Risk: Med — ships incomplete feature, trend detection becomes permanent tech debt
- Pros: Fast.
- Cons: The entire point is answering "which skill is getting more expensive?" Without trends, you're manually comparing dashboards across days.

## Recommended Approach

**Approach A: Complete Feature.** Trend detection is ~30 lines of pure function code. Same S effort bucket. Shipping tests for incomplete code is backwards — complete the feature first, then test all of it.

## Implementation Plan

### Step 1: Add `computeSkillCostTrends()` to dashboard.ts (~30 lines)

New pure function alongside `aggregateSkillCostStats`. New exported types + constants:

```typescript
export interface SkillCostTrend {
  skillName: string;
  recentAvgCostUsd: number;   // avg cost in recent window
  previousAvgCostUsd: number; // avg cost in previous window
  changePercent: number;       // ((recent - previous) / previous) * 100
  flagged: boolean;            // true if changePercent > TREND_THRESHOLD_PERCENT
  recentRunCount: number;
  previousRunCount: number;
}

export const TREND_THRESHOLD_PERCENT = 15;
export const TREND_WINDOW_SIZE = 10;  // jobs per window
```

Function logic:
1. Filter to complete jobs with skillCosts (all time, not just today)
2. Sort by completedAt descending (fallback to enqueuedAt)
3. Split into recent (first TREND_WINDOW_SIZE) and previous (next TREND_WINDOW_SIZE). A partial previous window (1 to TREND_WINDOW_SIZE-1 jobs) is valid and used for comparison — the window doesn't need to be full.
4. If previous window is empty (0 jobs), return [] (not enough history for any comparison)
5. Aggregate per-skill avg cost in each window via helper function
6. For skills appearing in both windows, compute changePercent = ((recent - previous) / previous) * 100. **Division-by-zero guard:** if previousAvgCostUsd is 0, set changePercent to 0 and flagged to false (a skill that was free before can't meaningfully have a "percentage increase")
7. Flag if changePercent > TREND_THRESHOLD_PERCENT
8. Sort by changePercent descending

Helper `aggregateWindow(jobs)` groups jobs by skill name, computes avg cost and count per skill. Returns `Map<string, { avg: number; count: number }>`. Same pattern as `aggregateSkillCostStats` internals.

### Step 4: Update formatDashboard() to show flagged trends

After the existing skill cost table (dashboard.ts:671-685), add a "Cost Trends" sub-section that only appears when flagged trends exist:

Note: `trends?.` uses optional chaining as a forward-compat guard. Existing `DashboardData` objects persisted in daemon-state.json before this change won't have `trends`. The type says it's always present, but at runtime during the upgrade window it may be missing.

```typescript
const flaggedTrends = data.skillCosts.trends?.filter(t => t.flagged) ?? [];
if (flaggedTrends.length > 0) {
  lines.push(
    "",
    "### Cost Trends (flagged >15% increase)",
    "",
    "| Skill | Recent Avg | Previous Avg | Change | Runs (R/P) |",
    "|-------|-----------|-------------|--------|------------|",
  );
  for (const t of flaggedTrends) {
    lines.push(
      `| ${t.skillName} | $${t.recentAvgCostUsd.toFixed(2)} | $${t.previousAvgCostUsd.toFixed(2)} | +${t.changePercent.toFixed(1)}% | ${t.recentRunCount}/${t.previousRunCount} |`,
    );
  }
}
```

### Step 5: Integrate trends into computeHealthScore()

Add trend-aware concern: if any skill has >30% cost increase (2x the base threshold), reduce health score. Follows existing pattern (mergeHealth.revertRate > 0.3 triggers concern). The -2 per flagged skill magnitude matches the existing concern deduction scale: merge revert concern deducts ~5 points, circuit breaker deducts ~10 points. A single skill cost spike is less severe than a merge revert (which loses work) but worth flagging, so -2 per skill keeps it proportional.

```typescript
const flaggedCount = data.skillCosts?.trends?.filter(t => t.changePercent > 30).length ?? 0;
if (flaggedCount > 0) {
  concerns.push(`${flaggedCount} skill(s) with >30% cost increase`);
  score -= flaggedCount * 2;
}
```

8. `handles single-entry skillCosts` — job with only one skill in the record

**computeSkillCostTrends tests (8 tests):**
10. `returns empty when fewer than TREND_WINDOW_SIZE+1 eligible jobs` — 10 or fewer jobs means recent window is full but previous window is empty (0 jobs), so returns []. 11 jobs returns partial trends (recent=10, previous=1). 20 jobs fills both windows completely.
11. `computes trends across two full windows` — 20 jobs, verify changePercent math
12. `flags skills exceeding TREND_THRESHOLD_PERCENT` — recent avg $1.50, previous avg $1.00, changePercent = 50%
13. `does not flag skills below threshold` — recent avg $1.10, previous avg $1.00, changePercent = 10%
14. `handles skills appearing in only one window` — skill X in recent only, skill Y in previous only, both excluded from trends
15. `sorts by changePercent descending` — 3 skills with different trends
16. `handles zero previous avg without division error` — previous avg 0, verify no NaN/Infinity
17. `uses completedAt for sort order, falls back to enqueuedAt` — mix of jobs with/without completedAt

**formatDashboard + wiring tests (5 tests):**
18. `formatDashboard includes skill cost table when data exists` — call formatDashboard with skillCosts.skills populated, verify table headers and rows
19. `formatDashboard omits skill cost table when no data` — empty skills array, verify no "Skill Cost Breakdown"
