## Problem Statement

4. **Implement relay awareness** — Full step tracking: `detectCompletedSteps`, `matchCommitToStep`, `extractStepTokens` + 45 tests in `step-tracking.test.ts`

The TODOS.md entry reads: `## ~~P3: Implement Skill Hardening~~ — COMPLETE (2026-03-27)`

The design doc at `docs/designs/implement-skill-hardening.md` has Status: APPROVED and all implementation steps were executed.

`filterOpenTodos()` in `prioritize.ts:59` correctly strips `## ~~` headings from the TODOS content. So the filtered backlog sent to the LLM does NOT include this item. But the LLM is hallucinating a stale priority pick anyway. Two confirmed root causes:

1. **`loadUnresolvedReviewFindings()` (prioritize.ts:326)** — the primary cause. This function scans the last 10 job pipeline reports for review skill decisions containing action keywords (`fix`, `implement`, `add`, `build now`, `extract`, `validate`, `replace`). It parses `**Q:** ... → **A:** ... (N/10)` patterns from `pipeline-report.md` files. **It has NO mechanism to check whether those actions were subsequently implemented.** Old eng review decisions about "add loadDesignDoc tests" persist in `.garyclaw/daemons/default/jobs/*/pipeline-report.md` and get re-injected into the prioritize prompt every cycle as "unresolved review findings" with a +2 scoring bonus.
2. **Design doc on disk** — `docs/designs/implement-skill-hardening.md` still exists with "unresolved review findings" in its title. Combined with the review findings injection, this creates strong contextual signal for the LLM to invent a stale pick.

## What Makes This Cool

The daemon burned a pipeline cycle on a no-op. For a system that costs $0.50-$4 per cycle, this is a real efficiency bug. Fixing it means fewer wasted cycles, which compounds across hundreds of daemon runs.

## Constraints

- S effort: blocklist (XS) + validation gate (XS) = S combined
- Must not break `filterOpenTodos` backward compatibility
- Must preserve the existing prioritize prompt size budget system

## Premises

1. `filterOpenTodos()` correctly strips struck-through TODOS. The bug is NOT in filtering.
2. The LLM is inventing a priority pick from contextual signals (design docs, review findings, CLAUDE.md) rather than from the filtered backlog.
3. The fix needs defense-in-depth: prevent the hallucination at prompt time AND catch it deterministically at parse time.
4. `loadUnresolvedReviewFindings` is worth keeping (it surfaces genuinely unresolved findings) but a completed-items blocklist and validation gate are cheaper and more reliable than adding staleness detection to the review scanner.

## Approaches Considered

### Approach A: Inject Completed-Items Blocklist into Prioritize Prompt
Summary: After `filterOpenTodos`, extract the titles of completed items (struck-through `## ~~{title}~~` headings) and inject them as a negative constraint: "DO NOT select these items — they are ALREADY COMPLETE: [list]".

Effort: XS
Risk: Low
Pros: Direct, easy to test, addresses root cause (LLM doesn't know what's done), works regardless of which context source primes the hallucination
Cons: Adds tokens to an already budget-constrained prompt, completed list grows over time
Reuses: `filterOpenTodos` already splits on `## ` boundaries, extraction is 3-5 lines

### Approach B: Prune Stale Design Docs and Review Findings
Summary: After a TODOS item is marked complete, move or archive its design doc from `docs/designs/` to `docs/designs/archived/`. Also prune `.garyclaw/` review output directories for completed items.

Effort: S
Risk: Medium
Pros: Removes the contextual signal at the source, cleaner codebase over time
Cons: Requires new archival logic, design docs may still be referenced by other items, doesn't prevent hallucination from CLAUDE.md references, may break `loadUnresolvedReviewFindings` if it scans archived dirs

### Approach C: Post-Prioritize Validation Gate
Summary: After the prioritize LLM returns its pick, run a deterministic check: does the pick title fuzzy-match any completed TODOS heading? If yes, log a warning, skip the pick, and re-run prioritize or fall through to the next candidate.

Effort: S
Risk: Low
Pros: Catches hallucinations regardless of source, fail-safe even if prompt injection fails, can report the bug for dashboard stats
Cons: Requires fuzzy matching (Levenshtein, already used in reflection.ts), adds a post-hoc validation step, doesn't prevent the wasted LLM call
Reuses: `normalizedLevenshtein` from `reflection.ts`, `parseTodoItems` from `prioritize.ts`

## Recommended Approach

**Approach A (Completed-Items Blocklist) + Approach C (Validation Gate).** A alone might fail if the prompt budget truncates the blocklist. C alone wastes an LLM call. Together they form defense-in-depth: the blocklist prevents most hallucinations, and the validation gate catches any that slip through.

The cost: ~400 tokens for the blocklist (at 30 completed items, ~12-15 tokens each including bullet formatting) + ~20 lines for the validation gate. Combined effort: S.

## Implementation Order

1. **Extract completed item titles from TODOS.md** (~10 lines in `prioritize.ts`)
   - New exported function: `extractCompletedTitles(content: string): string[]`
   - NOTE: If this function already exists in `prioritize.ts` (check first), verify its behavior matches and add any missing tests. Otherwise create it.

2. **Inject blocklist into prioritize prompt** (~8 lines in `buildPrioritizePrompt`)
   - After the backlog section, add a budgeted section: "### Already Complete (DO NOT SELECT)"
   - List completed titles as bullet points
   - Budget: 500 tokens, dedicated key `completedItems` in the `PRIORITIZE_SECTION_BUDGETS` constant (aliased as `SB` in `buildPrioritizePrompt`). Separate from `claimedItems`, which tracks actively-claimed work by other instances, a different concern.
   - 2-3 tests: blocklist injected, empty when no completions, truncated under budget

3. **Post-prioritize validation gate** (~25 lines, new function)
   - New exported function: `validatePriorityPick(pickTitle: string, completedTitles: string[]): boolean` (returns `true` if acceptable, `false` if pick should be rejected)
   - Normalize both sides (lowercase, strip punctuation)
   - Reject if the pick matches a completed item by either: (a) case-insensitive substring containment (only when the completed title is 20+ characters, to avoid false positives from short titles like "QA"), or (b) `normalizedLevenshtein` distance < 0.3
   - Import `normalizedLevenshtein` from `reflection.ts` (already exported, returns 0-1 normalized distance)
   - **Data flow:** Import `extractCompletedTitles` from `prioritize.ts` in job-runner.ts. Read raw TODOS.md via `safeReadText(join(projectDir, 'TODOS.md'))` and pass to `extractCompletedTitles()`. This mirrors how `filterOpenTodos` is already called in `buildPrioritizePrompt`.
   - Validation gate runs only when `parsePriorityPickTitle()` returns a non-null title
   - Wire into job-runner.ts immediately after `parsePriorityPickTitle()` (see `test/job-runner.regression-4.test.ts` for the parse site)
   - **Fallback:** Add a separate exported function `parseAlternativeTitles(content: string): string[]` (do NOT change `parsePriorityPickTitle` return type, which is used by existing tests). Parse from the `## Alternatives` section using regex `/^### (?:\d+(?:st|nd|rd|th)):\s*(.+)/gm`. In job-runner.ts, after `parsePriorityPickTitle()` returns a title that fails validation, call `parseAlternativeTitles()` and iterate alternatives in order, stopping at the first that passes `validatePriorityPick()`. When no alternatives survive validation, emit `priority_pick_exhausted` event and skip the job (daemon goes idle until next trigger).
   - **Wiring pseudocode** (in job-runner.ts, after line ~1633):
     ```
     const completedTitles = extractCompletedTitles(rawTodos);
     if (title && !validatePriorityPick(title, completedTitles)) {
       const alts = parseAlternativeTitles(priorityContent);
       title = alts.find(t => validatePriorityPick(t, completedTitles)) ?? null;
       if (!title) { emit('priority_pick_exhausted'); return; }
     }
     ```
   - 6-7 tests in new `test/prioritize-validation-gate.test.ts`: exact match rejects, fuzzy match rejects, no match passes, empty completed list passes, short-title substring guard, alternative fallback, all-rejected exhaustion

4. **Dashboard stats for rejected picks** (~5 lines)
   - Emit `priority_pick_rejected` event when validation gate fires
   - Track in dashboard as a prioritize health metric
   - 1-2 tests: event emission, dashboard aggregation

## Open Questions

2. Should `normalizedLevenshtein` be extracted to a shared `src/string-utils.ts`? **Deferred** — acceptable as a cross-module import from `reflection.ts` for S effort. Extract if a third module needs it.
