## Problem Statement

GaryClaw's most impressive quality is invisible. The daemon built 22 of its own 43 modules, invented features when the backlog ran dry, shipped 782+ commits (96% daemon-generated), and makes autonomous decisions with 542 oracle calls at 9.5 average confidence. But you can only see this by reading JSON files and markdown reports scattered across `.garyclaw/`. There's no way to watch the daemon think.

The TODO item frames this as "Twitch for an AI coding agent." That's the right framing. The data already exists. Every oracle decision, every mutation cycle, every commit, every cost event, every pipeline skill transition is persisted in `.garyclaw/` files. This is pure presentation layer.

## What Makes This Cool

You open a browser tab and watch an AI build itself. You see oracle decisions appear in real-time with confidence scores and reasoning chains. You see the mutation timeline... ideas invented, critiqued by the pride test (score < 7 = pruned), and the survivors shipping to main. You see the growth chart animate from 8 modules on day 1 to 43 modules and 3,438 tests today, each module colored by who built it (human vs daemon). You see the daemon's taste profile and domain expertise evolving.

This is the thing that makes people star the repo.

## Constraints

- Must live inside the GaryClaw repo (TypeScript, same stack) so the daemon can build and improve it
- No external framework dependencies (the daemon needs to be able to edit the source files directly)
- No build step for the web assets (plain HTML/CSS/JS served as static files)
- Must work when the daemon isn't running (historical data mode)
- Must not modify the daemon process itself (read-only observer pattern)
- No new daemon instrumentation needed... all data comes from existing `.garyclaw/` files (growth cache is derived from git history, not new collection)
- The existing `DashboardData` interface in `src/dashboard.ts` already aggregates most of what we need

## Premises

1. **This lives inside the GaryClaw repo.** Same TypeScript stack. The daemon can dogfood it, improve it, QA it. The Tauri dashboard (garyclaw-hud) is a separate native macOS project. This is the web-based evolution feed.

2. **Plain HTML+JS, no framework.** Zero build step. The daemon can directly edit HTML/CSS/JS files. Served as static assets by a Node.js HTTP server. CSS variables for theming. Vanilla JS with `EventSource` for SSE.

3. **Standalone HTTP server with SSE, not bolted onto daemon IPC.** A new `garyclaw dashboard --serve` command starts a lightweight HTTP server (port 3333 default). It reads `.garyclaw/` files on startup and watches for changes via `fs.watch`. SSE pushes updates to connected clients. Works with or without a running daemon.

4. **All four views ship in v1.** Live feed, mutation timeline, growth over time, oracle mind. The wow factor requires all four. Cutting any one reduces the "whoa" reaction to "that's nice."

5. **No public hosting in v1.** Local server only. The `--serve` flag starts a local HTTP server. Public hosting (snapshot to GitHub Pages, Vercel deploy on job completion) is a natural Phase 2 but would scope-creep this into deployment pipeline and CI/CD design.

## Approaches Considered

### Approach A: Minimal — Static HTML Report Generator

Extend the existing `generateDashboard()` in `src/dashboard.ts` to output HTML instead of markdown. Single file, no server, open in browser. `garyclaw dashboard --html > report.html && open report.html`.

```
Summary: Generate a single HTML file from DashboardData, no server
Effort:  S
Risk:    Low
Pros:    - Smallest diff, reuses existing aggregation
         - No new dependencies or server code
         - Ships in hours
Cons:    - Not real-time (snapshot only)
         - No "watching the daemon think" feeling
         - Refreshing the page = re-running the command
Reuses:  dashboard.ts aggregation, DashboardData interface
```

### Approach B: Live Server with SSE (RECOMMENDED)

New module `src/dashboard-server.ts` starts an HTTP server. Serves static HTML/CSS/JS from `src/dashboard-web/`. SSE endpoint watches `.garyclaw/` files and pushes structured events. Four views: live feed, mutation timeline, growth chart, oracle mind.

```
Summary: HTTP server with SSE, static web assets, four real-time views
Effort:  M
Risk:    Medium (fs.watch reliability, SSE reconnection)
Pros:    - Real-time updates via SSE
         - Works with or without running daemon
         - Daemon can edit and improve the web assets
         - Zero external dependencies (node:http, node:fs)
         - The "whoa" factor is real
Cons:    - New module (~400 lines TS) + ~700 lines of HTML/CSS/JS
         - fs.watch behavior varies by OS (macOS = kqueue, reliable)
         - Port management (what if 3333 is taken)
Reuses:  dashboard.ts pure aggregation functions, DashboardData,
         all existing .garyclaw/ file formats
```

### Approach C: WebSocket Bidirectional

Like Approach B but with WebSocket instead of SSE. Enables the browser to send commands back (trigger skills, cancel jobs, modify pipeline).

```
Summary: WebSocket server with bidirectional control plane
Effort:  L
Risk:    High (security surface, daemon state mutation from browser)
Pros:    - Full interactive control from the browser
         - Could replace CLI for common operations
Cons:    - Security risk (browser can trigger daemon actions)
         - Requires WebSocket library or raw ws implementation
         - Bidirectional control is scope creep
         - The daemon shouldn't take orders from a web UI
Reuses:  dashboard.ts, daemon-ipc.ts patterns
```

## Recommended Approach

**Approach B: Live Server with SSE.** The wow factor requires real-time. SSE is the right primitive... unidirectional (server to client), built into every browser via `EventSource`, reconnects automatically, and requires zero dependencies beyond `node:http`. The dashboard observes, it doesn't control. That's the right boundary.

Approach A is too boring. Approach C is scope creep that introduces security surface. B hits the sweet spot.

### Module: `src/dashboard-server.ts`

New source module. ~400 lines. Responsibilities:

1. **HTTP server** (`node:http.createServer`) bound to `127.0.0.1` (localhost only, no network exposure). No authentication needed... localhost binding means only the local user can connect. No CORS headers (same-origin by default, no cross-origin consumers in v1). Configurable port (default 3333). On `EADDRINUSE`: try ports 3334-3343, log which port was used. Fail with clear message if all 10 ports taken.
2. **Static file serving** for `src/dashboard-web/` assets (Content-Type mapping for `.html`, `.css`, `.js`, `.svg`; 404 for unknown paths)
3. **REST endpoints** that call existing pure functions from `dashboard.ts`:
   - `/api/state` calls `buildDashboard()` (already exists, returns `DashboardData`). Uses `discoverInstances()` from `daemon-registry.ts` to aggregate across all daemon instances (default + named). Returns 200 with `DashboardData` JSON, or 200 with empty defaults if `.garyclaw/` doesn't exist yet.
   - `/api/decisions` reads `decision-outcomes.md`, parses into structured JSON (see Response Schemas below). Query params: `?limit=50&offset=0` for pagination.
   - `/api/growth` reads cached growth data from `.garyclaw/growth-cache.json` (see Growth Data Extraction). Returns date-indexed metrics.
   - `/api/mutations` reads `pipeline-outcomes.jsonl` + `todo-state/*.json`, correlates into timeline (see Mutation Timeline Algorithm). Query params: `?limit=20`.
4. **SSE endpoint** (`/api/events`):
   - On connect: send `event: init` with current `DashboardData` snapshot
   - `fs.watch` on `.garyclaw/` directory (recursive, macOS only... see Platform Notes)
   - On file change: debounce 500ms, re-read changed file using `safeReadJSON()` / `safeReadText()` from `safe-json.ts` (handles atomic writes correctly), send typed event
   - SSE event types with payloads (see SSE Event Schemas below)
   - Client tracking: maintain `Set<ServerResponse>` of connected clients, remove on `close` event
5. **Error handling**: all file reads wrapped in try/catch, fail-open with empty defaults. Missing `.garyclaw/` dir returns valid empty state (not 500). `buildDashboard()` errors caught and logged, return last-known-good state.
6. **Graceful shutdown** on SIGINT/SIGTERM: close watcher, end all SSE connections, close server

### SSE Event Schemas

Each SSE event has `event:` (type) and `data:` (JSON payload):

```
event: job_update
data: {"jobId":"...","status":"running","skill":"implement","todoTitle":"...","elapsed":142}

event: decision
data: {"id":"...","question":"...","chosen":"...","confidence":9,"principle":"P3"}

event: budget
data: {"dailySpentUsd":26.58,"dailyLimitUsd":1000,"dailyRemaining":973.42}

event: commit
data: {"sha":"abc123","message":"feat: ...","author":"daemon","timestamp":"..."}

event: pipeline_progress
data: {"skillIndex":2,"totalSkills":3,"currentSkill":"qa","todoTitle":"..."}

event: mutation
data: {"todoTitle":"...","outcome":"shipped","skills":["implement","qa"]}
```

The client uses `event.type` to route to the correct view updater. Only the affected DOM subtree re-renders.

### Parsing `decision-outcomes.md`

The file is written by `reflection.ts` and has a stable format. Each decision is a markdown section:

```markdown
### d-{timestamp}-{index}
- **Timestamp:** {ISO 8601}
- **Question:** {text}
- **Chosen:** {option label}
- **Confidence:** {1-10}
- **Principle:** {P1-P7 text}
- **Outcome:** {success|neutral|failure}
- **Job:** {job-id}
```

Parser logic:
1. Split file content on `/^### d-/m` to get individual decision blocks
2. For each block, extract fields via regex: `/\*\*Timestamp:\*\* (.+)/`, `/\*\*Chosen:\*\* (.+)/`, etc.
3. Map to `DecisionEntry` interface. Skip blocks that fail to parse (fail-open).
4. Handle the `[compact]` prefix added by oracle memory compaction (strip it before parsing).
5. Export as `parseDecisionOutcomes(content: string): DecisionEntry[]` in `dashboard-server.ts`.

This format has been stable since Phase 5b (reflection.ts). If the format evolves, the parser will fail-open (skip unparseable entries) and the test suite will catch regressions.

### Mutation Timeline Algorithm

A "mutation cycle" = one TODO item's journey through the pipeline, from prioritize pick to merge (or failure). This is a non-trivial correlation engine and should be its own exported function (`buildMutationTimeline()`) with dedicated tests.

Correlation algorithm:

1. Read `pipeline-outcomes.jsonl` via existing `readPipelineOutcomes()` from `pipeline-history.ts` (reuse, don't rewrite). Each record has `todoTitle`, `skills`, `outcome`, `timestamp`.
2. Read `todo-state/*.json` — each file has `state`, `artifacts` (commits, branches), timestamps. Use existing `readTodoState()` from `todo-state.ts`.
3. Read `daemon-state.json` jobs — each job has `skills`, `claimedTodoTitle`, `status`, `costUsd`
4. **Join key:** `todoTitle` (normalized via `slugify()` from `todo-state.ts` for fuzzy matching)
5. **Group:** all pipeline outcomes + jobs + todo-state for the same TODO slug into one `MutationCycle`
6. **Sort:** by `startedAt` descending (most recent first)
7. **Outcome mapping:**
   - todo-state `merged` or `complete` = **"shipped"** (the daemon's work landed on main)
   - todo-state `implementing` / `qa-running` / `pr-created` = **"in-progress"** (actively being worked)
   - todo-state `failed` or job status `failed` = **"failed"** (attempted but didn't pass QA or merge)
   - Item appeared in `priority.md` history but has no todo-state file AND no matching job = **"not-started"** (deprioritized or superseded, not the same as "pruned")
   - Note: "pruned" in the invention sense (pride test score < 7) is a TODOS.md concept, not a pipeline state. The mutation timeline shows pipeline outcomes, not backlog triage.

### Platform Notes

`fs.watch({ recursive: true })` is reliable on macOS (kqueue-backed) and Windows (ReadDirectoryChangesW). On Linux, `recursive: true` support was added in Node.js 19+ via `inotify` but may miss events in deeply nested directories.

**Strategy:** Use `fs.watch({ recursive: true })` unconditionally. If a Linux user reports missed events, the fallback is trivial: poll `.garyclaw/` file mtimes every 2 seconds (a `setInterval` fallback, ~10 lines). This is a local dev tool, not a production service... optimizing for the common case (macOS, where the daemon runs) is correct.

### Growth Data Extraction

The growth chart needs historical data. Strategy:

1. **On first `--serve`:** Run `git log --format="%H %aI" -- CLAUDE.md` to find all commits that touched CLAUDE.md. Cap at 200 most recent commits to bound execution time.
2. **For each commit:** Run `git show {sha}:CLAUDE.md` and extract module/test counts via regex patterns:
   - Primary: `/(\d+) source modules.*?(\d+) tests/` (current format)
   - Fallback: `/(\d+) source modules/` and `/(\d+) tests/` separately
   - If neither matches (early CLAUDE.md format didn't have these): skip that revision, interpolate from nearest neighbors
3. **Batch execution:** Run all `git show` commands in parallel (10 concurrent) with 5s timeout per command. Total cache build capped at 60s. Show progress: "Building growth cache... 45/120 commits"
4. **Cache results** in `.garyclaw/growth-cache.json` with `{ headSha: string, builtAt: string, snapshots: GrowthSnapshot[] }`. Invalidate when HEAD SHA changes. Incremental update: on cache hit where HEAD has moved, run `git log {cachedHeadSha}..HEAD --format="%H %aI" -- CLAUDE.md` to find only new commits, process those, prepend to cached snapshots, update headSha.
5. **Commit authorship:** Parse `git log --format="%aI %ae"` to classify human vs daemon commits. Daemon email: `GARYCLAW_DAEMON_EMAIL` constant from `sdk-wrapper.ts`. Commits without that email = human.
6. **Module attribution:** For each module in the current Module Map, run `git log --format="%ae" --diff-filter=A -- src/{module}.ts | tail -1` to find the creating commit's author. Cache in `growth-cache.json` as `moduleAttribution`.

This gives us the animated growth story: 8 modules → 43 modules, with each data point dated and attributed. Cache build is a one-time cost (~10-30s), subsequent loads are instant.

### File Watching Strategy

```typescript
// Watch .garyclaw/ recursively
const watcher = fs.watch(garyclawDir, { recursive: true }, (eventType, filename) => {
  // Debounce: accumulate changed filenames in a Set for 500ms, then batch-process
  // This handles atomic writes (safeWriteJSON does tmp + rename = 2 events per write)
  //
  // Map filename to SSE event type:
  //   daemon-state.json → job_update
  //   global-budget.json → budget
  //   oracle-memory/* → decision
  //   priority.md → priority
  //   pipeline-outcomes.jsonl → mutation
  //
  // IMPORTANT: Use safeReadJSON() / safeReadText() from safe-json.ts for all reads.
  // This handles the rename race from atomic writes (safeWriteJSON writes to .tmp then renames).
  // Never use raw readFileSync — it may read a partial/empty file mid-atomic-write.
  //
  // Re-read only the changed file, recompute affected DashboardData slice
  // Push typed SSE event to all connected clients
});
```

Each file change maps to a specific SSE event type so the client only re-renders the affected DOM subtree.

## Open Questions

1. **Growth data granularity:** Do we show daily snapshots or per-commit? Per-commit is more dramatic (you see the step function) but the 200-commit cap means older history is lost for repos with 2000+ CLAUDE.md-touching commits. For GaryClaw today (~100 CLAUDE.md commits), per-commit works fine. Revisit if the cap becomes limiting.

2. **`fs.watch` noise under heavy parallel load:** With 5 parallel daemon instances, `.garyclaw/` can see dozens of file writes per second. The 500ms debounce batches these, but each debounce cycle re-reads changed files and broadcasts to all SSE clients. Monitor for performance issues under real load. Mitigation: filter `fs.watch` events to only the ~10 known filenames listed in the Data Sources table, ignoring transient files (checkpoint.prev.json, job subdirectory churn).
