# Design: Auto-Fix Loop After Post-Merge Revert

## Problem Statement

When post-merge verification catches a regression and auto-reverts, the daemon creates a P2 bug TODO and idles until the next prioritize cycle picks it up. The daemon already has everything it needs to attempt an immediate fix: test output, branch name, revert SHA, original job context. But it throws that context away and waits 10-60 minutes for prioritize to rediscover it.

This is the last gap in the self-healing cycle. The operator should wake up to a clean main, not a reverted main with a pending bug TODO.

## What Makes This Cool

The daemon becomes a closed-loop self-healing system. Bad merge? Reverted in seconds, auto-fix attempted immediately, and if the fix works, merged back to main. The operator wakes up to a clean main branch with zero regressions. "The daemon merged 14 features overnight, caught 1 regression, auto-reverted, fixed the regression, re-merged, and kept going."

## Constraints

- Max 2 auto-fix attempts per original merge SHA (prevents infinite revert-fix loops)
- Budget cap: 2x the original job's cost (a $4 job can spend up to $8 on auto-fix)
- Opt-in: `config.merge.autoFixOnRevert: false` (default). This is a significant behavior change.
- Auto-fix retry count must be independent from crash-recovery `retryCount` (different semantics, different caps)
- Auto-fix jobs must skip composition entirely (pipeline is always `implement -> qa`, hardcoded)
- The bug TODO is still created on revert (as today). Auto-fix just also enqueues work immediately.
- If auto-fix succeeds and merges, the bug TODO should be auto-marked complete.
- If auto-fix fails (2 retries exhausted), the bug TODO remains for prioritize to handle.
- Named instances only (default instance has no worktree branch to fix in).

## Premises

1. Immediate re-enqueue after revert is the right response. The daemon has the test output, the branch, and the revert SHA. Waiting for prioritize is wasteful.
2. Retry cap of 2 prevents infinite loops. Fix introduces new regression -> revert -> fix -> revert -> STOP, leave bug TODO.
3. `implement -> qa` is the right pipeline shape. The code exists (just broken). No need for prioritize, office-hours, or eng-review.
4. Opt-in default is correct. Auto-enqueue after failure is a big behavior change.
5. Budget guard (2x original cost) prevents runaway spending.
6. Retry tracking must be separate from crash-recovery retryCount. Crash recovery retries the same job. Auto-fix creates a NEW job targeting a specific regression. Different lifecycle, different cap.

## Approaches Considered

### Approach A: Inline in handlePostMergeVerification

Add auto-fix logic directly inside `handlePostMergeVerification()` in job-runner.ts. After creating the bug TODO, call `enqueue()` with `implement -> qa`.

Effort: XS | Risk: Low
Pros: Fewest files changed. Leverages existing machinery.
Cons: handlePostMergeVerification grows past 100 lines. Retry tracking collides with crash-recovery retryCount.

### Approach B: Separate Auto-Fix Coordinator (CHOSEN)

New module `src/auto-fix.ts` (~80 lines) encapsulating retry tracking, budget guards, and enqueue decisions. `handlePostMergeVerification` calls `maybeEnqueueAutoFix()` after revert. Coordinator owns its own state file.

Effort: S | Risk: Low
Pros: Clean separation. Independent retry tracking. Testable budget guards. Survives daemon restart.
Cons: One new module + test file.

### Approach C: Event-Driven Daemon Hook

Emit `"merge_reverted"` event, daemon listens and decides. Most extensible but over-engineered for one consumer.

Effort: S | Risk: Medium
Pros: Extensible. Cons: Over-engineered, harder to test.

## Recommended Approach

**Approach B: Separate Auto-Fix Coordinator.** Clean separation, independent retry state, testable in isolation.

### New Module: `src/auto-fix.ts` (~80-100 lines)

```typescript
// ── Types ──────────────────────────────────────────────────────

export interface AutoFixState {
  /** Key: original merge SHA. Value: retry metadata. */
  entries: Record<string, AutoFixEntry>;
}

export interface AutoFixEntry {
  originalMergeSha: string;
  originalJobId: string;
  originalJobCost: number;
  bugTodoTitle: string;      // the P2 bug TODO title, used to claim the TODO for the auto-fix job
  retryCount: number;        // 0, 1, or 2
  totalAutoFixCost: number;  // accumulated across retries
  createdAt: string;         // ISO timestamp
  lastAttemptAt?: string;
}

export const MAX_AUTO_FIX_RETRIES = 2;
export const AUTO_FIX_BUDGET_MULTIPLIER = 2;  // 2x original job cost
```

### Core Function: `maybeEnqueueAutoFix()`

```typescript
export function maybeEnqueueAutoFix(ctx: {
  projectDir: string;
  mergeSha: string;
  jobId: string;
  jobCost: number;
  instanceName: string;
  skills: string[];          // original pipeline skills
  testOutput?: string;       // from verifyPostMerge
  revertSha?: string;
  bugTodoTitle: string;      // the P2 bug TODO title just created
  enqueue: (skills: string[], triggeredBy: Job["triggeredBy"], detail: string) => string | null;
  log: (level: string, msg: string) => void;
  config: { autoFixOnRevert?: boolean };
}): { enqueued: boolean; reason: string } {
  // 1. Gate: config check
  if (!ctx.config.autoFixOnRevert) {
    return { enqueued: false, reason: "autoFixOnRevert disabled" };
  }

  // 2. Read state (project-level, shared across instances)
  const state = readAutoFixState(ctx.projectDir);

  // 3. Check retry cap
  const entry = state.entries[ctx.mergeSha];
  if (entry && entry.retryCount >= MAX_AUTO_FIX_RETRIES) {
    ctx.log("info", `Auto-fix retry cap reached for ${ctx.mergeSha} (${entry.retryCount}/${MAX_AUTO_FIX_RETRIES})`);
    return { enqueued: false, reason: "retry_cap_reached" };
  }

  // 4. Check budget cap
  const budgetCap = ctx.jobCost * AUTO_FIX_BUDGET_MULTIPLIER;
  const spent = entry?.totalAutoFixCost ?? 0;
  if (spent >= budgetCap) {
    ctx.log("info", `Auto-fix budget cap reached for ${ctx.mergeSha} ($${spent.toFixed(2)}/$${budgetCap.toFixed(2)})`);
    return { enqueued: false, reason: "budget_cap_reached" };
  }

  // 5. Build enqueue detail
  const nextRetry = (entry?.retryCount ?? 0) + 1;
  const detail = [
    `auto-fix attempt ${nextRetry}/${MAX_AUTO_FIX_RETRIES}`,
    `for revert of ${ctx.mergeSha.slice(0, 8)}`,
    ctx.testOutput ? `\nTest output:\n${ctx.testOutput.slice(0, 1500)}` : "",
  ].join(" ");

  // 6. Enqueue implement -> qa with context
  //    IMPORTANT: enqueue BEFORE persisting state. If enqueue fails (dedup, budget),
  //    we don't consume a retry slot.
  const jobId = ctx.enqueue(
    ["implement", "qa"],
    "post-merge-revert",  // new triggeredBy variant
    detail,
  );

  if (!jobId) {
    return { enqueued: false, reason: "enqueue_failed" };
  }

  // 7. Persist state ONLY after successful enqueue
  const newEntry: AutoFixEntry = {
    originalMergeSha: ctx.mergeSha,
    originalJobId: ctx.jobId,
    originalJobCost: ctx.jobCost,
    bugTodoTitle: ctx.bugTodoTitle,
    retryCount: nextRetry,
    totalAutoFixCost: spent,
    createdAt: entry?.createdAt ?? new Date().toISOString(),
    lastAttemptAt: new Date().toISOString(),
  };
  state.entries[ctx.mergeSha] = newEntry;
  writeAutoFixState(ctx.projectDir, state);

  ctx.log("info", `Auto-fix enqueued: job ${jobId}, attempt ${nextRetry}`);
  return { enqueued: true, reason: "enqueued" };
}
```

State file lives at `{projectDir}/.garyclaw/auto-fix-state.json` (project-level, shared across all instances). Both `readAutoFixState` and `writeAutoFixState` accept `projectDir`, NOT `checkpointDir` (which resolves to per-instance dirs like `.garyclaw/daemons/{name}/`). Shared state is correct here because auto-fix retry caps apply per merge SHA regardless of which instance caused the revert. Entries keyed by merge SHA. Pruned on read: entries older than 24h are removed (regressions that old are no longer auto-fixable, they're prioritize's problem).

**Locking:** Use the existing `budget-lock.ts` mkdir-based advisory lock pattern around read+write in both `maybeEnqueueAutoFix` and the cost accumulation path. Lock dir: `{projectDir}/.garyclaw/auto-fix-lock/`. Same timeout/retry semantics as budget lock (500ms timeout, 50ms poll). Concurrent reverts of the same SHA are extremely unlikely (only one instance can be merging a given branch), but parallel instances merging different branches could race on the shared state file during cost accumulation.

### Auto-Fix Job Behavior

The auto-fix job is a normal `implement -> qa` pipeline with special properties:

1. **skipComposition: true** — pipeline is always `implement -> qa`, no composition override
2. **triggeredBy: "post-merge-revert"** — identifies the job source for dashboard/audit
3. **claimedTodoTitle: bugTodoTitle** — the enqueued job's `claimedTodoTitle` is set to the bug TODO title from the `AutoFixEntry.bugTodoTitle` field. This is how the job claims the bug TODO and how `catchUpCompletedTodos` matches the job back to the TODO on success.
4. **autoFixMergeSha: mergeSha** — typed field on Job for cost accumulation (no fragile string parsing from triggerDetail)

6. **On auto-fix job completion:** if the job succeeds and merges, the bug TODO auto-marks as complete via existing `catchUpCompletedTodos` (which scans TODO state for items in `merged` state and marks them in TODOS.md). The auto-fix job's `claimedTodoTitle` ensures the TODO state file maps to the bug TODO. If the fix fails or gets reverted again, `maybeEnqueueAutoFix` is called again with the SAME merge SHA, incrementing retryCount.

### Budget Tracking for Auto-Fix Attempts

The budget cap is **best-effort, checked pre-enqueue**. It cannot enforce mid-run because a job's cost is only known after completion. This means the actual total spend can exceed the cap by up to one job's cost (e.g., a $4 original job with 2x cap = $8 limit, but if attempt 1 costs $6, the cap is already exceeded before attempt 2 is checked). This is acceptable: the retry cap (max 2) is the hard safety limit, the budget cap is a soft guard.

When an auto-fix job completes (success or failure), the cost is accumulated back to the auto-fix state entry. Wire this in the job completion handler:

```typescript
// In processNext(), after job completes, if triggeredBy === "post-merge-revert":
// Read job.autoFixMergeSha (typed field on Job, no string parsing needed)
// Update state.entries[mergeSha].totalAutoFixCost += job.costUsd
const state = readAutoFixState(jobConfig.projectDir);
const sha = nextJob.autoFixMergeSha;
if (sha && state.entries[sha]) {
  state.entries[sha].totalAutoFixCost += nextJob.costUsd ?? 0;
  writeAutoFixState(jobConfig.projectDir, state);
}
```

## Open Questions

1. **Should auto-fix jobs run on the SAME instance that caused the revert, or any available instance?** Currently: enqueue goes to the same instance's queue (since `handlePostMergeVerification` runs in the job runner's processNext). This is probably correct: the instance has the worktree with the broken code.

2. **Should the implement prompt for auto-fix include the original job's design doc context?** The bug TODO has the test output, but the original design doc might help. Could thread `designDocPath` from the original job's TODO state. Deferring: test output alone should be sufficient for most regressions.

3. **24h prune window: is that right?** A regression from 12 hours ago that's been retried twice already is done. But what if the daemon was stopped for 20 hours and restarts? The entry is pruned and auto-fix becomes available again. This is probably fine: a fresh attempt after 20h might succeed with different context.

## Success Criteria

- Auto-fix triggers within 5 seconds of a post-merge revert (no waiting for next prioritize cycle)
- Retry cap prevents infinite loops (verified by test with 3+ consecutive reverts)
- Budget cap prevents runaway spending (verified by test with high-cost original job)
- Config gate prevents auto-fix when `autoFixOnRevert: false` (default)
- Dashboard shows auto-fix stats (attempts, success rate, cost)
- Doctor detects stale auto-fix state
