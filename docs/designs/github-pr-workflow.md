# Design: GitHub PR Workflow — Create PRs Instead of Direct Merge

## Problem Statement

GaryClaw's daemon instances currently merge their worktree branches directly to main via rebase + ff-only merge. This works for solo development but lacks:

1. **Reviewable audit trail** — no GitHub-visible record of what the daemon did and why
2. **Human intervention point** — no way to reject bad changes before they hit main
3. **CI integration** — no external quality gate beyond GaryClaw's own pre-merge tests
4. **Team adoption** — teams expect PR-based workflows; direct merge to main is a non-starter

The fix: add an optional PR-based merge strategy that creates structured GitHub PRs with pipeline summaries, oracle decisions, and test results. Configurable via `merge.strategy: "pr"`.

## What Makes This Cool

The daemon creates a PR that reads like a senior engineer's summary. Not just "here's the diff" but "here's what I was asked to do, what I decided along the way, what tests I ran, what the oracle said, and what issues I found and fixed." The PR body becomes the audit trail that makes autonomous work trustable.

Combined with GitHub auto-merge, this creates a fully autonomous loop where the daemon pushes code, CI validates it externally, and the merge happens without human involvement... but humans CAN intervene at any point by reviewing the open PR.

## Constraints

- **`gh` CLI required** for PR mode (already installed, v2.83.2 on this machine)
- **No new npm dependencies** — shell out to `gh`, don't add octokit
- **Backward compatible** — `merge.strategy: "direct"` (default) preserves exact current behavior
- **Named instances only** — default instance works on main directly, no branch to PR from
- **GitHub auto-merge requires repo setting** — must be enabled in GitHub repo settings for `--auto` flag to work. Design should handle the case where it's not enabled.

## Premises

1. `gh` CLI is the right tool. Already installed, no new deps, uses existing GitHub auth.
2. This is additive, not a replacement. Direct merge stays as the default.
3. The structured PR body is the primary value — it's what makes this worth doing.
4. Auto-merge after CI should be opt-in, not forced. Some users want PRs as human gates.
5. Existing merge flow (rebase + ff-only + pre-merge tests + post-merge verify) is untouched for `strategy: "direct"`.

## Approaches Considered

### Approach A: Minimal — gh CLI shell-out in job-runner

Add `createPullRequest()` to worktree.ts. Job-runner calls it instead of `mergeWorktreeBranch` when `strategy: "pr"`. PR body built from existing job/pipeline data.

- **Effort:** S
- **Risk:** Low
- **Pros:** No deps, ~150 lines, uses existing rebase flow, simple
- **Cons:** No CI wait, no auto-merge, PR just sits there

### Approach B: Full PR lifecycle — create + wait for CI + auto-merge

Same as A plus `waitForPrChecks()` that polls `gh pr checks`. Auto-merges via `gh pr merge` when checks pass.

- **Effort:** S-M
- **Risk:** Medium (polling complexity, timeout, rate limits)
- **Pros:** Full autonomous loop with visibility into CI status
- **Cons:** Polling code is new complexity, GitHub rate limits, timeout edge cases

### Approach C: Hybrid — PR creation + GitHub auto-merge flag (CHOSEN)

Create the PR via `gh pr create`, then enable GitHub auto-merge via `gh pr merge --auto --squash`. No polling. GitHub merges when checks pass. GaryClaw moves on to the next job immediately.

- **Effort:** S
- **Risk:** Low
- **Pros:** Simplest full-autonomy path, zero polling, ~120 lines of new code
- **Cons:** Requires repo-level auto-merge setting, TODO state update deferred, no post-merge verification (merge happens on GitHub's side)

## Recommended Approach

**Approach C: Hybrid.** Best balance of simplicity and autonomy. The daemon creates a well-structured PR, pushes the branch, enables auto-merge, and immediately starts the next job. GitHub handles CI gating asynchronously.

The one tradeoff (deferred TODO state) is handled by checking PR merge status at the start of the next job cycle, or via a lightweight `gh pr view` check.

#### Flow Changes in job-runner.ts

The post-job merge section (line ~880) gets a strategy check:

```
if merge.strategy === "pr":
  1. Push branch to remote: git push origin garyclaw/{instance} --force-with-lease
  2. Build PR body from job context (buildPrBody)
  3. Create PR: gh pr create --title "..." --body "..." --base main --head garyclaw/{instance}
  4. If prAutoMerge: gh pr merge {number} --auto --squash
  5. If prReviewers: gh pr edit {number} --add-reviewer {reviewers}
  6. Update TODO state to "pr-created" (new state)
  7. Log to merge-audit.jsonl with PR number/URL
  8. Notify: "PR #{number} created: {url}"

if merge.strategy === "direct" (or undefined):
  (existing flow — unchanged)
```

#### New TODO State: "pr-created"

Add `"pr-created"` to the TODO state lifecycle, between "qa-complete" and "merged":

```
claimed → implementing → qa-complete → pr-created → merged → complete
```

This captures the window where the PR exists but hasn't been merged yet. Reconciliation can check `gh pr view` to detect when GitHub auto-merge completes.

#### PR Body Size Guard

GitHub's PR body limit is 65,536 characters. `buildPrBody` must truncate:
- Oracle decisions: cap at 10 most interesting (highest/lowest confidence)
- Test output: cap at 1000 chars
- Issues: cap at 20
- Total body: hard cap at 60,000 chars with "... truncated" marker

#### Error Handling

- **`gh` not installed:** Log warning, fall back to direct merge. Event: `merge_strategy_fallback`.
- **`gh` not authenticated:** Log warning, fall back to direct merge.
- **Push fails (branch already has open PR):** Update existing PR body instead of creating new. `gh pr edit`.
- **Auto-merge not enabled on repo:** `gh pr merge --auto` fails gracefully. Log warning, PR stays open for manual merge. Not a job failure.
- **Network failure:** PR creation is best-effort after successful job. Job is still marked complete. Merge-audit logs the failure for dashboard visibility.

#### Pre-merge Tests with PR Strategy

When `strategy: "pr"`, pre-merge tests still run in the worktree before pushing. This catches obvious failures before creating a PR. The flow:

1. Rebase onto baseBranch (existing)
2. Run pre-merge tests in worktree (existing, if configured)
3. If tests fail: don't create PR, log failure (existing behavior)
4. If tests pass: push + create PR (new)

This means CI on the PR is a second quality gate, not the only one.

#### Dashboard Integration

Add PR stats to dashboard:
- PRs created (total, by instance)
- PRs auto-merged vs manually merged vs still open
- Avg time from PR creation to merge (if trackable via `gh pr view`)

## Open Questions

1. **PR merge status reconciliation:** When should GaryClaw check if auto-merged PRs actually merged? Options: (a) start of each job, (b) periodic check in daemon poll loop, (c) GitHub webhook. Recommend (a) for simplicity.
2. **Squash vs merge commit:** Default to squash for clean history, but some teams want merge commits for traceability. Config option covers this.
3. **Branch cleanup after PR merge:** GitHub can auto-delete branches after PR merge. Should GaryClaw also clean up its worktree? Probably yes, but deferred to a follow-up.

## Success Criteria

1. `merge.strategy: "pr"` creates a GitHub PR with structured body after successful job
2. PR body includes pipeline summary, oracle decisions, test results, and issues
3. `prAutoMerge: true` enables GitHub auto-merge on the PR
4. Existing `strategy: "direct"` behavior is completely unchanged
5. Fallback to direct merge when `gh` is unavailable
6. All new code has tests (target: 15-20 tests)
7. Dashboard shows PR stats
