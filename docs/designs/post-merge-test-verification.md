# Design: Post-Merge Test Verification + Auto-Revert Safety Net

## Problem Statement

The daemon auto-merges worktree branches to main after pipeline completion. A pre-merge test gate already exists (worktree.ts lines 402-448), running tests on the rebased branch before the ff-only merge. But there is no verification that main itself is healthy after the merge lands. If a regression slips through (pre-merge gate skipped, environment-dependent test, worktree vs main-repo behavioral difference), main stays broken and every subsequent pipeline cycle compounds the damage.

This is the single biggest safety gap in overnight autonomous operation. The daemon has no feedback loop between "merged to main" and "main is healthy." Defense-in-depth means testing twice: once before merge (existing), once after (this design).

## What Makes This Cool

The daemon becomes self-healing. Bad merge? Reverted in seconds, automatically. The operator wakes up to a clean main branch with a detailed revert log explaining what happened and a P2 bug TODO already queued for the next cycle. Zero human intervention required.

"The daemon merged 14 features overnight, caught 1 regression post-merge, auto-reverted it, filed a bug, and kept going."

## Constraints

- Must not duplicate work when pre-merge gate is active (if pre-merge tests passed, post-merge verification should be fast or skippable)
- Must not hold the merge lock during post-merge verification (other instances need to merge)
- Auto-revert must be atomic (`git revert HEAD --no-edit`) and must not break if another instance merged between our merge and our verification
- Must be configurable: `config.merge.skipPostMergeVerification` opt-out for repos with slow test suites that already trust the pre-merge gate
- Revert must create a JSONL audit trail and a P2 bug TODO for the next prioritize cycle
- Must extend existing dashboard with revert count (not a separate dashboard section)

## Premises

1. Post-merge verification adds meaningful safety as defense-in-depth. The pre-merge gate tests in the worktree; post-merge tests in the actual main repo directory. Environment differences (node_modules state, .env files, working directory assumptions) can cause divergent results.
2. Auto-revert is the correct response. Unlike pre-merge failure (where we just skip the merge), post-merge failure means broken code is already on main. Notify-only is insufficient for overnight operation, reverts are the only action that preserves main health.
3. The window between merge and verification is small (seconds), but another instance could merge in that window. The revert must target a specific commit SHA, not just `HEAD`.
4. Creating a P2 bug TODO closes the feedback loop. The daemon will pick it up on the next prioritize cycle and attempt to fix whatever broke.
5. This is S-effort. One new function, wiring in one existing file, extensions to two existing modules.

## Approaches Considered

### Approach A: Minimal Post-Merge Verify + Revert

**Summary:** New `verifyPostMerge()` function in worktree.ts. After successful merge, run test command on main repo. If tests fail, `git revert <merge-sha> --no-edit`. Return result. Wire into job-runner.ts.

**Effort:** XS (human: ~1 day / CC: ~10 min)
**Risk:** Low
**Pros:**
- Smallest diff (~50 lines in worktree.ts, ~20 in job-runner.ts)
- Prevents the worst failure mode (broken main overnight)
- Uses existing test command from merge config

**Cons:**
- No audit trail for reverts (only daemon log)
- No automatic bug TODO creation
- No dashboard visibility into revert rate
- Completeness: 4/10

### Approach B: Verify + Revert + Audit + Bug TODO + Dashboard (RECOMMENDED)

**Summary:** Full safety net. `verifyPostMerge()` with SHA-targeted revert, append to `merge-reverts.jsonl`, create P2 bug TODO in TODOS.md with failure details, extend `failure-taxonomy.ts` with `"test-regression"` category, add revert count to existing dashboard merge health section, notification on revert. Configurable via `config.merge.skipPostMergeVerification`.

**Effort:** S (human: ~3 days / CC: ~25 min)
**Risk:** Low
**Pros:**
- Complete safety net with full feedback loop
- Audit trail enables debugging "what was reverted and why?"
- Bug TODO means the daemon will try to fix it next cycle
- Dashboard shows revert rate alongside merge health
- Notification wakes you up only if something was reverted
- Completeness: 9/10

**Cons:**
- Touches 5 source files (worktree.ts, job-runner.ts, failure-taxonomy.ts, notifier.ts, dashboard.ts)
- ~60 new tests
- TODOS.md mutation from within job-runner is a new pattern (but markTodoCompleteInFile already does this)

### Approach C: Post-Merge Verify + Auto-Revert + Auto-Fix Loop

**Summary:** On revert, immediately enqueue a fix pipeline (`implement → qa`) targeting the specific regression. The daemon doesn't just revert, it tries to fix the problem.

**Effort:** M (human: ~1 week / CC: ~1 hour)
**Risk:** Medium
**Pros:**
- Fully self-healing, not just self-protecting
- Could resolve the regression before the operator wakes up

**Cons:**
- Risk of infinite loop (fix introduces new regression, revert, fix again...)
- Budget implications ($3-5 per fix attempt)
- Complex interaction with prioritize skill (fix job vs normal backlog)
- Premature, better to add after Approach B proves the revert pattern works
- Completeness: 10/10 (but disproportionate risk)

## Recommended Approach

**Approach B: Verify + Revert + Audit + Bug TODO + Dashboard.**

Approach A is too bare for a system that runs overnight. You need to know what was reverted, why, and the daemon needs to know it should fix it. Approach C is the right next step after B proves itself, but building the auto-fix loop before the basic revert works is premature.

## Implementation Order

### Step 1: `verifyPostMerge()` in worktree.ts

New exported function after `mergeWorktreeBranch()`:

```typescript
export interface PostMergeVerifyResult {
  verified: boolean;          // true = tests passed on main
  reverted: boolean;          // true = auto-revert executed
  revertSha?: string;         // SHA of the revert commit (if reverted)
  mergeSha: string;           // SHA of the merge commit being verified
  testOutput?: string;        // truncated stderr (max 2000 chars)
  testDurationMs?: number;
  reason?: string;            // human-readable failure reason
}

export function verifyPostMerge(
  repoDir: string,
  mergeSha: string,
  options?: {
    testCommand?: string;     // default: "npm test"
    testTimeout?: number;     // default: 120000
  },
): PostMergeVerifyResult {
```

Logic:
1. Verify we're on the base branch and `HEAD` matches or is ahead of `mergeSha` (another instance may have merged on top)
2. Run `options.testCommand ?? "npm test"` in `repoDir` (NOT worktree, this is the main repo)
3. If tests pass: return `{ verified: true, reverted: false, mergeSha, testDurationMs }`
4. If tests fail:
   a. Check if `HEAD === mergeSha`. If yes, `git revert <mergeSha> --no-edit`. If no (another merge landed on top), skip revert and return `{ verified: false, reverted: false, reason: "HEAD moved past merge SHA — manual revert needed" }`
   b. On successful revert: return `{ verified: false, reverted: true, revertSha: <new HEAD>, mergeSha, testOutput, testDurationMs }`
   c. On revert failure (conflict): return `{ verified: false, reverted: false, reason: "Revert had conflicts — manual intervention needed", testOutput, testDurationMs }`

**Key design decision: SHA-targeted revert.** We store the merge commit SHA immediately after `mergeWorktreeBranch()` returns successfully. If `HEAD` has moved past that SHA by the time we verify, we can't safely revert (another instance's work is on top). In that case, notify and leave for manual resolution. This is the safe default.

**Merge lock is NOT held during post-merge verification.** The existing merge lock protects rebase+test+merge atomicity. Post-merge verification is a separate concern that runs after the lock is released. This means another instance CAN merge between our merge and our verification. The SHA check handles this.

**Cascading failure scenario:** Instance A merges broken code, Instance B merges on top before A's verification runs. A's verification fails, but `HEAD !== mergeSha` so A skips revert and returns "manual revert needed." B's verification also fails (A's broken code plus B's code). B can revert B's commit, but A's broken commit remains. Main is still broken.

Mitigation: when `HEAD !== mergeSha` and tests fail, log a `git log --oneline mergeSha..HEAD` listing all commits between the merge and current HEAD. Include this in the notification so the operator has full context. The revert-all-to-known-good approach is tempting but too dangerous for v1 (it would revert other instances' valid work). This scenario is rare in practice (requires two broken merges in the same ~60s window) and resolves itself on the next pre-merge gate (Instance C will fail pre-merge tests and not merge). Accept this as a known limitation and revisit if it occurs in production.

### Step 3: Wire into job-runner.ts

In the auto-merge block (job-runner.ts ~line 741), after `mergeResult.merged` is true:

```typescript
if (mergeResult.merged && mergeResult.commitCount && mergeResult.commitCount > 0) {
  // Post-merge verification (defense-in-depth)
  const skipPostVerify = mergeConfig?.skipPostMergeVerification ?? false;
  // Also skip if pre-merge tests already passed (same test command = redundant)
  const preMergePassed = mergeResult.testsPassed === true;

  if (!skipPostVerify && !preMergePassed) {
    try {
      // Capture merge SHA from main repo HEAD. mergeWorktreeBranch() doesn't return
      // the merge SHA directly (MergeResult has no sha field), so we read HEAD here
      // immediately after merge while we know HEAD === the merge commit.
      const mainHead = execFileSync("git", ["rev-parse", "HEAD"], {
        cwd: jobConfig.projectDir, stdio: "pipe", encoding: "utf-8",
      }).trim();

      const verifyResult = verifyPostMerge(jobConfig.projectDir, mainHead, {
        testCommand: mergeConfig?.testCommand,
        testTimeout: mergeConfig?.testTimeout,
      });

      if (verifyResult.verified) {
        d.log("info", `Post-merge verified: tests pass on main` +
          (verifyResult.testDurationMs ? ` (${Math.round(verifyResult.testDurationMs / 1000)}s)` : ""));
      } else if (verifyResult.reverted) {
        d.log("warn", `POST-MERGE REVERT: ${verifyResult.reason}` +
          (verifyResult.testOutput ? `\n${verifyResult.testOutput.slice(0, 500)}` : ""));
        // Audit, notify, create bug TODO (Steps 4-5)
      } else {
        d.log("warn", `Post-merge verification failed but revert skipped: ${verifyResult.reason}`);
      }
    } catch (err) {
      d.log("warn", `Post-merge verification error: ${err instanceof Error ? err.message : String(err)}`);
    }
  }
}
```

**Smart skip: if pre-merge tests passed, skip post-merge verification by default.** When `mergeResult.testsPassed === true`, the pre-merge gate already ran the same tests on code that is byte-identical to main after the ff-only merge. In the overwhelming majority of cases, re-running is pure waste. The primary risk this feature addresses is when the pre-merge gate is skipped or not configured, not environment differences between worktree and main (which are theoretically possible but rare in practice since both share the same git objects and typically the same node_modules via symlink or install).

For operators who want maximum paranoia, `config.merge.forcePostMergeVerification: true` overrides the smart skip and always verifies post-merge, even when pre-merge passed. This is the escape hatch for repos with known worktree-vs-main divergence (e.g., environment-dependent tests, .env files that differ).

This is the key optimization that prevents doubling test time. In the common case (pre-merge gate active), post-merge adds zero overhead.

### Step 5: Bug TODO creation + notification

On successful revert, create a P2 bug TODO in TODOS.md:

```typescript
function createRegressionTodo(
  todosPath: string,
  instanceName: string,
  jobId: string,
  testOutput: string,
): boolean {
  // Append a new ## heading to TODOS.md
  const title = `P2: Fix post-merge regression from ${instanceName} (job ${jobId})`;
  const body = [
    `\n## ${title}\n`,
    `**What:** Post-merge test verification failed after auto-merge to main. The merge was auto-reverted.`,
    `**Priority:** P2 (auto-generated safety item)`,
    `**Effort:** S`,
    `**Test output (truncated):**`,
    "```",
    testOutput.slice(0, 1000),
    "```",
    `**Branch:** \`garyclaw/${instanceName}\` (still exists after revert, check out to debug)`,
    `**Added by:** Post-merge safety net on ${new Date().toISOString().slice(0, 10)}`,
    "",
  ].join("\n");

  // Use reflection lock to prevent concurrent TODOS.md appends from parallel instances.
  // Multi-line markdown appends can garble if two instances write simultaneously.
  // The reflection lock (mkdir-based, same as oracle-memory writes) is already imported
  // in job-runner.ts for post-job reflection. Pass the .garyclaw directory as the base;
  // acquireReflectionLock appends its own lock subdirectory name internally.
  const gcDir = join(dirname(todosPath), ".garyclaw");
  const acquired = acquireReflectionLock(gcDir, 5000); // 5s timeout, short
  try {
    appendFileSync(todosPath, body, "utf-8");
  } finally {
    if (acquired) releaseReflectionLock(lockDir);
  }
  return true;
}
```

This integrates with the existing prioritize skill. The next time any daemon instance runs `prioritize`, it will see this P2 item and score it highly (P2 priority, auto-generated safety item).

export function notifyMergeReverted(
  job: Job,
  verifyResult: PostMergeVerifyResult,
  config: DaemonConfig,

  sendNotification(title, message);
}
```

## Open Questions

1. **Should post-merge verification hold the merge lock?** Current design says no, to avoid blocking other instances. But this means another instance could merge between our merge and our verification, making the SHA check fail. In practice this is rare (verification takes ~30-60s, merge lock acquisition + merge takes ~5-10s). If it becomes a problem, we can add a brief post-merge lock.

2. **Should the bug TODO include the branch name for easy investigation?** Yes. The branch (`garyclaw/{instance}`) still exists after revert. The operator (or daemon) can check it out and debug.

3. **Should we skip post-merge verification when commitCount is 0?** Yes. Zero-commit merges ("already up to date") don't change main. This is already handled by the `commitCount > 0` guard.

## Success Criteria

- Zero broken mains overnight: post-merge regression is caught and reverted within 60s
- Revert is SHA-targeted: safe even with parallel instances merging concurrently
- Smart skip: zero overhead when pre-merge gate is active (the common case)
- Bug TODO created: the daemon will attempt to fix the regression on the next cycle
- Dashboard shows revert count alongside merge health
- Backward compatible: existing configs work exactly as before (post-merge verification defaults to enabled)
