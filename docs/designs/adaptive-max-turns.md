## Problem Statement

GaryClaw's orchestrator uses a fixed `maxTurnsPerSegment=15` for every segment. Browse-heavy phases (screenshots, WebFetch) consume 30-150K tokens per turn. Edit-only phases consume 5-20K tokens per turn. At 15 turns of browse-heavy work, the context can blow past the relay threshold (850K of 1M) before the deferred relay check fires at segment end. This wastes context, causes emergency relays with incomplete work, and makes autonomous runs less resilient.

The token monitor already has `computeGrowthRate()` which returns tokens/turn over a sliding window. It's defined, tested, and never called in the orchestrator. The orchestrator already extracts tool names from every message via `extractToolUse()`. Both signals are free. Adaptive maxTurns connects them to segment sizing.

## What Makes This Cool

The orchestrator becomes self-sizing. Browse-heavy skills like /qa and /design-review automatically get shorter segments (3-8 turns). Edit-heavy skills like /implement get longer segments (15-25 turns). No config tuning. No per-skill profiles.

The tool-use lookahead is the interesting bit. When the orchestrator sees a `WebFetch` or `Screenshot` tool call in segment N, it knows the next segment will likely be heavy too. This is like TCP's ECN (Explicit Congestion Notification) vs. pure loss-based detection. The tool invocation IS the ECN bit. The system reacts to mode shifts within 0 turns of delay instead of 2-3.

The delta between "burns through context and emergency-relays" and "precisely sizes every segment" is the difference between a dumb automation loop and something that feels like a senior engineer managing their own context window.

## Constraints

- SDK's `maxTurns` is set once per `startSegment()` call. Cannot be changed mid-segment.
- First segment of first session has no growth rate data (no prior turns to measure).
- The relay threshold (default 0.85) serves as the safety net. Adaptive maxTurns is a prediction, not a guarantee.
- Must not break existing `--max-turns` CLI behavior. The user's value becomes a ceiling.
- Must remain testable with synthetic data (no SDK calls in tests).
- XS effort scope: 1 new function, 1 constant set, 2-3 call site changes, ~60 lines production code.

## Premises

1. **Growth rate is predictive enough.** The average tokens/turn over the last 5 turns predicts the next 5 turns within 2x. Browse-heavy phases sustain high rates; edit phases sustain low rates. Phase transitions (browse -> edit) happen at segment boundaries, which is exactly when we recalculate. Note: `computeGrowthRate(state, windowSize)` slices the last `windowSize` entries from `turnHistory` and requires at least 2 data points. If fewer than 2 turns exist, it returns `null` (triggering the configured default fallback). If 2-4 turns exist with a window of 5, it uses all available turns. Short segments always produce a rate estimate after 2 turns.
2. **Tool names are leading indicators.** A `WebFetch` or `Screenshot` tool call means the next turn will be 3-5x heavier than an edit turn. This is a free signal already in the message stream. Using it eliminates the 1-2 turn lag of trailing-only growth rate.
3. **Per-segment adaptation is sufficient.** Mid-segment adaptation would be ideal but requires SDK changes. The relay threshold already catches mid-segment surprises.
4. **The first segment needs a fallback.** With no growth data, use the configured default (15). By segment 2, we have data.
5. **The safety margin matters.** We should target landing at ~72% of the context window (headroomFactor 0.85 * threshold 0.85 = 0.7225), not right at the relay threshold. This leaves ~13% headroom before the relay fires.
6. **Relay segments need it too.** But relay starts with fresh context, so the first relay segment uses the default. By relay segment 2, fresh growth data is available.

## Cross-Model Perspective

A Claude subagent provided an independent cold read:

- **Coolest version they hadn't considered:** Predictive segment sizing via tool-use lookahead. The tool name is a leading indicator (ECN bit), not a trailing one. A `WebFetch` call means the next turn is fat, and you know this before the tokens arrive.
- **Key insight:** "The coolest version learns per-skill growth profiles over time" reveals the real excitement is the learning system (matching GaryClaw's tagline: "gets smarter every run"), not the reactive controller. The reactive controller is table stakes.
- **Existing 50% solution:** Netflix's concurrency-limits library (`AIMDLimit`, `GradientLimit`) implements the same AIMD + gradient pattern. Domain translation needed: concurrency -> maxTurns, latency -> tokens/turn, timeout -> crossed relay threshold.
- **Prototype suggestion:** Include tool lookahead in v1 (not deferred). The cost is ~15 extra lines and it eliminates the trailing-indicator blind spot. This was incorporated into the recommended approach.

## Approaches Considered

### Approach A: Reactive AIMD Controller (Minimal Viable)

Pure growth-rate-based AIMD. Compute remaining context budget, divide by smoothed growth rate, clamp [3, configuredMax]. No tool awareness.

- **Effort:** XS (~40 lines)
- **Risk:** Low
- **Pros:** Simplest possible implementation. Uses only data already tracked.
- **Cons:** Trailing indicator only. Reacts 1-2 turns after a mode shift. Treats all turns equally.
- **Reuses:** `computeGrowthRate()`, `TokenMonitorState`
- **Completeness:** 6/10

### Approach B: Predictive AIMD + Tool Lookahead (Recommended)

Same AIMD core as A, but adds tool-use lookahead. When the orchestrator sees a heavy tool (`WebFetch`, `Screenshot`, `WebSearch`) in segment N, it applies a multiplicative growth rate boost when computing maxTurns for segment N+1. The trailing growth rate then confirms or corrects on subsequent segments.

- **Effort:** XS (~60 lines, adds ~15 lines to Approach A)
- **Risk:** Low
- **Pros:** Eliminates trailing-indicator blind spot. Tool name is a free signal. Still a pure function. The "ECN bit" analogy makes the design self-explanatory.
- **Cons:** Needs a "heavy tools" set that may need tuning over time. Slightly more complex signature.
- **Reuses:** `computeGrowthRate()`, `TokenMonitorState`, `extractToolUse()`
- **Completeness:** 8/10

### Approach C: Full Learning Controller (Deferred)

Approach B + persistent per-skill growth profiles in oracle memory. After each job, write the skill's growth curve to reflection data. On next run, pre-load the profile and set initial maxTurns before turn 1 fires.

- **Effort:** S (adds oracle-memory persistence, profile schema, cold-start handling)
- **Risk:** Medium (schema design, cold-start edge cases, profile staleness)
- **Pros:** True "learns from experience." Optimal from turn 1 on repeated runs.
- **Cons:** Needs 5+ runs per skill to build useful profiles. Profile staleness risk.
- **Reuses:** Everything from B + `oracle-memory.ts`, `reflection.ts`
- **Completeness:** 10/10

## Recommended Approach

**Approach B: Predictive AIMD + Tool Lookahead.** It eliminates the biggest weakness of pure reactive (trailing indicator lag) with minimal extra complexity. The function signature accepts an optional `lastHeavyToolSeen` parameter, so Approach C can layer on top later by seeding the initial growth rate from persisted profiles, with zero API changes.

## Implementation Details

### Heavy Tools Constant

Location: `src/token-monitor.ts`

```typescript
/**
 * Tools that produce large context growth per turn (30-150K tokens).
 * When one of these fires in a segment, the next segment's maxTurns
 * should account for the higher growth rate.
 */
export const HEAVY_TOOLS: ReadonlySet<string> = new Set([
  "WebFetch",
  "WebSearch",
  "Screenshot",
]);

/**
 * Multiplier applied to growth rate when a heavy tool was seen
 * in the current segment. Conservative: assume 2.5x the measured rate.
 */
export const HEAVY_TOOL_GROWTH_MULTIPLIER = 2.5;
```

### New Function: `computeAdaptiveMaxTurns()`

Location: `src/token-monitor.ts`

```typescript
export function computeAdaptiveMaxTurns(
  state: TokenMonitorState,
  relayThresholdRatio: number,
  configuredMaxTurns: number,
  options?: {
    headroomFactor?: number;        // Target this fraction of relay threshold (default: 0.85)
    minTurns?: number;              // Floor (default: 3)
    growthWindowSize?: number;      // Turns for growth rate calc (default: 5)
    lastHeavyToolSeen?: boolean;    // Was a heavy tool used in the previous segment?
  },
): { maxTurns: number; reason: string } {
  const headroomFactor = options?.headroomFactor ?? 0.85;
  const minTurns = options?.minTurns ?? 3;
  const growthWindowSize = options?.growthWindowSize ?? 5;
  const heavyToolSeen = options?.lastHeavyToolSeen ?? false;

  // No data yet — use configured default
  const growthRate = computeGrowthRate(state, growthWindowSize);
  if (growthRate === null || growthRate <= 0 || state.contextWindow === null) {
    return {
      maxTurns: configuredMaxTurns,
      reason: "no growth data yet, using configured default",
    };
  }

  const currentSize = state.turnHistory.length > 0
    ? state.turnHistory[state.turnHistory.length - 1].computedContextSize
    : 0;

  // Target: land at headroomFactor * relayThreshold * contextWindow
  // With defaults (0.85 * 0.85 = 0.7225), we target ~72% of the context window,
  // leaving ~13% headroom before the relay threshold fires.
  const targetSize = state.contextWindow * relayThresholdRatio * headroomFactor;
  const remainingBudget = targetSize - currentSize;

  if (remainingBudget <= 0) {
    return {
      maxTurns: minTurns,
      reason: `already at/past target (${currentSize} >= ${Math.round(targetSize)})`,
    };
  }

  // Apply heavy tool multiplier: if a heavy tool was seen, assume
  // the effective growth rate is higher than the trailing average.
  const effectiveRate = heavyToolSeen
    ? growthRate * HEAVY_TOOL_GROWTH_MULTIPLIER
    : growthRate;

  const predicted = Math.floor(remainingBudget / effectiveRate);
  const clamped = Math.max(minTurns, Math.min(predicted, configuredMaxTurns));

  return {
    maxTurns: clamped,
    reason: `growth ${Math.round(growthRate)} tok/turn` +
            (heavyToolSeen ? ` (heavy tool: x${HEAVY_TOOL_GROWTH_MULTIPLIER})` : "") +
            `, budget ${Math.round(remainingBudget)} tok` +
            `, predicted ${predicted}, clamped to ${clamped}`,
  };
}
```

### Call Site 1: Orchestrator Segment Loop

`src/orchestrator.ts`, inside `runSkillInternal()`'s segment-level `for` loop.

Track heavy tool usage across segments. The flag must be hoisted above the segment loop so that segment N's heavy tool observations influence segment N+1's maxTurns computation. A `previousHeavyToolSeen` / `currentHeavyToolSeen` pair ensures the flag from segment N is read at segment N+1's start, then reset for fresh tracking.

```typescript
// Hoist above the segment loop (alongside relayFlag, etc.)
let previousHeavyToolSeen = false;

// 3. Segment loop
for (let segmentIndex = 0; ; segmentIndex++) {
  // Read the flag from the previous segment, then reset for this segment
  const heavyToolFlag = previousHeavyToolSeen;
  previousHeavyToolSeen = false;  // reset for current segment's tracking

  // ... inside the message processing loop, after extractToolUse():
  if (toolUse && HEAVY_TOOLS.has(toolUse.toolName)) {
    previousHeavyToolSeen = true;  // will be read by the NEXT segment
  }
```

At segment start, compute adaptive maxTurns. No special guard for the first segment... the function itself returns the configured default when `computeGrowthRate()` returns `null` (no turn history yet).

```typescript
// Always call computeAdaptiveMaxTurns — it self-falls-back when no data.
const { maxTurns: adaptiveMaxTurns, reason: adaptiveReason } =
  computeAdaptiveMaxTurns(
    monitor,
    config.relayThresholdRatio,
    config.maxTurnsPerSegment,
    { lastHeavyToolSeen: heavyToolFlag },
  );

callbacks.onEvent({
  type: "adaptive_turns",
  maxTurns: adaptiveMaxTurns,
  reason: adaptiveReason,
  sessionIndex,
  segmentIndex,
});

const segment = startSegment({
  prompt: segmentIndex === 0 ? currentPrompt : "Continue.",
  maxTurns: adaptiveMaxTurns,
  cwd: config.projectDir,
  env: config.env,
  settingSources: config.settingSources,
  canUseTool: askHandler.canUseTool,
  ...(segmentIndex > 0 && sessionId ? { resume: sessionId } : {}),
});
```

**Scoping:** `previousHeavyToolSeen` is declared above the segment loop so it carries from segment N to segment N+1 within the same session. For the very first segment, `heavyToolFlag` is `false` and the function returns the default (no growth data). For relay sessions (fresh `TokenMonitorState`), the function also returns the default because `computeGrowthRate()` returns `null`. No special-case guards needed.

### Call Site 2: Relay Segment Builder

`src/relay.ts`, `buildRelaySegment()`: Uses `config.maxTurnsPerSegment` directly. No adaptive override parameter needed because relay creates a fresh `TokenMonitorState`, so `computeAdaptiveMaxTurns()` naturally falls back to the configured default for the first segment. By segment 2, fresh growth data is available.

```typescript

  };

```

### Relay Wiring

The orchestrator's relay path calls `executeRelay()` with 3 args (checkpoint, config, canUseTool). No adaptive override is passed:

```typescript
// In orchestrator, relay path (after relayFlag detected):
const { segmentOptions, prepareResult } = executeRelay(
  checkpoint,
  config,
  askHandler.canUseTool,
);
```

**Why no explicit adaptive override for relay:** The relay creates a fresh `TokenMonitorState` (at the top of the session loop in `runSkillInternal()`). The fresh state has empty `turnHistory`, so `computeGrowthRate()` returns `null`, triggering the fallback to the configured default. By segment 2 of the relay session, fresh growth data is available and adaptive sizing kicks in. This happens naturally with no special relay handling.

### New Event Type

Add to `OrchestratorEvent` discriminated union in `types.ts`:

```typescript
| { type: "adaptive_turns"; maxTurns: number; reason: string; sessionIndex: number; segmentIndex: number }
```

### Config: Enable/Disable

Add optional field to `GaryClawConfig` in `types.ts`:

```typescript
/** Disable adaptive maxTurns computation (use fixed maxTurnsPerSegment). Default: true. */
adaptiveMaxTurns?: boolean;
```

CLI flag: `--no-adaptive` sets `adaptiveMaxTurns: false`.

When disabled, the orchestrator skips `computeAdaptiveMaxTurns()` and uses `config.maxTurnsPerSegment` directly. This preserves backward compatibility and gives operators an escape hatch.

### CLI Display

In `cli.ts`, handle the new event with a dim info line:

```
  Adaptive turns: 7 (growth 98412 tok/turn (heavy tool: x2.5), budget 689000 tok, predicted 7, clamped to 7)
```

## Open Questions

1. **Should `HEAVY_TOOL_GROWTH_MULTIPLIER` be configurable?** Start hardcoded at 2.5. If operators need tuning, make it a config field later.
2. **Should adaptive turns be reported in the dashboard?** Probably yes (avg adaptive turns per job), but defer to a separate change.
3. **Should `Read` on large files count as a heavy tool?** The subagent suggested this. However, we don't know the file size from the tool_use message alone (only the path). Defer: the growth rate will catch large reads within 1 turn anyway.
4. **Should growth rate hints carry across relay sessions?** Deferred. The first relay segment uses the default, which is safe. Injecting rate hints is a future optimization for Approach C (learning controller).

## Success Criteria

1. Browse-heavy skills (/qa, /design-review) get segments of 3-8 turns instead of fixed 15.
2. Edit-heavy skills (/implement) get segments of 10-25 turns.
3. Given any synthetic growth rate and context size, `computeAdaptiveMaxTurns()` never returns a maxTurns that would exceed the target budget at the given rate (testable property). In integration (dogfood runs), zero emergency relays where context exceeds threshold by >5%.
4. Existing `--max-turns` flag behavior preserved (acts as ceiling).
5. All new code covered by unit tests with synthetic data (no SDK calls).
6. Heavy tool lookahead demonstrably reduces maxTurns when WebFetch/Screenshot seen.
7. `adaptive_turns` event emitted and displayed in CLI.
