## Problem Statement

The pipeline composition learning loop has a gap. When the daemon finishes a job, it records the outcome in `pipeline-outcomes.jsonl` via `appendPipelineOutcome()`. But two fields are hardcoded to useless defaults: `effort: null` and `priority: 3`. And `taskCategory` (the type-level extension point added in Mode 1) is never populated.

This means the Oracle's pipeline recommendations are flying blind on history. It can see general pipeline outcomes ("implement -> qa succeeded 8 times") but not category-specific patterns ("when we skipped design-review on a visual/UX task, QA found 8 issues; when we included it, QA found 0"). The skip-risk scoring in `computeSkipRiskScores()` treats all tasks equally, which is exactly the wrong abstraction. Skipping `design-review` on a refactor is fine. Skipping it on a UI redesign is not.

The prioritize skill already classifies tasks implicitly when it recommends a pipeline. But that classification dies in `priority.md` and never reaches the outcome history. The learning loop doesn't close.

## What Makes This Cool

After this ships, the daemon gets genuinely smarter about skill selection per task type. Run 10 visual/UX tasks and skip `design-review` on half of them. The ones where it was skipped and QA found visual bugs get recorded with `taskCategory: "visual-ux"`. The next time a visual/UX task comes through prioritize, the Oracle sees: "historical failure rate when design-review skipped on visual-ux tasks: 60%. When included: 10%." It recommends `design-review` for that task. The static table never could.

This is the difference between "always skip design-review for S-effort items" (static) and "skip design-review for S-effort refactors, include it for S-effort UI tasks" (learned).

## Constraints

- No new source files. All source changes are modifications to existing modules. New test files are fine (follows existing `test/job-runner-*.test.ts` naming pattern).
- `PipelineOutcomeRecord` schema change must be backward-compatible (existing JSONL records without `taskCategory` are valid).
- Task category classification must come from the prioritize output (LLM-driven), not regex heuristics. The LLM already has full TODO context.
- Pure functions, no SDK calls in new code. Testable with synthetic data.
- Prioritize prompt additions must stay within token budget (~1500 tokens for per-category outcome stats).
- Must not change the cold-start gate threshold (10 outcomes). Per-category data is injected as supplementary context, not as a gate.

## Premises

1. The existing plumbing is 80% done. `PipelineOutcomeRecord` has `taskCategory?`. `appendPipelineOutcome()` writes to JSONL. `computeSkipRiskScores()` does weighted decay. The prioritize prompt has the skill catalog. Wiring these together is the work.

2. Task category classification belongs in the prioritize output, parsed from `priority.md` alongside the existing `### Recommended Pipeline` section. The LLM is already reasoning about task nature to recommend pipelines. Emitting a category label is trivial.

3. Per-category outcome stats in the prioritize prompt are more valuable than per-category skip-risk scores. The Oracle can reason about "60% failure rate when design-review skipped on visual-ux tasks" better than a numeric risk score. Inject human-readable stats, not just numbers.

4. `effort` and `priority` should be populated from actual TODO state at outcome recording time, not left as null/3. This data is already available in `todo-state.ts` and `priority.md`.

## Approaches Considered

### Approach A: Minimal Wiring — Populate Fields + Inject Stats

**Summary:** Populate `taskCategory`, `effort`, and `priority` on `PipelineOutcomeRecord` from the prioritize output and TODO state. Add `computeCategoryStats()` to `pipeline-history.ts`. Inject per-category outcome stats into the prioritize prompt. Done.

**Effort:** S
**Risk:** Low
**Pros:**
- Smallest diff. ~150 lines across 4 files. Zero new modules.
- Closes the learning loop end-to-end: classify -> run -> record -> inject -> classify again.
- Fail-open everywhere. Missing category -> "unknown". Missing stats -> no injection.
- All pure functions, trivially testable.

**Cons:**
- Per-category skip-risk scoring not implemented. `computeSkipRiskScores()` still treats all categories equally. The Oracle gets human-readable stats but the programmatic skip-risk restoration doesn't benefit.
- Category classification accuracy is unverified. If the LLM miscategorizes tasks, per-category stats are noisy.

**Reuses:** `parsePipelineRecommendation()` pattern for parsing `### Task Category`. `buildPrioritizePrompt()` injection pattern. `appendPipelineOutcome()` existing call site in `job-runner.ts`. `PipelineOutcomeRecord.taskCategory` existing field.

### Approach B: Full Category-Aware Skip-Risk

**Summary:** Everything in Approach A, plus: extend `computeSkipRiskScores()` to accept a `taskCategory` filter. When composing a pipeline, pass the current task's category to get category-specific risk scores. The Oracle restoration path (`oracleRestoredSkills`) becomes category-aware.

**Effort:** M
**Risk:** Medium
**Pros:**
- Skip-risk scoring becomes precise. "design-review has 0.6 skip-risk for visual-ux tasks but 0.05 for refactors."
- The programmatic restoration in `composePipeline()` becomes smarter without needing the Oracle prompt at all.
- Enables future per-category circuit breaker.

**Cons:**
- Requires category at composition time, but category comes from the prioritize output which runs before composition. This creates a chicken-and-egg: the prioritize skill must classify the task, then composition uses that classification. The data flow works but is non-obvious.
- `computeSkipRiskScores()` API changes break existing tests (22 tests in `pipeline-history.test.ts`).
- Per-category skip-risk needs more samples per category to be meaningful (MIN_SKIP_SAMPLES=3 per category, not globally). With 7 categories and 100 max outcomes, some categories may never hit the minimum.

**Reuses:** Everything in A plus `computeSkipRiskScores()`, `composePipeline()` skip-risk path.

### Approach C: Structured Outcome Feedback in Prioritize Prompt

**Summary:** Approach A wiring, plus: instead of injecting raw per-category stats, build a structured "lessons learned" section that maps (category, skill-skipped) -> (failure rate, example TODO titles). The Oracle gets narratives, not just numbers.

**Effort:** S-M
**Risk:** Low
**Pros:**
- Most effective use of Oracle reasoning. "When design-review was skipped on visual tasks like 'Mobile vault redesign' and 'Dashboard color update', QA found issues 4/5 times" is more actionable than "visual-ux skip-design-review failure rate: 80%."
- Doesn't require changing any scoring functions. Pure prompt enrichment.
- Still closes the learning loop. The Oracle can reason about patterns humans would see.

**Cons:**
- Prompt token cost is higher per category (~200 tokens per category with examples vs ~50 for just stats).
- Formatting the narrative requires more code than a stats table.
- The "lessons" quality depends on having enough examples with real TODO titles.

**Reuses:** Everything in A plus `readPipelineOutcomes()` for narrative building.

## Recommended Approach

**Approach A: Minimal Wiring — Populate Fields + Inject Stats.**

Approach A closes the loop with the smallest diff and the lowest risk. The Oracle is already good at reasoning from context. Give it per-category outcome stats and let it figure out the patterns. Per-category skip-risk (Approach B) is a future optimization that matters at 200+ outcomes. Narrative formatting (Approach C) is nice but adds prompt tokens and code for marginal benefit over a stats table.

Ship A. If the Oracle makes bad skill recommendations despite having per-category stats, upgrade to B (category-aware skip-risk) as a targeted fix.

## Implementation Plan

### 1. Add `parseTaskCategory()` to `job-runner.ts` (~15 lines)

Parse `### Task Category` from `priority.md`, same pattern as `parsePipelineRecommendation()`. Note: `"unknown"` is a code-level fallback for missing/invalid values. It is not offered to the LLM as a classification option.

```typescript
const VALID_CATEGORIES = [
  "visual-ux", "architectural", "bug-fix", "refactor",
  "performance", "infra", "new-feature", "unknown",
] as const;

export function parseTaskCategory(content: string): string {
  const match = content.match(/^###\s*Task Category\s*\n+(\S+)/m);
  if (!match) return "unknown";
  const raw = match[1].toLowerCase().trim();
  return (VALID_CATEGORIES as readonly string[]).includes(raw) ? raw : "unknown";
}
```

### 4. Inject per-category stats into prioritize prompt (~25 lines)

In `buildPrioritizePrompt()`, after the skill catalog section. Note: `readPipelineOutcomes()` performs file I/O, but `buildPrioritizePrompt()` is already async and calls `safeReadText()`, `readOracleMemory()`, `readFailureRecords()`, etc. This follows the existing pattern. Pass the outcomes path as a parameter if a future refactor extracts I/O from prompt building.

```typescript
// Per-category pipeline outcome stats
const outcomeHistoryPath = join(projectDir, ".garyclaw", "pipeline-outcomes.jsonl");
const outcomes = readPipelineOutcomes(outcomeHistoryPath);
if (outcomes.length >= 10) {
  const stats = computeCategoryStats(outcomes);
  if (stats.length > 0) {
    lines.push("### Pipeline Outcome Patterns by Task Category");
    lines.push("");
    lines.push("| Category | Skill | When Skipped (fail%) | When Included (fail%) | Delta |");
    lines.push("|----------|-------|---------------------|----------------------|-------|");
    for (const s of stats.slice(0, 15)) { // cap at 15 rows
      const delta = s.skippedFailureRate - s.includedFailureRate;
      lines.push(`| ${s.category} | ${s.skill} | ${s.skippedCount} jobs (${s.skippedFailureRate.toFixed(0)}% fail) | ${s.includedCount} jobs (${s.includedFailureRate.toFixed(0)}% fail) | ${delta > 0 ? "+" : ""}${delta.toFixed(0)}pp |`);
    }
    lines.push("");
    lines.push("Use these patterns when recommending pipelines. High delta means the skill matters for that category.");
    lines.push("");
  }
}
```

  ├─ outputs: priority.md with:

  │   ├─ ### Recommended Pipeline: design-review -> implement -> qa

  ▼
job-runner (post-prioritize)

  ├─ parses: parseTaskCategory() -> "visual-ux"

  ├─ runs: design-review -> implement -> qa

### Migration

None needed. `taskCategory` is already an optional field on `PipelineOutcomeRecord`. Existing JSONL records without it are read as `undefined`, which `computeCategoryStats()` groups under "unknown". Backward-compatible.

### Cost Impact

- Per-category stats table in prioritize prompt: ~200-400 tokens (15 rows max, ~25 tokens/row).
- Task Category output instructions: ~150 tokens (category definitions).
- `parseTaskCategory()` is a regex, zero API cost.
- `computeCategoryStats()` is a loop over JSONL records, zero API cost.
- Net prompt size increase: ~350-550 tokens. Current prioritize prompt is ~5.5K. Total ~6K, well within 10K relay budget.

## Known Limitations

- **Category fragmentation at low volumes.** With 7 categories, MAX_PIPELINE_OUTCOMES=100, and a minimum of 3 outcomes per (category, skill) pair, rare categories like "performance" or "infra" may never accumulate enough data to produce meaningful stats. This is acceptable: the Oracle still gets the raw pipeline outcomes via `decision-outcomes.md`, and the stats table is supplementary context. If fragmentation proves problematic, lower the minimum from 3 to 2.
- **Existing fields on `PipelineOutcomeRecord`.** The `skippedSkills: string[]` and `skills: string[]` fields already exist on the type (see `types.ts` line ~644). `computeCategoryStats()` uses these directly. No type changes needed.
- **Effort/priority regex fragility.** The permissive regex (`Effort[\s:*]*\b(XS|S|M|L|XL)\b`) handles common LLM formatting variants. If parsing fails, the fallback is `effort: null` and `priority: 3`, which is the current behavior. No worse than today.

## What I noticed about how you think
