# Design: Step Tracking Across Relays

Generated by /office-hours on 2026-03-27
Branch: main
Repo: 0raclide/GaryClaw

## Problem Statement

When the implement skill runs a multi-step design (5+ numbered steps in the implementation order), it often hits the 85% context threshold mid-implementation. The relay fires, a fresh session starts, and the new session gets a generic checkpoint that says "continue the implement skill" with issue/decision history... but zero knowledge of which implementation steps are already done.

The fresh session re-reads the full design doc, sees all steps, and may:
1. **Repeat completed steps** — re-implementing types.ts that was already committed in step 1
2. **Produce duplicate commits** — same module committed twice with slightly different implementations
3. **Contradict prior work** — the new session's step 2 doesn't match the patterns established in step 1 by the previous session

This is a correctness bug, not a nice-to-have. The implement skill's Rule #1 says "follow the implementation order exactly," but after relay that rule becomes contradictory because the session doesn't know where "exactly" left off.

## What Makes This Cool

After this fix, GaryClaw can implement a 7-step design doc across 3 relay sessions and the result looks like one person wrote it in a single sitting. Each relay seamlessly picks up at the right step. The commit log is clean — no duplicates, no contradictions. That's the "feels like your work" promise actually delivered.

## Constraints

- **Backward compatible** — existing checkpoints without step tracking must still work (optional field)
- **Implement-skill only** — other skills (qa, design-review) don't have ordered steps to track
- **Git log is the source of truth** — the implement skill enforces "one commit per step," so git log is the reliable signal
- **Token budget** — step tracking data in the relay prompt must be compact (steps are short strings, not full diffs)
- **Minimize orchestrator coupling to implement internals** — the orchestrator should import only pure utility functions from implement.ts (`findDesignDoc`, `extractImplementationOrder`), not embed implement-specific business logic. The conditional (`skillName === "implement"`) is the one coupling point, kept intentional and minimal.

## Premises

1. **The relay prompt is the right injection point.** The checkpoint captures state that survives relay. The relay prompt is what the fresh session reads first. Injecting step progress here is the natural location.
2. **Git log analysis reliably detects completed steps.** GaryClaw already has `parseGitLog` in issue-extractor.ts, and the implement skill's Rule #3 ("one commit per step") makes commit-to-step matching feasible.
3. **Step matching works by commit message content.** Design doc steps are numbered (`1. Create types.ts`). Commit messages from the implement skill contain step references or file names that can be matched.
4. **The Checkpoint interface can be extended with optional fields.** `validateCheckpoint` checks required fields. A new `implementProgress?` field is backward-compatible — old checkpoints simply won't have it.
5. **Only the implement skill needs this.** QA finds/fixes bugs (unordered). Design-review produces findings (unordered). Ship has a fixed sequence but never relays (too short). Implement is the only skill with ordered multi-step work that can span relays.

## Approaches Considered

### Approach A: Checkpoint-Embedded Step Tracking (Minimal Viable)

**Summary:** Add an optional `implementProgress` field to the Checkpoint interface. At checkpoint time, scan the git log for commits matching implementation steps. In `generateRelayPrompt`, inject a "Steps Completed / Steps Remaining" section so the fresh session knows exactly where to resume.

**Effort:** S
**Risk:** Low

**Pros:**
- Minimal diff: 4-5 files, ~200-250 lines of new code + ~250 lines of tests
- Uses existing patterns (git log parsing, checkpoint extension, relay prompt building)
- Deterministic — step detection happens at checkpoint time, not at Claude's discretion

**Cons:**
- Step detection relies on commit message format (mitigated by implement skill's strict commit rules)
- Implement-specific optional field on the generic Checkpoint interface (mitigated by being optional, backward-compatible, and self-contained — no Checkpoint code needs to understand its internals except the relay prompt renderer)

**Reuses:** git log invocation pattern from issue-extractor.ts (execFileSync + line parsing, not the `parseGitLog` function itself), checkpoint rotation, relay prompt tiering

### Approach B: Skill-State Plugin Architecture (Ideal Architecture)

**Summary:** Add a generic `skillState?: Record<string, unknown>` extension to Checkpoint. Let skills register "checkpoint enricher" callbacks. The orchestrator calls enrichers at checkpoint time. Each skill writes its own state. `generateRelayPrompt` has a pluggable section for skill-specific context.

**Effort:** M
**Risk:** Medium (adds plugin architecture before there's a second consumer)

**Pros:**
- Clean separation of concerns — no implement-specific code in orchestrator
- Other skills could use the same pattern later (e.g., qa tracking browse state across relays)
- Extensible without modifying core types

**Cons:**
- Over-engineering for one skill — YAGNI until a second skill needs it
- Adds indirection (callback registration, dynamic dispatch)
- Plugin discovery mechanism adds complexity without current payoff

**Reuses:** Checkpoint infrastructure, issue-extractor patterns

### Approach C: Git-Log-Only Detection at Prompt Build Time (No State Changes)

**Summary:** Don't modify checkpoints at all. Instead, modify `buildImplementPrompt` to always scan the git log for evidence of completed steps. Add instructions to the relay prompt telling Claude to check git log before starting work.

**Effort:** XS
**Risk:** Medium

**Pros:**
- Zero checkpoint changes, zero type changes, smallest possible diff
- Works even for manual `garyclaw resume` (no checkpoint needed)

**Cons:**
- Each relay session wastes turns re-scanning git log
- Detection happens at Claude's discretion (not deterministic) — may miss commits or miscount
- Fails if commits were squashed, rebased, or amend-ed
- The fresh session has no structured data about progress — just instructions to "go check"

**Reuses:** Existing git log commands via Bash tool

## Recommended Approach

**Approach A: Checkpoint-Embedded Step Tracking.**

It's the right size for this problem. Deterministic step detection at checkpoint time (not Claude's discretion). Compact representation in the relay prompt. Backward-compatible optional field. The plugin architecture of Approach B can be extracted later if a second skill needs skill-specific relay state — that extraction is a 15-minute refactor, not a rewrite. Approach C puts too much trust in the fresh session correctly interpreting git log output.

## Implementation Order

1. **Add `implementProgress` to types.ts.** New optional field on the Checkpoint interface:
   ```typescript
   implementProgress?: {
     completedSteps: number[];    // 1-indexed step numbers that are done
     currentStep: number;         // next step to work on (1-indexed). When all steps done: totalSteps + 1
     totalSteps: number;          // total steps in design doc
     stepCommits: Record<number, string>;  // step number → commit SHA
     designDocPath: string;       // path to design doc (for relay prompt context)
   };
   ```
   When all steps are complete, `currentStep` = `totalSteps + 1`. The relay prompt renders "All implementation steps complete" instead of "Resume at step N."

   When steps are completed non-sequentially (e.g., 1 and 3 done but not 2), `currentStep` = `min(incomplete steps)` (i.e., 2). The relay prompt lists specific completed steps so the fresh session knows exactly what's done vs. remaining, regardless of order.

   Update `validateCheckpoint` to accept checkpoints with or without this field (backward compat).

2. **Add step detection logic to implement.ts.** New exported function `detectCompletedSteps(steps: string[], projectDir: string, sinceCommit?: string): ImplementProgress`. Scans `git log --oneline` for commits that match step descriptions.

   **When `sinceCommit` is absent** (first checkpoint, no prior gitHead): use `git log --oneline --max-count=50` to scan recent commits only. This prevents scanning the entire repo history on the first relay. 50 commits is generous — a typical implement session produces 5-15 commits.

   **Matching strategy** (two tiers, applied in order):
   - **Tier 1 (exact step number):** commit message matches pattern `/^(step\s+)?(\d+)[.:]/i` where the captured number matches a step index. Examples:
     - `"step 1: create types.ts"` → matches step 1 directly
     - `"1. Add DashboardData interface"` → matches step 1 via leading number
     - `"Step 3: wire dashboard into job-runner"` → matches step 3 directly
   - **Tier 2 (fuzzy token matching):** extract key tokens from step description (file names like `types.ts`, module names like `dashboard`, action verbs like `wire`). Match if commit message contains 2+ key tokens from the step. Score by token overlap count, pick highest-scoring step. Examples from real implement-skill dogfood runs (conventional commit format — no step number prefix):
     - `"feat: add DashboardData interface to types.ts"` → matches step "1. Add DashboardData to types.ts" via tokens `DashboardData` + `types.ts`
     - `"feat: add dashboard module with pure aggregation functions + 33 tests"` → matches step "2. Create dashboard.ts" via tokens `dashboard` + `module`
     - `"feat: wire dashboard generation into job-runner completion path"` → matches step "3. Wire into job-runner" via tokens `wire` + `job-runner`
   - Uses `execFileSync` for git log, not the SDK (this runs in the orchestrator process, not in Claude)

3. **Wire step detection into orchestrator checkpoint building.** In `buildCheckpoint()`, if `config.skillName === "implement"`:
   - Read the design doc (use `findDesignDoc` from implement.ts)
   - Extract implementation order (use `extractImplementationOrder`)
   - Run `detectCompletedSteps` against git log since last checkpoint's `gitHead`
   - Attach result as `implementProgress` on the checkpoint

4. **Inject step progress into relay prompt.** In `generateRelayPrompt()` (checkpoint.ts), if `checkpoint.implementProgress` exists:
   - Add a `## Implementation Progress` section after Session Context
   - List completed steps with commit SHAs
   - Highlight the current step to resume at
   - List remaining steps
   - Replace the generic "Continue the implement skill" instruction with "Resume implementation at step N"
   - If all steps complete (`currentStep > totalSteps`), render "All implementation steps complete — verify tests pass and finalize"

   **Prompt assembly order for relayed implement sessions:** The relay prompt (from `generateRelayPrompt` in checkpoint.ts) is the *only* prompt the fresh session receives. It contains the step progress section. The fresh session does NOT re-run `buildImplementPrompt` — that function is only called on the first session of an implement skill. So steps 4 and 5 are separate concerns: step 4 handles what the relay prompt says, step 5 handles what the first-session prompt says when resuming from a pipeline checkpoint (not a relay).

5. **Update `buildImplementPrompt` for pipeline resume awareness.** When the implement skill is invoked after a pipeline resume (detectable via checkpoint presence in the skill's checkpoint dir), modify the prompt to:
   - Show only remaining steps in the Implementation Order section (not all steps)
   - Reference completed steps briefly ("Steps 1-3 complete, see commits above")
   - Maintain the same strict rules but scoped to remaining work
   - This handles the case where a *pipeline* resumes (not a relay within implement), so `buildImplementPrompt` is called again with a checkpoint available from the previous run

6. **Tests.** New test file `test/step-tracking.test.ts`:
   - `detectCompletedSteps` with various commit message formats
   - `detectCompletedSteps` with partial completion (steps 1,2 done, 3-5 remaining)
   - `detectCompletedSteps` with no matches (fresh start)
   - `generateRelayPrompt` with `implementProgress` present
   - `generateRelayPrompt` without `implementProgress` (backward compat)
   - `buildCheckpoint` integration: verify implementProgress is populated for implement skill
   - `buildCheckpoint` integration: verify implementProgress is absent for non-implement skills
   - Edge cases: empty steps list, single step, all steps complete, commit message variations

## Open Questions

1. **How fuzzy should step matching be?** Exact step-number matching ("step 1:") is safest but fragile if Claude doesn't use that format. Fuzzy matching (file name overlap) catches more but risks false positives. Start with exact + file-name fallback, tune from dogfood data.

2. **Should step tracking persist across pipeline skills?** If `prioritize → implement → qa` runs, and implement relays, the step tracking is clear. But if the whole pipeline resumes from a checkpoint, should the qa skill see implement's step progress? Probably not — keep it implement-scoped.

3. **What if Claude amends a commit instead of creating a new one?** The git log would show fewer commits than expected. The step detector should handle this gracefully — match on content, not commit count.

## Success Criteria

- After relay, the fresh session starts at the correct step (not step 1)
- No duplicate commits produced across relay boundaries
- Zero regressions in existing checkpoint/relay tests
- Step detection works with real implement-skill commit messages from dogfood runs
- Relay prompt with step progress is < 500 tokens additional overhead
- Backward compatible: old checkpoints without `implementProgress` still work
