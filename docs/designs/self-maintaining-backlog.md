# Design: Self-Maintaining Backlog — Auto-Complete + Rate Limit Resilience

Generated by /office-hours on 2026-03-29
Branch: main
Repo: 0raclide/GaryClaw
Status: APPROVED
Mode: Builder

## Problem Statement

Two operational failures from Session 3 waste significant daemon time and money:

**1. TODOS.md doesn't auto-mark complete.** After a successful auto-merge, `todo-state.ts` advances the state file to "merged" but TODOS.md headings stay open. Next cycle, `parseTodoItems()` returns them as actionable. The daemon re-picks them, re-designs them, re-implements them. $30 wasted overnight rebuilding 3 features that were already merged to main.

**2. Rate limit spam-retry.** When Claude Max rate limits hit (02:19 AM), all 5 parallel instances get the error simultaneously. `failure-taxonomy.ts` classifies it as `infra-issue`, `retryable: true`. The job-runner crash-resume logic re-queues each job immediately. Each instance retries, gets rate-limited again, re-queues again. 25+ retries in 10 seconds. The dedup system breaks because each retry is a new job ID. 4 hours of idle time burned on $0 pipelines that never execute a single turn.

## What Makes This Cool

The daemon maintains its own backlog. You push code, go to sleep, and wake up to a TODOS.md where completed items are crossed off with summaries ("COMPLETE (2026-03-29) — Implemented in 7 commits, auto-merged from garyclaw/worker-2"). When rate limits hit, the daemon logs "Rate limited until 02:42 AM — holding all jobs" and goes quiet. At 02:42 it resumes automatically. Zero human intervention for either scenario.

## Constraints

- TODOS.md is the human-readable backlog. Must remain valid markdown after edits.
- `todo-state.ts` state files are the machine-readable state. Both systems must agree.
- Rate limit behavior depends on Claude Max error message format (needs spike or empirical data).
- Must work for both named instances (worktree) and default instance (main repo).
- Auto-mark must never mark an item complete that's currently running in another instance.
- Rate limit hold must propagate across all instances sharing the same Claude Max account.

## Premises

1. TODOS.md is the source of truth the daemon reads to pick work — if it doesn't reflect reality, the daemon does redundant work.
2. Rate limit errors from Claude Max include a parseable reset time or at minimum a predictable retry window. **Unverified — spike required before implementation. Design uses 30-minute fallback if parsing fails, so this premise is soft (nice-to-have precision, not a blocker).**
3. Editing TODOS.md happens on the main repo (not worktree) after auto-merge has already landed, so no merge conflicts.
4. A daemon-level hold (not per-job) is the right granularity for rate limits, since all instances share the same Claude Max account.

## Approaches Considered

### Approach A: Surgical Integration (Recommended)

**Summary:** Two targeted additions to existing modules. Auto-mark hooks into the post-merge success path in `job-runner.ts`. Rate limit hold adds a `rateLimitResetAt` field to `DaemonState` and a time-gate check at the top of `processNext()`.

**Effort:** S (human: ~3 days / CC: ~25 min)
**Risk:** Low

**Pros:**
- Minimal new code — ~160 lines across 6 files (core: ~120 across 3, plus notifications/dashboard/registry). No new modules.
- Both features hook into existing, well-tested code paths
- No new modules, no new dependencies
- Rate limit hold is daemon-level (correct granularity for shared account)

**Cons:**
- TODOS.md rewrite is string manipulation on markdown — fragile if heading format varies
- Rate limit reset time parsing depends on error message format (could change)
- Cross-instance rate limit coordination adds a write to shared `global-budget.json` (last-writer-wins is safe due to extend-only semantics)

**Reuses:** `parseTodoItems()` from prioritize.ts for heading detection, `safeReadText`/`safeWriteText` from safe-json.ts (both already exist, lines 96 and 109), `classifyError()` from failure-taxonomy.ts for rate limit detection, `slugify()` from todo-state.ts for matching.

### Approach B: Full State Sync Layer

**Summary:** New module `src/backlog-sync.ts` that bidirectionally syncs TODOS.md and todo-state/. Reads state files, rewrites TODOS.md with status badges, completion dates, and inline progress. Rate limit tracking goes into a shared `rate-limit-state.json` in the parent `.garyclaw/` dir for cross-instance coordination.

**Effort:** M (human: ~1 week / CC: ~45 min)
**Risk:** Medium

**Pros:**
- Full bidirectional sync — state files and TODOS.md always agree
- Cross-instance rate limit coordination via shared state file
- Could add inline status badges (e.g., `[implementing]`, `[in QA]`)
- Foundation for future dashboard integration

**Cons:**
- Significantly more code (~300 lines new module)
- Bidirectional sync creates reconciliation edge cases
- Inline badges change the TODOS.md format, may confuse human readers
- Cross-instance rate limit coordination adds shared state complexity

**Reuses:** Same as Approach A plus `readTodoState()`/`writeTodoState()` for bidirectional sync, `readGlobalBudget()` pattern for shared rate limit state.

### Approach C: Event-Driven with File Watcher

**Summary:** A file watcher on `todo-state/` directory triggers TODOS.md updates whenever state changes. Rate limit uses OS-level timer (setTimeout) with a recovery file that persists across daemon restarts.

**Effort:** M (human: ~1 week / CC: ~45 min)
**Risk:** High

**Pros:**
- Reactive — updates happen immediately on any state change, not just post-merge
- Clean separation of concerns (watcher is independent of job-runner)
- Timer-based rate limit is precise to the second

**Cons:**
- File watchers are unreliable across platforms (especially in worktrees)
- Additional process/thread complexity
- Over-engineered for the actual problem — state changes are already well-defined events
- setTimeout doesn't survive daemon restart

## Recommended Approach

**Approach A: Surgical Integration.** The problem is specific and the insertion points are clear. The post-merge hook in `job-runner.ts` (inside the `if (mergeResult.merged)` block, after the `writeTodoState` call) already writes todo state — adding a TODOS.md rewrite there is 30 lines. The `processNext()` function is the single entry point for all job execution — adding a rate limit time-gate at the top (before the `find(j => j.status === "queued")` call) is 15 lines. No new modules, no new patterns, minimal surface area for bugs. (Note: line numbers in this doc are approximate, based on commit 397c44c.)

Approach B is the right choice if we want inline progress badges or cross-instance rate limit coordination. But neither is needed today. The cross-cycle dedup in `getCompletedTodoTitles()` already prevents re-picking completed items even if TODOS.md is stale — auto-mark is a defense-in-depth + human-readability improvement, not the sole protection. Rate limit hits are correlated (same account, same time), so independent per-instance holds are functionally equivalent to coordinated holds.

## Implementation Plan

### Part 1: Auto-Mark TODOS.md on Merge

**New function: `markTodoCompleteInFile()`** in `src/todo-state.ts`

```typescript
/**
 * Rewrite a TODOS.md heading from open to ~~complete~~.
 * Matches heading by slug comparison (handles minor title edits).
 * Appends a completion summary line below the heading.
 *
 * Returns true if the file was modified, false if title not found.
 */
export function markTodoCompleteInFile(
  todosPath: string,
  title: string,
  summary: string,
): boolean
```

**Logic:**
1. `safeReadText(todosPath)` — read current TODOS.md. If file does not exist or is empty, return `false` (no-op).
2. Split into lines, find the `## ` heading whose slug matches `slugify(title)`
3. Guard: skip if heading already contains `~~`
4. Rewrite: `## P2: Foo Bar` → `## ~~P2: Foo Bar~~ — COMPLETE ({date})`
5. Insert summary line after heading: `\nImplemented by {instanceName}. {summary}\n`
6. `safeWriteText(todosPath, result)` — atomic write back (tmp+rename pattern)

**Guard against marking running items:** Before calling `markTodoCompleteInFile()`, check `getClaimedTodoTitles()` from daemon-registry. If the title is claimed by any running instance, skip.

**Hook point:** inside `job-runner.ts`'s `if (mergeResult.merged)` block, after the existing `writeTodoState()` call that advances state to "merged":

```typescript
// Auto-mark TODOS.md — defense-in-depth + human readability
try {
  const todosPath = join(jobConfig.projectDir, "TODOS.md"); // main repo, not worktree
  const summary = `${mergeResult.commitCount ?? 0} commit(s) auto-merged from garyclaw/${resolvedInstanceName}`;
  const marked = markTodoCompleteInFile(todosPath, nextJob.claimedTodoTitle!, summary);
  if (marked) {
    d.log("info", `Auto-marked TODO "${nextJob.claimedTodoTitle}" complete in TODOS.md`);
  }
} catch (err) {
  d.log("warn", `Auto-mark TODOS.md failed: ${err instanceof Error ? err.message : String(err)}`);
}
```

**Catch-up on daemon start: `catchUpCompletedTodos()`**

Called from `createJobRunner()` after the crash-resume loop (after the `for (const job of state.jobs)` block that re-queues crashed jobs).

```typescript
/**
 * Scan todo-state/ for items that reached "merged" or "complete"
 * but whose TODOS.md heading is still open. Marks them complete.
 * Runs once on daemon start to catch items completed before a crash.
 */
function catchUpCompletedTodos(
  checkpointDir: string,  // Use parentCheckpointDir (shared .garyclaw/) for named instances,
                          // since writeTodoState writes to parentCheckpointDir ?? checkpointDir.
                          // For default instance, this is just .garyclaw/.
  projectDir: string,     // main repo dir (not worktree)
): number                 // returns count of headings marked
```

**Logic:**
1. `readdirSync(join(checkpointDir, "todo-state"))` — list all `{slug}.json` files. If dir missing, return 0.
2. For each file, `safeReadJSON<TodoState>()` — read state.
3. Filter to states where `state === "merged" || state === "complete"`.
4. For each qualifying state, call `markTodoCompleteInFile(todosPath, state.title, summaryFromState)`.
5. `summaryFromState`: format from stored fields, e.g., `"Completed by ${state.instanceName}, job ${state.lastJobId}"`. If fields are missing, use generic `"Completed"`.
6. Guard: check `getClaimedTodoTitles()` — skip any title currently running in another instance.
7. Log count: `d.log("info", "Catch-up: marked N TODO heading(s) complete in TODOS.md")`.

| Location | Current behavior | `rate_limited` handling |
|----------|-----------------|----------------------|

| `daemon-registry.ts` `getCompletedTodoTitles()` | Scans for `complete` jobs | No change — `rate_limited` is not complete |
| `dashboard.ts` `aggregateJobStats()` | Counts by status | Add `rate_limited` count to dashboard output |
| `notifier.ts` summary formatting | Formats job status | Add `rate_limited` display string |

**Rate limit detection in error handler** (inside `processNext()`'s `catch (err)` block, after `classifyError()` and `appendFailureRecord()`):

Check if the failure category is `infra-issue` and the error message matches rate limit patterns:

```typescript
if (classification.category === "infra-issue" && isRateLimitError(nextJob.error ?? "")) {
  const resetAt = parseRateLimitResetTime(nextJob.error ?? "");
  if (resetAt) {
    state.rateLimitResetAt = resetAt.toISOString();
    nextJob.status = "rate_limited";
    d.log("info", `Rate limited until ${resetAt.toISOString()} — holding all jobs`);
  } else {
    // Can't parse reset time — use default 30-minute hold
    const fallback = new Date(Date.now() + 30 * 60 * 1000);
    state.rateLimitResetAt = fallback.toISOString();
    nextJob.status = "rate_limited";
    d.log("info", `Rate limited (unknown reset) — holding jobs for 30 min until ${fallback.toISOString()}`);
  }
  persistState(state, checkpointDir);
}
```

**Time-gate at top of `processNext()`** (insert after the `if (running) return` guard, before the `find(j => j.status === "queued")` call):

```typescript
async function processNext(): Promise<void> {
  if (running) return;

  // Rate limit hold: don't process any jobs until reset time passes
  if (state.rateLimitResetAt) {
    const resetAt = new Date(state.rateLimitResetAt);
    if (Date.now() < resetAt.getTime()) {
      return; // Still rate-limited — caller (poll interval) will retry later
    }
    // Reset time passed — clear the hold
    d.log("info", `Rate limit hold expired — resuming job processing`);
    state.rateLimitResetAt = undefined;
    // Re-queue any rate_limited jobs
    for (const job of state.jobs) {
      if (job.status === "rate_limited") {
        job.status = "queued";
        d.log("info", `Re-queued rate-limited job ${job.id}`);
      }
    }
    persistState(state, checkpointDir);
  }

  const nextJob = state.jobs.find((j) => j.status === "queued");
  if (!nextJob) return;
  // ... rest of processNext
```

**New helper functions** in `src/job-runner.ts`:

```typescript
/** Check if an error message indicates a rate limit. */
function isRateLimitError(message: string): boolean {
  const lower = message.toLowerCase();
  return lower.includes("rate limit") ||
    lower.includes("status 429") ||
    lower.includes("http 429") ||
    lower.includes("too many requests");
}

/**
 * Parse rate limit reset time from error message.
 * Claude Max format: "resets at 2:42 PM" or "try again in 23 minutes"
 * Returns null if unparseable.
 */
export function parseRateLimitResetTime(message: string): Date | null {
  // Pattern 1: "resets at HH:MM AM/PM"
  const atMatch = message.match(/resets?\s+at\s+(\d{1,2}):(\d{2})\s*(AM|PM)/i);
  if (atMatch) {
    const now = new Date();
    let hours = parseInt(atMatch[1], 10);
    const minutes = parseInt(atMatch[2], 10);
    const ampm = atMatch[3].toUpperCase();
    if (ampm === "PM" && hours < 12) hours += 12;
    if (ampm === "AM" && hours === 12) hours = 0;
    const reset = new Date(now);
    reset.setHours(hours, minutes, 0, 0);
    // If reset time is in the past, assume next day
    if (reset.getTime() < now.getTime()) {
      reset.setDate(reset.getDate() + 1);
    }
    return reset;
  }

  // Pattern 2: "try again in N minutes"
  const inMatch = message.match(/(?:try again|wait|retry)\s+in\s+(\d+)\s*min/i);
  if (inMatch) {
    const mins = parseInt(inMatch[1], 10);
    return new Date(Date.now() + mins * 60 * 1000);
  }

  // Pattern 3: "Retry-After: N" (seconds)
  const retryAfter = message.match(/retry-after:\s*(\d+)/i);
  if (retryAfter) {
    const secs = parseInt(retryAfter[1], 10);
    return new Date(Date.now() + secs * 1000);
  }

  return null; // Unparseable — caller uses 30-min fallback
}
```

**Cross-instance rate limit propagation:** When a rate limit is detected, write `rateLimitResetAt` to the global budget file (`global-budget.json`) so other instances can read it. Add a check in each instance's `processNext()`:

```typescript
// Check global rate limit hold (shared across all instances)
if (parentCheckpointDir) {
  const globalBudget = readGlobalBudget(parentCheckpointDir);
  if (globalBudget.rateLimitResetAt) {
    const resetAt = new Date(globalBudget.rateLimitResetAt);
    if (Date.now() < resetAt.getTime()) {
      return; // Another instance is rate-limited — hold this one too
    }
  }
}
```

**New function in `daemon-registry.ts`:**

```typescript
export function setGlobalRateLimitHold(
  parentDir: string,
  resetAt: string,
  instanceName: string,
): void {
  const budget = readGlobalBudget(parentDir);
  // Only extend the hold, never shorten it
  if (!budget.rateLimitResetAt || new Date(resetAt) > new Date(budget.rateLimitResetAt)) {
    budget.rateLimitResetAt = resetAt;
    safeWriteJSON(join(parentDir, "global-budget.json"), budget);
  }
}
```

### Edge Cases

1. **TODOS.md heading format varies:** Use slug comparison, not exact string match. `slugify("P2: Self-Maintaining Backlog")` matches regardless of minor edits.
2. **Rate limit error format changes:** 30-minute fallback if unparseable. Fallback is conservative (Claude Max resets are typically 15-25 min).
3. **Multiple instances hit rate limit simultaneously:** Each writes to global-budget.json. `setGlobalRateLimitHold()` uses extend-only semantics (latest reset time wins). `safeWriteJSON` handles concurrent writes.
4. **Daemon restart during rate limit hold:** `rateLimitResetAt` persists in DaemonState. On restart, `processNext()` checks the hold immediately. No lost state.
5. **TODOS.md edit conflict:** Auto-mark runs on main repo after merge. If a human is editing TODOS.md concurrently, `safeWriteText` atomic write prevents corruption. Worst case: human edit overwrites the auto-mark. Next catch-up cycle re-marks it.
6. **Rate-limited job that was mid-pipeline:** Status changes to `rate_limited` (not `failed`), so `retryCount` is not incremented. When hold expires, status flips back to `queued` in-place (preserving original FIFO array order, not inserted at front). Pipeline resume from `pipeline.json` picks up where it left off.
7. **Concurrent writes to global-budget.json (Part 3 only):** `setGlobalRateLimitHold()` uses extend-only semantics, so last-writer-wins is safe. Worst case: a slightly shorter hold (if an earlier reset time overwrites a later one between read and write). This is benign because the rate-limited instance will re-set the hold on its next retry. No advisory lock needed for this field.

## Open Questions

1. **What's the exact Claude Max rate limit error format?** Need to capture a real error message to verify the regex patterns in `parseRateLimitResetTime()`. If the format is different, the patterns need updating, but the 30-minute fallback covers us.
2. **Should `rate_limited` jobs count toward the daily job limit?** Current design: no, since they didn't execute. But they consumed an enqueue slot. Recommendation: don't count them — they'll be re-queued and counted when they actually run.
3. **Should auto-mark commit the TODOS.md change?** Current design: no, just write the file. The next daemon commit will pick it up. Alternative: auto-commit with a `[garyclaw]` prefix so the self-commit filter skips it.

## Success Criteria

- After auto-merge, TODOS.md heading is rewritten to `~~complete~~` within 5 seconds
- `parseTodoItems()` returns 0 items for auto-marked headings (verified by existing test pattern)
- Rate limit error triggers a hold, no job processing until reset time
- Zero spam-retries: a rate limit event produces exactly 1 log line + 1 notification, not 25+
- Dashboard shows rate limit hold status and history
- All existing tests pass (no regressions in job-runner, todo-state, or failure-taxonomy)
