# Design: Auto-Merge Dirty Working Tree Fix + Cross-Cycle Dedup

Mode: Builder

## Problem Statement

Two bugs combine to waste $30-60/overnight in daemon parallel operation:

**Bug 1 — Dirty working tree blocks every auto-merge.** `mergeWorktreeBranch()` in `worktree.ts:320-338` runs `git diff --cached --quiet` and `git diff --quiet` on the **main repo directory** before checking out the base branch. When any tracked file has uncommitted changes on main (e.g., a prior daemon instance edited something, or a relay session left a stash artifact, or CLAUDE.md was updated in-flight), the merge fails with "Working tree has uncommitted changes." On 2026-03-29, every auto-merge failed overnight due to this.

**Bug 2 — No cross-cycle dedup.** When merge fails, the next cron cycle re-assigns the same TODO and rebuilds the feature from scratch. The job-runner's pre-assignment in `job-runner.ts:258-297` only deduplicates against currently-running/queued `claimedTodoTitle` values across instances. It doesn't check whether a `garyclaw/*` branch already has the work done but just couldn't merge. On 2026-03-29, workers rebuilt the same features 2-3x each, burning ~$30 in duplicate work.

## What Makes This Cool

The daemon goes from "builds features and throws them away" to "builds features and lands them." Fix 1 is a surgical stash/pop around the checkout+merge block. Fix 2 teaches the pre-assignment to scan existing branches, recognizing "I already built this, I just need to merge it." Together, overnight runs go from ~20% merge success to ~95%.

## Constraints

- S effort budget. Fix 1 touches `worktree.ts` only. Fix 2 touches `job-runner.ts` + `worktree.ts`.
- Must be fully testable with synthetic data (mocks for git commands, no real repos in tests).
- Must not break the pre-merge validation gate (12 commits just landed for this).
- The stash/pop must handle the case where stash creates nothing (clean tree minus the diff-quiet failure edge case).
- Cross-cycle dedup must handle branch names that don't exactly match TODO titles (fuzzy match).
- Must preserve the merge-lock pattern (stash/pop happens inside the lock, not outside).

## Premises

1. **The dirty working tree check is too strict for daemon operation.** The check exists to protect a human's uncommitted work. In daemon mode, the main repo's working tree is often "dirty" with daemon artifacts that are safe to stash. The fix: stash before checkout, pop after merge. This is the same pattern `relay.ts:24` uses for relay segments.

2. **Existing `garyclaw/*` branches are reliable indicators of completed work.** If `garyclaw/worker-1` has 8 commits ahead of main, that's a feature that was built and just needs merging. Rebuilding from scratch wastes 100% of the prior compute. The branch is the evidence.

3. **Fuzzy matching on branch names is necessary because branch names are sanitized.** `branchName()` in `worktree.ts:91` runs through `sanitizeBranchComponent()`, which strips special chars. TODO titles like "Fix Auto-Merge Dirty Working Tree" become branch `garyclaw/worker-1` (named after the instance, not the TODO). So we can't match branch name to TODO title directly. Instead, we need to scan branch content (commits) or use the `claimedTodoTitle` from daemon-state.json history.

4. **The stash/pop is safe because the merge-lock serializes all merge attempts.** No two instances can stash/pop concurrently because `acquireMergeLock()` prevents it. The stash/pop is atomic within the lock window.

5. **Completed jobs in daemon-state.json are the most reliable cross-cycle dedup signal.** Rather than fuzzy-matching branch names to TODO titles, scan all instances' `daemon-state.json` for `completed` jobs whose `claimedTodoTitle` matches the candidate TODO. If the work was done (even if merge failed), skip it.

## Approaches Considered

### Approach A: Stash/Pop + Completed-Job Scan (recommended)

**Summary:** Fix 1: wrap the checkout+merge block in `git stash --include-untracked` / `git stash pop` inside the existing merge-lock. Fix 2: in job-runner pre-assignment, scan all instances' daemon-state.json for completed jobs with matching `claimedTodoTitle`. Skip TODOs where prior completion is detected.

**Effort:** S (human: ~2 days / CC: ~15 min)
**Risk:** Low

**Pros:**
- Stash/pop is the same proven pattern as `relay.ts` uses for relay segments.
- Completed-job scan uses existing `daemon-registry.ts` infrastructure (instance discovery).
- Branch scan uses existing `listWorktrees()` + `git rev-list --count`.
- Completed-job scan is the primary dedup signal. Branch scan deferred to follow-up if needed.
- Stash/pop inside merge-lock means no concurrent stash races.

**Cons:**
- `git stash pop` can conflict if the stashed changes touch files that the merge modified. Mitigation: pop with `--quiet` and if it fails, leave the stash (logged as warning, human can resolve).
- Completed-job scan depends on daemon-state.json surviving daemon restarts (it does, via `persistState()`).
- Adding two dedup signals (job history + branch) is slightly more complex than just one. But both are cheap O(n) scans.

**Reuses:** `acquireMergeLock/releaseMergeLock` (worktree.ts), `discoverInstances` (daemon-registry.ts), `parseTodoItems` (prioritize.ts), `listWorktrees` (worktree.ts), `safeReadJSON` (safe-json.ts).

### Approach B: Exclude .garyclaw from Diff Check + Branch-Only Dedup

**Summary:** Fix 1: change the diff check to `git diff --quiet -- ':!.garyclaw'` to exclude daemon artifacts instead of stashing. Fix 2: only scan branches (no job history), using `git log --format=%s` on the branch to extract TODO-like titles from commit messages.

**Effort:** S (human: ~2 days / CC: ~15 min)
**Risk:** Medium

**Pros:**
- Simpler Fix 1: pathspec exclusion is a one-line change, no stash lifecycle to manage.
- No risk of stash pop conflicts.

**Cons:**
- `.garyclaw/` is already in `.gitignore`, so `git diff --quiet` already ignores it. The real dirty files are tracked source files (e.g., CLAUDE.md, package.json modified during a relay). Excluding `.garyclaw/` doesn't fix the actual problem.
- Commit message parsing for TODO title matching is fragile. Commit messages are written by Claude, not structured.
- No job-history signal means if daemon-state.json is cleared (e.g., manual reset), dedup is lost.

### Approach C: Detached Merge (No Checkout Required)

**Summary:** Instead of `git checkout baseBranch && git merge --ff-only branch`, use `git merge-base --is-ancestor baseBranch branch` to verify ff-ability, then `git update-ref refs/heads/baseBranch branch` to advance the ref directly. No checkout needed, no dirty working tree problem.

**Effort:** S (human: ~2 days / CC: ~20 min)
**Risk:** Medium-High

**Pros:**
- Eliminates the dirty-tree problem entirely by never checking out the base branch.
- No stash/pop lifecycle.
- Conceptually cleaner: a fast-forward merge IS just a ref update.

**Cons:**
- `git update-ref` bypasses hooks and doesn't update the working tree. If the main repo's working tree is on baseBranch, it becomes out of sync with HEAD (tree says old commit, HEAD says new).
- The pre-merge test gate runs in the worktree, so it still works. But any process watching main's working tree (like a dev server) would see stale files until `git checkout`.
- Breaks the `restoreBranch()` pattern since we never left the original branch.
- More complex to reason about correctness. The current checkout+merge pattern is well-understood.

## Recommended Approach

**Approach A: Stash/Pop + Completed-Job Scan.** It's the safest, uses proven patterns already in the codebase, and fixes both problems with defense-in-depth dedup. The stash/pop inside the merge-lock is exactly how relay.ts handles the same problem.

## Implementation Plan

### Step 1: Stash/Pop in mergeWorktreeBranch (worktree.ts)

After acquiring the merge-lock but before `git checkout baseBranch`, stash any dirty state:

```typescript
// Inside the try block after acquireMergeLock, before checkout:
let stashed = false;
try {
  const stashOutput = execFileSync(
    "git", ["stash", "push", "--include-untracked", "-m", "garyclaw-merge-stash"],
    { cwd: repoDir, stdio: "pipe", encoding: "utf-8" },
  ).trim();
  // "No local changes to save" means nothing was stashed
  stashed = !stashOutput.includes("No local changes");
} catch {
  // Stash failed — continue anyway (best-effort)
}
```

After merge completes (success or failure), before `releaseMergeLock`:

```typescript
// Pop stash if we pushed one
if (stashed) {
  try {
    execFileSync("git", ["stash", "pop"], { cwd: repoDir, stdio: "pipe" });
  } catch {
    // Pop conflict — leave stash in place, log warning
    // Human can resolve with `git stash show` / `git stash drop`
  }
}
```

**Key detail:** The stash/pop wraps the ENTIRE checkout+rebase+test+merge+restore sequence inside the merge-lock. The finally block for `releaseMergeLock` should come AFTER the stash pop. The lock hold time is effectively unchanged, stash push/pop adds only a few milliseconds.

**Delete the old dirty-tree check.** Remove lines 318-338 of the current `worktree.ts` (the `git diff --cached --quiet` + `git diff --quiet` block and its catch returning "Working tree has uncommitted changes"). The stash/pop replaces this entirely.

### Step 2: Add getCompletedTodoTitles to daemon-registry.ts

New export that scans all instances' daemon-state.json for completed jobs:

```typescript
export function getCompletedTodoTitles(
  gcDir: string,
  excludeInstance?: string,
): Set<string> {
  const titles = new Set<string>();
  const instances = discoverInstances(gcDir);
  for (const inst of instances) {
    if (inst.name === excludeInstance) continue;
    try {
      const statePath = join(inst.dir, "daemon-state.json");
      const state = safeReadJSON<DaemonState>(statePath);
      if (!state?.jobs) continue;
      for (const job of state.jobs) {
        if (job.status === "completed" && job.claimedTodoTitle) {
          titles.add(job.claimedTodoTitle);
        }
      }
    } catch { /* skip unreadable instances */ }
  }
  return titles;
}
```

## Open Questions

1. **~~Should stash pop failure auto-drop the stash?~~** Resolved: leave the stash in place and log a warning. Dropping would lose the user's changes. The user can `git stash list` and resolve manually.

2. **Should cross-cycle dedup also attempt to re-merge existing branches?** When a TODO has a completed branch but failed merge, should the daemon retry the merge instead of skipping? Recommendation: no, for v1. The skip prevents wasted rebuilds. Re-merge can be a follow-up feature.

3. **How long should completed-job history be retained?** Currently daemon-state.json grows indefinitely. Should completed jobs older than N days be pruned? Recommendation: defer. The file is small (a few KB per day of operation). Pruning is a separate hardening task.
