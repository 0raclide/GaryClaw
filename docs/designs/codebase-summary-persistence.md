# Design: Codebase Summary Persistence Across Relays

## Problem Statement

When GaryClaw relays to a fresh session, the checkpoint carries structured data (issues, decisions, findings, implementation progress) but loses tacit knowledge. The previous session learned things about the codebase: naming conventions, file organization patterns, approaches that were tried and failed, architectural constraints discovered during exploration. The fresh session starts from zero on all of this.

The result: session 3 may re-explore an approach that session 1 already tried and rejected. It may apply a fix using camelCase in a file that uses snake_case. It may create a new utility function when one already exists two directories over. Each relay boundary is a mini-amnesia event for everything except the structured task list.

For multi-relay implement runs (the primary overnight use case), this directly degrades fix quality and wastes context on re-exploration.

## What Makes This Cool

The "whoa" moment: you're watching GaryClaw's relay prompt and it says:

```
## Codebase Context (carried from sessions 0-2)

**Approaches that failed (don't retry):**
- Tried using git diff --name-only for step detection, but it misses files changed across multiple commits. Use git log --name-only instead.
- Tried adding a direct fs.readFileSync call in the new module, but this project wraps all file I/O through src/safe-json.ts — use safeReadJSON() instead.

**Observations:**
- This project uses kebab-case for file names and camelCase for function names
- Test files in test/ mirror the src/ structure — colocated by module name
- All shared types live in types.ts with zero imports. Every other module imports from types.
- Git commands always use execFileSync, never execSync (shell injection prevention)
```

Session 3 reads that and immediately knows what session 1 learned the hard way. Zero re-exploration. Fixes match the project's conventions from the first attempt.

## Constraints

- Must fit within the existing 10K token relay prompt budget. Spike 4 showed typical prompts use ~2K tokens. Budget for codebase summary: 2K tokens max (leaves 6K headroom for growth).
- Extraction must not consume extra SDK turns. Mine from existing assistant messages, don't ask Claude to "write a summary."
- Must not slow down the checkpoint/relay hot path. Summary generation happens inline, not as a separate SDK call.
- Backward compatible: checkpoints without `codebaseSummary` must still work (the field is optional).
- Summary content must be safe: no secrets, no API keys, no user data. It's codebase observations only.

## Premises

1. **The primary value is preventing re-exploration of failed approaches and dead ends during multi-relay runs.** Secondary value: convention consistency across relay boundaries.
2. **The summary must fit within the 10K token relay prompt budget.** A 2K token cap for the summary section leaves comfortable headroom alongside existing sections.
3. **Passive extraction from assistant messages is sufficient.** Claude already narrates what it discovers ("I see this project uses...", "I tried X but it failed because..."). Mining these statements is cheaper than asking for a dedicated summary turn.
4. **This is a within-run feature first.** Cross-run persistence (writing to oracle-memory) is a natural Phase 2 extension, not in scope for this design.
5. **The checkpoint type must remain backward-compatible.** Optional `codebaseSummary` field, validated gracefully.

## Approaches Considered

### Approach A: Structured Extraction (Pattern-Matched Mining)

**Summary:** Define ~8 observation categories (naming conventions, file organization, test patterns, failed approaches, key utilities, architectural rules, performance notes, dependency notes). During the segment loop, regex-match assistant text against patterns that signal observations ("this project uses...", "I tried...but...", "there's already a..."). Store matched observations in a typed `CodebaseSummary` object on the checkpoint. Inject as a formatted section in the relay prompt.

**Effort:** S
**Risk:** Medium — regex patterns may miss observations or match noise. Requires tuning the pattern set.

**Pros:**
- Fully typed, testable, deterministic
- Zero extra SDK calls — mines existing assistant text stream
- Observation categories are explicit, easy to add/remove
- Token budget is predictable (fixed categories, truncation per category)

**Cons:**
- Regex-based extraction is brittle — Claude's phrasing varies across sessions
- May miss valuable observations that don't match any pattern
- Maintenance burden: patterns need updating as Claude's language evolves

**Reuses:** `extractAssistantText()` in orchestrator.ts (already processes assistant messages), `estimateTokens()` in checkpoint.ts, tiered truncation pattern from `generateRelayPrompt()`.

### Approach B: Free-Form Accumulator (Append + Deduplicate)

**Summary:** During each segment, accumulate raw "observation" strings extracted from assistant messages using lightweight heuristics (sentences containing signal words like "convention", "pattern", "avoid", "failed", "uses", "always", "never"). At checkpoint time, deduplicate near-duplicates (Levenshtein, already in reflection.ts) and truncate to token budget. Store as a flat `string[]` on the checkpoint. Inject as a bullet list in the relay prompt.

**Effort:** S
**Risk:** Low — simple accumulation, graceful degradation (worst case: slightly noisy observations, still better than nothing).

**Pros:**
- Captures a wider range of observations than pattern matching
- Simple to implement — no category taxonomy to maintain
- Near-duplicate dedup prevents bloat across segments
- Levenshtein distance already battle-tested in reflection.ts

**Cons:**
- Less structured output — relay prompt gets a grab-bag of observations
- Harder to enforce token budget per-category (everything is one pool)
- Signal-to-noise ratio depends on heuristic quality

**Reuses:** `extractAssistantText()` in orchestrator.ts, Levenshtein distance from reflection.ts, `estimateTokens()` from checkpoint.ts.

### Approach C: End-of-Segment Summary Prompt (Active Generation)

**Summary:** Before triggering a relay, inject a special "Continue." prompt that includes: "Before proceeding, write a brief codebase summary covering: conventions you've observed, approaches you tried that failed, key files/utilities discovered, architectural patterns." Parse Claude's response as the summary. Store on checkpoint.

**Effort:** S
**Risk:** Medium — consumes 1-2 extra turns per relay, which may push context over the relay threshold.

**Pros:**
- Highest quality summaries — Claude generates them intentionally
- No regex/heuristic brittleness
- Claude can synthesize observations rather than just extracting fragments

**Cons:**
- Costs 1-2 turns per relay (context growth + latency)
- Risks triggering relay before the summary is captured (chicken-and-egg with context threshold)
- More complex orchestrator logic — need to inject summary request at the right moment
- Summary quality varies with remaining context (rushed near threshold)

**Reuses:** Orchestrator segment loop, checkpoint infrastructure.

## Recommended Approach

**Approach B: Free-Form Accumulator** — because it's the simplest thing that works, captures the widest range of observations, and reuses existing infrastructure (Levenshtein dedup, token estimation). The relay prompt already has a tiered truncation strategy, so adding a budget-capped observations section is a natural extension.

Approach A's regex patterns would need constant tuning and still miss edge cases. Approach C's extra turns are wasteful — Claude already narrates its observations naturally, we just need to catch them.

Start with Approach B. If the signal-to-noise ratio proves too low in practice, upgrade to Approach A's categories as a filtering layer on top of B's extraction.

## Implementation Plan

### New Types (types.ts)

```typescript
export interface CodebaseSummary {
  observations: string[];      // Deduplicated observation strings
  failedApproaches: string[];  // Specifically "I tried X but Y" patterns
  lastSessionIndex: number;    // Highest session index that contributed observations (used in relay prompt header)
}
```

Add optional `codebaseSummary?: CodebaseSummary` to the `Checkpoint` interface.

### New Module: src/codebase-summary.ts

**Functions:**
1. `extractObservations(text: string): string[]` — Extract candidate observation sentences from assistant text.

   **Sentence splitting:** Split on `\n` to get lines, then split lines on `. ` (period-space) to get sentences. Discard sentences shorter than 20 characters or longer than 300 characters (too short = fragments, too long = full paragraphs).

   **Signal word filtering:** Require 2+ signal words from this set: "convention", "pattern", "uses", "avoid", "failed", "tried", "always", "never", "don't", "instead", "already exists", "naming", "structure", "architecture", "organized". The 2-word minimum filters out narration noise like "I always check the tests" (only 1 signal word: "always").

   **Negative patterns (exclude):** Sentences matching these patterns are narration, not observations: "I don't see", "I'll try", "let me try", "I always check", "I tried running", "let me check". These are Claude reasoning about its next action, not codebase observations.

   **Code-anchor bonus:** Sentences containing a file path (matches `/[a-zA-Z][\w.-]+\.\w+/`) or a function/class name (matches `/[a-zA-Z]\w+\(\)/`) are accepted with only 1 signal word, since the code reference makes them high-signal.

2. `extractFailedApproaches(text: string): string[]` — More targeted extraction for "tried X but Y" patterns. Match sentences containing ("tried" or "attempted") AND ("but" or "however" or "failed" or "doesn't work" or "broke"). These are the highest-value observations for preventing re-exploration.

3. `deduplicateObservations(observations: string[], threshold?: number): string[]` — Near-duplicate removal using two strategies:
   - **Levenshtein:** Normalized distance < 0.3 (normalized = editDistance / max(len(a), len(b)), matching reflection.ts).
   Keyword-overlap dedup is deferred to a future iteration if Levenshtein proves insufficient. Two dedup strategies in v1 is speculative. The 2K token budget acts as a backstop for any semantic duplicates that Levenshtein misses.

   Import Levenshtein from reflection.ts or extract to a shared utility.

4. `truncateToTokenBudget(entries: string[], maxTokens: number): string[]` — Drop oldest entries first until total estimated tokens are under budget. Same pattern as relay prompt truncation.

5. `buildCodebaseSummary(current: CodebaseSummary | undefined, newObservations: string[], newFailed: string[], sessionIndex: number): CodebaseSummary` — Merge previous summary with new observations, deduplicate, then enforce token budget. Merge order: previous observations first, then new observations appended. Since `truncateToTokenBudget` drops from the front (oldest), newer observations are retained when budget is tight. Priority allocation:
   - **Failed approaches:** 500 token budget (highest signal, preventing re-exploration is the #1 value)
   - **General observations:** 1,500 token budget (conventions, patterns, utilities)
   - Each array is truncated independently via `truncateToTokenBudget()`
   - `lastSessionIndex` is set to `sessionIndex`

6. `formatCodebaseSummaryForRelay(summary: CodebaseSummary): string` — Format as markdown section for the relay prompt. Output format:
   ```
   ## Codebase Context (carried from sessions 0-{lastSessionIndex})

   **Approaches that failed (don't retry):**
   - {failedApproach1}
   - {failedApproach2}

   **Observations:**
   - {observation1}
   - {observation2}
   ```
   Omit the "Approaches that failed" subsection if `failedApproaches` is empty. Omit the "Observations" subsection if `observations` is empty. Omit the entire section if both are empty.

### Orchestrator Changes (orchestrator.ts)

In the segment message loop, after `extractAssistantText(msg)`:
```typescript
if (text) {
  const obs = extractObservations(text);
  const failed = extractFailedApproaches(text);
  pendingObservations.push(...obs);
  pendingFailed.push(...failed);
}
```

Declare `pendingObservations: string[]` and `pendingFailed: string[]` at the session loop level (reset per session, accumulated across segments within a session).

In `buildCheckpoint()`, accept the pending arrays as parameters and add:
```typescript
const codebaseSummary = buildCodebaseSummary(
  lastCheckpoint?.codebaseSummary,
  pendingObservations,
  pendingFailed,
  sessionIndex,
);
```

Include `codebaseSummary` in the returned checkpoint object (same pattern as `implementProgress`).

**Important:** All `buildCheckpoint()` call sites in the orchestrator must pass the pending observation arrays. There are four call sites: relay (line ~441), success (line ~524), error (line ~595), and cost-exceeded (line ~425). Since `pendingObservations` and `pendingFailed` are declared at the session loop level, all four call sites can access them. Missing any call site means observations from the final session are lost on error/cost-exceeded paths.

### Checkpoint Changes (checkpoint.ts)

In `generateRelayPrompt()`, add a section after findings and before instructions:
```typescript
if (checkpoint.codebaseSummary) {
  text += formatCodebaseSummaryForRelay(checkpoint.codebaseSummary);
}
```

Token budget: The 2,000 token budget is enforced inside `buildCodebaseSummary()` (500 for failedApproaches, 1,500 for observations), not in the relay prompt generator. The relay prompt's `maxTokens` (10K) applies to the combined output. Since the summary is pre-truncated to 2K, and typical issue/decision sections use ~2K (per spike 4), total stays well under 10K.

### Validation (checkpoint.ts)

Update `validateCheckpoint()` to accept optional `codebaseSummary` field:
```typescript
// codebaseSummary is optional — backward compatible
if (d.codebaseSummary !== undefined) {
  if (typeof d.codebaseSummary !== "object" || d.codebaseSummary === null) return false;
  const cs = d.codebaseSummary as Record<string, unknown>;
  if (!Array.isArray(cs.observations) || !Array.isArray(cs.failedApproaches)) return false;
}
```

## Open Questions

1. **Should Levenshtein be extracted to a shared utility?** Currently lives in reflection.ts. Both codebase-summary and reflection need it. Could move to a `src/string-utils.ts`. Decision: do it if the implement skill wants to, defer if it adds scope.

## Future Work (out of scope)

- **Cross-run persistence:** Write accumulated codebase summaries to `.garyclaw/oracle-memory/codebase-knowledge.md` at job completion, so future runs on the same project start with prior knowledge. Natural extension once within-run summaries are validated.
- **Semantic dedup via embeddings:** If keyword-overlap dedup proves insufficient, could use a lightweight embedding model for true semantic similarity. Overkill for v1.

## Success Criteria

- **No re-exploration:** After relay, the new session references codebase summary observations instead of re-discovering them. Measurable by checking relay prompt content in checkpoint files.
- **Convention consistency:** Fixes in session N+1 match conventions documented in the summary from session N. Measurable by code review of multi-relay run outputs.
- **Token budget compliance:** Codebase summary section stays under 2K tokens. Measurable by unit test.
- **Backward compatibility:** Old checkpoints without `codebaseSummary` load and relay correctly. Measurable by unit test.
- **Zero extra SDK calls:** No new `startSegment` or `query` calls added to the relay path. Measurable by code review.
