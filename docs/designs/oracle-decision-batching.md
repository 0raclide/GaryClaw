## Problem Statement

Every Oracle decision is a separate SDK `query()` call with a ~43K token prompt. The prompt is: Decision Principles (500 tokens) + taste (4K) + domain expertise (20K) + decision outcomes (12K) + MEMORY.md (6K) + recent decisions (500) + the actual question (500). Of that, 42.5K tokens are identical across every call in a segment. Only the question changes.

At 20 decisions per job × 2-5s per call = 40-100s of serial Oracle overhead. With 274+ decisions tracked and growing, 5 parallel daemon instances, and 6 cron cycles per day, this compounds to 25+ minutes of wall-clock Oracle latency daily.

The Oracle's `createSdkOracleQueryFn` in `oracle.ts` creates a fresh `query()` session for every single decision. The full 43K context is sent, processed, and discarded each time.

## What Makes This Cool

The Oracle becomes conversational instead of amnesiac. Instead of 20 cold starts per job, it does 1 cold start and 19 warm follow-ups. The first question loads the full context (principles, memory, domain expertise). Every subsequent question says "here's another one" and the Oracle already knows everything. Context reuse means the Oracle can also see its own prior decisions natively (they're in the conversation history) instead of through a truncated `decisionHistory` injection.

The latency win is real: ~50% reduction from session reuse alone. The quality win might be even more interesting... an Oracle that sees its full decision thread makes more consistent choices than one that gets a 5-item summary each time.

## Constraints

- Must not require `ANTHROPIC_API_KEY`. GaryClaw uses Claude Max login (API key stripped in `buildSdkEnv`). All Oracle calls must go through the Agent SDK `query()`.
- Must preserve the per-decision audit trail in `decisions.jsonl`. Each decision needs its own timestamp, confidence, rationale, and principle.
- Must preserve escalation detection (security phrases, low confidence).
- Must be backward-compatible: `--no-memory` flag, non-autonomous mode, and the existing `OracleConfig.queryFn` interface must still work.
- Must handle session invalidation gracefully (fall back to full prompt).
- Effort budget: S (small). ~4 source files changed (new `oracle-session.ts`, modified `oracle.ts`, `ask-handler.ts`, dashboard wiring).

## Premises

1. **Oracle decisions arrive one-at-a-time, 12-400s apart.** Verified across 694 decisions in production logs. Zero rapid-fire (<2s) pairs observed. Cross-call time-window batching is not viable.
2. **The 43K token prompt prefix is identical across all calls in a segment.** Principles, memory, domain expertise, and outcomes don't change between decisions within the same job.
3. **The Agent SDK supports `resume` with a session ID.** Already used by the orchestrator for relay segments. The Oracle can reuse a session to avoid resending context.
4. **Multi-question AskUserQuestion calls are rare today.** But the code supports them (the `for...of` loop in ask-handler.ts), so batching within a single call is still worth supporting for future-proofing.
5. **Session reuse reduces per-call context to ~1K tokens (just the question).** The full conversation history is maintained by the SDK. No need to re-inject principles or memory.

## Approaches Considered

### Approach A: Oracle Session Reuse (recommended)

**Summary:** Maintain a persistent Oracle "conversation" per ask-handler instance. First decision sends the full 43K context. Subsequent decisions resume the session with just the new question (~1K tokens). The SDK's `resume` parameter carries the conversation context.

**Effort:** S (human: ~3 days / CC: ~25 min)
**Risk:** Medium (SDK `resume` behavior with `maxTurns: 1` needs spike verification)

**Pros:**
- Biggest latency win: ~50% reduction (1 cold start + N-1 warm calls)
- Better decision consistency: the Oracle sees its own prior decisions natively instead of through a 5-item summary
- Reduces total Oracle token cost by ~40% per job (context sent once, not N times)
- Clean separation: `OracleSession` class owns session lifecycle, `askOracle` becomes a thin wrapper

**Cons:**
- Session context grows linearly with decisions (each Q+A pair adds ~1-2K tokens). At 20 decisions, that's ~40K of history. Mitigated by a `maxSessionTokens` threshold (default: 80K estimated) that triggers session reset, falling back to a fresh full-context call. This caps worst-case growth while preserving the optimization for typical jobs (20-30 decisions).
- If the SDK invalidates the session (timeout, error), must fall back to full prompt. Needs a retry path.
- The `resume` behavior with `maxTurns: 1` hasn't been tested for Oracle-style use (no tools, just text completion). Needs a 5-minute spike.
- Changes the `OracleConfig` interface: `queryFn` becomes `queryFnWithResume` or the session management moves into the config.

**Concurrency note:** `OracleSession.ask()` calls are inherently serialized because `canUseTool` is called once per tool use and Claude waits for the response before proceeding. No mutex needed since the SDK never fires two AskUserQuestion callbacks concurrently within a single segment.

**Reuses:** `sdk-wrapper.ts` resume support, `oracle.ts` prompt building, `ask-handler.ts` decision loop.

**Implementation sketch:**

```typescript
// New: src/oracle-session.ts
export interface OracleSession {
  sessionId: string | null;
  callCount: number;
  totalTokensSaved: number;
}

export function createOracleSession(
  queryFn: (prompt: string, resume?: string) => Promise<{ text: string; sessionId: string }>,
): {
  ask: (input: OracleInput, config: OracleConfig) => Promise<OracleOutput>;
  getStats: () => OracleSession;
  reset: () => void;
} {
  let session: OracleSession = { sessionId: null, callCount: 0, totalTokensSaved: 0 };

  return {
    async ask(input, config) {
      if (!session.sessionId) {
        // First call: full context
        const prompt = buildOraclePrompt(input);
        const { text, sessionId } = await queryFn(prompt);
        session = { sessionId, callCount: 1, totalTokensSaved: 0 };
        // parseAndDecorate = parseOracleResponse + shouldEscalateForSecurity + isTaste logic
        // (same pipeline as existing askOracle() in oracle.ts)
        return parseAndDecorate(text, input, config);
      }

      // Subsequent: resume with just the question
      try {
        const prompt = buildFollowUpPrompt(input);
        const { text, sessionId } = await queryFn(prompt, session.sessionId);
        session.callCount++;
        session.totalTokensSaved += 42_000; // ~42K prefix avoided
        session.sessionId = sessionId; // Update in case it changes
        // Session growth check: reset if estimated context exceeds threshold
        const estimatedTokens = 43_000 + session.callCount * 1_500;
        if (estimatedTokens > 80_000) {
          session.sessionId = null; // Next call will start fresh
        }
        return parseAndDecorate(text, input, config);
      } catch {
        // Session expired or invalid — fall back to full prompt
        session.sessionId = null;
        return this.ask(input, config); // Retry with full context
      }
    },
    getStats: () => ({ ...session }),
    reset: () => { session = { sessionId: null, callCount: 0, totalTokensSaved: 0 }; },
  };
}
```

**Follow-up prompt (subsequent calls):**

```typescript
export function buildFollowUpPrompt(input: OracleInput): string {
  return `## New Decision Required

Skill: /${input.skillName}

## Question
${input.question}

## Options
${input.options.map((o, i) => `${i + 1}. **${o.label}**: ${o.description}`).join("\n")}

Apply the same Decision Principles from above. Respond with ONLY a JSON object:
{
  "choice": "<exact label>",
  "confidence": <1-10>,
  "rationale": "<one sentence>",
  "principle": "<which principle>"
}`;
}
```

### Approach B: Multi-Question Batching (minimal viable)

**Summary:** When a single AskUserQuestion contains N questions, batch them into one Oracle call instead of N separate calls. Modify `buildOraclePrompt` to accept an array of questions and `parseOracleResponse` to handle array responses.

**Effort:** XS (human: ~1 day / CC: ~10 min)
**Risk:** Low

**Pros:**
- Simple change: modify the for-loop in ask-handler to collect questions, make one Oracle call
- Preserves existing Oracle session model (no resume needed)
- Zero risk to existing single-question behavior

**Cons:**
- Near-zero impact today. Zero multi-question AskUserQuestion calls observed in 694 decisions.
- Later questions in a batch can't see earlier answers (dependent questions lose context)
- Only optimizes the rare multi-question case, not the common single-question case

**Reuses:** Existing `buildOraclePrompt`, `parseOracleResponse`.

### Approach C: Prompt Compression

**Summary:** Reduce the Oracle prompt size without session reuse. Aggressively truncate domain expertise (20K → 5K, keeping only sections relevant to the current skill), compress decision outcomes to a summary, and use a shorter principles format. Target: ~15K token prompt instead of ~43K.

**Effort:** S (human: ~2 days / CC: ~15 min)
**Risk:** Medium (quality regression if important context is truncated)

**Pros:**
- Each call is ~65% cheaper and faster, no SDK behavior changes needed
- Simpler than session reuse, no new state to manage
- Compounds with any future session reuse

**Cons:**
- Reduces Oracle context quality. The Oracle might make worse decisions with less domain expertise.
- Hard to measure impact without A/B testing on decision quality.
- Doesn't address the fundamental redundancy (same context sent N times).

## Recommended Approach

**Approach A: Oracle Session Reuse.**

Session reuse is the only approach that addresses the root cause: sending 43K tokens of identical context 20 times per job. Prompt compression treats the symptom (large prompts) not the disease (redundant prompts). Approach B (multi-question batching) has zero observed impact today and is deferred as a future follow-up if multi-question AskUserQuestion calls become common.

The implementation order:
1. **Spike** (5 min): Verify `resume` works with Oracle-style queries (maxTurns: 1, no tools)
2. **Core** (15 min): `oracle-session.ts` with create/ask/reset/getStats
3. **Wire** (5 min): Update `ask-handler.ts` to use OracleSession instead of raw askOracle
4. **Metrics** (5 min): Add `oracle_session_stats` event for dashboard integration

**Prerequisite spike:** Before implementing, run a 5-minute spike to verify:
- Can `query()` with `resume: sessionId` continue a `maxTurns: 1` session?
- Does the resumed session retain the original prompt context?
- Does the resumed session accept a new prompt as the "next message"?
- Which SDK result message field carries the session ID? (Expected: `session_id` on result message, per `extractResultData` in `sdk-wrapper.ts`)
- Is the `canUseTool: deny` callback re-applied on resume, or inherited from the original session?

If the spike fails (resume doesn't work for Oracle-style queries), fall back to Approach C.

**Approach C fallback implementation:** If the spike fails, Approach C is a direct modification to `buildOraclePrompt` in `oracle.ts`. Truncate domain expertise from 20K to 5K tokens (keep only sections whose headings match the current `skillName`), compress decision outcomes to the most recent 10 entries, and remove MEMORY.md from the Oracle prompt (it's the least decision-relevant context). This reduces the prompt from ~43K to ~15K tokens per call with no new files or SDK behavior changes. The `ORACLE_MEMORY_BUDGETS` in `types.ts` gains a `oracleCompressed` variant.

## Open Questions

1. **Does `resume` with `maxTurns: 1` work as expected?** The SDK resume feature is designed for multi-turn agent sessions. Using it for single-turn Oracle queries is an untested pattern. The 5-min spike will answer this.

2. **Session lifetime vs. segment lifetime.** Should the Oracle session persist across relay boundaries? Recommendation: no. Reset on each new segment. The memory/context might change between relays (new decisions, updated outcomes).

3. **Dashboard integration.** Should the dashboard show Oracle session stats (tokens saved, cache hits, session resets)? Recommendation: yes, via `adaptive_turns`-style event emission. Low effort, high observability.

4. **Fallback behavior.** When session resume fails, should the Oracle retry with full context silently or emit a warning? Recommendation: retry silently (to the caller), but log a `session_reset` event for observability.
