# Design: TODO State Tracking — Artifact Detection + State Files

Generated by /office-hours on 2026-03-29
Branch: main
Repo: 0raclide/GaryClaw
Status: APPROVED
Mode: Builder

## Problem Statement

On 2026-03-29, 5 parallel daemon workers rebuilt the same features 2-3x each, wasting $30+ in compute. The root cause: the daemon has no persistent memory of what pipeline stage a TODO item has reached. Design docs exist in `docs/designs/`, branches have committed code, QA may have already passed, but every new pipeline cycle starts from `open` state and re-runs the entire pipeline from scratch.

The existing cross-cycle dedup (`getCompletedTodoTitles` in `daemon-registry.ts`) only tracks binary completion (done or not). It can't express "this TODO was designed yesterday, skip to implement." The pipeline state in `pipeline.json` is per-job and ephemeral, not per-TODO and persistent.

## What Makes This Cool

The daemon becomes truly incremental. Push 5 TODO items overnight, each one progresses through design, implement, review, QA, merge on its own timeline. If a worker crashes mid-implement, the next cycle picks up at implement, not prioritize. If an instance gets stopped and cleaned up, the state file survives and another instance continues. If the state file itself gets deleted, artifact detection reconstructs the state from git history. Zero wasted compute, zero lost progress.

## Constraints

- Must use existing `safeWriteJSON`/`safeReadJSON` for atomic persistence (corruption recovery built in)
- State transitions only at skill boundaries (never mid-skill, skills are idempotent)
- Must not break existing claiming infrastructure (`claimedTodoTitle`, `getClaimedTodoTitles`, file conflict prevention)
- State directory `.garyclaw/todo-state/` must be gitignored (instance-local, not committed)
- Slug function must be pure and deterministic (no Date.now, no randomness)
- Must fail-open: if state file is corrupt or missing, fall back to full pipeline (never block work)

## Premises

1. The $30 waste is caused by lack of state tracking. Claiming partially works (prevents two workers from picking the same item in a single cycle), but has a known race condition where parallel instances can claim before persistence (see Session 3 claim timing bug). State tracking addresses the *consequence* (redundant work across cycles) rather than the *cause* (claim timing within a cycle). Even with perfect claiming, without state tracking each cycle restarts completed items from scratch.
2. State transitions map cleanly to skill boundaries: `open` -> `designed` (after office-hours) -> `implemented` (after implement) -> `reviewed` (after eng-review) -> `qa-complete` (after QA) -> `merged` (after auto-merge succeeds) -> `complete` (after TODOS.md item is struck through or removed). The `merged`/`complete` distinction exists because auto-merge success doesn't mean the TODOS.md housekeeping happened. Both map to `"skip"` in `getStartSkill()`, but `complete` is the terminal state used by `getCompletedTodoTitles` for cross-cycle dedup. In practice, the daemon writes `merged` after auto-merge, and a future housekeeping pass (deferred, out of scope) would advance to `complete`.
3. A file-per-TODO approach (`.garyclaw/todo-state/{slug}.json`) is the right persistence model. Per-TODO is the natural key, unlike per-job (ephemeral) or per-instance (dies on cleanup).
4. Slug stability is achievable with deterministic slugification + Levenshtein fallback for minor title edits.
5. System B (state files) + System A (artifact detection) together deliver self-healing that's worth the ~100 extra lines over state files alone.
6. The existing `safeWriteJSON`/`safeReadJSON` infrastructure handles atomic persistence needs without new dependencies.

## Approaches Considered

### Approach A: State Files Only (Minimal Viable)

**Summary:** New `src/todo-state.ts` with `slugify()`, `readTodoState()`, `writeTodoState()`, `getStartSkill()`. Wire into job-runner and pipeline. ~150 lines new + ~50 lines wiring.

**Effort:** S
**Risk:** Low
**Pros:**
- Simple, fast to implement
- Solves the core $30 waste problem
- Uses existing `safeWriteJSON` infrastructure

**Cons:**
- No self-healing if state files get corrupted or deleted
- State only as good as the writes. Stale state from a crashed instance stays stale until timeout.
- Can't recover after `--cleanup` removes a worktree and its branch

**Reuses:** `safeWriteJSON`/`safeReadJSON`, `parseTodoItems`, existing claiming in `job-runner.ts`

### Approach B: State Files + Artifact Detection (Ideal Architecture)

**Summary:** Everything in A, plus `reconcileState()` that cross-validates state files against real artifacts (design docs, branches, commits on main). Runs at processNext() time. Self-heals when state is stale or missing. ~250 lines total.

**Effort:** S (slightly larger, same order of magnitude)
**Risk:** Medium (artifact detection heuristics need careful testing)
**Pros:**
- Self-healing: delete every state file, it reconstructs from git
- Catches stale states from crashed instances without waiting for 2h timeout
- "Truth is in the artifacts" philosophy matches the project ethos
- Only ~100 lines more than Approach A

**Cons:**
- Artifact detection heuristics could misfire (design doc for a different TODO matches)
- Git operations in processNext() add latency (~50-100ms for branch list + log scan)
- More test surface area (reconciliation logic needs thorough coverage)

**Reuses:** Same as A, plus `getGitHead` from pipeline.ts, `execFileSync` for git branch/log queries

### Approach C: Extend Pipeline State (No New Module)

**Summary:** Add `todoTitle` and `todoSlug` fields to `PipelineState`. Let `daemon-registry.ts` aggregate pipeline states across instances. No new files or state directory.

**Effort:** XS
**Risk:** Medium
**Pros:**
- Zero new infrastructure
- Leverages existing pipeline persistence

**Cons:**
- Pipeline state is per-job, not per-TODO. Doesn't survive job cleanup.
- Harder to query "what state is TODO X in?" across multiple pipeline runs
- No self-healing capability
- Muddies the pipeline abstraction (pipelines are about skill sequences, not TODO lifecycle)

## Recommended Approach

**Approach B: State Files + Artifact Detection.** The self-healing property is the difference between "state tracking that mostly works" and "state tracking you never have to think about." Artifact detection is ~100 lines of git queries with well-defined semantics. The cost is minimal. The protection against $30 overnight surprises is high.

### `slugify()` — Pure, Deterministic

```
Input:  "TODO State Tracking — Artifact Detection + State Files"
Output: "todo-state-tracking-artifact-detection-state-files"
```

Rules:
1. Lowercase
2. Replace non-alphanumeric with hyphens
3. Collapse multiple hyphens
4. Trim leading/trailing hyphens
5. Truncate to 80 chars (at word boundary)

No Date.now, no randomness, no external state. Same input always produces same output.

### `detectArtifacts()` — Evidence-Based Verification

Scans three artifact sources:

1. **Design docs in `docs/designs/`:** Two-tier matching: (a) slug substring match in filename (e.g., slug `todo-state-tracking` matches `todo-state-tracking.md`), or (b) if no slug match, require >= 2 non-stopword title keywords found in the first 5 lines of candidate files. Stopwords: "the", "a", "an", "for", "and", "or", "in", "of", "to", "with". This prevents false positives like "oauth" matching unrelated OAuth design docs. Cost: ~10ms (local SSD, small repo; may reach ~50ms on large repos or network filesystems).

2. **Git branches:** `git branch --list "garyclaw/*"` + grep for slug. Also check for branches containing key title words. Cost: ~20ms.

3. **Commits on main:** `git log --oneline -50 main` + match against slug (substring in commit message) or >= 2 non-stopword title keywords in the same commit message. Same stopword list as design doc matching. This prevents false positives from common words like "fix" or "validation" that appear in many commits. Cost: ~30ms.

Total: ~60ms per TODO item on local SSD with small repo. May reach ~200ms on large repos or network-mounted worktrees. Acceptable for processNext() hot path (runs once per job, not per turn).

### `reconcileState()` — Self-Healing Rules

The TODO description already specifies these rules clearly:

| State File (B) | Artifacts (A) | Action |
|----------------|---------------|--------|
| null | design doc exists | Create state as `designed` |
| null | branch with commits | Create state as `implemented` |
| null | commits on main | Create state as `merged` |
| `designed` | branch with commits | Promote to `implemented` |
| `implemented` | commits on main | Promote to `merged` |
| `implemented` | no branch, no main commits | Check updatedAt: if > 2h + instance PID dead (via `isPidAlive` from `pid-utils.ts`, looking up PID from `state.instanceName`'s PID file at `.garyclaw/daemons/{instanceName}/daemon.pid`; if PID file is missing, treat instance as dead), reset to `designed`. If instance PID is alive, trust B (work in progress). |
| Any | A shows MORE advanced | Promote B (evidence trumps records) |
| Any | A shows LESS advanced, B recent (<2h) | Trust B (work in progress) |
| `open` only | A shows LESS advanced, B stale (>2h) | Trust A (safe demotion). Demoting from `open` is a no-op anyway. Demoting from `designed` to `open` would re-run office-hours, which is acceptable cost (~$0.50) but not worth the false positive risk, so `designed` is excluded from demotion. |
| `designed` or later | A shows LESS advanced, B stale (>2h) | Trust B but log WARNING. Never demote from `designed` or later states based on missing artifact evidence alone. Rationale: overnight runs can take 4-8h, and a merged branch deletion or `--cleanup` removes artifact evidence while the state is legitimately advanced. The fail-open principle here means "slightly wasteful skip is better than re-running a $5 implement cycle." |

### `getStartSkill()` — State to Pipeline Mapping

```typescript
function getStartSkill(state: TodoState): string {
  switch (state.state) {
    case "open":         return "prioritize";
    case "designed":     return "implement";
    case "implemented":  return "plan-eng-review";  // or QA if no review in pipeline
    case "reviewed":     return "qa";
    case "qa-complete":  return "skip";             // auto-merge happens post-pipeline, not as a skill
    case "merged":
    case "complete":     return "skip";             // skip entirely
  }
}
```

Note: `qa-complete` maps to `"skip"` because auto-merge is handled by job-runner post-pipeline completion, not as a pipeline skill. The job-runner already has auto-merge wiring that runs after `runPipeline()` resolves. If QA passed, the TODO is ready for merge and the pipeline has no more skills to run.

**Pipeline skill matching:** `getStartSkill()` returns a *preferred* start skill. The job-runner wiring uses `findNextSkill()` (not raw `indexOf`) to find the first skill in the configured pipeline that is at or after the preferred skill in lifecycle order:

```typescript
const LIFECYCLE_ORDER = ["prioritize", "office-hours", "implement", "plan-eng-review", "qa"];

function findNextSkill(pipelineSkills: string[], preferredStart: string): number {
  const preferredIdx = LIFECYCLE_ORDER.indexOf(preferredStart);
  if (preferredIdx === -1) return 0; // unknown skill, start from beginning

  // Find the first pipeline skill at or after the preferred lifecycle position
  for (let i = 0; i < pipelineSkills.length; i++) {
    const skillIdx = LIFECYCLE_ORDER.indexOf(pipelineSkills[i]);
    if (skillIdx >= preferredIdx) return i;
  }
  return pipelineSkills.length; // all skills are before preferred, skip all
}
```

This handles partial pipelines: if `getStartSkill` returns `plan-eng-review` but the pipeline is `[prioritize, implement, qa]`, it skips to `qa` (the next skill at or after `plan-eng-review` in lifecycle order).

### Wiring: `job-runner.ts` processNext()

After the existing pre-assignment claiming block (~line 360), add:

```typescript
// Read TODO state to determine start skill
if (preAssignedTitle) {
  const slug = slugify(preAssignedTitle);
  const storedState = findTodoState(checkpointDir, preAssignedTitle);
  const artifacts = detectArtifacts(
    jobConfig.worktreePath ?? jobConfig.projectDir,
    preAssignedTitle,
    slug,
  );
  const reconciledState = reconcileState(storedState, artifacts);

  const startSkill = getStartSkill(reconciledState);
  if (startSkill === "skip") {
    d.log("info", `TODO "${preAssignedTitle}" already complete — skipping`);
    // Mark job complete without running
    nextJob.status = "complete";
    nextJob.completedAt = new Date().toISOString();
    running = false;
    persistState(state, checkpointDir);
    processNext(); // Try next job
    return;
  }

  // Trim pipeline skills to start from the right point
  const pipelineSkills = nextJob.skills;
  const startIndex = findNextSkill(pipelineSkills, startSkill);
  if (startIndex > 0 && startIndex < pipelineSkills.length) {
    const skippedSkills = pipelineSkills.slice(0, startIndex);
    nextJob.skills = pipelineSkills.slice(startIndex);
    d.log("info", `TODO "${preAssignedTitle}" at state "${reconciledState.state}" — skipping [${skippedSkills.join(", ")}]`);
  } else if (startIndex >= pipelineSkills.length) {
    // All pipeline skills are before the current state — skip entirely
    d.log("info", `TODO "${preAssignedTitle}" at state "${reconciledState.state}" — all pipeline skills already complete`);
    nextJob.status = "complete";
    nextJob.completedAt = new Date().toISOString();
    running = false;
    persistState(state, checkpointDir);
    processNext();
    return;
  }

  // Pass design doc path to implement if available
  if (reconciledState.designDocPath) {
    nextJob.designDocPath = reconciledState.designDocPath;
  }
}
```

### Wiring: `pipeline.ts` executePipelineFrom()

After each skill completes successfully (~line 458), add a state write:

The `skillToTodoState` mapping converts skill names to lifecycle states:

```typescript
const SKILL_TO_STATE: Record<string, TodoLifecycleState> = {
  "office-hours":     "designed",
  "implement":        "implemented",
  "plan-eng-review":  "reviewed",
  "qa":               "qa-complete",
};

function skillToTodoState(skillName: string): TodoLifecycleState | null {
  return SKILL_TO_STATE[skillName] ?? null;
}
```

Note: `prioritize` is intentionally absent. It does not advance state because it merely selects a TODO item. The state advances to `designed` only after office-hours produces a design doc. This means `prioritize` is expected to always be followed by `office-hours` in the same pipeline run (e.g., `[prioritize, office-hours, implement, qa]`). A standalone `prioritize` run (without office-hours) is a valid no-op from the state tracking perspective: the item is claimed but not yet designed.

Skills not in this map (e.g., `bootstrap`, `research`, `evaluate`) don't affect TODO lifecycle state.

```typescript
// Advance TODO state after skill completion
if (config.todoTitle) {
  const slug = slugify(config.todoTitle);
  const newState = skillToTodoState(skillName);
  if (newState) {
    // Extract design doc path after office-hours: scan docs/designs/ for most recent file
    // matching the slug. This mirrors how the implement skill discovers design docs.
    let designDocPath = existingState?.designDocPath;
    if (skillName === "office-hours") {
      const designDir = join(config.projectDir, "docs", "designs");
      try {
        const files = readdirSync(designDir)
          .filter(f => f.includes(slug) || f.includes(slug.slice(0, 20)))
          .sort(); // alphabetical, most recent filename wins
        if (files.length > 0) {
          designDocPath = join("docs", "designs", files[files.length - 1]);
        }
      } catch { /* docs/designs/ may not exist yet */ }
    }

    const currentBranch = (() => {
      try { return execFileSync("git", ["branch", "--show-current"], { cwd: config.projectDir, encoding: "utf-8" }).trim(); }
      catch { return undefined; }
    })();

    writeTodoState(config.checkpointDir, slug, {
      title: config.todoTitle,
      slug,
      state: newState,
      designDocPath,
      branch: currentBranch,
      instanceName: config.instanceName,
      lastJobId: state.pipelineId,
      updatedAt: new Date().toISOString(),
    });
  }
}
```

### Slug Stability

The slug is deterministic from the title. If a title changes slightly (e.g., "TODO State Tracking" becomes "Todo State Tracking"), the exact slug match fails. Fallback: scan all state files, compute Levenshtein distance between stored `title` and query `title`, match if normalized distance < 0.3 (same threshold used by `reflection.ts` for reopened issue detection).

**Tie-breaking:** Exact slug match always takes priority over Levenshtein. If multiple state files match via Levenshtein (both within 0.3 threshold), the closest match wins (lowest normalized distance). If distances are equal, the most recently updated file wins (`updatedAt` comparison). This should be rare in practice since TODO titles tend to be distinct.

## Resolved Questions

1. **Partial pipeline definitions:** Resolved by `findNextSkill()` with `LIFECYCLE_ORDER`. If `getStartSkill` returns a skill not in the configured pipeline, `findNextSkill` skips to the next pipeline skill at or after that lifecycle position. See `getStartSkill()` section above.

## Open Questions

1. **Should state files be per-project or per-project-per-instance?** Current proposal: per-project (`.garyclaw/todo-state/`), not per-instance (`.garyclaw/daemons/{name}/todo-state/`). This means all instances share state, which is the correct behavior for "don't redo work that another instance already did."

2. **Should `detectArtifacts()` run on every processNext() call, or only when the state file is missing/stale?** Running every time adds ~60ms but provides continuous self-healing. Running only on miss saves time in the common case. Recommendation: run every time. 60ms is noise compared to job execution time.

## Success Criteria

- Parallel daemon instances never re-run a pipeline stage that's already complete for a given TODO item
- A TODO item that was `designed` yesterday starts at `implement` today (not `prioritize`)
- Deleting all state files and restarting causes artifact detection to reconstruct correct states
- State file corruption falls back to full pipeline (fail-open, no blocked work)
- `garyclaw doctor` detects orphaned and stale state files
- Overnight 5-worker run shows $0 in redundant compute (vs $30+ before)
