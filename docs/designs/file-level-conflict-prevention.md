# Design: File-Level Conflict Prevention for Parallel Instances

Generated by /office-hours on 2026-03-29
Branch: main
Repo: 0raclide/GaryClaw
Status: APPROVED
Mode: Builder

## Problem Statement

When 5 parallel daemon instances run overnight, each picks a different TODO item, implements it in its own worktree branch, and attempts auto-merge back to main. On 2026-03-29, 5 workers spent $49.60 but only 2 features merged cleanly. The other 3 conflicted because they independently modified the same core files (`oracle.ts`, `types.ts`). That's ~$30 of compute wasted on work that couldn't land.

The existing pre-assignment system (`job-runner.ts:260-308`) prevents two instances from claiming the same TODO title, and cross-cycle dedup prevents re-doing completed items. But there's zero awareness of which *files* each TODO item will touch. Two different TODO items can both require changes to `types.ts`, and the system has no way to know.

## What Makes This Cool

GaryClaw is a daemon that builds itself. Parallel instances autonomously pick backlog items, design solutions, implement them, and merge. File-level conflict prevention is the missing piece that makes parallel self-improvement actually reliable. Instead of a 40% merge success rate, you get close to 100%. Same overnight budget, 2.5x more value delivered.

## Constraints

- Must integrate into the existing pre-assignment flow in `processNext()` (job-runner.ts)
- Must use the existing cross-instance coordination pattern in `daemon-registry.ts`
- Fail-open: if no file paths can be extracted from a TODO, allow the claim (don't block work over missing metadata)
- No new dependencies. Pure TypeScript, filesystem-based coordination (same as everything else in GaryClaw)
- Must not add latency to job startup. File extraction and overlap checks should complete in <10ms

## Premises

1. The primary cost of merge conflicts is wasted compute ($30/run), not wasted time (daemon runs overnight anyway)
2. TODO descriptions + design docs contain enough file path hints to predict ~80% of conflicts. The 20% that can't be predicted should fail-open
3. The solution extends the existing pre-assignment and claiming infrastructure in job-runner.ts and daemon-registry.ts
4. False positives (blocking a non-conflicting item) are cheaper than false negatives (letting a conflict through). A blocked item just gets picked up in the next cycle
5. A static module dependency map captures the co-modification patterns that caused the 2026-03-29 failures. Files like `types.ts` are always co-modified with the module that adds new interfaces

## Approaches Considered

### Approach A: Static File Extraction (Minimal Viable)

Extract file paths from TODO descriptions and design docs using regex patterns. Maintain a `claimedFiles` string array on each Job in `daemon-state.json`. Before claiming a TODO, check if its predicted files overlap with any running instance's claimed files.

**Effort:** S
**Risk:** Low

**Pros:**
- Simple regex extraction (~50 lines), no new dependencies
- Extends existing claiming infrastructure naturally
- Fail-open when no files detected

**Cons:**
- Only catches files explicitly mentioned in TODO text
- Can't predict indirect modifications (e.g., "add new Oracle principle" doesn't mention `types.ts`)
- The exact failure mode from 2026-03-29 (indirect `types.ts` co-modification) would NOT be caught

**Reuses:** `getClaimedTodoTitles()` pattern, `parseTodoItems()`, pre-assignment block

### Approach B: Static Extraction + Module Dependency Map (Recommended)

Approach A plus a configurable module dependency map. When a TODO mentions `oracle.ts`, the system also claims `types.ts` and `oracle-memory.ts` because they're known co-modification targets.

**Effort:** S
**Risk:** Low

**Pros:**
- Catches indirect modifications (the exact 2026-03-29 failure mode)
- Map is small (~15 entries for GaryClaw), explicit, and auditable
- Lives in `.garyclaw/file-deps.json` or hardcoded as a default
- Explicit > implicit: you can see exactly why a claim was blocked

**Cons:**
- Map needs manual updates when new modules are added
- Module-level granularity (can't distinguish which parts of `types.ts` are touched)

**Reuses:** Same as A, plus `safeReadJSON` for the config file

### Approach C: Keyword-to-File Heuristic (Creative/Lateral)

Instead of extracting file paths, extract semantic keywords from TODO titles ("Oracle" -> `oracle.ts`, "dashboard" -> `dashboard.ts`). Works even when TODOs don't mention file paths at all.

**Effort:** S
**Risk:** Medium (keyword ambiguity)

**Pros:**
- Works when TODOs don't mention file paths
- Captures intent rather than explicit references

**Cons:**
- Keyword ambiguity ("implement" matches `implement.ts` or is a generic verb)
- Harder to debug why a claim was blocked
- Needs tuning and stop-word filtering

**Reuses:** `extractTopicKeywords()` pattern from auto-research.ts

## Recommended Approach

**Approach B: Static Extraction + Module Dependency Map.** It directly addresses the exact failure mode from 2026-03-29 with an explicit, auditable dependency map. The map costs ~15 lines to maintain and catches the indirect `types.ts` co-modifications that killed 3 out of 5 overnight workers.

## Implementation

### New Module: `src/file-conflict.ts`

Three pure functions:

#### 1. `extractPredictedFiles(todoDescription: string, designDocContent?: string): string[]`

Extract file paths from TODO markdown. Patterns to match:
- Explicit paths: `src/oracle.ts`, `oracle.ts`, `test/oracle.test.ts`
- Backtick-wrapped: `` `oracle.ts` ``, `` `src/job-runner.ts` ``
- `**Files:**` sections in TODO descriptions
- `**Implementation notes:**` sections mentioning paths

Does NOT attempt to map function names to files (e.g., "In `processNext()`"). That would require a separate function-to-file lookup table, adding complexity for marginal benefit. The backtick-path and section-based patterns are sufficient.

Returns a deduplicated array of file basenames (e.g., `["oracle.ts", "types.ts"]`). Uses basenames to avoid path format inconsistencies (`src/oracle.ts` vs `oracle.ts`).

**Basename limitation:** Basename matching assumes a flat module structure (all source files have unique names). Projects with duplicate basenames across directories (e.g., `src/utils/types.ts` and `src/models/types.ts`) would need path-prefix disambiguation. GaryClaw has a flat `src/` structure, so this is fine today. Deferred for external repo support.

#### 2. `expandWithDependencies(files: string[], depMap: FileDependencyMap): string[]`

Given a list of predicted files, expand using the dependency map. If `oracle.ts` is predicted and the map says `oracle.ts -> [types.ts, oracle-memory.ts]`, the expanded set is `[oracle.ts, types.ts, oracle-memory.ts]`.

The dependency map type:
```typescript
export interface FileDependencyMap {
  [file: string]: string[];  // file basename -> co-modified files
}
```

Default map (derived from 2026-03-29 overnight run analysis):
```typescript
export const DEFAULT_FILE_DEPS: FileDependencyMap = {
  "oracle.ts": ["types.ts", "oracle-memory.ts"],
  "oracle-memory.ts": ["types.ts", "oracle.ts"],
  "dashboard.ts": ["types.ts"],
  "job-runner.ts": ["types.ts", "daemon-registry.ts"],
  "daemon.ts": ["types.ts", "daemon-registry.ts"],
  "pipeline.ts": ["types.ts"],
  "orchestrator.ts": ["types.ts", "token-monitor.ts"],
  "evaluate.ts": ["types.ts"],
  "bootstrap.ts": ["types.ts"],
  "worktree.ts": ["types.ts"],
  "daemon-registry.ts": ["types.ts"],
  "prioritize.ts": ["types.ts"],
  "implement.ts": ["types.ts"],
};
```

The user can override with `.garyclaw/file-deps.json` (loaded via `safeReadJSON`, falls back to default).

#### 3. `hasFileOverlap(predictedFiles: string[], claimedFiles: string[]): { overlaps: boolean; conflictingFiles: string[] }`

Set intersection. Returns which specific files conflict (for logging).

### Changes to Existing Modules

#### `src/types.ts`

Add to `Job` interface:
```typescript
claimedFiles?: string[];  // Predicted files this job will modify (for conflict prevention)
```

#### `src/daemon-registry.ts`

New function `getClaimedFiles()`:
```typescript
export function getClaimedFiles(
  checkpointDir: string,
  excludeInstance?: string,
): Map<string, string[]>  // instanceName -> claimed files
```

Scans all instance `daemon-state.json` for running/queued jobs with `claimedFiles` set. Same pattern as `getClaimedTodoTitles()`.

#### `src/job-runner.ts`

In the pre-assignment block (`processNext()`, around line 262-308):

Change the current "pick first actionable item" to a loop that iterates through actionable items until one passes the conflict check:

1. Load the dependency map once: `safeReadJSON<FileDependencyMap>('.garyclaw/file-deps.json', validator)` falling back to `DEFAULT_FILE_DEPS`
2. Load other instances' claimed files once: `getClaimedFiles(parentCheckpointDir, resolvedInstanceName)` returns `Map<string, string[]>` (instanceName -> files)
3. Build a reverse map `fileToInstances: Map<string, string[]>` from the claimed files map (maps each file to which instance(s) claim it). This preserves instance attribution for logging.
4. For each actionable item (sorted by priority):
   a. If the TODO description references a design doc (look for `**Design doc:**` field in the parsed TODO block), read it via `safeReadText(designDocPath)` from `safe-json.ts` (this function already exists, see module map). Wrap in try/catch for fail-open on missing/unreadable files.
   b. Call `extractPredictedFiles(item.description, designDocContent)`
   c. Call `expandWithDependencies(predicted, depMap)`
   d. Check overlap: for each file in the expanded set, check `fileToInstances.has(file)`. Collect all conflicting files and their owning instance names.
   e. If any overlap: log `"Skipped TODO '${item.title}': file conflict on [${conflictingFiles}] with instance(s) [${instanceNames}]"` and continue to next item
   f. If no overlap (or no predicted files, i.e., fail-open): claim it and set `nextJob.claimedFiles = expanded`
5. If ALL actionable items conflict with running instances, the instance idles for this cycle (same behavior as when the backlog is empty or exhausted)

## Open Questions

1. **Do shared dependencies cause conflict detection?** If `oracle.ts -> [types.ts]` and `dashboard.ts -> [types.ts]`, will two items touching `oracle.ts` and `dashboard.ts` respectively be detected as conflicting? **Yes, by design.** Both expanded sets include `types.ts`, so `hasFileOverlap` catches this. This is single-level expansion (not graph transitivity), but it handles the shared-dependency case that caused the 2026-03-29 failures.

2. **Should we read the design doc file for file extraction?** The TODO `designDoc` field sometimes points to a design doc with more detail. Reading it adds I/O but catches more paths. **Recommendation: Yes, but fail-open.** If the design doc doesn't exist or can't be read, just use the TODO description.

3. **Should the dependency map learn from past jobs?** After each successful merge, record which files were actually modified and update the map. **Recommendation: Defer.** The static map is good enough for v1. Learning can be added later when we have more job history.

## Success Criteria

- 5 parallel workers can run overnight with 0 file-level merge conflicts (up from 40% success rate)
- Pre-assignment adds <10ms latency to job startup
- Fail-open behavior: if a TODO has no detectable file paths, it's still claimable
- Dashboard or log shows when items are skipped due to file conflicts (observability)
- ~43 new tests, all passing

## What I noticed about how you think

- The TODO description in TODOS.md already has the right implementation sketch. The heuristic approach (extract paths from descriptions, fallback to allowing claims) shows good instinct for fail-open design.
- The $30 wasted / $49.60 total framing grounds the problem in real dollars, not abstract "merge conflicts are bad." That's the kind of specificity that makes good engineering decisions.
- The priority pick correctly identified this as the highest-ROI item. Five parallel workers at 40% success is worse than three workers at 100% success, and this fix makes the five-worker setup viable.
- The existing pre-assignment infrastructure (claiming, cross-instance scanning, cross-cycle dedup) was well-designed for extensibility. Adding file-level awareness slots in cleanly because the patterns are already there.
