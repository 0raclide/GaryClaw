# Design: Auto-Research Trigger (Daemon Integration)

Generated by /office-hours on 2026-03-27
Branch: main
Repo: 0raclide/GaryClaw
Status: APPROVED
Mode: Builder

## Problem Statement

The Oracle makes decisions during autonomous daemon jobs. When it encounters unfamiliar territory (WebSocket libraries, OAuth 2.1 flows, ARM intrinsics), confidence scores drop below 6. These low-confidence decisions are weaker, more likely to escalate, and more likely to produce outcomes that get flagged as failures in post-job reflection.

Currently, the only way to teach the Oracle about new domains is `garyclaw research <topic>` run manually by a human. The daemon has no mechanism to detect its own knowledge gaps and fill them autonomously.

The auto-research trigger closes this feedback loop: after a job completes, scan its decisions for low-confidence clusters in the same topic area. If 3+ decisions cluster around a topic, auto-enqueue a research job before the next job runs. Next time the Oracle encounters that topic, domain-expertise.md has real knowledge instead of general training data.

This makes GaryClaw genuinely self-improving. Every overnight run teaches it about domains it struggled with.

## What Makes This Cool

The "whoa" moment: your daemon was bad at OAuth decisions on Tuesday night. By Wednesday morning it had researched OAuth 2.1 best practices on its own. Wednesday night's run has higher confidence scores and better decisions in that area, with zero human intervention.

This is a compounding loop. Each job that struggles produces research that helps future jobs. Over time, the Oracle's knowledge base grows proportionally to the variety of work it does.

## Constraints

- Must not break existing behavior. Feature is gated behind `autoResearch.enabled` config flag (default: false).
- Must respect budget limits. Research jobs count against daily cost and job count budgets.
- Must respect freshness. If a topic was researched within 14 days, skip it (the existing researcher already handles this).
- Must not block the job queue. Research jobs are enqueued like any other job, processed by the same FIFO queue.
- Must work with parallel daemon instances. Research enqueue uses existing cross-instance dedup.
- Topic extraction must be best-effort, not perfect. A simple heuristic is better than a complex NLP pipeline. Track extraction quality post-launch: `topics_extracted / low_confidence_decision_count` ratio in the dashboard. If consistently below 0.5, upgrade to Approach C.

## Premises

1. **Topic extraction from decision questions is tractable.** Oracle questions contain the domain context ("Which WebSocket library should we use?", "Should we use PKCE for this OAuth flow?"). Keyword extraction from the question text is sufficient to infer the topic area.

2. **3+ low-confidence decisions in one topic area within a single job signals a genuine knowledge gap.** One low-confidence decision could be an ambiguous question. Three in the same area means the Oracle is consistently uncertain about that domain.

3. **Research should happen between jobs, not during.** Injecting research mid-job would interrupt the current task and waste context. Post-job research jobs are queued and processed normally.

4. **The existing researcher module needs no changes.** `runResearch(config: ResearchConfig, startSegmentFn)` (researcher.ts:65) already accepts a topic string, handles freshness, web search, section merging, and token budgets. The `startSegmentFn` parameter matches the SDK wrapper's `startSegment` function available in the orchestrator. The new code only determines *when and what* to research.

5. **This feature belongs in the post-job hook, not the orchestrator.** The orchestrator runs within a single job. The auto-research trigger fires after a job completes, which is the job-runner's domain. Specifically, it hooks into the same post-completion path where reflection and dashboard generation already run.

## Approaches Considered

### Approach A: Post-Job Topic Analyzer (Minimal Viable)

**Summary:** A new `src/auto-research.ts` module (~100 lines) with two pure functions: `extractTopicsFromDecisions()` groups low-confidence decisions by topic keywords, and `getResearchTopics()` returns topics that exceed the threshold. The job-runner calls these after reflection and enqueues research jobs for each topic.

**Effort:** S (human: ~2 days / CC: ~20 min)
**Risk:** Low
**Pros:**
- Minimal new code. Two pure functions + one hook in job-runner.
- Topic extraction is simple keyword clustering (extract 2-3 word noun phrases from question text, group by overlap).
- Fully testable with synthetic data, no SDK calls in tests.
- Reuses existing `runResearch()` and job-runner enqueue.

**Cons:**
- Keyword extraction is naive. "Which authentication flow?" and "OAuth token refresh?" might not cluster because they share no keywords.
- No learning about which research actually helped (that's the 10x version).

**Reuses:** `researcher.ts` (runResearch), `job-runner.ts` (enqueue), `reflection.ts` (reads decisions), existing DaemonConfig schema.

### Approach B: Oracle-Assisted Topic Extraction (Ideal Architecture)

**Summary:** Instead of keyword clustering, use a single Oracle query post-job to analyze all low-confidence decisions and extract research topics. The Oracle sees the full decision context and can infer higher-level topic areas that keyword matching would miss.

**Effort:** S (human: ~3 days / CC: ~25 min)
**Risk:** Medium
**Pros:**
- Much better topic extraction. The Oracle understands that "authentication flow" and "PKCE" and "token refresh" are all "OAuth 2.1".
- Single API call to analyze all low-confidence decisions at once (cheaper than per-decision analysis).
- The Oracle already has a query function (`createSdkOracleQueryFn`) ready to use.

**Cons:**
- Adds API cost per job (one Oracle query, ~40K tokens). Negligible relative to job cost but non-zero.
- Oracle might hallucinate topics. Need a validation step.
- Slightly more complex to test (need to mock Oracle query).

**Reuses:** `oracle.ts` (createSdkOracleQueryFn), `researcher.ts`, `job-runner.ts`, `reflection.ts`.

### Approach C: Hybrid (keyword fast-path + Oracle fallback)

**Summary:** Try keyword clustering first (Approach A). If it produces fewer topics than expected (e.g., 5+ low-confidence decisions but only 1 topic cluster), fall back to Oracle-assisted extraction (Approach B). Best of both worlds: free when keywords work, smart when they don't.

**Effort:** S (human: ~3 days / CC: ~25 min)
**Risk:** Low
**Pros:**
- Zero API cost when keyword clustering works (most cases).
- Falls back to Oracle intelligence for ambiguous cases.
- Clear upgrade path: start with keyword-only, add Oracle fallback later.

**Cons:**
- More code paths to test.
- The fallback threshold is another tunable parameter.

**Reuses:** Everything from A and B.

## Recommended Approach

**Approach A: Post-Job Topic Analyzer (Minimal Viable).**

The keyword extraction heuristic should handle most cases well. Low-confidence decisions tend to explicitly name the technology or domain they're uncertain about ("Should we use ws or Socket.IO?", "Is this the right OAuth flow?"). The question text is already highly specific because Claude formats the questions when using tools, and technical terms survive the split-and-filter algorithm easily. We'll track extraction quality via the `topics_extracted / low_confidence_decision_count` ratio (see Constraints) and upgrade to Approach C if the ratio drops below 0.5 over 10+ jobs.

Approach B is the better long-term solution, but it adds API cost and complexity for marginal improvement in the initial version. The upgrade path from A to C is clean: `extractTopicsFromDecisions()` is a pure function that can be replaced or augmented without changing any callers.

Ship A now. If topic extraction quality is poor (measurable: research jobs enqueued for irrelevant topics, or low-confidence decisions don't produce any topics), upgrade to C.

## Open Questions

1. **Should research jobs have lower budget priority than regular jobs?** Currently all jobs share the same budget. A research-heavy night could crowd out QA/implement jobs. Possible mitigation: cap auto-research at 2 jobs per post-job trigger.

2. **Should we track which research actually improved confidence?** The 10x version would compare Oracle confidence on same-topic decisions before and after research. This is valuable but a separate feature (P3 item for later).

3. ~~**What about topic dedup across jobs?**~~ **Resolved:** `getResearchTopics()` checks freshness via `isTopicStale()` on `existingDomainExpertise` before returning topics. If a previous job already triggered research for the same topic, the freshness check prevents re-enqueue. No queue slot wasted.

## Success Criteria

1. After a job with 3+ low-confidence decisions in the same topic area, a research job is auto-enqueued.
2. The research job runs successfully and updates domain-expertise.md.
3. The next job encountering that topic has access to the researched knowledge (Oracle prompt includes it).
4. Feature is gated behind `autoResearch.enabled` config flag. Default: false.
5. Research jobs respect all budget limits (daily cost, per-job cost, max jobs/day).
6. No changes to existing `researcher.ts` module.
7. All new code has unit tests with synthetic data (no SDK calls).

## Distribution Plan

This is an internal module addition to GaryClaw. No distribution changes needed. Existing `npx tsx src/cli.ts daemon start` picks up the new behavior when `autoResearch.enabled` is set in daemon config.

### New Code

#### 1. `src/auto-research.ts` (~120 lines)

Imports `AutoResearchConfig` from `types.ts` (following the zero-imports-in-types convention). Imports `isTopicStale` from `researcher.ts` for freshness checks. Imports `readDecisionsFromLog` from `reflection.ts` (already exported there).

```typescript
import type { AutoResearchConfig, Decision } from "./types.js";
import { isTopicStale } from "./researcher.js";

/** Stop words filtered from topic keyword extraction */
const STOP_WORDS = new Set([
  "should", "would", "could", "which", "what", "that", "this", "have",
  "with", "from", "they", "been", "were", "will", "does", "about",
  "into", "more", "some", "than", "other", "approach", "option",
  "decide", "choose", "best", "right", "use", "using", "used",
  "garyclaw", "oracle", "decision", "question", "answer",
]);

export const DEFAULT_AUTO_RESEARCH_CONFIG: AutoResearchConfig = {
  enabled: false,
  lowConfidenceThreshold: 6,
  minDecisionsToTrigger: 3,
  maxTopicsPerJob: 2,
};

/**
 * Extract topic keywords from a decision question.
 * Returns 2-5 significant lowercased words (nouns, technical terms).
 * No stemming in v1 — exact lowercased match only.
 *
 * Algorithm:
 * 1. Lowercase the entire question string
 * 2. Split on whitespace and punctuation: question.toLowerCase().split(/[\s,;:?!.()\[\]{}"']+/)
 * 3. Filter out words <= 3 chars
 * 4. Filter out STOP_WORDS (common English + Oracle/GaryClaw jargon)
 * 5. Deduplicate
 * 6. Return first 5 remaining words (preserves original word order)
 *
 * Example: "Should we use PKCE for this OAuth flow?" → ["pkce", "oauth", "flow"]
 * Example: "Which WebSocket library should we use for real-time updates?" → ["websocket", "library", "real-time", "updates"]
 */
export function extractTopicKeywords(question: string): string[];

/**
 * Group low-confidence decisions by topic similarity.
 * Two decisions are in the same topic group if they share 2+ lowercased keywords.
 *
 * Clustering strategy: greedy single-pass assignment.
 * For each decision, iterate existing groups in order. If the decision shares
 * 2+ keywords with ANY member of a group, add it to that group (single-linkage).
 * If no group matches, create a new group. This is O(n*k) where k = number of
 * groups, which is fine for typical decision counts (10-30 per job).
 *
 * NOT transitive merge: groups don't merge after creation. This prevents
 * distant topics from chaining together (e.g., "OAuth token" + "token bucket"
 * + "bucket storage" would NOT all merge into one group).
 *
 * Returns groups sorted by size (largest first).
 */
export function groupDecisionsByTopic(
  decisions: Decision[],
  config: AutoResearchConfig,
): TopicGroup[];

export interface TopicGroup {
  /** Synthesized topic label: top 3 keywords by frequency, title-cased, space-joined.
   *  Example: "OAuth Pkce Authentication" */
  topic: string;
  keywords: string[];        // the shared lowercased keywords
  decisions: Decision[];     // the decisions in this group
  avgConfidence: number;     // average confidence across decisions
}

/**
 * Determine which topics need research based on decision analysis.
 * Returns topic strings suitable for passing to runResearch().
 *
 * Rules:
 * - Only considers decisions with confidence < lowConfidenceThreshold
 * - Requires minDecisionsToTrigger decisions in the same topic group
 * - Returns at most maxTopicsPerJob topics
 * - Skips topics that are already fresh in domain-expertise.md (via isTopicStale())
 *
 * Freshness check: calls isTopicStale(existingDomainExpertise, topic) for each
 * candidate. If the topic was researched within 14 days, it is excluded. This
 * prevents enqueuing research jobs for topics that were already researched by
 * a previous auto-research trigger or manual `garyclaw research` command.
 */
export function getResearchTopics(
  decisions: Decision[],
  existingDomainExpertise: string | null,
  config: AutoResearchConfig,
): string[];
```

```typescript
// Auto-research trigger: analyze low-confidence decisions and enqueue research
if (nextJob.status === "complete" && currentConfig.autoResearch?.enabled) {
  try {
    const decisions = readDecisionsFromLog(join(jobDir, "decisions.jsonl"));
    const memConfig = defaultMemoryConfig(currentConfig.projectDir);
    const memoryFiles = readOracleMemory(memConfig, currentConfig.projectDir);
    const topics = getResearchTopics(
      decisions,
      memoryFiles.domainExpertise,
      currentConfig.autoResearch,
    );

    if (topics.length > 0) {
      d.log("info", `Auto-research: extracted ${topics.length} topic(s) from ${decisions.length} decisions: [${topics.join(", ")}]`);
    } else {
      d.log("debug", `Auto-research: no topics extracted from ${decisions.length} decisions`);
    }

    for (const topic of topics) {
      // Use researchTopic field (not designDoc) to pass topic to orchestrator
      const jobId = enqueueWithTopic(["research"], "auto_research", `low-confidence: ${topic}`, topic);
      if (jobId) {
        d.log("info", `Auto-research: enqueued research job ${jobId} for topic "${topic}"`);
      } else {
        d.log("info", `Auto-research: skipped enqueue for "${topic}" (budget/dedup)`);
      }
    }
  } catch (err) {
    d.log("warn", `Auto-research trigger failed: ${err instanceof Error ? err.message : String(err)}`);
  }
}
```

Add `enqueueWithTopic` as a nested function inside `createJobRunner()` (same scope as `enqueue`, `state`, `persistState`, and `checkpointDir`):

```typescript
function enqueueWithTopic(
  skills: string[],
  triggeredBy: Job["triggeredBy"],
  triggerDetail: string,
  researchTopic: string,
): string | null {
  const jobId = enqueue(skills, triggeredBy, triggerDetail);
  if (jobId) {
    const job = state.jobs.find(j => j.id === jobId);
    if (job) job.researchTopic = researchTopic;
    persistState(state, checkpointDir);
  }
  return jobId;
}
```

#### 5. `src/orchestrator.ts` modifications (~20 lines)

Add a research skill dispatch at the top of `runSkill()`, before the normal SDK session flow. The dispatch is gated on BOTH `skillName === "research"` AND `config.researchTopic` being set. This is the disambiguation mechanism: a user running `garyclaw run research qa` will NOT trigger auto-research dispatch because `researchTopic` is only set by the job-runner for auto-research jobs.

```typescript
import { runResearch, type ResearchConfig } from "./researcher.js";
import { defaultMemoryConfig } from "./oracle-memory.js";

// In runSkill(), before the main orchestrator loop:
if (config.skillName === "research" && config.researchTopic) {
  callbacks.onEvent({ type: "segment_start", sessionIndex: 0, segmentIndex: 0 });

  const researchConfig: ResearchConfig = {
    topic: config.researchTopic,
    projectDir: config.projectDir,
    maxSearches: 10,
    timeoutMs: 300_000,
    force: false,
    oracleMemoryConfig: defaultMemoryConfig(config.mainRepoDir ?? config.projectDir),
  };

  // runResearch() signature: (config: ResearchConfig, startSegmentFn: ...) => Promise<ResearchResult>
  // The startSegmentFn is already available in runSkill() via sdk-wrapper.startSegment.
  // Verified: researcher.ts line 65-74 shows the expected function signature matches.
  const result = await runResearch(researchConfig, startSegment);
  callbacks.onEvent({
    type: "skill_complete",
    totalSessions: 1,
    totalTurns: result.searchesUsed,
    costUsd: 0, // research cost tracked by SDK internally
  });

  return; // Skip normal orchestrator loop
}
```
