## Problem Statement

The dashboard backend shipped (commit 9fcbb45: `dashboard-server.ts`, 846 lines). The frontend skeleton shipped (commit 1c1640b: `index.html`, `style.css`, `app.js`, `charts.js`, 1307 lines total). All four views render data, SSE connects, tabs route. But three gaps prevent this from being the "Twitch for AI agents" experience the TODO describes:

1. **Growth view is empty.** The `/api/growth` endpoint returns `{ snapshots: [], moduleAttribution: {} }` because no `buildGrowthData()` function exists. The growth tab shows "Growth data not yet cached." This is the most dramatic view (8 modules → 43 modules animated over time) and it's dead.

2. **Live view is static cards, not a live feed.** The current Live tab shows health/budget/oracle stat cards that update via SSE. But there's no streaming activity log showing events as they happen. No "now building" card with elapsed time. No commit stream. The design doc calls for a live feed; what shipped is a live dashboard. Different energy.

3. **Oracle Mind is missing taste and expertise.** The Mind tab shows decision stats, confidence sparkline, and searchable decision log (all working). But the daemon's taste profile (`taste.md`) and domain expertise (`domain-expertise.md`) aren't rendered. These are the "daemon's mind" content that makes the tab name honest.

## What Makes This Cool

The growth chart is the hero. Watching the bar chart animate from 8 modules to 43, each bar labeled by date, each module colored human (blue) or daemon (green), with the daemon-built count climbing from 0 to 22... that's the screenshot that gets shared. The live activity feed is the hook for repeat viewers. The Oracle Mind taste/expertise section is the "this thing has a personality" moment.

## Constraints

- No new modules. All work happens in existing files: `dashboard-server.ts`, `app.js`, `charts.js`, `style.css`
- No external dependencies (vanilla JS, node builtins only)
- No build step for web assets
- Growth data must be computed from git history (not manually curated)
- Must work when daemon isn't running (historical data mode)
- All reads are fail-open (missing files = empty state, not crash)

## Premises

1. **The backend REST + SSE infrastructure is complete and working.** No new endpoints needed. The existing `/api/growth`, `/api/state`, `/api/decisions`, `/api/mutations`, `/api/events` endpoints cover all data needs. The gap is data population (growth) and frontend rendering (all three gaps).

2. **Growth data comes from git history.** `git log` on `CLAUDE.md` gives us historical module/test counts. `git log --format="%ae"` classifies human vs daemon commits. `git log --diff-filter=A -- src/*.ts` gives module attribution. This is a one-time cache build (~10-30s), stored in `.garyclaw/growth-cache.json`.

3. **The live activity feed is an append-only event log in the browser.** SSE events (`job_update`, `decision`, `mutation`, `budget`) each generate a timestamped line in a scrolling feed. Max 100 entries, oldest pruned. This is the "Twitch chat" equivalent.

4. **Taste and expertise are markdown rendered as simple HTML.** No full markdown parser needed. `taste.md` and `domain-expertise.md` have predictable structure (headings, bullets, YAML frontmatter). A 30-line renderer handles the common cases.

5. **The implement skill handles all coding. This design doc covers what to build, not how to code it.**

## Architecture

### 1. Growth Data Pipeline

New exported function in `dashboard-server.ts`:

```typescript
export async function buildGrowthData(
  projectDir: string,
  checkpointDir: string,
): Promise<{ snapshots: GrowthSnapshot[]; moduleAttribution: Record<string, string> }>
```

**Algorithm:**
1. Check cache at `.garyclaw/growth-cache.json`. If `headSha` matches current HEAD, return cached data.
2. Run `git log --format="%H %aI %s" -- CLAUDE.md` (cap at 200 commits).
3. For each commit SHA, run `git show {sha}:CLAUDE.md` and extract module/test counts via regex:
   - Primary: `/(\d+) source modules.*?(\d+) test/` patterns
   - Fallback: separate `/(\d+) source modules/` and `/(\d+) tests?/` searches
   - Skip unparseable commits (fail-open). `git show {sha}:CLAUDE.md` will fail for commits before the file existed; these are silently skipped.
   - If git is unavailable or the repo is shallow-cloned (no full history), return empty snapshots array (fail-open, consistent with all other GaryClaw data loading)
4. Classify commits: `git log --format="%aI %ae" --since="2026-03-25"` (project start). Commits from `GARYCLAW_DAEMON_EMAIL` = daemon, others = human.
5. Module attribution: `git log --format="%ae" --diff-filter=A --name-only -- 'src/*.ts'` in a single call to get all module attributions at once, then parse output (one subprocess instead of 43 per-module calls). Each `src/*.ts` filename maps to the first author email in the output for that file.
6. Write cache to `.garyclaw/growth-cache.json` with `{ headSha, builtAt, snapshots, moduleAttribution }`.
7. Wire as `loadGrowthDataFn` parameter to `createRequestHandler()`.

**Incremental update:** On cache hit where HEAD has moved, run `git log --format="%H %aI %s" {cachedHeadSha}..HEAD -- CLAUDE.md` for only new commits. Append new snapshots to the existing array (newest last). Re-compute module attribution only for new `src/*.ts` files added since cache. Total execution capped at 60s.

**Performance note:** The `git log -p -- CLAUDE.md` approach (one command, parse diffs) is preferred over per-SHA `git show` calls (200 subprocess spawns). If the implementer finds `git log -p` parsing unreliable, fall back to batched `git show` with concurrency of 10 and 5s per-command timeout. Realistic cold-cache time: 30-60s for 200 commits.

### 2. Live Activity Feed

New section in the Live tab rendered by `renderLiveFeed()` in `app.js`:

```
+--------------------------------------------------+
|  LIVE ACTIVITY                                    |
|  ● 10:32:15  Oracle decided: "Use vitest" (9/10) |
|  ● 10:31:42  Job started: implement (worker-3)   |
|  ● 10:30:58  Budget: $2.14 spent on qa skill     |
|  ● 10:30:01  Mutation shipped: "Oracle Cache"     |
|  ...                                              |
+--------------------------------------------------+
```

**Implementation:**
- Add `state.activityLog = []` (max 100 entries)
- Each SSE event handler appends a `{ time, icon, text, type }` entry
- `renderActivityFeed()` builds a scrollable `<div>` with timestamped lines
- CSS classes: `.activity-line`, `.activity-time`, `.activity-icon`
- Event-to-text mapping:
  - `job_update` → "Job {status}: {skill} ({instance})"
  - `decision` → "Oracle decided: \"{chosen}\" ({confidence}/10)"
  - `budget` → "Budget: ${amount} spent"
  - `mutation` → "Mutation {outcome}: \"{todoTitle}\""
- Auto-scroll to bottom on new entries (unless user has scrolled up)
- Pulsing green dot next to "LIVE ACTIVITY" header when connected
- On SSE disconnect: pulse stops (dot turns amber), existing entries preserved, "Reconnecting..." line appended. On reconnect: dot turns green, feed resumes appending without clearing history

**Taste Profile rendering:**
- Simple markdown-to-HTML: headings → `<h3>`, bullets → `<li>`, bold → `<strong>`
- Displayed in a card with "Taste Profile" title

**Domain Expertise rendering:**
- Parse YAML frontmatter per topic for freshness dates
- Display as a grid of topic badges with freshness indicators:

## Design Decisions

1. **Growth cache builds asynchronously.** The server starts immediately and serves the Growth tab with a "Building growth cache..." skeleton. When the cache build completes, a `growth_ready` SSE event fires and the Growth tab re-renders with data. This avoids blocking server startup for the 30-60s cold cache build.

## Open Questions

1. **Taste.md format stability:** The simple markdown renderer handles headings, bullets, bold, and paragraphs. Unrecognized lines are wrapped in `<p>` tags with `white-space: pre-wrap` to preserve basic readability. If taste.md evolves to use tables or code blocks, the renderer will need updates, but won't crash.

2. **Domain expertise YAML parsing in vanilla JS:** The expertise file has `---` YAML frontmatter per topic. A simple regex splitter (`/^---$/m`) + key-value parser (~20 lines) handles this without a YAML library. Edge cases (multi-line values, nested objects) won't parse but won't crash.
