## Problem Statement

`analyzeBootstrapQuality()` in evaluate.ts scores bootstrap output on a 0-100 scale, but the scoring is structural, not semantic. It checks whether CLAUDE.md has an "Architecture" heading (10 points) but not whether that section describes the actual architecture. It checks framework coverage via `computeFrameworkCoverage()` (deps in package.json mentioned in CLAUDE.md), but that's one-directional: it catches omissions, not hallucinations.

A CLAUDE.md that says "uses Express and Jest" when the repo actually uses Fastify and Vitest would score full marks on structural completeness (40/40) and lose nothing on accuracy (the check only asks "are deps mentioned?" not "are mentions correct?"). That's a score of 70+ for a document that would actively mislead downstream skills.

The pipeline quality gate at `pipeline.ts:334` uses this score to decide whether to re-bootstrap. A false high score means the gate passes bad output through. Prioritize and implement then work from wrong assumptions, wasting $2-5 of compute per run.

NihontoWatch was dogfooded in Session 3. The bootstrap quality gate and evaluate skill exist. What's missing is verification that CLAUDE.md's factual claims match reality.

## What Makes This Cool

The system fact-checks its own generated documentation. Bootstrap produces CLAUDE.md claiming "entry point is src/index.ts, tests use Vitest, database is PostgreSQL via Prisma." The verification pass reads the filesystem: src/index.ts exists (pass), vitest is in devDependencies (pass), but there's no prisma in package.json and no .prisma directory (FAIL). Quality score drops. If below threshold, the enrichment loop fires and the second bootstrap pass knows to not hallucinate Prisma.

The per-claim truth table is the artifact. Not just a score, but a list of what bootstrap got right and wrong. That's debuggable. That feeds back into the enrichment prompt.

## Constraints

- Extends `analyzeBootstrapQuality()` in evaluate.ts. No new modules.
- Must remain pure TypeScript, no Claude calls. The whole point is deterministic verification.
- Must not increase quality gate latency by more than ~50ms (filesystem reads are fast).
- Must work on any repo, not just Node/TypeScript. Python, Go, Rust repos have different signals.
- Backward compatible: existing `BootstrapEvaluation` type gets new optional fields. Existing callers don't break.
- Claim extraction uses regex/heuristics, not NLP. Good enough for tech stack, file paths, test frameworks. Won't catch "the architecture is event-driven" but that's acceptable for v1.

## Premises

1. CLAUDE.md contains extractable factual claims: tech stack names, file paths, test framework references, entry point locations. These appear as plain text or in code blocks.
2. The filesystem is ground truth. If CLAUDE.md says "src/server.ts is the entry point" and that file doesn't exist, that's a verifiable error.
3. One-directional checking (deps → doc) is insufficient. Reverse checking (doc → repo) catches hallucinations, which are the more dangerous failure mode for downstream skills.
4. The existing `BootstrapEvaluation` type in types.ts is the right place to add claim verification results. Both pipeline quality gate and evaluate skill read this type.
5. Rebalancing the quality score (reducing structural weight, adding claim verification weight) is safe because the threshold is 50 and the enrichment loop is capped at 1 retry.

## Approaches Considered

### Approach A: Claim Verification Pass (Minimal Viable)

**Summary:** Add `verifyClaudeMdClaims()` to evaluate.ts. Extracts tech stack names, file paths, and test framework references from CLAUDE.md content. Verifies each against filesystem and package.json. Returns per-claim pass/fail. Integrates as a new sub-score in `analyzeBootstrapQuality()`.

**Effort:** S
**Risk:** Low
**Pros:**
- ~120-150 lines across evaluate.ts (~100 lines), types.ts (~20 lines), and bootstrap.ts (~10 lines)
- Reuses `extractDependencies()`, `detectTechStack()`, `KNOWN_FRAMEWORKS`, existing infrastructure
- Improves scoring for both quality gate and evaluate skill in one change
- Per-claim truth table is immediately useful for enrichment prompts

**Cons:**
- Only catches claims it knows how to parse (tech stack, paths)
- Natural language architecture descriptions can't be verified deterministically
- Regex-based claim extraction may miss some patterns

**Reuses:** `extractDependencies()`, `detectTechStack()`, `computeFrameworkCoverage()`, `walkFileTree()`

### Approach B: Full Semantic Audit

**Summary:** Three verification layers: (1) claim extraction + filesystem check (same as A), (2) reverse coverage check (important repo features not mentioned in CLAUDE.md), (3) architecture pattern validation (check if described patterns match actual import graph).

**Effort:** M
**Risk:** Medium
**Pros:**
- Catches both hallucinations and omissions comprehensively
- Import graph analysis is genuinely novel
- Would catch "claims event-driven but all code is synchronous"

**Cons:**
- Import graph parsing is ~150 extra lines and fragile across languages
- Over-engineers for current needs (we have 1 external dogfood target)
- Medium risk of false positives from import analysis heuristics

### Approach C: Ground Truth Diff

**Summary:** Generate a deterministic "reference CLAUDE.md" from `analyzeCodebase()`, diff against bootstrap output. Score = overlap ratio.

**Effort:** S
**Risk:** Low
**Pros:**
- Elegant single-metric approach
- Catches hallucinations and omissions in one pass

**Cons:**
- Reference doc is shallow (tech stack + structure only)
- Diff-based comparison struggles with paraphrasing
- Doesn't produce actionable per-claim feedback for enrichment

## Recommended Approach

**Approach A: Claim Verification Pass.** Ships in S effort, directly fixes the scoring gap, and the per-claim truth table is immediately useful for enrichment prompts. The pipeline quality gate benefits on the next run. Layer B's import graph analysis on top later if A's false-negative rate proves too high in practice.

## Implementation Details

### 1. Claim Types to Extract

```typescript
interface ClaudeMdClaim {
  type: "tech_stack" | "file_path" | "test_framework" | "entry_point";
  claimed: string;        // what CLAUDE.md says
  evidence: string;       // what the filesystem shows
  verified: boolean;      // does the claim hold?
}
```

### 2. Extraction Heuristics

**Tech stack claims:** Scan for names in the `KNOWN_FRAMEWORKS` map from evaluate.ts (React, Express, Vitest, Prisma, etc.) appearing in CLAUDE.md prose. Skip any text inside fenced code blocks that begin with `npm install`, `pip install`, `brew install`, or `cargo add` (installation instructions, not usage claims). Also skip claims that appear within 20 words of "considered," "alternative," "instead of," "not using," "replaced," or "migrated from" to avoid false positives from comparison/migration context. Cross-reference each mentioned framework against actual package.json dependencies via `extractDependencies()`. If CLAUDE.md mentions "Prisma" but no prisma package exists in any dependency field, that's a failed claim.

**V1 scope: Node/TypeScript repos only.** `extractDependencies()` reads package.json. For Python (pyproject.toml, requirements.txt), Go (go.mod), and Rust (Cargo.toml) repos, tech stack claim verification is skipped in v1 (claims still extracted but marked `verified: true` with `evidence: "non-Node repo, verification deferred"`). Multi-language dependency parsing is a follow-up enhancement. File path and entry point claims work for all languages since they use `existsSync()`.

**File path claims:** Regex for paths matching `src/...`, `lib/...`, `test/...`, `app/...` patterns in prose paragraphs and inline code. Skip paths inside fenced code blocks detected as directory tree listings. A fenced block is a tree listing if >50% of its non-empty lines match `/^\s*[├└│─┬]\s/` or `/^\s{2,}\S+\.\w+$/` (indented filenames). For tree listing blocks, take the first 5 leaf-node paths in document order and verify as a single "file tree accuracy" claim. The claim passes if >= 4/5 sampled paths exist (80% threshold). Check `existsSync()` relative to project root.

**Test framework claims:** Look for mentions of test runners (vitest, jest, mocha, pytest, go test) and verify against package.json devDependencies or file presence (e.g., `pytest.ini`, `jest.config.*`, `vitest.config.*`).

**Entry point claims:** Look for "entry point", "main file", "starts at" patterns near file paths. Verify the referenced file exists.

### 3. Reverse Coverage (Lightweight)

In addition to forward verification (doc claims → repo check), add one reverse check: are there major frameworks in package.json that CLAUDE.md never mentions? This is what `computeFrameworkCoverage()` already does, but surface it as individual `ClaudeMdClaim` entries with `verified: false` for missing mentions. **These reverse coverage claims feed into the claim verification sub-score (Section 4), not the framework coverage sub-score.** The framework coverage sub-score retains its existing forward-direction behavior (what % of deps are mentioned). The claim verification sub-score covers both directions: forward claims that are wrong + reverse claims that are missing.

### 4. Score Rebalancing

Current scoring (100 points total):
- Structural completeness: 40 pts (section headings present)
- Framework coverage: 30 pts (deps mentioned in doc)
- TODOS viability: 20 pts (items above 5.0 threshold)
- Token efficiency: 10 pts (size in sweet spot)

Proposed scoring (100 points total):
- Structural completeness: 30 pts (reduced from 40)
- Claim verification: 20 pts (NEW — % of claims verified, includes both forward and reverse claims)
- Framework coverage: 20 pts (reduced from 30, forward-direction only via existing `computeFrameworkCoverage()`)
- TODOS viability: 20 pts (unchanged)
- Token efficiency: 10 pts (unchanged)

**Worked example — all headings, wrong content:** CLAUDE.md has all 4 expected sections (30 pts structural). Claims "uses Prisma, Express, Jest" but repo has Drizzle, Fastify, Vitest. Forward claim verification: 3 hallucinated frameworks fail (Prisma/Express/Jest not in deps). Reverse claim verification: 3 real frameworks missing from doc (Drizzle/Fastify/Vitest). Total: 0/6 claims verified → 0 pts claim verification. Forward framework coverage via `computeFrameworkCoverage()`: checks if real deps (drizzle-orm, fastify, vitest) are mentioned in doc text, they aren't → 0 pts framework coverage. Note: these are disjoint failure modes, not double-counting. Claim verification penalizes mentions of non-existent dependencies; framework coverage penalizes missing mentions of real dependencies. TODOS viability 20 pts (assume decent TODOS). Token efficiency 10 pts. **Total: 30 + 0 + 0 + 20 + 10 = 60.** With a few missing sections or thin TODOS, this drops below 50, triggering enrichment.

### 5. Type Changes

```typescript
// Add to BootstrapEvaluation in types.ts
interface BootstrapEvaluation {
  // ... existing fields ...
  claims?: ClaudeMdClaim[];           // per-claim verification results
  claimsVerified?: number;            // count of verified claims
  claimsTotal?: number;               // total claims extracted
  claimVerificationScore?: number;    // 0-20 sub-score
}
```

Optional fields for backward compatibility. Existing code that reads `BootstrapEvaluation` without claim fields continues to work.

### 6. Enrichment Prompt Enhancement

When the quality gate triggers re-bootstrap, the enrichment prompt (`buildEnrichedBootstrapPrompt()` in bootstrap.ts) should include failed claims:

```
The previous CLAUDE.md contained these factual errors:
- Claimed "uses Prisma" but no prisma found in package.json or .prisma/ directory
- Referenced "src/server.ts" but file does not exist (actual entry: src/app.ts)
Please correct these in the new version.
```

This is a ~10-line addition to `buildEnrichedBootstrapPrompt()`.

## Open Questions

1. **Claim extraction false positives:** If CLAUDE.md mentions "React" in a "we considered React but chose Vue" context, that would be extracted as a tech stack claim. Should we skip claims that appear in negative/comparison contexts? (Probably not for v1 — keep it simple.)
2. **Multi-language repos:** For Python repos, should we check `requirements.txt` / `pyproject.toml` instead of `package.json`? (`detectTechStack()` already handles this, so yes, reuse it.)
3. **Score rebalancing migration:** Existing dogfood runs have scores under the old formula. Dashboard history will show a score discontinuity. Acceptable? (Yes — the scores aren't persisted cross-run, only used within a pipeline execution.)

## Success Criteria

- `analyzeBootstrapQuality()` returns per-claim verification results in `BootstrapEvaluation`
- A CLAUDE.md with hallucinated tech stack scores at least 15 points lower than before
- The quality gate correctly triggers enrichment on a CLAUDE.md with >30% failed claims
- Enrichment prompt includes failed claims, improving second-pass accuracy
- All existing tests pass (score rebalancing doesn't break threshold logic)
- 15+ new tests cover claim extraction, verification, and scoring

## What I noticed about how you think

- The priority pick correctly identified that this has multiplicative impact: fixing bootstrap quality improves every downstream skill. That's systems thinking, not feature thinking.
- The existing `analyzeBootstrapQuality()` was a good first pass — structural checks ship fast. But recognizing that structural != semantic and coming back to add verification is the kind of iterative quality improvement that separates good systems from great ones.
- The bootstrap quality gate (score < 50 triggers enrichment) was designed with exactly this extension in mind — the threshold stays the same, the scoring gets smarter. Clean architecture.
