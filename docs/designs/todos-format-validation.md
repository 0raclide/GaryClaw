## Problem Statement

The bootstrap skill has an idempotency gate: if TODOS.md exists, skip generation (`bootstrap.ts:649`). The evaluate skill checks TODOS.md item count using `## P\d:` regex matching (`evaluate.ts:543`). The prioritize skill parses items with `parseTodoItems()` which requires `## P{N}: Title` format (`prioritize.ts:42`).

When GaryClaw dogfoods an external repo that already has a human-written TODOS.md in a different format (like NihontoWatch, which uses freeform headings like `## Activate Inbound Email Reply Gateway`), three things go wrong silently:

1. Bootstrap sees TODOS.md exists, skips generation. No error.
2. Evaluate counts 0 items (`todosMdItemCount: 0`), scores viability at 0/15. No warning about format mismatch.
3. Prioritize calls `parseTodoItems()`, gets an empty array, and has nothing to prioritize. The pipeline continues with a "no items found" result.

The consequence: 15 quality points silently lost in the evaluate score, and the entire prioritize/implement/qa pipeline runs on zero actionable items. Budget wasted on a no-op pipeline.

This was discovered by examining NihontoWatch's actual TODOS.md during this office-hours session. The file exists, has real content, but zero items match the `## P{N}:` format.

## What Makes This Cool

GaryClaw detects when its own format assumptions are wrong. Instead of silently scoring 0 and moving on, it says "this TODOS.md has 5 items but none match my parser. Format mismatch." Then the bootstrap skill can offer to generate a companion file or reformat.

## Constraints

- Pure TypeScript extension to `evaluate.ts` and `bootstrap.ts`. No new modules.
- Zero Claude calls for validation. Deterministic format checking.
- Must not overwrite existing TODOS.md files. Format conversion is opt-in.
- Must extend existing `BootstrapEvaluation` interface (backward-compatible optional fields).
- Must work on any repo, not just NihontoWatch.

## Premises

1. The `## P{N}: Title` format is required by `parseTodoItems()` and cannot be relaxed without breaking the prioritize skill's scoring dimensions and downstream pipeline.
2. Repos with pre-existing TODOS.md in non-standard format are a real and common case. Most human-written TODO files don't use GaryClaw's format.
3. The bootstrap idempotency gate (skip if file exists) is correct for avoiding destructive overwrites, but should be format-aware: "exists AND parseable" vs "exists but incompatible."
4. The evaluate skill should surface format mismatch as a quality note, not silently score 0.
5. The fix is small (XS effort) but high-leverage: it closes the gap between "validators exist" and "validators produce useful signals on real repos."

## Approaches Considered

### Approach A: Format Validation + Quality Note (Minimal Viable)

**Summary:** Add a `validateTodosFormat()` function to `evaluate.ts` that checks whether TODOS.md items match the `## P{N}:` pattern. When items exist but none match, add a quality note explaining the format mismatch. Extend `BootstrapEvaluation` with a `todosFormatValid` boolean.

**Effort:** XS
**Risk:** Low
**Pros:**
- 5 lines of validation logic, ~8 tests
- Makes the 0/15 viability score explainable instead of silent
- No behavioral changes to bootstrap or prioritize

**Cons:**
- Detection only. Doesn't fix the problem, just surfaces it.
- Pipeline still runs on zero items after detection.

**Reuses:** `analyzeBootstrapQuality()`, existing `## P\d:` regex, `BootstrapEvaluation` interface

### Approach B: Format-Aware Bootstrap Gate (Ideal Architecture)

**Summary:** Extend the bootstrap idempotency gate to distinguish "TODOS.md exists AND is parseable" from "TODOS.md exists but is incompatible." When incompatible, bootstrap generates a `TODOS-garyclaw.md` companion file with properly formatted items derived from codebase analysis, and the prioritize skill reads from that file when the primary TODOS.md has zero parseable items.

**Effort:** S
**Risk:** Med
**Pros:**
- Actually fixes the pipeline: repos with non-standard TODOS.md get actionable items
- Non-destructive: original TODOS.md untouched
- Prioritize fallback is a 3-line change (check TODOS-garyclaw.md if TODOS.md yields 0 items)

**Cons:**
- Two TODOS files is confusing. Which is the source of truth?
- Bootstrap prompt needs adjustment to generate the companion file
- Adds a new naming convention that downstream skills must know about

**Reuses:** `buildBootstrapPrompt()`, `parseTodoItems()`, `analyzeCodebase()`

### Approach C: Format Conversion Prompt (Creative/Lateral)

**Summary:** When evaluate detects format mismatch, inject a "reformat TODOS.md" task into the next pipeline run. The implement skill gets a prompt saying "convert existing TODOS.md items to `## P{N}: Title` format, preserving all content." This uses Claude to do the format conversion, keeping the original content but making it pipeline-compatible.

**Effort:** S
**Risk:** Med
**Pros:**
- One TODOS.md file, properly formatted, with original content preserved
- Claude handles the ambiguity of mapping freeform items to priority tiers
- Most natural UX: "I fixed your TODO format so the pipeline can read it"

**Cons:**
- Modifies the user's file (even if content is preserved, format changes)
- Costs a Claude turn for format conversion
- Priority tier assignment by Claude is subjective

**Reuses:** `buildBootstrapPrompt()` pattern, `parseTodoItems()` for validation

## Recommended Approach

**Approach A first, then Approach B as follow-on.** Approach A is 15 minutes of work and immediately makes the evaluate score meaningful on real repos. It surfaces the problem cleanly. Approach B is the right long-term fix but adds complexity (companion file, fallback logic) that should be designed after seeing how often format mismatch actually occurs across multiple repos.

## Implementation Details

### 1. New function: `validateTodosFormat()` in `evaluate.ts`

**Interface location:** `TodosFormatValidation` goes in `types.ts` alongside `BootstrapEvaluation` (since it's referenced from that interface). The `validateTodosFormat()` function lives in `evaluate.ts`.

```typescript
// In types.ts:
export interface TodosFormatValidation {
  totalHeadings: number;      // actionable ## headings (excludes whole-line ~~completed~~ markup)
  parsableItems: number;      // headings matching ## P{N}: pattern (colon required)
  unparsableItems: string[];  // actionable headings that DON'T match (first 5)
  formatValid: boolean;       // true if parsableItems > 0 OR totalHeadings === 0 (nothing to fix)
}

// In evaluate.ts:
export function validateTodosFormat(todosContent: string): TodosFormatValidation {
  const allHeadings = todosContent.match(/^## .+$/gm) ?? [];
  // "Completed" = whole heading is strikethrough: "## ~~Title~~ — COMPLETE"
  // Partial strikethrough ("## P1: ~~fix thing~~") is NOT completed, still actionable.
  const actionableHeadings = allHeadings.filter(h => !h.match(/^## ~~.+~~/));
  // Match existing evaluate.ts:543 regex exactly: /^## P\d:/ (trailing colon required,
  // no trailing .+ — a bare "## P1:" with no title is technically parseable)
  const parsableHeadings = actionableHeadings.filter(h => h.match(/^## P\d:/));
  const unparsable = actionableHeadings
    .filter(h => !h.match(/^## P\d:/))
    .slice(0, 5);

  return {
    totalHeadings: actionableHeadings.length,
    parsableItems: parsableHeadings.length,
    unparsableItems: unparsable,
    // Valid if at least one matches format, OR file has no actionable items (nothing to fix)
    formatValid: parsableHeadings.length > 0 || actionableHeadings.length === 0,
  };
}
```

### 2. Wire into `analyzeBootstrapQuality()`

Inside the existing `if (result.todosMdExists)` block, after the priority.md score-counting logic (after the closing brace of `if (existsSync(priorityPath))`), where `todosContent` is already in scope from `const todosContent = safeReadText(todosMdPath) ?? ""`. Add:

```typescript
// Validate TODOS.md format compatibility with parseTodoItems()
const formatCheck = validateTodosFormat(todosContent);
result.todosFormatValidation = formatCheck;

if (!formatCheck.formatValid && formatCheck.totalHeadings > 0) {
  result.qualityNotes.push(
    `TODOS.md format mismatch: ${formatCheck.totalHeadings} actionable headings, ` +
    `${formatCheck.parsableItems} parseable by prioritize skill. ` +
    `Examples: ${formatCheck.unparsableItems.slice(0, 2).join(", ")}`
  );
}
```

Note: this code is INSIDE the existing `if (result.todosMdExists)` block, not wrapped in a new one.

### 3. Extend `BootstrapEvaluation` in `types.ts`

Add optional field:

```typescript
todosFormatValidation?: TodosFormatValidation;
```

### 4. Update `formatEvaluationReport()` in `evaluate.ts`

Add a TODOS Format row. **Important:** this row should appear in the main Bootstrap Quality section (near the existing `todosMdItemCount` output), NOT inside the "Bootstrap Semantic Validation" sub-section that is gated behind `semanticAccuracy !== undefined`. Format validation is independent of semantic validation and should always render when TODOS.md exists.

```
| TODOS Format | {parsableItems}/{totalHeadings} parseable | {formatValid ? "PASS" : "MISMATCH"} |
```

### 5. Quality score adjustment

No score rebalancing needed. The existing `viabilityScore` already correctly scores 0 when `todosMdItemCount === 0`. The format validation just explains WHY it's 0 instead of leaving it unexplained.

| `analyzeBootstrapQuality` format valid no note | No format mismatch note when TODOS.md uses correct format |
| `formatEvaluationReport` includes format row | Report markdown includes TODOS Format validation row in Bootstrap Quality section (not semantic section) |

**Total: ~10 tests.** All use tmp directories with synthetic files. No SDK calls.

## Open Questions

1. Should Approach B (companion file) be a separate TODO item? Probably yes, gated behind "format mismatch detected on 2+ repos."
