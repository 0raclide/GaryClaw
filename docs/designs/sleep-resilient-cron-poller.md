## Problem Statement

The cron poller in `src/triggers.ts` checks every 60 seconds: "does the current minute match the cron expression?" If macOS sleeps through a matching minute, the trigger is permanently lost. On 2026-03-27, the self-improvement pipeline (`0 */2 * * *`) missed 3-4 triggers overnight, costing 3-4 improvement cycles that could have shipped features or fixes.

The daemon's core promise is "push code, go to sleep, GaryClaw keeps working." Sleep-resilient cron is the difference between that promise being real and it being marketing.

## What Makes This Cool

After this fix, the daemon truly becomes "start and forget." Close your laptop at midnight, open it at 8am, and the daemon fires immediately, catching up on the cron window it missed. One timestamp + one scan loop. The overnight reliability gap just... closes.

## Constraints

- Must fire at most once per wake event, regardless of how many cron windows were missed during sleep. Firing 4 missed triggers at once wastes API overhead (job dedup would collapse them anyway, but the enqueue calls and log noise are unnecessary).
- Must not change behavior when the system is awake. Same cron windows, same timing, same dedup.
- Must be testable with the existing `CronPollerDeps` injection pattern (injectable `now()`, `setInterval`).
- No external dependencies. No new files. Change stays within `createCronPoller` in `src/triggers.ts`.
- The `lastFiredMinute` dedup guard must still prevent double-fire within the same minute.

## Premises

1. **macOS sleep causes missed triggers.** The 60s `setInterval` doesn't fire during sleep. When the system wakes, `check()` runs with the current time, missing any cron windows that occurred during sleep. Observed on 2026-03-27.
2. **Fire-once-per-wake is the right catch-up strategy.** Firing all missed windows would create duplicate enqueues that dedup collapses. Fire the most recent match only.
3. **The existing `CronPollerDeps` injection is sufficient for testing.** Simulate sleep by jumping `now()` forward past multiple cron windows.
4. **This is backward-compatible.** When awake, the poller still fires at the same times. The only new behavior is catch-up after sleep gaps.

## Approaches Considered

### Approach A: lastCheckedAt Scan (Minimal Viable) -- RECOMMENDED

**Summary:** Add a `lastCheckedAt` timestamp to `createCronPoller`. On each tick, scan every minute from `lastCheckedAt` to `now` against the cron schedule. If any match, fire once (the most recent match). Update `lastCheckedAt = now` after every tick.

**Effort:** XS (human: ~1 day / CC: ~10 min)
**Risk:** Low

**Pros:**
- Smallest diff. ~15-20 lines changed in `createCronPoller`, zero new files.
- Reuses `matchesCronSchedule()` directly for each scanned minute.
- Testable with existing `CronPollerDeps.now()` injection.

**Cons:**
- Scanning minute-by-minute for long sleeps (24h = 1440 iterations) is O(n) per tick. Not a real concern at this scale.
- No persistence across daemon restarts. If the daemon process dies, `lastCheckedAt` resets to "now."

### Approach B: Next-Match Precomputation (Ideal Architecture)

**Summary:** After each fire (or on start), precompute the next matching timestamp from the cron schedule. On each tick, check `now >= nextMatch`. After sleep, the comparison naturally catches the missed window.

**Effort:** S (human: ~2 days / CC: ~15 min)
**Risk:** Low-Med

**Pros:**
- O(1) per tick.
- Cleaner mental model: "when is the next fire?" vs "did I miss anything?"
- Could power a "next scheduled run" display in dashboard.

**Cons:**
- Requires a new `computeNextMatch(schedule, fromDate)` function (~30 lines).
- More code to maintain. The scanning approach in A is simpler and correct.
- DST transitions and month boundaries make next-match computation fiddly.

## Recommended Approach

**Approach A: lastCheckedAt Scan.** It's the smallest change that fully solves the problem. The minute-scanning "inefficiency" is irrelevant (1440 iterations = <1ms). The code reads clearly: "for every minute I missed, did any match the cron schedule? If yes, fire once." No new abstractions needed.

## Implementation Plan

### Step 1: Add `lastCheckedAt` state to `createCronPoller` (~5 lines)

In `src/triggers.ts`, inside `createCronPoller`:

```typescript
let lastCheckedAt: number = 0; // initialized in start()
```

The value is set to 0 here and initialized properly in `start()` via `d.now().getTime()`, ensuring test determinism (the injectable clock is always used, never `Date.now()` directly).

### Step 2: Replace the `check()` function (~15 lines)

Current logic:
```typescript
function check(): void {
  const now = d.now();
  const minuteKey = `...`;
  if (minuteKey === lastFiredMinute) return;
  if (matchesCronSchedule(schedule, now)) {
    lastFiredMinute = minuteKey;
    onTrigger(config.skills, detail);
  }
}
```

New logic:
```typescript
function check(): void {
  const now = d.now();
  const nowMs = now.getTime();

  // Scan every minute from lastCheckedAt to now (inclusive of current minute).
  // Floor lastCheckedAt to minute boundary to include the boundary minute.
  // The lastFiredMinute guard prevents double-fire if a boundary minute was already checked.
  // This scan is O(minutes-slept) per tick — 1440 iterations max for 24h sleep, <1ms.
  const scanStart = lastCheckedAt - (lastCheckedAt % 60_000);
  let latestMatch: Date | null = null;

  for (let t = scanStart; t <= nowMs; t += 60_000) {
    const candidate = new Date(t);
    if (matchesCronSchedule(schedule, candidate)) {
      latestMatch = candidate;
    }
  }

  if (latestMatch) {
    const minuteKey = `${latestMatch.getFullYear()}-${latestMatch.getMonth()}-${latestMatch.getDate()}-${latestMatch.getHours()}-${latestMatch.getMinutes()}`;
    if (minuteKey !== lastFiredMinute) {
      lastFiredMinute = minuteKey;
      const detail = `Cron matched: ${config.expression} at ${latestMatch.toISOString()}`;
      onTrigger(config.skills, detail);
    }
  }

  lastCheckedAt = nowMs;
}
```

Key behaviors:
- **Normal operation (awake):** `scanStart` is ~60s ago, loop runs 1 iteration, matches current minute if applicable. Same behavior as before.
- **After sleep (8h gap):** `scanStart` is 8h ago, loop scans 480 minutes, finds the latest match, fires once.
- **Single-fire cap:** Only the `latestMatch` fires. Even if 4 cron windows were missed, one trigger.
- **Dedup guard:** `lastFiredMinute` still prevents double-fire if the same minute is checked twice within 60s.

### Step 3: Initialize `lastCheckedAt` on `start()` (~2 lines)

```typescript
start() {
  lastCheckedAt = d.now().getTime();
  check();
  timer = d.setInterval(check, 60_000);
},
```

This ensures the first check doesn't scan from epoch. If the poller starts at 8:00 and a cron window was at 7:00, it does NOT catch up (the daemon wasn't running then). Catch-up only applies to windows missed while the poller was running but the system slept.

## Open Questions

1. **Should `lastCheckedAt` be injectable via `CronPollerDeps`?** Probably not needed since `now()` already controls the clock in tests. But if a test needs to simulate "poller started 2 hours ago," it would need to control the initial value. Decision: use `d.now().getTime()` for initialization so it's automatically testable via the existing `now()` injection.

2. **Edge case: what if the system clock jumps backward?** (NTP correction, timezone change.) If `nowMs < lastCheckedAt`, the scan loop doesn't execute (start > end). The poller just waits for the next tick. This is correct behavior, no special handling needed.

3. **Should the catch-up trigger detail say "caught up" vs normal fire?** Useful for debugging but adds complexity. Decision: include the matched time in the detail string. If the matched time is more than 2 minutes before `now`, it's clearly a catch-up. No special flag needed.
