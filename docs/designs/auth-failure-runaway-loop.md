# Design: Auth Failure Runaway Loop Fix

Generated by /office-hours on 2026-03-30
Branch: main
Repo: 0raclide/GaryClaw
Status: APPROVED
Mode: Builder

## Problem Statement

When Claude auth expires, continuous mode creates a tight spin loop burning through 50+ job IDs in seconds. Each job starts, fails auth verification in ~2 seconds at $0 cost, gets marked `failed`, and immediately a new job is triggered because the dedup check (line 375 of job-runner.ts) only blocks `queued/running/rate_limited` jobs, not `failed` ones.

The result: polluted job history, lost queued work (legitimate jobs get buried), and wasted daemon cycles. Discovered 2026-03-30 during NihontoWatch operation.

**Root cause:** The rate limit hold mechanism in `job-runner.ts` processNext error handler only triggers for `classification.category === "infra-issue" && isRateLimitError(nextJob.error)`. Auth failures classify as `auth-issue` and bypass this check entirely.

## What Makes This Cool

It's not cool, it's just correct. A daemon that detects it can't authenticate and backs off for 30 minutes instead of burning through 50 job IDs is basic reliability. What's satisfying is that every piece of infrastructure already exists... this is pure wiring.

## Constraints

- XS effort: under 20 lines of production code changed
- All infrastructure exists: `classifyError()` returns `auth-issue`, `rateLimitResetAt` holds work, `RATE_LIMIT_FALLBACK_MS` = 30 min
- Must not break existing rate limit behavior
- Must be testable with synthetic data (no SDK calls in tests)

## Premises

1. Auth failures need the same hold mechanism as rate limits, not a new system.
2. A 30-minute hold (`RATE_LIMIT_FALLBACK_MS`) is the right default. Re-authenticating requires human intervention, so 30 minutes is a floor.
3. A minimum-cost guard on continuous re-enqueue prevents $0 jobs from creating infinite chains for any failure category, not just auth.

## Approaches Considered

### Approach A: Minimal Wire (extend existing condition)

Add `|| classification.category === "auth-issue"` to the rate limit hold check on line 957.

- **Effort:** XS
- **Risk:** Low
- **Pros:** 1-line change, uses proven infrastructure
- **Cons:** Doesn't protect against future unknown failure categories that also spin

### Approach B: Category-based hold + minimum cost guard (CHOSEN)

Two changes:

1. **Extend hold condition** to cover `auth-issue` alongside rate limits
2. **Add minimum cost guard** on continuous re-enqueue paths: skip re-enqueue if the just-completed job cost < $0.01

This provides defense in depth. The hold catches known spin categories. The cost guard catches unknown future categories where a job fails so fast it costs nothing.

- **Effort:** XS
- **Risk:** Low
- **Pros:** Defense in depth, prevents entire class of future spin bugs
- **Cons:** ~15 lines vs 1 line (still XS)

## Recommended Approach

**Approach B: Category-based hold + minimum cost guard.**

### Change 1: Extend hold condition (job-runner.ts processNext error handler)

Find the existing rate limit detection block (search for `isRateLimitError`):

Current:
```typescript
if (classification.category === "infra-issue" && isRateLimitError(nextJob.error)) {
```

New:
```typescript
const shouldHold =
  (classification.category === "infra-issue" && isRateLimitError(nextJob.error)) ||
  classification.category === "auth-issue";

if (shouldHold) {
```

This makes auth failures set `rateLimitResetAt` and mark the job `rate_limited` instead of `failed`. The existing hold infrastructure handles everything else: the 30-minute wait, global budget propagation, re-queue on expiry, notification, dashboard display.

### Change 2: Minimum cost guard on continuous re-enqueue (proactive hardening)

This is proactive hardening beyond the auth fix. It prevents $0 spin loops from any failure category, not just auth. Find the continuous re-enqueue block (search for `"auto re-enqueue after successful pipeline"`):

Current:
```typescript
if (nextJob.status === "complete" && nextJob.skills.length > 1 && nextJob.skills.includes("prioritize")) {
```

New:
```typescript
const MIN_COST_FOR_REENQUEUE = 0.01;
if (
  nextJob.status === "complete" &&
  nextJob.skills.length > 1 &&
  nextJob.skills.includes("prioritize") &&
  nextJob.costUsd >= MIN_COST_FOR_REENQUEUE
) {
```

Export `MIN_COST_FOR_REENQUEUE` for testability.

A job that costs $0 didn't do any real work. If a pipeline "completes" at $0 (e.g., all TODOs skipped because they were previously completed), re-enqueueing just creates another $0 cycle. The cost guard breaks this loop.

**Note:** The skip-completed re-enqueue paths on lines 754 and 770 should NOT get the cost guard. Those paths fire when TODO state says "already done" and exist specifically to advance to the next TODO. They'll naturally stop when `enqueue()` returns null (budget/dedup/exhausted backlog).

### Change 3: Auth-specific notification

Reuse the existing `notifyRateLimitHold` callback but pass an auth-specific message. The hold block already calls `d.notifyRateLimitHold?.(holdUntil, resolvedInstanceName, jobConfig)`. Add a `reason` parameter (default `"rate_limit"`) so the notification can distinguish auth holds from rate limit holds and display "Auth expired, holding for 30 min. Run `claude` to re-authenticate." instead of the generic rate limit message.

If adding the `reason` parameter touches too many call sites, just reuse the existing notification as-is. The hold behavior is what matters; the notification text is polish.

### Test Plan

1. **Auth hold trigger:** Verify that `classifyError` returning `auth-issue` triggers the hold mechanism (sets `rateLimitResetAt`, marks job `rate_limited`)
2. **Rate limit unchanged:** Existing rate limit tests still pass (no regression)
3. **Cost guard blocks $0 re-enqueue:** Continuous re-enqueue returns null when `costUsd < MIN_COST_FOR_REENQUEUE`
4. **Cost guard allows normal re-enqueue:** Jobs with `costUsd >= 0.01` still re-enqueue
5. **Skip-completed paths unaffected:** The skip-completed re-enqueue paths still fire regardless of cost
6. **Global budget propagation:** Auth hold propagates via `setGlobalRateLimitHold` for cross-instance coordination
7. **Notification:** Auth hold sends notification (either auth-specific or generic rate limit)

Estimated: 8-12 tests, all synthetic.
