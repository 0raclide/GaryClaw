## Problem Statement

GaryClaw's self-improvement loop has a non-deterministic gap. The evaluate skill has 8 tested TypeScript analysis functions that produce structured improvement candidates:

- `analyzeBootstrapQuality()` (reads CLAUDE.md, checks sections, framework coverage)
- `analyzeOraclePerformance()` (reads decisions.jsonl, clusters topics)
- `analyzePipelineHealth()` (reads pipeline.json, computes relays/growth)
- `extractObviousImprovements()` (threshold-based candidate extraction)
- `parseClaudeImprovements()` (parses `<improvements>` JSON from Claude output)
- `deduplicateImprovements()` (Levenshtein-based title dedup)
- `writeEvaluationReport()` (writes evaluation-report.json, .md, and improvement-candidates.md)
- `formatImprovementCandidates()` (formats candidates as TODOS.md entries)

These functions are tested (72 tests in `test/evaluate.test.ts` plus regression files) but **only called if Claude happens to write the files during its SDK segment**. The `buildEvaluatePrompt()` function runs the analysis to inject data into Claude's prompt, then *asks Claude* to write improvement-candidates.md. But Claude might not do it, or might format it wrong.

The `cli.ts` post-pipeline hook (`appendEvaluateCandidates()`) reads `improvement-candidates.md` from disk and appends to GaryClaw's TODOS.md. If that file doesn't exist, no improvements flow back. The entire autonomous self-improvement cycle depends on Claude's compliance with a prompt instruction.

## What Makes This Cool

After this wiring, running `garyclaw run bootstrap prioritize implement qa evaluate --autonomous --project-dir /path/to/target` **always** produces improvement candidates. Not "probably." Not "if Claude follows instructions." Always. The deterministic TS functions are the source of truth. Claude's additional insights (from the `<improvements>` block) are a bonus that gets merged in when available. The self-improvement loop becomes a mechanical guarantee.

## Constraints

- Must not change the evaluate skill's public API or break the 72+ existing tests
- Must work with both pipeline dispatch (`pipeline.ts`) and standalone (`garyclaw run evaluate`)
- Must handle the case where Claude's output contains zero, partial, or malformed `<improvements>` blocks
- Must not add significant token cost (the TS analysis functions are pure file reads, no SDK calls)
- The fix lives in pipeline.ts (post-skill hook) and optionally in orchestrator.ts (assistant text collection), not in evaluate.ts itself

## Premises

1. **The gap is pure wiring, not missing functionality.** All 8 analysis functions exist, are exported, and have 72+ tests. Zero new analysis logic needed.
2. **The deterministic path should run after the evaluate SDK segment completes.** Running before would miss Claude's additional improvement candidates. Running after means we can attempt to parse Claude's output, merge with obvious improvements, and write the canonical file.
3. **The right hook point is pipeline.ts.** Orchestrator handles single-skill lifecycle. Pipeline handles cross-skill concerns and has access to the target project directory. The CLI hook reads from the target project's `.garyclaw/` dir, which pipeline.ts controls.
4. **Claude's assistant text must be collected during the skill run to parse `<improvements>` blocks.** The checkpoint does not store raw assistant text. To get Claude's extra improvements, pipeline.ts needs to wrap callbacks and collect text during the evaluate segment.
5. **The CLI hook (`appendEvaluateCandidates`) doesn't need changes.** It correctly reads `improvement-candidates.md` and appends to TODOS.md. The fix ensures that file always exists.
6. **Daemon pipelines use `pipeline.ts`.** The daemon's job runner (`src/job-runner.ts`) calls `runPipeline()` from `pipeline.ts` for multi-skill jobs. The post-skill hook in pipeline.ts covers both CLI and daemon code paths.

## Approaches Considered

### Approach A: Pipeline Post-Skill Hook with Text Collection (Recommended)

**Summary:** Add a callback wrapper in `pipeline.ts` that collects assistant text during the evaluate skill's SDK segments. After the skill completes, run the deterministic analysis, parse Claude's `<improvements>` from collected text, merge with obvious improvements, dedup, and call `writeEvaluationReport()`.

**Effort:** S
**Risk:** Low
**Pros:**
- Captures Claude's extra improvements AND guarantees the deterministic baseline
- Single hook point in pipeline.ts, clean separation of concerns
- Follows the existing pattern (orchestrator runs `runReflection()` post-skill, pipeline.ts runs evaluate post-processing)
- Collected text is scoped to evaluate skill only, no memory bloat

**Cons:**
- Callback wrapper adds ~20 lines to pipeline.ts
- Collected text could be large if the evaluate segment is verbose (mitigated: only collect during evaluate, discard after)

**Reuses:** All 8 evaluate.ts functions, pipeline.ts skill dispatch pattern, orchestrator callback interface

### Approach B: Deterministic-Only (No Claude Text Parsing)

**Summary:** After the evaluate skill completes in `pipeline.ts`, run `analyzeBootstrapQuality()`, `analyzeOraclePerformance()`, `analyzePipelineHealth()`, `extractObviousImprovements()`, and `writeEvaluationReport()`. Ignore Claude's `<improvements>` block entirely.

**Effort:** XS
**Risk:** Low
**Pros:**
- Simplest possible change (~15 lines in pipeline.ts)
- No callback wrapping needed
- Guaranteed deterministic output

**Cons:**
- Loses Claude's subtle improvements that heuristics can't detect
- Claude's SDK segment still asks for `<improvements>` but they're never read... confusing prompt/behavior mismatch
- Misses "skill" and "relay" category improvements that require Claude's judgment to surface

**Reuses:** 6 of 8 evaluate.ts functions (skips `parseClaudeImprovements`, `deduplicateImprovements`)

### Approach C: Orchestrator-Level Hook (Evaluate-Aware Post-Completion)

**Summary:** Add evaluate-specific post-completion logic in `orchestrator.ts`, alongside the existing `runReflection()` hook. When `config.skillName === "evaluate"`, collect assistant text during the segment loop and run the deterministic path after success.

**Effort:** S
**Risk:** Med
**Pros:**
- Works for standalone `garyclaw run evaluate` without pipeline
- Keeps all post-skill logic in orchestrator

**Cons:**
- Orchestrator shouldn't know about evaluate-specific business logic (separation of concerns violation)
- Orchestrator already has 843 lines, adding skill-specific hooks makes it harder to maintain
- Standalone evaluate is a rare case (evaluate almost always runs in a pipeline)

**Reuses:** All 8 evaluate.ts functions, orchestrator post-completion pattern

## Recommended Approach

**Approach A: Pipeline Post-Skill Hook with Text Collection.** It captures everything (deterministic + Claude extras), follows existing patterns, and keeps the concern in the right module. The XS savings of Approach B aren't worth losing Claude's improvements, and Approach C violates separation of concerns.

## Open Questions

1. **Should collected text have a size cap?** The evaluate prompt asks Claude to write reports + improvements. Assistant text could be 10K+ tokens. A reasonable cap (e.g., last 50K chars) prevents memory issues while ensuring the `<improvements>` block is captured.
2. **Should the deterministic path also run for standalone evaluate?** Approach A only covers pipeline dispatch. For standalone (`garyclaw run evaluate`), the current behavior (Claude writes files) would persist. This is acceptable for now since standalone evaluate is rare, but could be addressed later by adding a thin wrapper in the evaluate skill prompt that explicitly calls `writeEvaluationReport()` instructions.

## Success Criteria

- `improvement-candidates.md` exists in `.garyclaw/` after every evaluate skill completion in a pipeline
- The file contains at minimum the deterministic improvements from `extractObviousImprovements()`
- When Claude produces an `<improvements>` block, those candidates are merged and deduplicated
- The CLI hook `appendEvaluateCandidates()` successfully appends to TODOS.md on every pipeline run that includes evaluate
- All existing evaluate tests continue to pass (72+ tests)
- Standalone evaluate (no pipeline) remains prompt-dependent; see Open Question #2
- New tests cover: callback wrapper text collection, post-skill hook execution, merge with Claude improvements, empty Claude output handling

## Distribution Plan

This is an internal module change to an existing npm project. No distribution changes needed. Existing `npx tsx src/cli.ts` entry point covers it.

## Next Steps

### Step 1: Add assistant text collector to pipeline.ts evaluate dispatch

In `executePipelineFrom()`, when `skillName === "evaluate"`, wrap the callbacks object to collect `assistant_text` events into a string buffer. Pass the original callback through for all events (so CLI display still works).

```typescript
// In pipeline.ts, inside the evaluate skill branch
const collectedText: string[] = [];
const wrappedCallbacks: OrchestratorCallbacks = {
  ...callbacks,
  onEvent: (event) => {
    if (event.type === "assistant_text") {
      collectedText.push(event.text);
    }
    callbacks.onEvent(event);
  },
  onAskUser: callbacks.onAskUser,
};
// Pass wrappedCallbacks (not callbacks) to runSkillWithPrompt:
await runSkillWithPrompt(skillConfig, wrappedCallbacks, evalPrompt);
```

### Step 2: Add post-completion deterministic analysis

After the evaluate skill's `runSkillWithPrompt()` returns (before `buildSkillReport()`), run:

```typescript
import {
  analyzeBootstrapQuality,
  analyzeOraclePerformance,
  analyzePipelineHealth,
  extractObviousImprovements,
  parseClaudeImprovements,
  deduplicateImprovements,
  writeEvaluationReport,
} from "./evaluate.js";

// After evaluate skill completes:
// Note: these functions also run inside buildEvaluatePrompt() to prime Claude's prompt.
// The duplication is intentional and cheap (pure filesystem reads, sub-millisecond).
const bootstrap = analyzeBootstrapQuality(config.projectDir);
const oracle = analyzeOraclePerformance(config.projectDir);
const pipeline = analyzePipelineHealth(config.projectDir);
const report = {
  targetRepo: config.projectDir,
  timestamp: new Date().toISOString(),
  bootstrap,
  oracle,
  pipeline,
  improvements: [],
};

const obvious = extractObviousImprovements(report);
const claudeImprovements = parseClaudeImprovements(collectedText.join(""));
report.improvements = deduplicateImprovements(obvious, claudeImprovements);

// Writes evaluation-report.json, evaluation-report.md, AND improvement-candidates.md
writeEvaluationReport(config.projectDir, report);
```

### Step 3: Add error boundary

Wrap the post-completion hook in try/catch. If the deterministic analysis fails, emit an error event but don't crash the pipeline. The evaluate skill already completed successfully at this point.

### Step 4: Add tests

- Test callback wrapper collects assistant text correctly
- Test deterministic path runs and writes files when Claude produces no `<improvements>`
- Test merge path when Claude produces valid `<improvements>` block
- Test error boundary when analysis functions throw
- Test that existing pipeline tests still pass (evaluate dispatch, events, full pipeline)

Target: 8-12 new tests in `test/pipeline-evaluate-deterministic.test.ts`

## What I noticed about how you think

- The priority pick nailed the diagnosis: "All analysis functions already exist and are tested; this is pure wiring work." That's exactly right. The 72 tests aren't decoration... they're a complete API surface waiting for a caller.
- The scoring breakdown weighted "autonomous run quality" at 3x, which is correct. A self-improvement loop that works 80% of the time is worth approximately zero percent of a loop that works 100% of the time. Non-determinism in feedback loops compounds into drift.
- Naming this "pure wiring work" kept the scope honest. This is S effort because there's no new algorithm to design, no new data model to invent. It's connecting tested outputs to tested inputs across a 15-line gap in pipeline.ts.
