# Design: garyclaw doctor — Self-Diagnostic Command

Generated by /office-hours on 2026-03-26

Repo: 0raclide/GaryClaw

## Problem Statement

When GaryClaw's daemon misbehaves — won't start, jobs fail silently, worktrees pile up, oracle memory corrupts — there's no diagnostic tool. Users read raw logs and manually inspect state files across 6 subsystems (PID files, oracle memory, worktrees, reflection locks, global budget, auth). This takes minutes per subsystem and requires knowing where each file lives. A 5-second `garyclaw doctor` command that checks all common failure modes and optionally self-heals would make overnight runs debuggable by anyone.

Additionally, the PID liveness check pattern (`readPidFile` + `isPidAlive`) is duplicated across daemon.ts, daemon-registry.ts, and will be needed by doctor — extracting it to a shared `pid-utils.ts` module unblocks cleaner code across all three consumers and adds process-name verification to prevent PID reuse false positives.

## What Makes This Cool

One command. Five seconds. Every failure mode diagnosed. Fixable ones already fixed. Think `brew doctor` for GaryClaw — color-coded PASS/WARN/FAIL per subsystem, with a `--fix` flag that auto-heals stale PIDs, stuck locks, and orphaned worktrees. Plus `--json` for programmatic consumption by the dogfood dashboard.

## Constraints

- Non-auth checks must complete in ≤ 5 seconds. Auth check runs in parallel with a 10s timeout and does not block the other checks.
- No side effects in default mode (diagnose-only). `--fix` flag required for self-healing.
- Must work with zero daemon instances running (checks state files on disk)
- Must not require `ANTHROPIC_API_KEY` (uses Claude Max login like everything else)
- Shared PID utility must be backward-compatible with existing daemon.ts/daemon-registry.ts callers
- Auth check: WARN on timeout, FAIL on actual error (per eng review)

## Premises

1. The 6 subsystems (PIDs, oracle memory, worktrees, reflection locks, budget, auth) cover all known failure modes
2. A shared PID utility with process-name verification is the right extraction
3. `--fix` flag for opt-in self-healing is the right UX (not always-fix, not never-fix)
4. Auth check needs a 10-second timeout with WARN on timeout, FAIL on error
5. No new dependencies needed — pure Node.js + existing GaryClaw utilities
6. `--json` output enables dogfood dashboard integration without extra work

## Approaches Considered

### Approach A: Minimal Viable — Single Module

One `src/doctor.ts` with 6 check functions, `runDoctor()` orchestrator, CLI command. PID utility exported from doctor.ts.

- **Effort:** S
- **Completeness:** 7/10
- **Pros:** Ships fastest. Single file.
- **Cons:** PID utility stuck in doctor.ts — awkward import path for daemon/registry. No `--json`. No `--fix`.

### Approach B: Ideal Architecture — Shared PID Utility + Doctor + JSON/Fix

Extract PID utilities to `src/pid-utils.ts` (shared). Doctor module with typed check results. CLI `garyclaw doctor [--fix] [--json]`. Refactor daemon.ts and daemon-registry.ts imports. Process-name verification on PID checks.

- **Effort:** S
- **Completeness:** 10/10
- **Pros:** Clean architecture. Shared PID utility unblocks 3 modules. `--json` enables dashboard. `--fix` enables self-healing. Process-name check prevents false positives.
- **Cons:** More files to create/modify. Requires updating daemon.ts/daemon-registry.ts imports.

### Approach C: Doctor-as-Middleware

Same as B, plus wire checks into daemon job runner as pre-flight before each job.

- **Effort:** M
- **Completeness:** 10/10
- **Pros:** Prevents failures before they happen.
- **Cons:** Adds latency. False positives could block jobs. Scope creep.

## Recommended Approach

**Approach B: Ideal Architecture.** Delivers the complete solution without middleware scope creep. Daemon-startup integration (Approach C) can be added later as a 1-line call.

### New Files

#### `src/pid-utils.ts` — Shared PID Liveness Utility

```typescript
export interface PidCheckResult {
  alive: boolean;
  pid: number;
  processName?: string;       // actual process name (if alive)
  expectedName?: string;      // expected process name (e.g., "node")
  nameMatch: boolean;         // true if processName matches expectedName
  stale: boolean;             // true if PID file exists but process is dead
}

/**
 * Read PID from a PID file.
 * Returns null if file doesn't exist or is unparseable.
 */
export function readPidFile(pidPath: string): number | null;

/**
 * Check if a PID is alive AND optionally verify process name.
 * Process-name check prevents PID reuse false positives —
 * if PID 12345 was our daemon but now it's a completely different process,
 * we detect that.
 */
export function isPidAlive(pid: number, expectedProcessName?: string): PidCheckResult;

/**
 * Write PID to file atomically (write to .tmp, rename).
 */
export function writePidFile(pidPath: string, pid: number): void;

/**
 * Remove a PID file. Best-effort, never throws.
 */
export function removePidFile(pidPath: string): void;

/**
 * Get process name for a PID.
 * Uses `ps -p <pid> -o comm=` on macOS/Linux.
 * Returns undefined if process not found or ps fails.
 */
export function getProcessName(pid: number): string | undefined;
```

**Process-name verification:** When daemon.ts writes a PID file, the process name is always `node` (or `tsx` during dev). `isPidAlive` checks that the PID's actual process name matches the expected name. If PID 12345 exists but is `Safari` instead of `node`, it's a stale PID from reuse — not our daemon.

**Implementation:** Use `process.kill(pid, 0)` for liveness (existing pattern), add `child_process.execFileSync('ps', ['-p', String(pid), '-o', 'comm='])` for process-name check. Wrap in try/catch — if ps fails, return `nameMatch: true` (optimistic fallback, don't block on ps failure).

#### `src/doctor.ts` — Diagnostic Engine

```typescript
export type CheckStatus = "PASS" | "WARN" | "FAIL" | "INFO";

export interface CheckResult {
  name: string;               // e.g., "stale-pids"
  status: CheckStatus;
  message: string;            // human-readable description
  details?: string[];         // additional detail lines
  fixable: boolean;           // can --fix resolve this?
  fixed?: boolean;            // was it fixed? (only set when --fix is true)
}

export interface DoctorReport {
  checks: CheckResult[];
  timestamp: string;
  durationMs: number;
  summary: {
    pass: number;
    warn: number;
    fail: number;
  };
}

export interface DoctorOptions {
  projectDir: string;
  fix: boolean;               // --fix flag
  skipAuth: boolean;          // --skip-auth for faster runs
  timeoutMs?: number;         // auth check timeout (default: 10000)
  dailyCostLimitUsd?: number; // budget limit override (default: read from daemon config, fallback: 50)
  maxJobsPerDay?: number;     // job limit override (default: read from daemon config, fallback: 20)
}

export async function runDoctor(options: DoctorOptions): Promise<DoctorReport>;
```

### The 6 Checks

Each check is a pure async function: `(options: DoctorOptions) => Promise<CheckResult>`.

#### Check 1: Stale PID Files

**What:** Scan all daemon instance directories for PID files. For each, verify the process is alive AND is a `node` process (process-name verification).

**Logic:**
```
for each instance in .garyclaw/daemons/*/daemon.pid:
  pid = readPidFile(path)
  if pid is null → WARN "unreadable PID file" (fixable: delete file)
  result = isPidAlive(pid, "node")
  if result.stale → WARN "stale PID {pid} for instance {name}" (fixable: delete PID + socket)
  if result.alive && !result.nameMatch → WARN "PID {pid} is {processName}, not node — likely reused" (fixable: delete PID + socket)
  if result.alive && result.nameMatch → PASS
if no PID files found → PASS "no daemon instances running"
```

**Fix action:** Delete stale PID file + associated socket file. Same as existing `cleanupDaemonFiles()` in daemon.ts.

#### Check 2: Oracle Memory Integrity

**What:** Verify oracle memory files are readable, well-formed, and not corrupted. Check both global (`~/.garyclaw/oracle-memory/`) and per-project (`.garyclaw/oracle-memory/`).

**Logic:**
```
for each layer in [global, project]:
  for each file in [taste.md, domain-expertise.md, decision-outcomes.md, metrics.json]:
    if file missing → INFO (not an error — gets created on first use)
    if file exists but empty → WARN "empty {file}" (fixable: write template)
    if metrics.json → try JSON.parse
      if parse fails → WARN "corrupt metrics.json" (fixable: rename to .bak, write fresh)
      if circuit breaker tripped → WARN "circuit breaker tripped — oracle memory disabled (accuracy: {X}%)"
    if .md file → check for injection patterns (sanitizeMemoryContent finds any)
      if injection found → WARN "potential injection in {file}" (not auto-fixable, just warn)
all readable and valid → PASS
```

**Fix action:** Rename corrupt files to `.bak`, write fresh templates (same pattern as `safe-json.ts` corruption recovery).

#### Check 3: Orphaned Worktrees

**What:** Compare git worktree list against running daemon instances. A worktree with no corresponding running daemon is "orphaned" (may be intentional if daemon was stopped but branch kept for manual merge).

**Logic:**
```
worktrees = listWorktrees(projectDir)
for each worktree:
  instanceName = extract from branch name (garyclaw/{name})
  pidPath = .garyclaw/daemons/{instanceName}/daemon.pid
  if no PID file → WARN "orphaned worktree for {instanceName} (branch: garyclaw/{name})"
    check if branch has unmerged commits:
      if yes → WARN "orphaned worktree with unmerged commits" (fixable: no — needs manual merge)
      if no → WARN "orphaned worktree, safe to remove" (fixable: removeWorktree)
  if PID file exists + alive → PASS
if no worktrees → PASS
```

**Fix action:** Remove orphaned worktrees with no unmerged commits. Leave worktrees with unmerged commits (WARN only — user must merge manually or use `--cleanup`).

#### Check 4: Stuck Reflection Locks

**What:** Check if a reflection lock exists and whether the holding process is still alive.

**Logic:**
```
for each oracle-memory dir in [global, project]:
  lockDir = {dir}/.reflection-lock
  if lockDir doesn't exist → PASS
  pidFile = lockDir/pid
  if pidFile missing → WARN "lock dir exists but no PID" (fixable: rmdir)
  pid = read pidFile
  if isPidAlive(pid, "node").alive → PASS "lock held by active process {pid}"
  else → WARN "stuck reflection lock (holder PID {pid} is dead)" (fixable: rmdir -r)
```

**Fix action:** Remove the lock directory (same as existing stale lock recovery in reflection-lock.ts, which uses a 30s timeout).

#### Check 5: Global Budget Status

**What:** Read the global budget file and check if limits are approaching or exhausted.

**Logic:**
```
budget = readGlobalBudget(projectDir)
if budget file corrupt → WARN "corrupt global budget" (fixable: write fresh)
if budget.date !== today → PASS "budget resets today"
remaining = dailyCostLimitUsd - budget.totalUsd
if remaining <= 0 → WARN "daily budget exhausted (${budget.totalUsd} spent)"
if remaining < 20% of limit → WARN "daily budget low (${remaining} remaining)"
jobsRemaining = maxJobsPerDay - budget.jobCount
if jobsRemaining <= 0 → WARN "max jobs reached ({budget.jobCount}/{maxJobsPerDay})"
else → PASS "budget healthy (${remaining} remaining, {jobsRemaining} jobs left)"
```

**Fix action:** Budget exhaustion is not fixable — it's a limit, not a bug. Report only. Corrupt budget file IS fixable (reset to fresh).

**Note:** Budget limits (`dailyCostLimitUsd`, `maxJobsPerDay`) must come from somewhere. Doctor reads the default daemon config or accepts `--daily-limit` and `--max-jobs` CLI flags. If no config exists, use sensible defaults (e.g., $50/day, 20 jobs/day) and WARN that no config was found.

#### Check 6: Auth Verification

**What:** Run a 1-turn SDK query to verify Claude Max login is active. This is the slowest check (~2-5s), so it runs last.

**Logic:**
```
try:
  result = await verifyAuth(buildSdkEnv()) with 10s timeout
  if result.sessionId → PASS "auth OK (session: {sessionId})"
catch timeout:
  → WARN "auth check timed out (10s) — may be network issue"
catch error:
  → FAIL "auth failed: {error.message}"
```

**Fix action:** Not auto-fixable. Tell the user to run `claude` to re-authenticate.

**`--skip-auth` flag:** Skips this check entirely for faster doctor runs (useful when iterating on fixes for other checks).

### CLI Interface

```

Options:
  --fix          Auto-fix fixable issues (stale PIDs, stuck locks, corrupt files)
  --json         Output as JSON (for programmatic consumption)

  --project-dir  Project directory (default: cwd)

Output (default):
  🔍 GaryClaw Doctor — running 6 checks...
```

### Refactoring Scope

**daemon.ts changes:**
- Remove private `readPidFile()`, `isPidAlive()`, `writePidFile()` functions
- Import from `src/pid-utils.ts` instead
- Update `isPidAlive` calls to pass `"node"` as expected process name

**daemon-registry.ts changes:**
- Import `readPidFile`, `isPidAlive` from `src/pid-utils.ts`
- Replace any inline PID checking logic

**No breaking changes:** The public API of daemon.ts and daemon-registry.ts doesn't change — only internal imports shift.

## Open Questions

1. **Default budget limits when no daemon config exists** — should doctor assume $50/day and 20 jobs/day, or skip the budget check entirely? Leaning toward sensible defaults + WARN.
2. **Should `--fix` require confirmation for destructive actions?** For stale PID cleanup and lock removal, probably not (they're clearly stale). For orphaned worktree removal, maybe yes — but `--fix` implies consent. Resolution: `--fix` fixes everything fixable without confirmation. Users who want granular control use default mode (diagnose) then fix manually.

## Success Criteria

- Non-auth checks complete in ≤ 5 seconds; total wall-clock time ≤ 10 seconds (auth timeout bound)
- Detects stale PIDs with process-name verification (no false positives from PID reuse)
- `--fix` resolves stale PIDs, stuck locks, corrupt oracle memory files, and empty worktrees
- `--json` output is valid JSON consumable by dogfood dashboard
- Shared PID utility used by daemon.ts, daemon-registry.ts, and doctor.ts (DRY)
- All existing tests pass after daemon.ts/daemon-registry.ts refactor
- New tests: ~45 tests across pid-utils.test.ts and doctor.test.ts
