Project Type Awareness is 98% implemented. The detection module (`src/project-type.ts`, 285 lines, 34 tests) works. All 7 integration points from the original design are wired:

Three cleanup items remain:

### 1. Duplicate injection bug in `evaluate.ts`

`buildEvaluatePrompt()` injects `## Project Type` twice: once at line 1700 (correct, before Instructions) and again at line 1737 (duplicate, after Instructions). The second block emits the same section a second time into every evaluate prompt. This wastes ~50 tokens per evaluation and may confuse the LLM with duplicated context.

**Fix:** Remove the duplicate block at lines 1737-1748.

### 2. Missing CLAUDE.md entries

The module map has no entry for `src/project-type.ts`. The test table has no entry for `test/project-type.test.ts`. This makes the module invisible to the daemon's self-improvement loop (prioritize reads CLAUDE.md to understand the codebase).

**Fix:** Add both entries.

### 3. DRY refactor: extract `buildProjectTypeSection()` helper (optional)

Four files copy the same 8-line try/catch/format block: `implement.ts`, `prioritize.ts`, `evaluate.ts` (twice!), and the prompt builders all follow the same pattern. (Note: `orchestrator.ts` uses a different format for the skill prompt prefix, so it is NOT a candidate for this helper. The orchestrator injects "Project type: ..." as a one-liner prefix, while prompt builders inject a `## Project Type` markdown section. Different formats, different call sites.) A shared `buildProjectTypeSection(projectDir)` helper in `project-type.ts` would reduce the prompt builder pattern to one function called in three places. The duplicate bug in evaluate.ts would have been impossible with a shared helper.

**Fix:** Add the helper, refactor prompt builder callers. Mark as optional because the copy-paste approach works correctly once the duplicate is removed.

## Premises

1. **The duplicate injection in evaluate.ts is a bug, not intentional.** The same section injected twice in the same prompt wastes tokens and adds no information. The first injection (before Instructions, line 1700) is correctly placed. The second (after Instructions, line 1737) adds nothing.

2. **CLAUDE.md entries are required for discoverability.** The daemon's prioritize and implement skills read CLAUDE.md to understand what modules exist and what tests cover them.

## Recommended Approach

Remove the duplicate, add CLAUDE.md entries. Optionally extract the shared helper for DRY.

### 3. (Optional) Extract shared helper

Add to `src/project-type.ts`:
```typescript
export function buildProjectTypeSection(projectDir: string): string {
  try {
    const pt = ensureProjectType(projectDir);
    if (pt.type === "unknown") return "";
    return `## Project Type\n\n${formatProjectContext(pt)}\n`;
  } catch {
    return "";
  }
}
```

Refactor the try/catch/ensureProjectType blocks in `implement.ts`, `prioritize.ts`, and `evaluate.ts` (the first, correct block) to call `buildProjectTypeSection()` instead. The orchestrator's prefix at lines 240-248 uses a different format (one-liner, not markdown section) and is NOT refactored.
