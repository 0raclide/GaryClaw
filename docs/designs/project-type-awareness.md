# Design: Project Type Awareness — Skill Routing by Project Nature

Generated by /office-hours on 2026-03-30

Status: DRAFT
Mode: Builder

## Problem Statement

20+ of the last 50 Oracle decisions answer the identical question: "GaryClaw is a CLI tool with no web UI, what should QA do?" Each costs ~$0.05 and 2-5s latency. The root cause: `orchestrator.ts` line 276-300 builds the `createAskHandler` call without `projectContext`, even though the Oracle interface (`oracle.ts:43`) already has the field. Every Oracle decision operates with zero project type awareness.

This is not just an Oracle problem. Skills themselves waste their first 1-3 turns probing for project type. The QA skill asks "is there a web UI?" The implement skill asks "what framework?" The bootstrap skill already runs `detectTechStack()` but doesn't persist the result for other skills.

With daemon costs trending up 4%, this is the highest-leverage single optimization available: a one-time classification eliminates an entire category of redundant decisions and wasted turns.

## What Makes This Cool

Every skill instantly knows what kind of project it's working on. QA on a CLI tool runs the test suite without asking. QA on a Next.js app opens the browser. Implement on a Go API knows to write handler tests. The daemon stops having the same conversation with itself 20 times and just does the right thing.

The detection is deterministic (no LLM call needed), the storage is a single JSON file, and the plumbing already exists. This is one of those rare changes where the fix surface is tiny but the impact ripples through every skill run.

## Constraints

- Detection must work without an LLM call (pure TS analysis of CLAUDE.md + package.json + file patterns)
- Must not break existing behavior when `.garyclaw/project-type.json` doesn't exist (fail-open)
- Must not add latency to job startup (detection takes <50ms, reading files already on disk)
- Must not change the Oracle's decision logic or prompt structure (only fills existing `projectContext` field)
- Must be testable with synthetic data (no SDK calls), matching existing test strategy
- `projectContext` string in Oracle prompt is capped at 500 chars (existing truncation in `oracle.ts:165`)

## Premises

1. **The repeated decisions are caused by missing project context, not skill prompt design.** The Oracle's `projectContext` field exists but is never set. Filling it eliminates the repeated "what kind of project is this?" questions.

2. **A one-time detection stored in `.garyclaw/project-type.json` is sufficient.** Projects don't change type between runs. CLI tools don't become web apps mid-pipeline.

3. **Detection is deterministic from CLAUDE.md + package.json + file patterns.** No LLM call needed. CLAUDE.md keywords, package.json dependencies, and file presence (e.g., `pages/`, `app/`, `src/cli.ts`) give high-confidence classification.

4. **Dual injection (Oracle + skill prompt) is needed.** Oracle context prevents repeated questions. Skill prompt context prevents wasted turns probing.

5. **The existing `projectContext` plumbing requires zero changes.** `oracle.ts`, `ask-handler.ts`, and `types.ts` already define the field. We just need to fill it.

## Approaches Considered

### Approach A: Minimal — Wire projectContext Only

Add `detectProjectType()`, store result, pass as `projectContext` string to Oracle. 3 files touched. Oracle stops asking, but skills still probe on turn 1.

- Effort: XS
- Risk: Low
- Pro: Smallest diff, zero architectural change
- Con: Skills still waste 1-3 turns probing

### Approach B: Full Injection — Oracle + Skill Prompts (CHOSEN)

Same detection as A, plus inject project type into every skill's initial prompt. Richer detection using CLAUDE.md keywords + package.json dependencies + file patterns.

- Effort: S
- Risk: Low
- Pro: Eliminates redundant decisions AND wasted skill turns
- Con: Touches orchestrator prompt construction (but well-tested path)

### Approach C: Oracle Decision Cache

Cache all repeated oracle decisions by normalizing question text. General solution but complex, fragile (false positive risk from question normalization), and doesn't help skill routing.

- Effort: M
- Risk: Medium
- Pro: Catches all repeated questions
- Con: Hard normalization, false positives, no skill routing benefit

## Recommended Approach

**Approach B: Full Injection.** The marginal effort over A is ~30 minutes of CC time, but it eliminates both the Oracle cost AND the wasted skill turns. The detection logic partially exists in `bootstrap.ts:detectTechStack()`.

## Implementation

### 1. New module: `src/project-type.ts` (~80 lines)

```typescript
export type ProjectType = "cli" | "web-app" | "api" | "library" | "monorepo" | "unknown";

export interface ProjectTypeResult {
  type: ProjectType;
  confidence: number;        // 0-1
  evidence: string[];        // human-readable reasons
  frameworks: string[];      // detected frameworks (next, express, commander, etc.)
  hasWebUI: boolean;         // shortcut for skills: "should I open a browser?"
  hasTestSuite: boolean;     // shortcut for QA: "can I run npm test?"
  testCommand?: string;      // detected test command from package.json scripts
}
```

**`detectProjectType(projectDir: string): ProjectTypeResult`**

Detection priority (first match wins within each tier, evidence accumulates):

| Signal | Source | Type | Confidence |
|--------|--------|------|------------|
| "CLI tool", "command-line" in CLAUDE.md | CLAUDE.md | cli | 0.9 |
| "web app", "web application", "frontend" in CLAUDE.md | CLAUDE.md | web-app | 0.9 |
| "API server", "REST API", "GraphQL API" in CLAUDE.md | CLAUDE.md | api | 0.9 |
| "library", "SDK", "package" in CLAUDE.md | CLAUDE.md | library | 0.9 |
| next, nuxt, remix, sveltekit in deps | package.json | web-app | 0.8 |
| express, fastify, hono, koa in deps (no frontend) | package.json | api | 0.8 |
| commander, yargs, inquirer, oclif in deps | package.json | cli | 0.8 |
| react, vue, svelte, angular in deps (no SSR) | package.json | web-app | 0.7 |
| `pages/` or `app/` dir + framework | file patterns | web-app | 0.7 |
| `src/cli.ts` or `bin/` dir | file patterns | cli | 0.6 |
| workspaces in package.json | package.json | monorepo | 0.6 |

`hasWebUI` = `type === "web-app"` or (type === "api" AND react/vue/svelte in deps).

`hasTestSuite` = `true` if package.json has `scripts.test` that isn't the default npm placeholder (`echo "Error: no test specified"`).

`testCommand` = package.json `scripts.test` value if it exists and isn't the placeholder.

**`loadProjectType(projectDir: string): ProjectTypeResult | null`** — reads `.garyclaw/project-type.json`, returns null if missing.

**`saveProjectType(projectDir: string, result: ProjectTypeResult): void`** — writes via `safeWriteJSON`.

**`ensureProjectType(projectDir: string): ProjectTypeResult`** — load from cache, detect if missing, save, return. This is the main entry point.

### 2. Wire into orchestrator: `src/orchestrator.ts`

Two injection points:

**a) Oracle projectContext** (in `createAskHandler` call, ~line 285):

```typescript
const projectType = ensureProjectType(config.projectDir);
// ... inside the oracle config:
oracle: {
  // ... existing fields
  projectContext: formatProjectContext(projectType),
},
```

**`formatProjectContext(pt: ProjectTypeResult): string`** returns a compact string:
```
CLI tool (confidence: 0.9). No web UI. Test suite: vitest (npm test).
Evidence: CLAUDE.md says "CLI tool", commander in deps.
```

Capped at 500 chars (existing truncation in `oracle.ts:165`).

**b) Skill prompt prefix** (in `currentPrompt` construction, ~line 231):

```typescript
const projectTypePrefix = projectType.type !== "unknown"
  ? `\n\nProject type: ${formatProjectContext(projectType)}\n` +
    (projectType.hasWebUI
      ? "This project has a web UI. Browser testing is appropriate.\n"
      : "This project has NO web UI. Do NOT attempt browser testing. Run the test suite instead.\n") +
    (projectType.testCommand
      ? `Test command: ${projectType.testCommand}\n`
      : "")
  : "";

let currentPrompt = initialPromptOverride
  ?? `Run the /${config.skillName} skill. Follow all SKILL.md instructions completely.${projectTypePrefix}`;
```

When `initialPromptOverride` is provided (pipeline context handoff), the project type gets injected by the pipeline-level `buildXxxPrompt()` functions instead, so it's not doubled.

### 3. Wire into pipeline prompt builders

Each `buildXxxPrompt()` function gets project type context appended:

- `buildImplementPrompt()` in `src/implement.ts` — append project type so implement knows the framework
- `buildBootstrapPrompt()` in `src/bootstrap.ts` — already has `detectTechStack()`, wire in `saveProjectType()` as side effect
- `buildPrioritizePrompt()` in `src/prioritize.ts` — append so priority scoring accounts for project nature
- `buildEvaluatePrompt()` in `src/evaluate.ts` — append so evaluation knows what to check

Pattern for each:
```typescript
import { ensureProjectType, formatProjectContext } from "./project-type.js";
// ... inside buildXxxPrompt():
const pt = ensureProjectType(projectDir);
if (pt.type !== "unknown") {
  prompt += `\n## Project Type\n${formatProjectContext(pt)}\n`;
}
```

### 4. Bootstrap integration

In `bootstrap.ts`, after `detectTechStack()` completes in `analyzeCodebase()`, also run `detectProjectType()` and save the result. This means any repo that gets bootstrapped automatically has project type cached for all future runs.

### 5. Types extension

Add to `GaryClawConfig` in `src/types.ts`:
```typescript
projectType?: ProjectTypeResult;  // Cached project type, loaded at startup
```

This is optional — the orchestrator calls `ensureProjectType()` which handles the cache-or-detect logic.

## Open Questions

1. **Should detection re-run periodically?** Current design: detect once, cache forever. If a project adds a web UI later, the cache is stale. Option: re-detect if `.garyclaw/project-type.json` is older than 7 days. Low priority since project types rarely change.

2. **Should monorepo detection trigger per-package classification?** Current design treats monorepo as a single type. For GaryClaw's use case (single-project daemon), this is fine. Defer per-package classification.

## Success Criteria

- Zero "is there a web UI?" Oracle decisions on GaryClaw after this ships
- QA skill routes to test-suite mode on first turn for CLI projects
- Detection adds <50ms to job startup time
- All existing tests pass unchanged (fail-open design)
