# Design: Prioritize Prompt Size Guard — Token Budgeting

Generated by /office-hours on 2026-03-31
Branch: main
Repo: 0raclide/GaryClaw
Status: APPROVED
Mode: Builder

## Problem Statement

The prioritize skill (`src/prioritize.ts`) assembles 8+ dynamic sections into a single prompt with no total token budget. `decision-outcomes.md` alone is 34K chars (~10K tokens) and growing. The `loadOracleContext()` function injects the full decision-outcomes content (capped at 12K tokens by oracle-memory, but that's still enormous). Combined with CLAUDE.md capabilities (~3K), TODOS.md (variable), skill catalog, pipeline outcome stats, failure patterns, quality trends, impact measurement, review findings, scored rules, and a worked example, the prompt can easily exceed practical limits.

Result: 15 sdk-bug failures total, 11 of which occur in the prioritize skill. This is the #1 failure category. Costs are trending up 19%.

**Critical finding (2026-03-31 analysis):** TODOS.md is 68,949 chars (~19,700 tokens), but 55 of 59 items are struck-through (completed). The 4 open items are only 5,019 chars (~1,434 tokens). The prompt carries **13x more dead weight than useful content**. Filtering to open items only is the single biggest optimization: -18,264 tokens before any budgeting kicks in.

## What Makes This Cool

Self-healing infrastructure. The daemon's most-used skill stops crashing because it now enforces its own resource limits. Every prioritize run costs less (shorter prompt = fewer input tokens), and the daemon can run longer between failures. Two birds, one stone: reliability + cost.

## Constraints

- Must use existing `estimateTokens()` from `checkpoint.ts` (chars / 3.5 heuristic)
- Must not change the behavior of other modules that build sections (aggregateFailurePatterns, loadOracleContext, etc.)
- Section truncation must preserve the most recent/relevant data (not random truncation)
- Must be backward-compatible: if all sections are small, the prompt should be identical to today
- ~120 lines of changes, S effort
- No new dependencies

## Premises

1. The prioritize prompt has no total token cap and will exceed SDK practical input limits as historical data accumulates.
2. The existing `estimateTokens()` / `truncateToTokenBudget()` pattern (used in checkpoint.ts, oracle-memory.ts, bootstrap.ts, codebase-summary.ts) is proven and reusable.
3. A total budget of ~40K tokens is appropriate for the prioritize prompt (leaves room for SDK system prompt, tool definitions, and response generation within the effective context window).
4. Sections should be prioritized: scoring rules and TODOS.md are essential (never truncated), while growing historical data (decision-outcomes via oracle context, pipeline outcomes) should be truncated first.
5. This can be implemented entirely in `src/prioritize.ts` by wrapping section assembly with budget tracking, importing `estimateTokens` from checkpoint.ts. A local `truncateSection` helper handles truncation with correct direction per section type. Note: oracle-memory.ts's `truncateToTokenBudget` internally uses `maxTokens * 4` for char budget while `estimateTokens` uses `chars / 3.5`, creating a ~14% overrun. To avoid this inconsistency, the local helper uses `estimateTokens` consistently for both measurement and truncation target calculation.
6. TODOS.md and capabilities sections need beginning-preserving truncation (keep P1/P2 items at top, keep config file headers). Historical sections (decision outcomes, failure patterns) need end-preserving truncation (keep newest data). The helper accepts a `keepEnd` flag.

## Approaches Considered

### Approach A: Static Per-Section Caps

Each section gets a fixed token budget constant. If a section exceeds its cap, it's truncated. If a section is empty, its budget is wasted (not redistributed).

```
const SECTION_BUDGETS = {
  todosContent: 8_000,
  capabilities: 3_500,
  oracleContext: 6_000,
  failurePatterns: 2_000,
  ...
};
```

**Effort:** S
**Risk:** Low
**Pros:**
- Simplest to implement (~50 lines)
- Predictable per-section behavior
- Easy to tune individual sections

**Cons:**
- Wasted budget when sections are empty (common: overnight goal, review findings, pipeline context often null)
- Over-constrains small sections that occasionally spike
- Doesn't adapt to prompt composition

### Approach B: Proportional Waterfall Budget

Total budget cap (40K tokens). Sections assembled in priority order, each with an individual soft cap. Track `tokensUsed` as sections are added. When a section would exceed `totalBudget - tokensUsed`, truncate it to fit. Unused budget from empty/small sections naturally flows to later sections.

Conceptual priority (what survives truncation, highest to lowest):
1. System instruction (fixed ~50 tokens)
2. TODOS.md backlog (cap: 8K, essential)
3. Scoring rules + worked example (fixed ~5.5K, essential)
4. Pre-assigned/claimed items (small, essential for correctness)

Note: Assembly order follows the existing `buildPrioritizePrompt()` structure (TODOS first, then context sections, then rules at the end). The priority order above describes what gets protected from truncation, not the order sections appear in the prompt.

Budget-managed sections (truncated when over budget):
5. CLAUDE.md vision + capabilities (cap: 4K)
6. Oracle context with decision outcomes (cap: 6K)
7. Failure patterns (cap: 2K)
8. Decision quality trends (cap: 1.5K)
9. Impact measurement (cap: 1K)
10. Skill catalog (cap: 2K)
11. Pipeline outcome stats (cap: 2K)
12. Review findings (cap: 1.5K)
13. Pipeline context from previous skills (cap: 1.5K)
14. Overnight goal (cap: 1K)
15. Project type section (cap: 500)

**Effort:** S
**Risk:** Low
**Pros:**
- Budget-efficient: empty sections donate their tokens to sections that need them
- Same pattern as checkpoint.ts relay prompt (tiered assembly with remaining budget)
- Predictable priority ordering means essential data is never lost
- Total prompt size is guaranteed to fit

**Cons:**
- Slightly more complex than static caps
- Per-section caps need tuning (but can be exported as constants for testing)

### Approach C: Build-Then-Shrink

Build the full prompt with no limits, measure total tokens, then if over budget proportionally shrink the N largest sections until within budget.

**Effort:** S
**Risk:** Medium
**Pros:**
- Simplest mental model ("build everything, then trim")
- Proportional shrinking feels fair

**Cons:**
- Always does full work even when truncation is needed
- Proportional shrinking may cut essential sections (TODOS.md is often large)
- Harder to predict which sections get cut
- Two passes over the data

## Recommended Approach

**Approach B: Proportional Waterfall Budget**

This is the same pattern used by `checkpoint.ts` relay prompt generation (assemble sections in priority order, track token budget, truncate when over). It handles the real-world distribution naturally: most sections are small (null or a few hundred tokens), but decision-outcomes and capabilities are huge. The waterfall means the huge sections get whatever budget remains after the small essential sections are assembled.

## Implementation Spec

### Pre-filter: `filterOpenTodos()`

Before any budgeting, filter TODOS.md to open items only. This is the biggest single win: -18K tokens.

```typescript
/**
 * Filter TODOS.md content to open items only.
 * Removes struck-through headings (## ~~P{N}:) and their full blocks.
 * Keeps preamble text (e.g. "# TODOS" heading) before the first ## block.
 */
export function filterOpenTodos(content: string): string {
  const blocks = content.split(/^(?=## )/m);
  const open = blocks.filter(b => !b.match(/^## ~~/));
  return open.join("\n").trim();
}
```

Wire into `buildPrioritizePrompt()` at line 713:
```typescript
const filteredTodos = todosContent ? filterOpenTodos(todosContent) : null;
```

Then use `filteredTodos` instead of `todosContent` for the TODOS.md section.

**Edge case:** If `filterOpenTodos` returns an empty or whitespace-only string (all items struck-through), fall through to the existing "No TODOS.md found" fallback message rather than pushing an empty section. This preserves the existing prompt behavior for the backlog-exhausted path.

### Core helpers

**Important: do NOT import `truncateToTokenBudget` from oracle-memory.ts.** That function
uses `maxTokens * 4` for char budget while `estimateTokens` uses `chars / 3.5`, causing
~14% overrun. Use the local `truncateSection` helper which uses `estimateTokens`
consistently for both measurement and truncation.

```typescript
import { estimateTokens } from "./checkpoint.js";

/**
 * Truncate text to fit within a token budget.
 * Uses estimateTokens (chars/3.5) consistently for both measurement and target.
 * keepEnd=true: drop from beginning (keep newest, for historical data like outcomes)
 * keepEnd=false: drop from end (keep beginning, for TODOS.md P1/P2 items, capabilities)
 */
export function truncateSection(content: string, maxTokens: number, keepEnd = true): string {
  if (estimateTokens(content) <= maxTokens) return content;
  const maxChars = Math.floor(maxTokens * 3.5);
  if (keepEnd) {
    const sliced = content.slice(-maxChars);
    const nl = sliced.indexOf("\n");
    return nl >= 0 && nl < sliced.length - 1
      ? "[...truncated oldest]\n" + sliced.slice(nl + 1)
      : sliced;
  } else {
    const sliced = content.slice(0, maxChars);
    const nl = sliced.lastIndexOf("\n");
    return nl > 0 ? sliced.slice(0, nl) + "\n[...truncated]" : sliced;
  }
}

/**
 * Add a section to the prompt lines if it fits within the remaining budget.
 * Returns tokens consumed. Truncates content if exceeds sectionCap or remaining budget.
 * keepEnd: truncation direction (true=keep newest, false=keep beginning)
 */
export function addBudgetedSection(
  lines: string[],
  header: string,
  content: string,
  sectionCap: number,
  remainingBudget: number,
  keepEnd = true,
): number {
  if (!content || content.trim().length === 0) return 0;
  const effectiveCap = Math.min(sectionCap, remainingBudget);
  if (effectiveCap <= 0) return 0;
  const tokens = estimateTokens(content);
  const finalContent = tokens > effectiveCap
    ? truncateSection(content, effectiveCap, keepEnd)
    : content;
  if (header) { lines.push(header); lines.push(""); }
  lines.push(finalContent);
  lines.push("");
  const block = header ? `${header}\n\n${finalContent}\n` : `${finalContent}\n`;
  return estimateTokens(block);
}
```

**Truncation direction per section:**

| Section | keepEnd | Reason |
|---------|---------|--------|
| TODOS.md | `false` | P1/P2 headings at top must survive |
| Capabilities | `false` | Module map headers, status at top |
| Oracle context | `true` | Recent outcomes most relevant |
| Failure patterns | `true` | Recent failures matter most |
| Quality trends | `true` | Recent trends |
| Impact measurement | `true` | Recent cost data |
| Pipeline outcome stats | `true` | Recent outcomes |
| Review findings | `true` | Recent reviews |
| Pipeline context | `true` | Recent skill findings |
| Overnight goal | `false` | Goal text reads top-down |
| Skill catalog | `false` | Structured reference, keep beginning |
| Project type | `false` | Short, structured |

### Changes to `buildPrioritizePrompt()`

The existing function assembles sections with direct `lines.push()`. The change wraps each dynamic section with `addBudgetedSection()` and tracks `tokensUsed`.

Fixed-cost sections (system instruction, scoring rules, worked example, pre-assigned/claimed items) are pushed directly and their tokens counted. Dynamic sections use `addBudgetedSection()`.

```typescript
export async function buildPrioritizePrompt(...): Promise<string> {
  const lines: string[] = [];
  let tokensUsed = 0;
  const remaining = () => PRIORITIZE_PROMPT_BUDGET - tokensUsed;

  // Fixed: system instruction
  const sysInstruction = "You are a technical product manager...";
  lines.push(sysInstruction);
  tokensUsed += estimateTokens(sysInstruction);

  // Budgeted: TODOS.md (keepEnd=false: keep P1/P2 at top)
  tokensUsed += addBudgetedSection(lines, "### Backlog (TODOS.md)",
    todosContent, PRIORITIZE_SECTION_BUDGETS.todosContent, remaining(), false);

  // Budgeted: capabilities (keepEnd=false: keep module map headers at top)
  // NOTE: Remove the existing inline `capabilities.slice(0, 12000)` guard (line 752).
  // addBudgetedSection now handles truncation via the 4K section cap.
  tokensUsed += addBudgetedSection(lines, "### Current Capabilities",
    capabilities, PRIORITIZE_SECTION_BUDGETS.capabilities, remaining(), false);

  // Budgeted: overnight goal (keepEnd=false: goal text reads top-down)
  tokensUsed += addBudgetedSection(lines, "### Overnight Goal",
    goal, PRIORITIZE_SECTION_BUDGETS.overnightGoal, remaining(), false);

  // Budgeted: oracle context (metrics + decision outcomes)
  tokensUsed += addBudgetedSection(lines, "### Oracle Intelligence",
    oracleCtx, PRIORITIZE_SECTION_BUDGETS.oracleContext, remaining());

  // Budgeted: failure patterns
  tokensUsed += addBudgetedSection(lines, "",
    failurePatterns, PRIORITIZE_SECTION_BUDGETS.failurePatterns, remaining());

  // Budgeted: decision quality trends
  tokensUsed += addBudgetedSection(lines, "",
    qualityTrends, PRIORITIZE_SECTION_BUDGETS.qualityTrends, remaining());

  // Budgeted: impact measurement
  tokensUsed += addBudgetedSection(lines, "",
    impact, PRIORITIZE_SECTION_BUDGETS.impactMeasurement, remaining());

  // Budgeted: pipeline context from previous skills
  tokensUsed += addBudgetedSection(lines, "### Previous Skill Findings",
    pipelineCtx, PRIORITIZE_SECTION_BUDGETS.pipelineContext, remaining());

  // Budgeted: unresolved review findings
  tokensUsed += addBudgetedSection(lines, "### Unresolved Review Findings",
    reviewFindingsStr, PRIORITIZE_SECTION_BUDGETS.reviewFindings, remaining());

  // Fixed: pre-assigned/claimed items (small, essential for correctness — not budgeted)

  // Budgeted: skill catalog (keepEnd=false: structured reference)
  tokensUsed += addBudgetedSection(lines, "## Available Skills",
    formatSkillCatalogForPrompt(), PRIORITIZE_SECTION_BUDGETS.skillCatalog, remaining(), false);

  // Budgeted: pipeline outcome stats (keepEnd=true: recent outcomes)
  tokensUsed += addBudgetedSection(lines, "### Pipeline Outcome Patterns by Task Category",
    pipelineOutcomeTable, PRIORITIZE_SECTION_BUDGETS.pipelineOutcomeStats, remaining());

  // Budgeted: project type (keepEnd=false: structured)
  tokensUsed += addBudgetedSection(lines, "",
    ptSection, PRIORITIZE_SECTION_BUDGETS.projectType, remaining(), false);

  // Fixed: scoring rules + worked example (always included, counted)
  lines.push(PRIORITIZE_RULES);
  tokensUsed += estimateTokens(PRIORITIZE_RULES);
  lines.push(WORKED_EXAMPLE);
  tokensUsed += estimateTokens(WORKED_EXAMPLE);

  return lines.join("\n");
}
```

### Key behaviors

1. **Empty sections cost zero.** If `loadOvernightGoal()` returns null, no tokens consumed, budget flows to the next section.
2. **Large sections get truncated.** `decision-outcomes.md` at 10K tokens gets truncated to 6K (its cap) or less (remaining budget).
3. **Total prompt never exceeds 40K tokens.** Even if every section is at max, the sum of all section caps (~31K) plus fixed sections (~6K: system instruction ~50 tokens, Phase 2/3/4 headings ~200 tokens, scoring rules ~3K, worked example ~2.5K, pre-assigned/claimed items ~200 tokens) fits well within 40K. For budget-managed sections, assembly order equals priority order: sections assembled earlier consume budget first and are therefore protected from truncation.
4. **Truncation direction is per-section.** `truncateSection(content, cap, keepEnd)` preserves the right end of each section: `keepEnd=true` for historical data (decision outcomes, failure patterns), `keepEnd=false` for structured content (TODOS.md, capabilities). See the direction table above.
5. **Backward compatible.** If all sections are small (total < 40K), the prompt is identical to today.
