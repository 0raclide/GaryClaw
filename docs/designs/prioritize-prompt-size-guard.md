# Design: Prioritize Prompt Size Guard

Generated by /office-hours on 2026-03-31
Branch: main
Repo: 0raclide/GaryClaw
Status: APPROVED
Mode: Builder

## Problem Statement

The prioritize skill's `buildPrioritizePrompt()` assembles 8+ dynamic sections with zero total token budgeting. The input files are massive and growing:

- **TODOS.md:** 69KB (~19,700 tokens)
- **CLAUDE.md:** 64KB (~18,300 tokens), capabilities section trimmed to 12K chars (~3,400 tokens)
- **decision-outcomes.md:** 34KB (~9,700 tokens), injected in full via `loadOracleContext()`
- **Skill catalog, rules, worked example, pipeline stats:** ~3,000 tokens fixed overhead
- **Failure patterns, quality trends, impact measurement, review findings:** variable, ~500-2,000 tokens each

Total prompt can exceed 40K+ tokens before the SDK adds its own overhead (system prompt, tool definitions, conversation history). This causes the #1 failure category: `sdk-bug` with 15 total failures, 11 in the prioritize skill. Costs are also trending up 19% because larger prompts = more input tokens billed.

## What Makes This Cool

The pattern already exists. `checkpoint.ts` has `estimateTokens()` and tiered truncation for relay prompts. `oracle-memory.ts` has per-file token caps. This is about applying the same discipline to `buildPrioritizePrompt()`, which is currently the only prompt builder with no budget at all.

Two wins from one fix: fewer SDK failures AND lower costs per prioritize run.

## Constraints

- Must be backward-compatible: when all sections fit within budget, output is identical
- Cannot lose critical information (TODOS.md items, rules, worked example)
- Must reuse `estimateTokens()` from checkpoint.ts (no new estimation logic)
- ~80 lines of implementation (S effort)
- No new modules, no new interfaces beyond a `SectionBudget` type

## Premises

1. The sdk-bug failures are caused by prompt size exceeding SDK limits, not by content quality
2. `estimateTokens()` at ~3.5 chars/token is accurate enough for budgeting
3. A total budget of ~40,000 tokens is safe (leaves headroom for SDK overhead, system prompt, tool defs, and response)
4. Sections have different value densities: TODOS.md and rules are essential, historical outcomes are low-value per token
5. The fix is backward-compatible: when everything fits, nothing changes

## Approaches Considered

### Approach A: Priority-Ordered Section Assembly with Per-Section Caps

**Summary:** Assign each section a hard token cap. Sections added in priority order. Each section self-truncates to its cap using the simplest strategy that preserves meaning (tail-truncate for historical data, head-truncate for recent context).

**Effort:** S
**Risk:** Low

**How it works:**

```typescript
const SECTION_BUDGETS: Record<string, number> = {
  rules:              5000,   // PRIORITIZE_RULES + WORKED_EXAMPLE (fixed, never truncated)
  todos:              12000,  // TODOS.md content
  capabilities:       3500,   // CLAUDE.md Current Capabilities (already trimmed to 12K chars)
  vision:             1500,   // CLAUDE.md product vision
  oracleContext:      4000,   // Oracle metrics + decision outcomes
  failurePatterns:    1500,   // Failure pattern aggregation
  qualityTrends:      1000,   // Decision quality trends
  impactMeasurement:  800,    // Recent cost impact
  skillCatalog:       1500,   // Available skills for pipeline recommendation
  pipelineStats:      1500,   // Per-category pipeline outcome patterns
  reviewFindings:     1500,   // Unresolved review findings
  pipelineContext:    1000,   // Previous skill findings in pipeline
  projectType:        500,    // Project type context
  claimedItems:       500,    // Parallel instance claimed items
  preAssigned:        500,    // Pre-assigned TODO title
};
// Total: ~36,300 tokens max, leaving headroom for Phase 2-4 instructions + overhead
```

Each section builder gains an optional `maxTokens` parameter. Truncation strategy varies:

| Section | Truncation Strategy |
|---------|-------------------|
| rules, worked example | Never truncated (fixed content) |
| todos | Simple line-by-line tail-truncate: keeps content from the top, drops from the bottom |
| capabilities | Already trimmed to 12K chars, further tail-truncate if needed |
| vision | Tail-truncate (simple line-by-line, keep from top) |
| oracleContext | Truncate decision-outcomes.md from the top (oldest outcomes dropped first) |
| failurePatterns, qualityTrends, impactMeasurement | Reduce to top-N items (top 2 instead of top 3) |
| skillCatalog | Remove `useWhen` field (least critical for pipeline recommendation) |
| pipelineStats | Reduce rows (top 10 instead of top 15) |
| reviewFindings | Truncate question context more aggressively (100 chars instead of 200) |

**Pros:**
- Simple, predictable, no runtime calculation
- Each section knows its own budget, truncation is local
- Easy to tune: bump a number, observe results
- Follows the exact pattern from `generateRelayPrompt()` in checkpoint.ts

**Cons:**
- Rigid allocation: if TODOS.md is small, its unused budget is wasted
- Requires manual tuning as sections grow

### Approach B: Dynamic Budget with Proportional Allocation

**Summary:** Set a total budget (40K tokens). Fixed sections claim their actual size. Remaining budget is distributed proportionally among dynamic sections by importance weight.

**Effort:** S (slightly more complex than A)
**Risk:** Low-Med (proportional math adds edge cases)

**How it works:**

```typescript
const TOTAL_BUDGET = 40_000;
const FIXED_SECTIONS = ['rules', 'workedExample', 'phaseInstructions'];
const SECTION_WEIGHTS: Record<string, number> = {
  todos: 5,            // highest priority
  oracleContext: 3,
  capabilities: 2,
  failurePatterns: 2,
  skillCatalog: 1,
  // ...
};

// 1. Measure fixed sections, subtract from budget
// 2. Measure each dynamic section at full size
// 3. If total fits, use everything (backward-compatible)
// 4. If over budget, allocate remaining proportionally by weight
// 5. Each section truncates to its allocation
```

**Pros:**
- Adapts to actual content sizes: small TODOS.md means more room for outcomes
- Single tuning knob (TOTAL_BUDGET) plus weights
- Backward-compatible by design (step 3)

**Cons:**
- More code (~120 lines vs ~80 for Approach A)
- Proportional allocation can produce odd results (200-token section getting 50 more tokens while 15K section gets cut to 8K)
- Harder to reason about: "why did my section get truncated?" requires understanding the math

### Approach C: Two-Pass Assembly with Overflow Spillover

**Summary:** First pass assembles everything. If over budget, second pass truncates sections in reverse priority order until budget is met.

**Effort:** XS
**Risk:** Med (aggressive truncation of low-priority sections could drop useful context)

**Pros:**
- Simplest implementation (~40 lines)
- Most backward-compatible (nothing changes when under budget)
- Clear mental model: lowest-priority sections get cut first

**Cons:**
- All-or-nothing per section (section is either full or gone)
- Low-priority sections might have critical signals that get dropped entirely
- No graceful degradation within sections

## Recommended Approach

**Approach A: Priority-Ordered Section Assembly with Per-Section Caps.**

Why: it's the simplest approach that actually solves the problem with fine-grained control. The per-section budgets are immediately tunable via constants. The truncation strategies are section-specific (oldest outcomes dropped vs. reduced item count). It follows the exact same pattern as `generateRelayPrompt()` in checkpoint.ts, so any contributor who's read that code will immediately understand this.

Approach B is more elegant but overkill for S effort. Approach C is too blunt, dropping entire sections rather than trimming them.

## Implementation Plan

### Step 1: Import `estimateTokens` from checkpoint.ts

```typescript
import { estimateTokens } from "./checkpoint.js";
```

The token estimation logic already exists. No new estimation code needed.

### Step 2: Unify into `truncateSection()` with direction parameter

A single function handles both tail-truncation (TODOS.md, capabilities) and head-truncation (decision outcomes). Uses char-slice for O(1) instead of O(n^2) line-shifting:

```typescript
/**
 * Truncate text to fit within a token budget.
 * @param keepEnd - if true, keeps the END of the text (drops oldest/top lines).
 *                  if false (default), keeps the START (drops newest/bottom lines).
 */
export function truncateSection(text: string, maxTokens: number, keepEnd = false): string {
  if (estimateTokens(text) <= maxTokens) return text;
  // estimateTokens uses ~3.5 chars/token, so charBudget = maxTokens * 3.5
  const charBudget = Math.floor(maxTokens * 3.5);
  if (keepEnd) {
    const sliced = text.slice(-charBudget);
    // Find first newline to avoid partial line
    const nl = sliced.indexOf('\n');
    return '[...older entries truncated]\n' + (nl >= 0 ? sliced.slice(nl + 1) : sliced);
  } else {
    const sliced = text.slice(0, charBudget);
    // Find last newline to avoid partial line
    const nl = sliced.lastIndexOf('\n');
    return (nl >= 0 ? sliced.slice(0, nl) : sliced) + '\n[...truncated to fit token budget]';
  }
}
```

### Step 4: Apply budgets in `buildPrioritizePrompt()`

Wrap each section with its truncation call. Key changes:

```typescript
// TODOS.md content
if (todosContent) {
  lines.push("### Backlog (TODOS.md)");
  lines.push("");
  lines.push(truncateSection(todosContent, SECTION_BUDGETS.todos));
  lines.push("");
}
```

Same pattern for every other section: wrap the content string with `truncateSection(content, SECTION_BUDGETS.sectionName)`. All sections use `keepEnd=false` (the default) except oracle context, which uses `keepEnd=true` as shown in Step 5. The sum of all per-section budgets (~36,300) is the mechanism that enforces the 40K total ceiling, no runtime total-prompt guard is needed since the per-section caps are individually enforced.

### Step 5: Truncate oracle context in `buildPrioritizePrompt()` only

**Important:** Do NOT modify `loadOracleContext()`. It is called by other consumers (`buildImplementPrompt`, etc.) and adding truncation there would be a cross-cutting change. Instead, truncate the oracle context output in `buildPrioritizePrompt()` using `truncateSection` with `keepEnd=true` (oldest outcomes dropped first, recent outcomes preserved):

```typescript
if (!config.noMemory) {
  const oracleCtx = loadOracleContext(projectDir);
  if (oracleCtx) {
    lines.push("### Oracle Intelligence");
    lines.push("");
    // keepEnd=true: drop oldest decision outcomes, preserve recent ones
    lines.push(truncateSection(oracleCtx, SECTION_BUDGETS.oracleContext, /* keepEnd */ true));
    lines.push("");
  }
}
```

### Step 6: Add total prompt size logging

After assembly, emit the total estimated token count for observability. Track section sizes during assembly using a simple record:

```typescript
// At the top of buildPrioritizePrompt(), alongside `const lines`:
const sectionTokens: Record<string, number> = {};

// After each section is assembled (example for todos):
const todosText = truncateSection(todosContent, SECTION_BUDGETS.todos);
sectionTokens.todos = estimateTokens(todosText);
lines.push(todosText);

// After full assembly:
const prompt = lines.join("\n");
const totalTokens = estimateTokens(prompt);
if (config.onEvent) {
  config.onEvent({
    type: 'prioritize_prompt_size',
    tokens: totalTokens,
    sections: sectionTokens,
  });
}
```
